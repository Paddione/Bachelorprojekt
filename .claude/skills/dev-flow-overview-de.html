Schritt vorab

Aufräumen vor dem Start

Bevor irgendetwas anderes passiert, prüft der Assistent, ob noch Arbeitskopien aus früheren Aufgaben herumliegen, die längst abgeschlossen sind — und bietet an, sie wegzuräumen. So sammeln sich keine Altlasten an. Wenn ihr es eilig habt: einfach überspringen, das Aufräumen kann später nachgeholt werden.

Aufräum-Durchgang Nur Anschauen — verändert nichts ungefragt
a
Alle laufenden Arbeitskopien auflisten
Der Assistent fragt das Werkzeugsystem: „Welche separaten Arbeitskopien existieren gerade?"
Was das bedeutet
Eine „Arbeitskopie" ist ein eigener Ordner, in dem an einer bestimmten Aufgabe gearbeitet wird — getrennt vom Haupt-Projekt, damit nichts durcheinander gerät.
b
Erkennen, welche schon erledigt sind
Für jede Arbeitskopie wird geprüft: „Gehört diese zu einer Aufgabe, deren Änderungen bereits im Hauptzweig zusammengeführt wurden?" Wenn ja, ist sie überflüssig — fertig, aber nicht weggeräumt.
c
Wegräumen — nur auf Wunsch
Findet der Assistent verwaiste Arbeitskopien, meldet er sie und fragt, ob er sie löschen soll.
Kein Zwang: wenn nichts gefunden wird — still und weiter mit Phase 1. Wenn etwas gefunden wird und ihr nicht aufräumen wollt — auch ok, dann eben weiter wie sonst.

Neue Funktion

Etwas Neues bauen

Eine neue Seite, ein neuer Bereich, eine neue Möglichkeit für Nutzer — irgendetwas, das vorher nicht existiert hat und von Besuchern bemerkt wird.

Arbeitsbereich: Neue-Funktion / …
Fehler beheben

Etwas Kaputtes reparieren

Eine Seite lädt falsch, ein Knopf tut nicht was er soll, Daten werden falsch angezeigt — irgendetwas verhält sich nicht so wie es sollte.

Benötigt eine Aufgabenzettel-Nummer
Wartung

Interna aufräumen

Für Nutzer unsichtbare Pflege-Arbeiten: Software-Bausteine aktualisieren, internen Code aufräumen, Dokumentation verbessern. Soll es regelmäßig wiederholt werden? Dann wird es als Routine hinterlegt.

Kein Aufgabenzettel nötig

Bevor es losgeht: Der Assistent fragt kurz nach, welchen Weg er einschlagen soll — und ob bei einer Fehler-Reparatur bereits ein Aufgabenzettel existiert.

Phase 1

Planung

Neue Funktion Neue-Funktion / … Endet mit Pause — Umsetzung folgt separat
1
Eigene Arbeitskopie einrichten Erst prüfen
Vor dem Anlegen schaut der Assistent nach, ob bereits eine Arbeitskopie für genau diese Aufgabe existiert (zum Beispiel weil eine ältere Sitzung sie schon vorbereitet hat). Falls ja: er wechselt einfach hinein, statt eine zweite, parallele Kopie anzulegen.

Falls nicht: er legt eine neue Arbeitskopie an. So kann er nach Belieben ausprobieren, ohne den laufenden Betrieb zu gefährden.
Was das bedeutet
Stellt euch vor, ihr wollt ein Zimmer neu streichen — ihr mietet euch erst eine identische Übungswohnung, probiert die Farben dort aus und streicht erst dann das echte Zimmer, wenn ihr mit dem Ergebnis zufrieden seid. Bevor ihr eine zweite Übungswohnung mietet, schaut ihr nach, ob ihr nicht schon eine habt.
2
Ideen-Werkzeug öffnen
Ein interaktives Brainstorming-Werkzeug wird gestartet und ist für kurze Zeit unter einer öffentlichen Web-Adresse erreichbar. Ihr könnt dem Assistenten dort in Echtzeit beim Denken zuschauen.
Was das bedeutet
Eine Art „Tafel", auf der Claude seine Überlegungen aufzeichnet — erreichbar unter brainstorm.mentolder.de während dieser Phase.
3
Brainstorming — Was soll es wirklich tun?
Der Assistent denkt die Funktion von allen Seiten durch: Was braucht der Nutzer? Welche Szenarien müssen abgedeckt sein? Welche Randfälle gibt es?
  • Nutzer-Erfahrung: Wie soll sich die Funktion anfühlen?
  • Inhalt: Welche Daten werden gezeigt oder gespeichert?
  • Grenzen: Was gehört nicht dazu?
Das Ergebnis wird als Anforderungsbeschreibung gespeichert.
4
Umsetzungsplan schreiben
Aus der Anforderungsbeschreibung wird ein konkreter Schritt-für-Schritt-Plan: Welche Dateien werden wie verändert? In welcher Reihenfolge? Wie wird geprüft, ob alles korrekt ist?
Was das bedeutet
Der Plan landet in einem dafür vorgesehenen Ordner und erhält einen eindeutigen Namen. Dazu schreibt der Assistent eine kleine Markierung in den Kopf der Datei (z. B. zu welchem Bereich die Änderung gehört) — daran erkennen andere Werkzeuge später automatisch, worum es geht.
4.5
Aufgabenzettel anlegen Neu
In der echten, live laufenden Datenbank wird automatisch ein Aufgabenzettel erstellt. Er bekommt eine eindeutige Nummer wie T000301, taucht sofort im Verwaltungsbereich der Website auf, und seine Nummer wird in den Plan eingetragen — damit Plan und Aufgabenzettel später eindeutig zueinander gehören.
Meldung an euch: „Aufgabenzettel T000301 angelegt — sichtbar unter web.mentolder.de/admin/bugs"
5
Plan sichern & hochladen Pause
Plan und Anforderungsbeschreibung werden gespeichert und auf den Server hochgeladen. Damit sind sie für jeden sichtbar und können nicht mehr versehentlich verloren gehen.
Was das bedeutet
Kein Code wird geschrieben. Kein Knopf gedrückt. Die Umsetzung beginnt erst in Phase 2 — absichtlich getrennt, damit ihr die Planung prüfen könnt.
Fehler beheben Reparatur / … Erst beweisen, dann reparieren · Aufgabenzettel ist Pflicht
1
Aufgabenzettel-Nummer klären Pflicht
Falls bereits ein Zettel existiert: Nummer übernehmen — fertig.
Falls noch kein Zettel existiert: Der Assistent fragt nach Titel, wie schwer der Fehler wiegt (kritisch, groß, klein, geringfügig) und einer kurzen Beschreibung, dann legt er den Zettel selbst an (Art: Fehler, Priorität: hoch).
Ohne Aufgabenzettel-Nummer geht es nicht weiter. Das ist bewusst so — jede Reparatur muss verfolgbar sein.
2
Eigene Arbeitskopie einrichten Erst prüfen
Wie bei einer neuen Funktion: Der Assistent prüft erst, ob bereits eine passende Arbeitskopie existiert, und wechselt nur hinein. Andernfalls legt er eine neue an — diesmal mit einem beschreibenden Namen für die Reparatur.
3
Den Fehler zuerst beweisbar machen
Bevor irgendwas repariert wird, schreibt der Assistent eine automatische Probe (einen Test), der den Fehler eindeutig nachweist. Dieser Test muss zunächst fehlschlagen — sonst hätte er den Fehler nicht wirklich gefunden.
Warum das wichtig ist
Eine Reparatur gilt nur dann als erfolgreich, wenn dieselbe Probe danach besteht. So kann niemand versehentlich den Fehler „reparieren", ohne ihn wirklich behoben zu haben.
4
Reparaturplan schreiben (bei komplexen Fehlern)
Lässt sich der Fehler mit einer einzigen kleinen Änderung beheben, reicht eine kurze Begründung im Speicherprotokoll.
Bei komplizierteren Fehlern: vollständigen Schritt-für-Schritt-Plan erstellen und die Aufgabenzettel-Nummer dort vermerken.
5
Plan sichern & hochladen Pause
Probe und Plan werden gespeichert und hochgeladen — bereit für Phase 2.
Wartung Wartung / … Vollständig in einem Schritt — kein Aufgabenzettel
0.5
Einmalig oder wiederkehrend? Neu
Allererste Frage: „Soll diese Wartung regelmäßig wiederholt werden — wöchentlich, monatlich, täglich?"
  • Klassische Kandidaten: Software-Bausteine aktualisieren, Zertifikate erneuern, Protokolle aufräumen, Sicherungen prüfen.
  • Erkennungsmerkmale: jede Formulierung mit „regelmäßig", „jede Woche", „jeden Monat", „automatisch".
Wiederkehrend: Der Assistent richtet eine Routine ein, die ab sofort selbstständig zur gewünschten Zeit läuft — keine Arbeitskopie, kein einmaliger Lauf, hier endet die Wartung.
Einmalig: weiter mit dem nächsten Schritt.
0.6
Gibt es schon offene Wartungs-Arbeitsbereiche?
Zuerst prüfen, ob bereits ein passender Wartungszweig offen ist. Falls ja: Änderung dort einbauen, statt einen neuen zu öffnen. So wird verhindert, dass viele kleine Wartungs-Änderungen das Protokoll unübersichtlich machen.
1
In einem Satz beschreiben, was zu tun ist
Beispiele: „Astro auf Version 5 aktualisieren", „Variablenname korrigieren", „Schreibfehler in der Dokumentation beheben".
2
Arbeitskopie einrichten & Änderung durchführen Erst prüfen
Wie bei den anderen Pfaden: erst prüfen, ob bereits eine Arbeitskopie existiert, sonst eine neue anlegen. Kein ausführlicher Plan, kein Anforderungsdokument, keine Test-zuerst-Methode nötig — direkt machen.
3
Prüfen, ob nichts kaputt gegangen ist
Alle automatischen Prüfroutinen laufen durch:
  • Allgemeine Probe-Suite — muss komplett grün sein
  • Konfigurationsdatei-Prüfung — falls Systemkonfigurationen betroffen
  • Website-Vorschau — falls Webseiten-Dateien betroffen
4
Freigabe einholen, zusammenführen & veröffentlichen
Änderungsantrag erstellen, nach automatischer Prüfung zusammenführen und den passenden Veröffentlichungs-Befehl ausführen. Fertig.
Phase 2

Umsetzung

Ausführung Neue Funktion · Fehler-Reparatur Startet auf der in Phase 1 vorbereiteten Arbeitskopie
0
Bin ich am richtigen Ort? Sicherheit
Bevor irgendetwas verändert wird, prüft der Assistent zwei Dinge:
  • Bin ich in der richtigen Arbeitskopie? — Wenn er versehentlich im Haupt-Ordner statt in der für die Aufgabe vorbereiteten Kopie steht, würde er den falschen Bereich verändern.
  • Arbeitet gerade jemand anderes hier? — Wenn parallel eine andere KI-Sitzung in derselben Arbeitskopie aktiv ist, gäbe es Datensalat. In dem Fall: kurz warten oder Rücksprache halten.
Findet der Assistent etwas Verdächtiges, meldet er es — keine Änderungen ohne Bestätigung.
1
Plan laden & verstehen
Der neueste Plan aus dem Planordner wird geladen und vollständig gelesen. Aus dem Namen der Arbeitskopie ergibt sich, ob es sich um eine neue Funktion oder eine Reparatur handelt. Falls ein Aufgabenzettel verknüpft ist, wird die Nummer für spätere Schritte vorgemerkt.
Was das bedeutet
Wenn mehrere Pläne offen sind, kann der Assistent auch eine Liste anzeigen und euch wählen lassen — falls die Reihenfolge wichtig ist.
1.5
Aufgabenzettel auf „In Bearbeitung" setzen Neu
Falls ein Aufgabenzettel verknüpft ist, wird er in der Datenbank automatisch auf „In Bearbeitung" gesetzt — damit jeder im Team sieht, dass gerade daran gearbeitet wird.
Meldung an euch: „Aufgabenzettel T000301 → In Bearbeitung"
2
Eigentliche Umsetzung
Bei neuen Funktionen: Mehrere spezialisierte Hilfs-Assistenten arbeiten parallel — einer kümmert sich um die Datenbankseite, ein anderer um die Benutzeroberfläche, ein dritter um die Systemkonfiguration. Das spart Zeit und nutzt das Fachwissen der jeweiligen Spezialisten.

Bei Reparaturen: Schritt für Schritt umsetzen, bis die in Phase 1 geschriebene Probe erstmals besteht statt fehlzuschlagen. Dann noch aufräumen und vereinfachen.
3
Lokale Prüfung
Bevor irgendetwas veröffentlicht wird, laufen alle automatischen Proben durch:
  • Konfigurationsdateien auf Fehler prüfen
  • Relevante Funktions- und Sicherheitstests ausführen
  • Gesamte Test-Suite starten — alles muss grün sein
4
Vorab-Vorschau Optional
Aktueller Stand: Die Vorschau-Umgebung läuft derzeit nicht. Änderungen werden lokal geprüft und danach direkt in der echten Umgebung veröffentlicht. Sobald die Vorschau wieder läuft, kann der Assistent die Änderung dort gefahrlos in Aktion zeigen, bevor sie live geht.
5
Änderungsantrag stellen
Der Assistent erstellt einen formalen Änderungsantrag — eine Zusammenfassung aller gemachten Änderungen mit Begründung, die vor dem Veröffentlichen nicht mehr von Hand geprüft werden muss (ihr habt das so vereinbart).

Bei Reparaturen muss der Antrag die Aufgabenzettel-Nummer enthalten, damit der Zettel später automatisch geschlossen wird, sobald der Antrag durchgeht.
5.5
Änderungsantrag mit dem Aufgabenzettel verknüpfen Neu
Direkt nach dem Stellen des Änderungsantrags wird in der Datenbank notiert: „Aufgabenzettel T000301 gehört zu Antrag Nummer 723." So kann man später vom Aufgabenzettel direkt zum Antrag springen — und umgekehrt.
Warum das wichtig ist
Beim Nachschauen, was zu einem bestimmten Fehler unternommen wurde, ist sofort sichtbar: Welcher Antrag hat ihn behoben? Welche Dateien wurden geändert?
6
Automatische Freigabe nach bestandenen Prüfungen
Alle automatischen Prüfungen (Tests, Sicherheitsscans) laufen in der Cloud. Sind sie grün, wird der Antrag automatisch übernommen.
6.5
Aufgabenzettel abschließen Neu
Nach erfolgreichem Zusammenführen wird der Aufgabenzettel automatisch abgehakt:
  • Neue Funktion: Auflösung „Ausgeliefert"
  • Reparatur: Auflösung „Behoben"
Zusätzlich wird ein interner Kommentar hinterlassen, der auf den Änderungsantrag und den ausgeführten Plan verweist.
Meldung an euch: „Aufgabenzettel T000301 → Erledigt (Ausgeliefert)"
7
Plan in der Datenbank ablegen & Datei löschen Geändert
Neuer Ablauf: Der ausgeführte Plan wird nicht mehr in einen Erledigt-Ordner verschoben, sondern vollständig in der Datenbank archiviert (zusammen mit Aufgabenzettel-Nummer und Antrags-Nummer). Anschließend wird die Plan-Datei aus dem Projekt entfernt.
Warum das so ist
Der Plan-Ordner bleibt schlank — nur was gerade in Arbeit ist, ist dort sichtbar. Wer nachsehen möchte, was historisch geplant und umgesetzt wurde, schaut in die Datenbank über den Verwaltungsbereich der Website. Der Plan ist also nicht weg, nur an einen sinnvolleren Ort umgezogen.
7.5
Arbeitskopie wegräumen Neu
Nach Abschluss wird die nicht mehr benötigte Arbeitskopie sauber entfernt — sowohl lokal als auch auf dem Server. So bleibt für die nächste Aufgabe alles aufgeräumt und der Aufräum-Durchgang aus Schritt vorab hat weniger zu tun.
Was bleibt: der Eintrag im Verlaufsprotokoll des Hauptzweigs. Der ist unveränderlich und zeigt nachvollziehbar, wann was geändert wurde.
8
Auf den echten Servern veröffentlichen
Je nachdem, welche Dateien verändert wurden, wird der passende Veröffentlichungsbefehl ausgeführt. Die Tabelle unten zeigt, was wann passiert:
Was wurde verändert? Was passiert dann? Wie prüft man es?
Website-Seiten oder -Bilder Website auf beiden Servern neu bauen und starten web.mentolder.de + web.korczewski.de besuchen
Das 3D-Brett (Systembrett) Brett-Dienst auf beiden Servern neu starten brett.mentolder.de + brett.korczewski.de besuchen
Dokumentations-Inhalte Dokumentations-Seite auf beiden Servern aktualisieren docs.mentolder.de + docs.korczewski.de besuchen
Video-/Livestream-Konfiguration Livestream-Dienst neu verbinden und DNS-Einträge aktualisieren Livestream-Status prüfen

Nur Dokumentation, Tests oder interne Skripte Kein Veröffentlichen nötig

Schlägt eine Prüfung nach dem Veröffentlichen fehl: Kein Notfall-Fix auf dem Hauptzweig. Stattdessen neuen Reparatur-Auftrag über Phase 1 starten.

Was passiert bei Problemen?

Automatische Prüfung schlägt fehl

Der Assistent sucht die Ursache, behebt sie auf derselben Arbeitskopie und lädt die korrigierte Version erneut hoch. Kein zweiter Antrag wird gestellt. Nach zwei erfolglosen Versuchen wird ein interner Kommentar am Aufgabenzettel hinterlassen — der Zettel bleibt auf „In Bearbeitung".

Veröffentlichen schlägt fehl

Das Problem wird protokolliert, Patrick wird benachrichtigt, und die Server bleiben im aktuellen Zustand. Kein automatisches Zurücksetzen. Der Aufgabenzettel bleibt auf „In Bearbeitung" — Patrick entscheidet manuell, was als nächstes passiert.

Prüfung nach Veröffentlichen schlägt fehl

Über Phase 1 wird sofort ein neuer Reparatur-Auftrag gestartet. Der Rückschritt wird wie ein normaler Fehler behandelt. Am betroffenen Aufgabenzettel wird ein interner Kommentar mit der Nummer des neuen Zettels hinterlassen.

Spezialistenteams

Bei größeren Aufgaben arbeitet Claude nicht allein — es gibt spezialisierte Hilfs-Assistenten für verschiedene Bereiche. Bevor einer davon eingesetzt wird, liest der Haupt-Assistent alle aktiven Pläne, damit der Spezialist den vollständigen Kontext kennt.

Fachbegriff-Übersetzungen

Diese Begriffe tauchen häufig in technischen Unterhaltungen auf — hier in verständlichem Deutsch.

Branch / Arbeitszweig
Eine separate Kopie des gesamten Projekts, in der Änderungen gefahrlos ausprobiert werden können, ohne den laufenden Betrieb zu beeinflussen. Wie eine Übungswohnung.
Worktree / Arbeitskopie
Der konkrete Ordner auf dem Computer, in dem an einem Arbeitszweig gearbeitet wird. Mehrere Arbeitskopien können parallel existieren — eine pro offener Aufgabe.
Stale / Verwaist
Eine Arbeitskopie, deren Aufgabe längst abgeschlossen ist, die aber noch nicht weggeräumt wurde. Der „Aufräumen vor dem Start"-Schritt findet solche und bietet das Wegräumen an.
Commit / Änderung speichern
Ein Schnappschuss des aktuellen Zustands aller Dateien — mit Datum, Uhrzeit und einer kurzen Beschreibung, was geändert wurde. Unveränderlich, immer nachvollziehbar.
Push / Hochladen
Die lokalen, gespeicherten Änderungen werden auf den gemeinsamen Server übertragen, damit sie für andere sichtbar und gesichert sind.
Pull Request / Änderungsantrag
Ein formaler Antrag, die Änderungen aus einer Arbeitskopie in den Hauptzweig zu übernehmen. Enthält eine Zusammenfassung und kann von anderen geprüft werden.
Merge / Zusammenführen
Die Änderungen aus der Arbeitskopie werden in den Hauptzweig integriert. Danach sind sie offiziell Teil des Projekts und können veröffentlicht werden.
CI / Automatische Prüfung
Ein automatisches System, das bei jeder Änderung sofort alle Tests ausführt und prüft, ob der Code fehlerfrei und sicher ist. Läuft in der Cloud, ohne menschliches Zutun.
Deploy / Veröffentlichen
Geprüfte Änderungen werden auf die echten, öffentlich zugänglichen Server übertragen. Erst dann sind sie für alle Nutzer sichtbar.
Routine / Cron / Schedule
Eine Aufgabe, die automatisch zu festgelegten Zeiten wiederholt wird (z. B. wöchentlich Abhängigkeiten prüfen). Statt einer einmaligen Reparatur wird das ganze Vorgehen einmal hinterlegt — der Server kümmert sich danach selbst darum.
Kustomize / Konfigurationsdateien
Beschreibungen, wie die verschiedenen Dienste (Website, Datenbank, Chat usw.) auf dem Server eingerichtet und miteinander verbunden sein sollen.
Namespace / Bereich
Eine isolierte Abteilung auf dem Server — ähnlich wie verschiedene Stockwerke in einem Gebäude. Jeder Bereich hat seine eigenen Dienste und Einstellungen.
kubectl / Clusterverwaltung
Das Werkzeug, mit dem der Assistent dem Server-Cluster Befehle gibt: Dienste starten, Protokolle abrufen, Konfigurationen anpassen.
Sealed Secret / Versiegeltes Geheimnis
Ein verschlüsseltes Paket mit Passwörtern und Zugangsdaten. Nur der richtige Server kann es öffnen — selbst wer den Code liest, sieht nur unlesbaren Text.
TDD / Test-zuerst-Methode
Eine Methode, bei der man zuerst eine automatische Probe schreibt, die den gewünschten Zustand beschreibt (und zunächst fehlschlägt), und erst danach den Code schreibt, der sie bestehen lässt.
Spec / Anforderungsbeschreibung
Ein Dokument, das beschreibt, was eine neue Funktion können soll, wie sie sich anfühlen soll und was sie ausdrücklich nicht tun soll.
Plan / Umsetzungsplan
Eine geordnete Liste konkreter Schritte, die ausgeführt werden müssen, um die Anforderungsbeschreibung umzusetzen. Wird zunächst als Datei gespeichert und nach Abschluss in die Datenbank verschoben.
Playwright / Browser-Test
Ein automatisches Werkzeug, das einen echten Browser steuert und Benutzerinteraktionen nachspielt — um zu prüfen, ob die Website korrekt funktioniert.
