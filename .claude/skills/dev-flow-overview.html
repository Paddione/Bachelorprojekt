Schritt 0 — Pfad-Wahl

feature

Neue Funktion

Neues Verhalten, neuer Endpunkt, neue UI-Sektion, neuer Task — alles, was Nutzer bemerken.

Branch: feature/<slug>
fix

Bug-Reparatur

Etwas ist kaputt; Output/Verhalten passt nicht zur Erwartung.

Erfordert T-###### Ticket
chore

Wartungsarbeit

Keine Verhaltensänderung — Dependency-Bumps, Refactors, Doku-Updates, CI-Tweaks.

Kein Plan, kein Ticket nötig

Bestätigung beim User einholen, bevor der gewählte Pfad startet. Beim Fix-Pfad: "Das klingt nach einem fix — Hast du eine T-###### Ticket-ID?"

Skill 2

dev-flow-execute

Ausführung feature/* · fix/* Startet nach dev-flow-plan auf bestehendem Branch
1
Plan finden & lesen
Neuesten Plan laden: ls -t docs/superpowers/plans/*.md | grep -v '/executed/' | head -1
Plan vollständig lesen. Pfad aus Branch-Prefix ableiten (feature/ → Feature, fix/ → Fix).
Ticket-ID aus Frontmatter extrahieren: 
TICKET_ID=$(awk '/^ticket_id:/{print $2; exit}' "$PLAN_FILE")
# z.B. TICKET_ID=T000301 — leer wenn kein Ticket im Plan
1.5
Ticket → in_progress Neu
Falls $TICKET_ID gesetzt: 
kubectl exec "$PGPOD" -n workspace --context mentolder -- \
  psql -U website -d website -At -c \
  "UPDATE tickets.tickets SET status = 'in_progress'
   WHERE external_id = '$TICKET_ID';"
Meldung: „Ticket $TICKET_ID → in_progress"
2
Implementierung
Feature: bevorzugt superpowers:subagent-driven-development (parallele Agents);
alternativ superpowers:executing-plans (sequenziell).
Backend/k8s: TDD via superpowers:test-driven-development · UI: frontend-design + Playwright

Fix: implementieren bis der Failing Test grün ist. Pflicht: red → green → refactor.
./tests/runner.sh local <test-id> → Ziel: PASS
3
Lokale Verifikation
task workspace:validate
./tests/runner.sh local <FA-XX | SA-XX | NFA-XX> — falls relevant
task test:all
4
Pre-Merge Preview auf dev k3d Optional
Status: k3d-Dev-Cluster läuft derzeit nicht. Lokal verifizieren und direkt auf Prod deployen.
5
Pull Request
Ruft commit-commands:commit-push-pr auf.
Feature: feat(<scope>): <kurze-beschreibung>
Fix: fix(<scope>): <kurze-beschreibung> — Body muss Closes T###### enthalten
6
Auto-Merge wenn CI grün
CI muss grün sein vor dem Merge.
6.5
Ticket abschließen Neu
Falls $TICKET_ID gesetzt, nach erfolgreichem Merge:
  • Feature: resolution = 'shipped'
  • Fix: resolution = 'fixed'
UPDATE tickets.tickets
  SET status = 'done', resolution = '$RESOLUTION'
WHERE external_id = '$TICKET_ID';

INSERT INTO tickets.ticket_comments (ticket_id, body, visibility)
SELECT id,
  'PR #$PR_NUM merged. Plan executed: docs/superpowers/plans/<slug>.md',
  'internal'
FROM tickets.tickets WHERE external_id = '$TICKET_ID';
Meldung: „Ticket $TICKET_ID → done ($RESOLUTION)"
7
Plan archivieren
mkdir -p docs/superpowers/plans/executed
mv docs/superpowers/plans/<slug>.md docs/superpowers/plans/executed/
git add docs/superpowers/plans/
git commit -m "chore(plans): mark <slug> as executed"
git push
8
Post-Merge Deploy
Geänderte Dateien prüfen via gh pr view <pr> --json files, dann passenden Task ausführen:
Geänderte Dateien Task Verify
website/src/**, website/public/** task feature:website web.mentolder.de + web.korczewski.de
brett/** task feature:brett brett.mentolder.de + brett.korczewski.de
k3d/docs-content/** task docs:deploy docs.mentolder.de + docs.korczewski.de
k3d/livekit*.yaml task feature:livekit task livekit:status ENV=mentolder + korczewski
k3d/**, prod/**, environments/sealed-secrets/** task feature:deploy task workspace:verify:all-prods + task health
docs/, *.md, tests/, .claude/, scripts/ kein Deploy

Priorität bei mehreren Matches: workspace → website → brett → livekit → docs.
Verify scheitert? → Keinen Fix auf main. Neuen fix/-Branch via dev-flow-plan öffnen.

Failure-Handling

CI rot vor Merge

Diagnose, Fix auf demselben Branch, neu pushen. Keinen zweiten PR aufmachen. Nach 2 erfolglosen Versuchen: internen Ticket-Kommentar hinterlassen. Ticket bleibt auf in_progress.

Deploy scheitert post-merge

Loggen, Patrick benachrichtigen, Cluster wie ist lassen. Kein Auto-Rollback. Ticket bleibt auf in_progress — Patrick muss manuell auf done oder blocked setzen.

Verify scheitert post-merge

Neuen fix/<slug>-Branch via dev-flow-plan Fix-Pfad öffnen. Regression als Bug behandeln. Am aktuellen Ticket internen Kommentar mit dem neuen Ticket-Link hinterlassen.

Agent-Routing

Beide Skills delegieren Spezialarbeit an Sub-Agents. Pflicht vor jedem Dispatch: bash scripts/plan-context.sh <role> ausführen und Output in <active-plans>-Tags voranstellen.
