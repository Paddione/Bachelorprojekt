Backup-Audit & Restore-Test

Vollständiger Audit des Backup-Systems: Setup-Verifikation, Live-Backup auslösen, verschlüsselte Dateien verifizieren und sicheren Restore-Test auf Temp-DB durchführen. Immer auf beiden Clustern — außer der Nutzer gibt explizit einen an.

Auslöser: Backup-Audit, Restore-Test, oder Backup-System-Diagnose
⚠
Immer auf beiden Clustern (mentolder + korczewski) — außer der Nutzer gibt explizit einen Cluster an. Jeder Cluster hat seine eigene shared-db und eigene Backups.
  • Datenbankenkeycloak, nextcloud, vaultwarden, website, docuseal
  • Nicht im BackupNextcloud-File-PVC, VW-Attachments, LiveKit-Recordings
  • Backup-ScheduleTäglich 02:00 UTC (CronJob)
  • VerschlüsselungAES — BACKUP_PASSPHRASE ≥ 32 Bytes
Phasen
1 Setup-Audit CronJob, PVC, Passphrase, ConfigMap 
# CronJob prüfen (Name: db-backup)
kubectl get cronjob -n workspace

# Schedule sollte "0 2 * * *" sein
# Suspended sollte false sein
kubectl describe cronjob db-backup -n workspace

# PVC-Status
kubectl get pvc -n workspace | grep backup

# Letzte Job-History
kubectl get jobs -n workspace --sort-by=.metadata.creationTimestamp | tail -5
⚠
Bekannter Name-Drift: CronJob heißt manchmal backup-postgres statt db-backup. Wenn alter Name: task workspace:deploy ausführen und alten CronJob löschen.
2 Live-Backup auslösen Sofortiges Backup triggern 
task workspace:backup ENV=mentolder
task workspace:backup ENV=korczewski

# Logs beobachten bis "✓ Backup complete"
kubectl logs -n workspace job/<backup-job-name> -f

Erfolgskriterium: Log zeigt "✓ Backup complete" — 5 .dump.enc-Dateien erstellt.

3 Verschlüsselte Dateien verifizieren Busybox-Pod für PVC-Inspektion 
# Busybox-Pod gegen backup-pvc starten
kubectl run backup-inspect --rm -it \
  --image=busybox \
  --overrides='{
    "spec": {
      "volumes": [{"name":"backup","persistentVolumeClaim":{"claimName":"backup-pvc"}}],
      "containers": [{"name":"busybox","image":"busybox",
        "command":["sh"],
        "volumeMounts":[{"name":"backup","mountPath":"/backup"}]
      }]
    }
  }' \
  -n workspace -- sh

# Im Pod:
# ls -lah /backup/  (sollte .dump.enc Dateien zeigen)
# Jede Datei sollte > 1 KB sein

Erwartetes Ergebnis: 5 Dateien — keycloak.dump.enc, nextcloud.dump.enc, vaultwarden.dump.enc, website.dump.enc, docuseal.dump.enc

4 Sicherer Restore-Test NIEMALS auf Produktion — nur Temp-DB
⚠
NIEMALS den Restore-Test auf der Produktions-Datenbank ausführen. Immer eine temporäre Test-DB verwenden. 
# 1. Temp-DB erstellen
task workspace:psql ENV=mentolder -- postgres
-- CREATE DATABASE website_restore_test;

# 2. Neuesten Backup-Timestamp ermitteln
task workspace:backup:list ENV=mentolder

# 3. Website-DB in Temp-DB restoren
# (backup-restore.sh mit --target-db website_restore_test)
bash scripts/backup-restore.sh restore website <timestamp> \
  --target-db website_restore_test \
  --env mentolder

# 4. Tabellen-Anzahl prüfen
task workspace:psql ENV=mentolder -- website_restore_test
-- SELECT schemaname, COUNT(*) FROM pg_tables
-- WHERE schemaname NOT IN ('pg_catalog','information_schema')
-- GROUP BY schemaname;

# 5. Temp-DB aufräumen
-- DROP DATABASE website_restore_test;

Echter Restore-Runbook (Notfall)
# 1. Verfügbare Backups anzeigen
task workspace:backup:list ENV=<env>

# 2. Betroffenen Service auf 0 skalieren
kubectl scale deployment/<service> --replicas=0 -n workspace

# 3. Restore ausführen
task workspace:restore -- <db> <timestamp> ENV=<env>
# db: keycloak|nextcloud|vaultwarden|website|docuseal|all

# 4. Service wieder hochfahren
kubectl scale deployment/<service> --replicas=1 -n workspace
