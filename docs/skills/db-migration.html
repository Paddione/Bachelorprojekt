Schema-Migration

Schreibt, testet und wendet Schema-Migrationen auf beiden Prod-Clustern an. Beide Cluster sind unabhängig — eine Migration auf mentolder propagiert nicht automatisch nach korczewski.

Auslöser: Schema-Änderungen in PostgreSQL — neue Tabellen, Spalten, Views, Schema-Grants
⚠
Beide Cluster sind unabhängig. Eine Migration auf mentolder propagiert NICHT automatisch nach korczewski. Immer explizit auf beiden Clustern anwenden.
  • DB-TypPostgreSQL 16 — shared-db in Namespace workspace
  • Migration-Pfadscripts/datamodel/<NNN>_<name>.sql
  • ER-Diagrammtask db:diagram
  • Datenbankenkeycloak, nextcloud, vaultwarden, website, docuseal
Phasen
1 Migration SQL schreiben Idempotent und transaktional

Dateiname: scripts/datamodel/<NNN>_<beschreibung>.sql

Regeln:

  • IF NOT EXISTS für alle DDL-Statements
  • In BEGIN/COMMIT wrappen
  • Niemals DROP ohne vorherigen Backup
  • Migrations sind idempotent — mehrfaches Ausführen sicher
-- scripts/datamodel/042_add_coaching_topics.sql
BEGIN;

CREATE TABLE IF NOT EXISTS coaching.topics (
  id       uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug     text NOT NULL UNIQUE,
  title    text NOT NULL,
  created  timestamptz DEFAULT now()
);

ALTER TABLE coaching.chunks
  ADD COLUMN IF NOT EXISTS topic_id uuid
  REFERENCES coaching.topics(id);

COMMIT;

Idempotenz-Check mit DO $$:

DO $$ BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM information_schema.columns
    WHERE table_schema = 'public'
      AND table_name = 'my_table'
      AND column_name = 'new_col'
  ) THEN
    ALTER TABLE public.my_table ADD COLUMN new_col text;
  END IF;
END $$;
2 Auf Dev testen Niemals zuerst auf Prod 
# Via task (sicher)
task workspace:psql ENV=dev -- website < scripts/datamodel/<NNN>_<name>.sql

# Oder interaktiv
task workspace:psql ENV=dev -- website
-- \i scripts/datamodel/<NNN>_<name>.sql

# Schema verifizieren
-- \d tablename
-- SELECT * FROM information_schema.columns WHERE table_name='...';
3 Auf mentolder anwenden Backup bei destruktiven Änderungen 
# Bei destruktiven Änderungen: Backup zuerst
task workspace:backup ENV=mentolder

# Migration anwenden
task workspace:psql ENV=mentolder -- website < scripts/datamodel/<NNN>_<name>.sql

# Verifizieren
task workspace:psql ENV=mentolder -- website
-- \d neues_schema
-- SELECT COUNT(*) FROM neues_schema.tabelle;
4 Auf korczewski anwenden Separater Cluster — separat anwenden!
⚠
Korczewski hat eine eigene shared-db. Die Migration von mentolder gilt NICHT automatisch hier. 
task workspace:psql ENV=korczewski -- website < scripts/datamodel/<NNN>_<name>.sql

# Verifizieren
task workspace:psql ENV=korczewski -- website
-- SELECT COUNT(*) FROM neues_schema.tabelle;
5 Grants neu setzen Service-Rollen brauchen Rechte auf neuen Objekten 
# Auf beiden Clustern!
task workspace:fix-tickets-grants ENV=mentolder
task workspace:fix-tickets-grants ENV=korczewski

Grants manuell (falls Schema-Grant nötig):

GRANT USAGE ON SCHEMA neues_schema TO website;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA neues_schema TO website;
ALTER DEFAULT PRIVILEGES IN SCHEMA neues_schema
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO website;
6 ER-Diagramm aktualisieren 
task db:diagram

Generiertes Diagramm in docs/db-schema-diagram.md committen.

7 Migration & Diagramm committen 
git add scripts/datamodel/<NNN>_<name>.sql docs/db-schema-diagram.md
git commit -m "chore(db): add <NNN> migration — <kurzbeschreibung>"
Häufige SQL-Muster
-- Spalte hinzufügen mit Default
ALTER TABLE schema.table
  ADD COLUMN IF NOT EXISTS neue_spalte text DEFAULT '' NOT NULL;

-- Schema mit Rollen-Zugriff erstellen
CREATE SCHEMA IF NOT EXISTS neues_schema;
GRANT USAGE ON SCHEMA neues_schema TO website;

-- View erstellen (idempotent)
CREATE OR REPLACE VIEW schema.meine_view AS
  SELECT id, title, created_at FROM schema.tabelle;

-- Index hinzufügen
CREATE INDEX IF NOT EXISTS idx_tabelle_spalte
  ON schema.tabelle(spalte);
Rollback
ℹ
DDL ist in PostgreSQL transaktional. Wenn die Migration in BEGIN/COMMIT gewickelt ist und fehlschlägt, wird automatisch zurückgerollt. Bei bereits angewendeten Migrationen: Backup-Restore (task workspace:restore). 
# Backup-Liste anzeigen
task workspace:backup:list ENV=mentolder

# Einzelne DB restoren
task workspace:restore -- website <timestamp> ENV=mentolder
