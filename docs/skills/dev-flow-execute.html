Dev-Flow dev-flow-execute

Plan-Ausführung & PR

Nimmt den von dev-flow-plan committeten Plan auf, führt die Implementierung durch, erstellt den PR, merged automatisch bei grünem CI und deployt post-merge. Nur auf feature/* oder fix/* Branches aufrufen.

Auslöser: Auf feature/* oder fix/* Branch — nachdem dev-flow-plan Plan committed hat
⚠
Voraussetzung: Ein committed Plan muss in docs/superpowers/plans/ existieren. Der aktuelle Branch muss mit dem Worktree-Verzeichnis übereinstimmen (Konsistenzcheck Schritt 0).
  • Vorgänger-Skilldev-flow-plan
  • Nachfolger-Skilldev-flow-e2e — nach Merge
  • PR-Formatfeat: ... / fix(T######): ...
  • Auto-MergeJa — wenn CI grün
Ausführungsschritte
0 Worktree-Konsistenzcheck Vor allem anderen

Sicherstellen, dass der aktuelle Branch mit dem Worktree-Verzeichnis übereinstimmt. Wenn nicht: Nutzer informieren, nicht stillschweigend weitermachen.

git branch --show-current
git worktree list --porcelain
0.5 Sync mit main Merge-Konflikte vermeiden 
git fetch origin main
git rebase origin/main
⚠
Bei Konflikten: Rebase abbrechen, Nutzer informieren. Niemals Konflikte blind auflösen — Nutzer muss entscheiden.
1 Plan lesen Neuesten Plan in docs/superpowers/plans/ finden

Den neuesten Plan-File lesen. Bei mehreren Plänen: den zum aktuellen Branch gehörenden wählen (Frontmatter enthält Branch-Referenz).

ls -lt docs/superpowers/plans/ | head -5
1.5 Ticket auf in_progress setzen Tracking aktuell halten 
-- In mentolder Postgres (website DB)
UPDATE tickets.tickets
SET status = 'in_progress', updated_at = now()
WHERE id = 'T######';
2 Implementierung Nach Plan — subagent-driven-development bevorzugt
ℹ
Subagent CWD muss immer explizit gesetzt werden. Subagenten starten im Repo-Root — bei Worktrees den Pfad übergeben.
  • Subagent-driven-development für unabhängige Teilaufgaben (Skill: superpowers:subagent-driven-development)
  • TDD für Backend-Änderungen (Skill: superpowers:test-driven-development)
  • frontend-design für UI-Komponenten (Skill: frontend-design)
  • Parallel fan-out für voneinander unabhängige Tasks
3 Lokale Verifikation Vor PR zwingend 
# Manifest-Validierung (kein Cluster nötig)
task workspace:validate

# Alle Offline-Tests
task test:all

# BATS Unit-Tests
task test:unit

# Kustomize-Manifest-Strukturprüfung
task test:manifests
4 Pre-Merge Preview (optional) dev k3d — aktuell nicht verfügbar

Dev-Cluster auf gekko-hetzner-2. Wenn verfügbar:

task dev:deploy
task dev:cluster:status

Aktuell nicht immer verfügbar — optional überspringen wenn CI-Tests ausreichen.

5 PR erstellen Squash-Merge, CI muss grün sein

PR-Titel-Format:

  • Feature: feat: <kurzbeschreibung>
  • Fix: fix(T######): <kurzbeschreibung>
  • Chore: chore: <kurzbeschreibung>

Fix-PRs enthalten im Body: Closes T######

gh pr create \
  --title "feat: feature-beschreibung" \
  --body "## Summary
- Was wurde implementiert
- Warum

## Test plan
- [ ] task test:all grün
- [ ] Manifeste validiert

Closes T######"
5.5 PR-Nummer notieren Für Ticket-Kommentar 
gh pr view --json number
6 Auto-Merge bei grünem CI Squash-and-merge 
gh pr merge --squash --auto

CI-Checks: BATS unit tests, kustomize Struktur, Taskfile dry-run, Test-Inventory-Check, Security-Scan, Arena-Build + Tests.

6.5 Ticket schließen Status: shipped / fixed 
UPDATE tickets.tickets
SET
  status = 'done',
  resolution = 'shipped',  -- oder 'fixed' bei fix-Branch
  comment = 'PR #<NR> gemergt',
  updated_at = now()
WHERE id = 'T######';
7 Plan archivieren Postgres + Datei löschen

Plan-Inhalt in die Postgres-Tabelle superpowers.plans archivieren, dann die Datei aus docs/superpowers/plans/ entfernen und committen.

7.5 Worktree & Branch aufräumen Nach erfolgreichem Merge 
git worktree remove <worktree-path>
git branch -d <branch-name>
git fetch --prune
