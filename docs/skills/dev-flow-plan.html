Dev-Flow dev-flow-plan

Planung & Branch-Setup

Entry-Point für alle Repo-Änderungen. Ermittelt den Pfad (feature/fix/chore), richtet den Worktree ein, führt Brainstorming durch, schreibt den Plan und committet ihn auf den Branch — dann wird gestoppt. Chores laufen vollständig inline.

Auslöser: Jede Änderungsanfrage im Repository (Feature, Bugfix, Chore)
Übersicht
⚠
Immer dev-flow-plan zuerst. Kein direktes Coden ohne Plan. Dieser Skill ist der einzige Einstiegspunkt für alle Entwicklungsarbeiten — er stellt sicher, dass jede Änderung nachverfolgt, begründet und auf einem isolierten Branch entwickelt wird.

Pfad-Auswahl
F Feature-Pfad Neue Funktionalität
  • 1
    Stale Worktree Audit Bestehende Worktrees prüfen. Veraltete bereinigen oder dem Nutzer melden. 
    git worktree list
git worktree list --porcelain
  • 2
    Pfad-Deklaration Branch-Typ festlegen: feature/<slug> für neue Features, fix/<slug> für Bugfixes, chore/<slug> für Verwaltungsaufgaben.
  • 3
    Worktree erstellen Isolierten Workspace aufsetzen via superpowers:using-git-worktrees. Jeder Feature-Branch lebt in einem eigenen Verzeichnis.
  • 4
    Asset Collection Alle relevanten Dateien lesen, bestehenden Code verstehen, Abhängigkeiten identifizieren. Keine Annahmen treffen ohne Quellcode gelesen zu haben.
  • 5
    Brainstorm-Tunnel aufbauen (optional) Für komplexe Features: Brainstorming-Session mit dem Nutzer via superpowers:brainstorming. Fokus auf User Experience, nicht Implementierungsdetails. 
    bash scripts/superpowers-helper-patch.sh
  • 6
    Plan schreiben Detaillierten Implementierungsplan in docs/superpowers/plans/<slug>.md erstellen. Skill: superpowers:writing-plans. Frontmatter-Hook danach ausführen: 
    bash scripts/plan-frontmatter-hook.sh docs/superpowers/plans/<file>.md
  • 7
    Ticket erstellen In der mentolder tickets-DB einen Eintrag anlegen. Status: backlog, attention_mode: ai_ready.
  • 8
    Assets anhängen & Commit Plan-Datei commiten und auf den Branch pushen. Danach: STOP. dev-flow-execute übernimmt die Implementierung.
F Fix-Pfad Bugfix mit Ticket-Pflicht
⚠
Ticket erforderlich. Fix-Branches benötigen zwingend eine Ticket-ID (T######). Ohne Ticket kein Fix-Branch.
  • 1
    Ticket-ID validieren — Ticket aus DB abrufen, Status auf in_progress setzen
  • 2
    Asset Collection — betroffene Dateien lesen, Fehlerursache verstehen
  • 3
    Worktree erstellen — Branch: fix/T######-<kurzbeschreibung>
  • 4
    Failing Test schreiben — TDD: Zuerst Test der den Bug reproduziert (für Backend-Fixes)
  • 5
    Plan (bei nicht-trivialen Fixes) — Ursache, Fix-Ansatz, Verifikation dokumentieren. Commit & STOP.
C Chore-Pfad Inline — kein separater Execute-Schritt

Chores (Dokumentation, Config-Updates, Dependency-Updates, Refactoring ohne Feature-Änderung) werden vollständig inline ausgeführt — kein separater dev-flow-execute nötig.

  • 1
    Recurring Check — Hat ähnliche Chores in der Vergangenheit gegeben? Muster identifizieren.
  • 2
    Worktree — Branch chore/<slug> erstellen
  • 3
    Änderung durchführen — Direkt implementieren
  • 4
    Verify & PR — task workspace:validate, dann PR erstellen und mergen

Regeln & Constraints
ℹ
Parallel-Sessions: Der Nutzer betreibt bis zu 8–10 gleichzeitige Claude-Prozesse. Ein anderer Agent kann jederzeit den Working Tree ändern. Vor git restore: immer git stash list + git log --all --grep prüfen.
  • Kein git commit ohne vorangehenden Plan
  • Branch-Naming: feature/*, fix/*, chore/*
  • Plan-Datei muss plan-frontmatter-hook.sh durchlaufen (domains + status Frontmatter)
  • Nach Plan-Commit: STOP — dev-flow-execute übernimmt
  • Kein direkter Push auf main
