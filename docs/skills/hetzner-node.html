Infrastruktur hetzner-node

Hetzner-Node Provisioning

Provisioniert neue Hetzner-Nodes oder stellt bestehende Nodes nach einem Rescue-Mode-Reset wieder her. WireGuard-Key-Management ist der kritische Pfad — ein einmal generierter Key wird in .secrets/<env>.yaml gespeichert und überlebt jeden Reset.

Auslöser: Neuen Hetzner-Node provisionieren oder nach Rescue-Mode-Reset wiederherstellen
  • Cluster mentolder3 CP auf gekko-hetzner-2/3/4, 6 Worker k3s-1..3 + k3w-1..3
  • Cluster korczewski1 CP pk-hetzner-4, 2 Worker pk-hetzner-6/8
  • WireGuardwg-mesh — alle mentolder-Nodes müssen Mitglied sein
  • SSH mentolderroot@gekko-hetzner-*
  • SSH korczewskipatrick@pk-hetzner-4/6/8 (AllowUsers patrick)
⚠
WireGuard-Key-Design: Der private Key wird einmalig generiert und in environments/.secrets/<env>.yaml gespeichert. Bei Recovery eines bestehenden Nodes den gleichen Key verwenden — dann sind keine Peer-Updates nötig.

0b WireGuard-Key-Entscheidung Bestehend oder neu generieren?

Recovery-Pfad (Node existiert bereits in .secrets): Den bestehenden Key aus environments/.secrets/<env>.yaml verwenden — keine Peer-Updates nötig.

Neuer Node: Key generieren:

# Primär
wg genkey

# Fallback (wenn wg nicht verfügbar)
python3 -c "
from cryptography.hazmat.primitives.asymmetric.x25519 import X25519PrivateKey
import base64
k = X25519PrivateKey.generate()
print(base64.b64encode(k.private_bytes_raw()).decode())
"

Privaten Key sofort in environments/.secrets/<env>.yaml speichern.

1 Cloud-Config oder Setup-Script generieren Je nach Modus

Neuer Server (cloud-init YAML):

  • WireGuard-Peer-Block: 6-Leerzeichen-Einrückung in YAML
  • k3s-Install-Script mit korrekter Rolle
  • node-ip auf öffentliche IP setzen
  • flannel-iface=wg-mesh für mentolder-Nodes

Rescue Mode (Bash-Script):

  • Kein cloud-init — Script manuell nach installimage ausführen
  • WireGuard konfigurieren, dann k3s installieren
# k3s control-plane-init
curl -sfL https://get.k3s.io | sh -s - server \
  --cluster-init \
  --node-ip=<PUBLIC_IP> \
  --flannel-iface=wg-mesh \
  --disable=traefik  # nur wenn eigenes Traefik

# k3s worker join
curl -sfL https://get.k3s.io | K3S_URL=https://<CP_IP>:6443 \
  K3S_TOKEN=<TOKEN> sh -s - agent \
  --node-ip=<PUBLIC_IP> \
  --flannel-iface=wg-mesh
2a Neuen Server erstellen (New Server-Pfad) hcloud CLI 
hcloud server create \
  --name <node-name> \
  --type <server-type> \
  --image ubuntu-24.04 \
  --location <hel1|nbg1|fsn1> \
  --user-data-from-file cloud-init-<env>-<node>.yaml \
  --ssh-key <key-name>
2b Rescue Mode (Rescue-Pfad) Kein cloud-init! 
# 1. Rescue Mode aktivieren
hcloud server enable-rescue --type linux64 <server-id>

# 2. Server neu starten
hcloud server reboot <server-id>

# 3. In Rescue booten und installimage ausführen
ssh root@<ip> "installimage -a -r no -i /root/.oldroot/nfs/install/../images/Ubuntu-2404-noble-amd64-base.tar.gz -p /,ext4,all -d sda -f yes -t yes -n <hostname>"

# 4. Nach Reboot: Setup-Script ausführen
# (KEIN cloud-init in Rescue Mode)
scp setup-script.sh root@<ip>:/root/
ssh root@<ip> "bash /root/setup-script.sh"
3 Setup-Script ausführen (Rescue) / cloud-init abwarten (New) 
# cloud-init Status prüfen (New Server)
ssh root@<ip> "cloud-init status --wait"
ssh root@<ip> "journalctl -u cloud-init -f"
4 WireGuard Peer-Sync Recovery=auto, Neuer Key=alle Peers aktualisieren

Recovery-Pfad: Existierender Key → kein Update nötig.

Neuer Key-Pfad: Alle bestehenden Nodes mit dem neuen Public Key aktualisieren:

# Neuen Public Key aus Private Key ableiten
echo "<PRIVATE_KEY>" | wg pubkey

# Auf jedem bestehenden Node:
wg set wg-mesh peer <NEW_PUBLIC_KEY> \
  allowed-ips <NEW_WG_IP>/32 \
  endpoint <NEW_PUBLIC_IP>:51820

# Persistent speichern
wg-quick save wg-mesh
⚠
wg-mesh-Mitgliedschaft ist load-bearing für mentolder. Einen Node ohne wg-mesh-Join hinzufügen bricht stillschweigend den Pod-zu-Pod-Traffic von diesem Node.
5 k3s Join verifizieren Node muss Ready sein 
kubectl get nodes
kubectl get nodes -o wide

# Node labels setzen
kubectl label node <node-name> \
  kubernetes.io/role=worker \
  node.kubernetes.io/instance-type=<type>
6 Post-Provisioning Checkliste
  • kubeconfig aktualisieren (kubectl config merge wenn nötig)
  • Firewall-Regeln prüfen (ufw: 7880/7881/tcp, 50000-60000/udp, 30000-40000/udp)
  • Janus-Ports falls Nextcloud Talk auf diesem Node: 20000-20200/udp
  • LiveKit-Pinning wenn benötigt: task livekit:dns-pin APPLY=true ENV=<env>
  • task workspace:deploy ENV=<env> um neue Node-Affinitäten zu aktivieren
