Keycloak Realm-Synchronisation

Synchronisiert den Keycloak-Realm-Zustand mit den JSON-Definitionen. Kein direktes UI-Bearbeiten ohne gleichzeitiges JSON-Update — sonst driften Realm-State und IaC-Definition auseinander.

Auslöser: Neue OIDC-Clients, geänderte Redirect-URIs, Mapper-Probleme, SSO-Login-Fehler
⚠
Niemals Realm-State in Keycloak UI editieren ohne JSON zu aktualisieren. UI-Only-Änderungen werden beim nächsten keycloak:sync überschrieben.

Phasen
1 Pre-Sync Check Keycloak Pod muss Running sein 
kubectl get pods -n workspace -l app=keycloak
task workspace:logs ENV=<env> -- keycloak
2 Realm JSON bearbeiten Wenn Änderungen nötig

Häufig geänderte Felder:

  • redirectUris — nach URL-Änderungen
  • webOrigins — für CORS
  • protocolMappers — User-Attribute in Token
  • groups — Gruppen-Definitionen
# JSON validieren vor Sync
python3 -c "import json; json.load(open('prod-mentolder/realm-workspace-mentolder.json'))"

3 Sync ausführen Realm-JSON auf Cluster anwenden 
task keycloak:sync ENV=<env>
4 Protocol Mappers sicherstellen Immer nach keycloak:sync 
bash scripts/keycloak-ensure-mappers.sh <env>

Dieser Script stellt sicher, dass alle erforderlichen Protocol Mapper (User-Attribute, Gruppen-Mitgliedschaften) in den Clients konfiguriert sind.

5 OIDC-Clients verifizieren Redirect-URIs aller Clients prüfen

Im Keycloak Admin-UI oder via API prüfen:

# Keycloak Admin-API (Token holen)
KC_TOKEN=$(curl -s -X POST \
  https://auth.<domain>/realms/master/protocol/openid-connect/token \
  -d "grant_type=password&client_id=admin-cli&username=admin&password=<pw>" \
  | jq -r .access_token)

# Clients listen
curl -s -H "Authorization: Bearer $KC_TOKEN" \
  "https://auth.<domain>/admin/realms/workspace/clients" | jq '.[].clientId'
6 SSO-Flow testen Incognito-Browser-Login

Im Incognito-Fenster testen:

  • https://web.<domain> → Login-Redirect zu Keycloak → Zurück zur Website
  • https://files.<domain> → Nextcloud OIDC-Login
  • https://vault.<domain> → Vaultwarden SSO
7 Gruppen-Mitgliedschaften prüfen Besonders /dev-access für dev.mentolder.de

Dev-Cluster-Zugriff wird über /dev-access-Gruppe enforced. Nutzer ohne diese Gruppe bekommen 403-Loop beim Login auf dev.mentolder.de.
