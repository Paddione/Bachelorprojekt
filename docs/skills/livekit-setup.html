Infrastruktur livekit-setup

LiveKit WebRTC Setup

Diagnosiert und repariert ICE-Verbindungsprobleme, broken Streams, RTMP-Ingestion-Fehler und Recording-Ausfälle. LiveKit läuft nur auf mentolder — korczewski nutzt einen separaten WebRTC-Pfad über arena-server.

Auslöser: ICE fails, Stream broken, RTMP ingestion broken, Recording failing
⚠
Mentolder-only. Arena-server auf korczewski verwendet einen separaten WebRTC-Pfad und wird hier nicht behandelt. LiveKit-Server läuft mit hostNetwork: true — Pod-Security ist privileged.
Architektur
  • livekit-servergekko-hetzner-3 (46.225.125.59), hostNetwork: true, NodeAffinity
  • livekit-ingressRTMP-Ingestion, gleicher Node
  • livekit-egressRecording auf PVC, beliebiger Node
  • livekit.<domain>DNS MUSS auf 46.225.125.59 gepinnt sein
  • stream.<domain>DNS MUSS auf 46.225.125.59 gepinnt sein
  • Admin-Seite/admin/stream
  • Zuschauer-Seite/portal/stream
Phasen
1 Status-Check Erster Überblick 
task livekit:status ENV=mentolder

# Detailliert
kubectl get pods -n workspace -l app=livekit-server
kubectl get pods -n workspace -l app=livekit-ingress
kubectl get pods -n workspace -l app=livekit-egress
kubectl get ingress -n workspace | grep livekit

# Logs
task livekit:logs ENV=mentolder
task livekit:logs ENV=mentolder -- ingress
task livekit:logs ENV=mentolder -- egress
2 DNS-Pinning verifizieren Kritisch für ICE
⚠
Browser treffen ~66% der Zeit einen Nicht-LiveKit-Node wenn DNS nicht gepinnt ist. ICE schlägt dann stillschweigend fehl. 
# DNS-Status prüfen
dig livekit.mentolder.de
dig stream.mentolder.de

# Soll auf 46.225.125.59 zeigen!
# Wenn falsch: DNS korrigieren
task livekit:dns-pin ENV=mentolder          # Zeigt API-Calls
task livekit:dns-pin ENV=mentolder APPLY=true # Führt aus
3 Firewall-Regeln prüfen UDP-Ports für ICE, TCP für Signaling

Auf dem Pin-Node (gekko-hetzner-3, 46.225.125.59) müssen folgende Ports offen sein:

PortProtokollVerwendung
7880TCPLiveKit-Server HTTP/WebSocket
7881TCPLiveKit-Server gRPC
50000–60000UDPICE/TURN (LiveKit)
30000–40000UDPICE/TURN (LiveKit Ingress)
20000–20200UDPJanus/TURN (separat — nicht in livekit:firewall-open)
 
task livekit:firewall-open NODE=46.225.125.59

# Manuell auf Node (via SSH)
ssh root@46.225.125.59 "ufw status verbose"
⚠
Janus 20000-20200/udp ist NICHT in livekit:firewall-open enthalten. Muss separat geöffnet werden wenn Nextcloud Talk auf diesem Node läuft.
4 Node-Affinity verifizieren Pod muss auf gekko-hetzner-3 laufen 
kubectl get pod -n workspace -l app=livekit-server -o wide

# Wenn Pod auf falschem Node: NodeAffinity-Label setzen
kubectl label node gekko-hetzner-3 \
  livekit.io/pin-node="true"

# Pod neu starten
kubectl rollout restart deployment/livekit-server -n workspace
5 Stream-Test OBS RTMP + Browser WebRTC

OBS RTMP-Test:

  • Stream-Key von /admin/stream holen
  • OBS: Server rtmp://stream.mentolder.de/live
  • Stream starten, RTMP-Logs prüfen

Browser WebRTC-Test:

  • Muss aus einem User-Gesture heraus aufgerufen werden
  • Chrome blockiert AudioContext ohne User-Gesture
  • /portal/stream im Browser öffnen — auf Klick warten
task livekit:logs ENV=mentolder -- ingress # RTMP-Verbindung beobachten
task livekit:logs ENV=mentolder           # Room-Joins beobachten
6 Recording Egress + PVC-Speicher 
# MP4-Liste im Egress PVC
task livekit:recordings ENV=mentolder

# PVC-Speicher prüfen
kubectl get pvc -n workspace | grep livekit
kubectl exec -n workspace deploy/livekit-egress -- df -h /recordings
Notfall-Prozedur
⚠
Hängender Stream / keine Möglichkeit den Raum zu schließen: 
# LiveKit-Server neu starten — schließt ALLE aktiven Räume
task livekit:end-stream ENV=mentolder
