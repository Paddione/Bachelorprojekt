Querschnitt mishap-tracker

Mishap-Log → Tickets

Konsumiert MISHAP_LOG-Einträge aus anderen Skills und materialisiert sie als Tickets in der mentolder Postgres-Datenbank. Passiver Side-Effect — wird niemals direkt aufgerufen.

Auslöser: Automatisch aus Post-Execution-Section anderer Skills — NIEMALS direkt aufrufen
⚠
NIEMALS direkt aufrufen. Dieser Skill ist ausschließlich für die Post-Execution-Section anderer Skills vorgesehen. Er hat keine eigenen Trigger-Bedingungen und kein direktes Interface.
Rolle im System
Produzenten

Alle anderen Skills pflegen während der Ausführung ein MISHAP_LOG. Sie notieren dort Anomalien, Abweichungen und potenzielle Probleme.

Konsument

mishap-tracker ist der einzige Konsument. Er liest das MISHAP_LOG am Ende jeder Skill-Ausführung und erstellt Tickets aus den Einträgen.

Ablauf
1 MISHAP_LOG prüfen Gibt es Einträge?

Falls MISHAP_LOG leer ist:

Keine Mishaps gefunden. ✓

Dann sofort stoppen — kein Ticket wird erstellt.

Falls Einträge vorhanden: Weiter mit Schritt 2.

2 Severity-Mapping MISHAP_LOG-Typ → Ticket-Felder
MISHAP_LOG type tickets.type tickets.severity
broken bug major
degraded bug minor
suspicious task minor
security bug critical
drift task trivial
3 Tickets in Postgres INSERT Für jeden MISHAP_LOG-Eintrag 
-- Verbindung: kubectl exec auf shared-db, mentolder Cluster
-- psql -U website -d website

INSERT INTO tickets.tickets (
  type,        -- per Mapping-Tabelle
  brand,       -- 'mentolder'
  status,      -- 'triage'
  severity,    -- per Mapping-Tabelle
  title,       -- aus MISHAP_LOG-Eintrag
  description, -- Eintrag + betroffene Komponente
  attention_mode -- 'ai_ready' oder 'needs_human'
) VALUES (
  <type>,
  'mentolder',
  'triage',
  <severity>,
  '[Auto] <mishap-titel>',
  'Automatisch erkannt während <skill-name>.
Komponente: <betroffene-komponente>
Details: <mishap-beschreibung>',
  'ai_ready'
);

Verbindungs-Methode:

kubectl exec -n workspace deploy/shared-db -- \
  psql -U website -d website -c "INSERT INTO ..."
MISHAP_LOG Format (für Skill-Autoren)

Jeder Skill, der Anomalien während der Ausführung erkennt, schreibt ins MISHAP_LOG:

# MISHAP_LOG Eintrag-Struktur:
MISHAP_LOG += {
  type: "broken" | "degraded" | "suspicious" | "security" | "drift",
  component: "keycloak" | "shared-db" | "livekit" | ...,
  title: "Kurzer Titel",
  description: "Ausführliche Beschreibung was beobachtet wurde"
}
Beispiel-Einträge
SkillTypBeispiel-Titel
backup-check broken CronJob db-backup suspended auf korczewski
livekit-setup drift DNS livekit.mentolder.de zeigt auf falschen Node
secret-rotation security Anthropic API-Key gibt 401 auf mentolder
deployment-assist suspicious Pod OOMKilled 3x in letzten 24h
db-migration drift Schema-Version auf korczewski hinter mentolder
ℹ
Design-Philosophie: mishap-tracker entkoppelt Anomalie-Erkennung (in den Skills verteilt) von der Ticket-Erstellung (zentralisiert hier). Skills müssen sich nicht um Datenbankverbindungen kümmern — sie loggen nur. mishap-tracker macht daraus strukturierte, trackbare Tickets.
