OpenClaw Local AI Gateway

Bootstrap, Restart, Debugging und Reset von OpenClaw auf dem WSL-Host — dem lokalen AI-Gateway das direkt mit Ollama auf der GPU-Box kommuniziert, ohne den Cluster-llm-router zu verwenden.

Auslöser: OpenClaw reagiert nicht, GPU-Verbindung bricht ab, oder Ersteinrichtung nötig
ℹ️
Läuft auf dem WSL-Host, nicht in Kubernetes. OpenClaw verbindet sich direkt mit Ollama auf 10.10.0.3:11434/v1 über das wg-mesh-VPN. llm-router hat kein Ingress — OpenClaw ist der einzige externe Pfad zu Ollama.
SettingWert
Ollama Base URLhttp://10.10.0.3:11434/v1
ZugangspfadDirekt über wg-mesh VPN
GPU-HostRTX 5070 Ti, 16 GB VRAM
Modell-ManagementOllama auf der GPU-Box

Bootstrap (Ersteinrichtung)

task openclaw:install    # OpenClaw Binary / Service installieren
task openclaw:configure  # Config auf 10.10.0.3:11434/v1 setzen

Täglicher Betrieb

task openclaw:start   # OpenClaw-Daemon (neu)starten
task openclaw:status  # Health-Probe: Ollama-Erreichbarkeit + Service-Status
task openclaw:logs    # journalctl-Tail für den openclaw-Service

Backup / Restore / Wipe

task openclaw:backup            # Snapshot ~/.openclaw → timestamped Archive
task openclaw:restore           # Aus dem neuesten Snapshot wiederherstellen
task openclaw:wipe CONFIRM=yes  # Destruktiv: ~/.openclaw löschen

wipe erfordert CONFIRM=yes um versehentlichen Datenverlust zu verhindern.

Architektur

WSL-Host → OpenClaw → Ollama (10.10.0.3:11434/v1)   [dieser Skill]

Cluster-Pods → llm-router → {embed:8081, rerank:8082, chat:11434}   [Cluster-Pfad]

Die beiden Pfade sind unabhängig. Embedding-Collections im Cluster (bge-m3) schlagen geschlossen fehl wenn der GPU-Host nicht erreichbar ist — OpenClaw ist davon unbeeinträchtigt.
