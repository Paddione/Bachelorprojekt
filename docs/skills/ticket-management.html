Betrieb ticket-management

Ticket-Triage & Auflösung

Triagiert und löst offene Tickets in der mentolder Tickets-Datenbank. Kategorisiert nach Actionability, schließt Resolved/Obsolete in Batches, behebt AI-fixbare Tickets autonom und leitet Human-needed mit konkreten Anfragen weiter.

Auslöser: Offene Tickets durcharbeiten, triagieren oder lösen
  • Verbindungtask workspace:psql ENV=mentolder -- website
  • Tabelletickets.tickets
  • UIhttps://web.mentolder.de/admin/bugs
  • ID-FormatBR-YYYYMMDD-xxxx
Gültige Feldwerte — NUR diese verwenden
FeldGültige Werte
status triage backlog in_progress in_review blocked done archived
resolution fixed | shipped | wontfix | duplicate | cant_reproduce | obsolete
attention_mode auto | ai_ready | needs_human
priority hoch | mittel | niedrig
severity critical major minor trivial
⚠
Freie Textwerte für status/resolution sind ungültig. Immer exakt einen der oben gelisteten Werte verwenden.
Session-Workflow
1 Offene Tickets holen Aktuelle Ticket-Liste 
SELECT id, type, status, severity, priority,
       attention_mode, title, created_at
FROM tickets.tickets
WHERE status NOT IN ('done', 'archived')
ORDER BY
  CASE severity
    WHEN 'critical' THEN 1
    WHEN 'major'    THEN 2
    WHEN 'minor'    THEN 3
    ELSE 4
  END,
  created_at DESC;
2 In 4 Buckets kategorisieren Vor jeder Aktion zeigen und bestätigen lassen
BucketKriteriumAktion
Bereits gelöst In production deployed, kein Ticket-Update Batch-close: status=done, resolution=fixed/shipped
Obsolet Nicht mehr relevant, Stack-Entfernung, Duplicate Batch-close: resolution=obsolete/duplicate/wontfix
AI-fixbar Klare Beschreibung, attention_mode=ai_ready, kein Hardware/Auth-Zugriff nötig Fan-out: dev-flow-plan + dev-flow-execute
Human needed Erfordert physischen Zugriff, Nutzer-Entscheidung, externe Credentials attention_mode=needs_human + spezifische Anfrage
⚠
Kategorisierung immer zuerst zeigen — dann Nutzer-Bestätigung abwarten. Niemals blind schließen oder fixen.
3 Batch-Close bereits gelöster Tickets 
UPDATE tickets.tickets
SET
  status = 'done',
  resolution = 'shipped',
  comment = 'Deployed in PR #<NR>',
  updated_at = now()
WHERE id IN ('T######', 'T######');
4 AI-fixbare Tickets parallel fixen Fan-out mit dev-flow-plan

Für jedes ai_ready-Ticket: dev-flow-plan auf einem fix/T######-...-Branch starten. Bei unabhängigen Fixes: parallel ausführen (Skill: superpowers:dispatching-parallel-agents).

-- Ticket auf in_progress setzen bevor Fix beginnt
UPDATE tickets.tickets
SET status = 'in_progress', updated_at = now()
WHERE id = 'T######';
5 Human-needed Tickets kommentieren Spezifische Anfrage — keine vagen Notizen 
UPDATE tickets.tickets
SET
  attention_mode = 'needs_human',
  comment = 'Benötige: <spezifische Information/Entscheidung/Zugriff>.
Dann kann ich: <was danach automatisch passiert>.',
  updated_at = now()
WHERE id = 'T######';
Häufige Fehler
FehlerWarum falsch
In-Progress Plan-Tickets schließen dev-flow-execute schreibt Status zurück — verursacht Konflikte
Freie Resolution-Texte verwenden DB-Constraint-Fehler; nur die 6 validen Werte erlaubt
Vage needs_human-Notizen "Nutzer muss entscheiden" ist nutzlos — spezifisch welche Entscheidung
Tests durch Assertions-Änderungen "fixen" Rote Tests zeigen echte Bugs an — nie Symptom behandeln
Ohne Nutzer-Bestätigung handeln Kategorisierung zuerst zeigen, dann auf Bestätigung warten
