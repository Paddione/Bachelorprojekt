# korczewski pvc-backup Longhorn Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the korczewski `pvc-backup` CronJob succeed by migrating `vaultwarden-data-pvc` (5Gi) and `docuseal-data-pvc` (2Gi) from `local-path` to `longhorn`, mirroring the mentolder setup (T000317).

**Architecture:** korczewski's `pvc-backup` mounter is stuck `Pending` because its three local-path data volumes span two nodes (`nextcloud`+`docuseal`+`backup-pvc` on `pk-hetzner-8`, `vaultwarden` on `pk-hetzner-6`) and local-path PVs are node-pinned, so no single node can mount all of them. PR #1296 ("local-path-aware") does NOT fix this — its mounter `requiredDuringScheduling` podAffinity demands every live local-path source's owning pod on one node. The fix is the mentolder pattern: pin `vaultwarden`+`docuseal` to `longhorn` (placement-independent CSI clone — #1296's code already clones longhorn-backed PVCs), leaving only single-node `nextcloud-data` (50Gi, excluded for lack of Longhorn headroom) and `backup-pvc` on local-path, both already co-located on `pk-hetzner-8`, so the mounter schedules on `pk-hetzner-8`. `storageClassName` is immutable on a bound PVC, so each service is migrated one at a time via a temporary Longhorn staging PVC.

**Tech Stack:** Kustomize, kubectl (`--context fleet`, namespace `workspace-korczewski`), alpine:3 migration pods, BATS (`tests/unit/manifests.bats`).

**Cluster facts (verified 2026-06-01):**
- Context `fleet`, namespace `workspace-korczewski`.
- fleet HAS `longhorn` storageClass (all 6 nodes `allowScheduling=true`). The `prod-korczewski/kustomization.yaml` comment "Longhorn is NOT installed on korczewski" is stale standalone-era lore — correct it.
- PVC → node map: `backup-pvc`→pk-8 (local-path, 1Gi), `nextcloud-data-pvc`→pk-8 (local-path), `docuseal-data-pvc`→pk-8 (local-path), `vaultwarden-data-pvc`→pk-6 (local-path).
- db-backup half of T000403 is ALREADY fixed live (GRANT drift) — out of scope here.

**Footguns (do NOT skip):**
- **Do NOT use `kubectl apply -k prod-fleet/korczewski`** to (re)create a PVC. That re-applies the whole overlay — including `pvc-backup-cronjob.yaml`, whose `$$`-escaped shell vars are only reduced to `$` by the push-deploy `sed -E 's/\$\$([a-zA-Z0-9_]|\{)/$\1/g'` (Taskfile ~1843). A raw `apply -k` leaves doubled `$$` and the orchestrator aborts with `SRC: parameter not set` under `set -u`. Create the new PVC with a **direct minimal `kubectl apply -f`** (PVCs carry no `$$` vars), and run the full overlay reconcile ONCE at the end via `task workspace:deploy ENV=korczewski`.
- **korczewski `backup-pvc` is local-path/1Gi** — NOT usable as the migration staging area (mentolder's is longhorn). Use a dedicated temporary Longhorn staging PVC per service.
- **`vaultwarden` holds the only copy** of its data (its data PVC, not a shared DB). The staging copy is the sole safety net — verify the copy before deleting the source PVC.

```bash
# two data PVCs the pvc-backup mounter must reach placement-independently.
# local-path PVCs are node-pinned RWO; the mounter cannot mount PVCs from two
# nodes at once, and korczewski's vaultwarden lives on pk-hetzner-6 while
# nextcloud/docuseal/backup-pvc live on pk-hetzner-8. Longhorn is distributed —
# pvc-backup clones the longhorn-backed PVCs (placement-independent), so the
# mounter only needs to co-locate with the remaining single-node local-path
# source (nextcloud, on pk-hetzner-8). [T000403]
#
# NOTE: storageClass is immutable on an existing bound PVC. Apply this patch
# AFTER the manual data migration in Tasks 3–4 (scale down → copy to a temp
# Longhorn staging PVC → delete old PVC → create new Longhorn PVC → restore).
#
# nextcloud-data-pvc (50Gi) is intentionally EXCLUDED — the Longhorn nodes lack
# headroom for 3×50Gi replicas; nextcloud-data stays on local-path (pk-hetzner-8)
# and is mounted LIVE by the mounter via podAffinity, same as mentolder.
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: vaultwarden-data-pvc
spec:
  storageClassName: longhorn
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: docuseal-data-pvc
spec:
  storageClassName: longhorn
EOF
```

For each service, copy the live data to your workstation BEFORE touching any PVC. vaultwarden source is on pk-hetzner-6, docuseal on pk-hetzner-8.

```bash
mkdir -p /tmp/t000403-premigrate

kind: Pod
metadata: { name: premigrate-vaultwarden, namespace: workspace-korczewski }

  nodeName: pk-hetzner-6
  restartPolicy: Never
  containers:
  - name: c
    image: alpine:3
    command: ["/bin/sh","-c","sleep 3600"]
    volumeMounts: [{ name: src, mountPath: /source, readOnly: true }]
  volumes:
  - name: src
    persistentVolumeClaim: { claimName: vaultwarden-data-pvc }

kubectl --context fleet -n workspace-korczewski wait pod/premigrate-vaultwarden --for=condition=Ready --timeout=90s
kubectl --context fleet -n workspace-korczewski exec premigrate-vaultwarden -- sh -c 'cd /source && tar czf - .' > /tmp/t000403-premigrate/vaultwarden-$(date +%Y%m%d-%H%M%S).tar.gz
```
