## Task Breakdown

### T1 — free-fly-camera.ts (neues Kernmodul)

**Target file:** `brett/src/client/free-fly-camera.ts`

Erstellt das neue, in sich gekapselte Free-Fly-Kamera-Modul. Enthält:
- State: Position (`THREE.Vector3`), Yaw/Pitch (Zahlen), Geschwindigkeitsvektor, gedrückte-Tasten-Set, `_active`-Flag, Pointer-Lock-Status.
- `enterFreeFly(camera, domElement)` — setzt Position/Yaw/Pitch aus aktueller Kamera, fordert Pointer-Lock an, registriert `mousemove`- und `pointerlockchange`-Listener.
- `exitFreeFly()` — verlässt Pointer-Lock, räumt Listener auf, setzt `_active = false`.
- `isFreeFly()` — gibt `_active` zurück.
- `tickFreeFly(dt, camera)` — integriert Tastatureingaben (WASD/QE/Space/Ctrl, Shift-Boost) mit exponentiellem Smoothing (`1 - exp(-k*dt)`), schreibt `camera.position` + `camera.lookAt`, klemmt Position in die Arena-Box (`x,z ∈ [-30,30]`, `y ∈ [0.3,25]`). Schreibt Kamera **nur** wenn `_active`.
- Interne Keyboard-State-Verwaltung (`keydown`/`keyup` auf `window`).
- Pointer-Lock-Fehler-Handling: bei `pointerlockerror` läuft der Modus weiter, Maus-Look inaktiv.
- Pitch-Klemmung auf ±85° (kein Gimbal-Flip).

**Acceptance Criteria:**
- Datei existiert und exportiert `enterFreeFly`, `exitFreeFly`, `isFreeFly`, `tickFreeFly`.
- Kein Import aus `scene.ts` oder anderen Modulen mit DOM/WebGL-Abhängigkeit.
- TypeScript kompiliert ohne Fehler (`tsc --noEmit` aus `brett/`).

### T2 — scene.ts: Orbit-State-Getter/Setter

**Target file:** `brett/src/client/scene.ts`

Erweitert `SceneApi` um zwei neue Methoden für den nahtlosen Orbit↔Free-Fly-Übergang:
- `getOrbitState()` — gibt aktuellen Orbit-Zustand zurück (theta, phi, dist, target).
- `setCameraToOrbit(position: THREE.Vector3)` — rekonstruiert Orbit-Parameter (theta/phi/dist) aus einer beliebigen Kamera-Position relativ zu `(0,1,0)`, sodass der erste Orbit-Frame exakt der letzten Free-Fly-Pose entspricht (kein Sprung). Orbit-Mathematik selbst bleibt unverändert.

**Acceptance Criteria:**
- `SceneApi`-Interface und Implementierung enthalten beide neuen Methoden.
- Keine bestehende Orbit-Logik oder Szenen-Initialisierung wurde verändert.
- TypeScript kompiliert ohne Fehler.

### T3 — board-boot.ts: Tick-Loop + Input-Handler

**Target file:** `brett/src/client/board-boot.ts`

Verdrahtet das neue `free-fly-camera`-Modul in den Tick-Loop und die Input-Verarbeitung:
- Tick-Loop-Priorität (Single-Writer pro Frame):
  1. `if (isInPov() || povLerpActive) → tickPov(); return;`
  2. `else if (isFreeFly()) → tickFreeFly(dt, camera); return;`
  3. `else updateCameraFromOrbit();`
- Keydown-Handler: Taste `F` → `enterFreeFly()` / `exitFreeFly()` (toggle), nur wenn kein Possession aktiv.
- Esc-Handler: wird **vor** bestehender Esc-Logik registriert; bei aktivem Free-Fly `exitFreeFly()` + `stopImmediatePropagation()`.
- Pointer-Lock-`change`-Listener: Lock-Verlust pausiert nur Maus-Look, verlässt Free-Fly **nicht** automatisch.
- POV-Eintritt (`figure_possess`-Event) ruft synchron `exitFreeFly()`.
- Orbit-`mousedown`-Handler prüft `isFreeFly()` am Anfang und kehrt früh zurück (kein Doppel-Schreiber).
- `enterFreeFly` erhält aktuelle Orbit-Position via `scene.getOrbitState()`; `exitFreeFly` ruft `scene.setCameraToOrbit(camera.position)`.

**Acceptance Criteria:**
- Exakt ein Kamera-Modus schreibt pro Frame (nachweisbar durch Code-Review der Tick-Loop-Struktur).
- `F`-Taste funktioniert nur im Observer-/Nicht-Besitzer-Zustand.
- Esc verlässt Free-Fly vor allen anderen Esc-Aktionen.
- TypeScript kompiliert ohne Fehler.

### T5 — Unit-Tests: free-fly-camera.ts

**Target file:** `brett/test/free-fly-camera.test.ts`

Headless Unit-Tests für das neue `free-fly-camera`-Modul (node:test / tsx, Three.js ohne WebGL-Kontext):
- `tickFreeFly` Framerate-Unabhängigkeit: gleiche Eingabe, identische Gesamtzeit, 2×/0.5× dt → Endposition ±ε.

- Orbit→Free-Fly→Orbit Übergang: `setCameraToOrbit(finalPos)` rekonstruiert Orbit-Parameter so, dass erster Orbit-Frame ≡ letzter Free-Fly-Frame (±ε).

**Acceptance Criteria:**
- Alle Tests laufen mit `task brett:test` (oder entsprechendem Vitest-Befehl) durch.
- Kein WebGL-Kontext oder DOM erforderlich (headless).
