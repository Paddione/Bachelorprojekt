# Plan: Infra: Automatisches Rollback bei Failed Deploy

**Ticket:** 1e1c2151
**Branch:** feature/1e1c2151-auto-rollback
**Datum:** 2026-06-10
**Status:** staged

---

## Ziel

Post-Deploy-Monitoring mit automatischem Rollback und Pushover-Benachrichtigung. Erkennt CrashLoopBackOff innerhalb von 5 Minuten und rollt auf das letzte erfolgreiche Image zurueck, wenn der Healthcheck 3x hintereinander scheitert.

## Architektur

### Neue Dateien

- `scripts/lib/notify.sh` — Sourcable Library mit `notify_pushover <title> <message> [priority]`. Liest `PUSHOVER_TOKEN` und `PUSHOVER_USER` aus Environment. Silent-skip wenn nicht gesetzt. HTTP POST via `curl` an `https://api.pushover.net/1/messages.json`.
- `scripts/post-deploy-watch.sh` — Standalone-Script fuer Post-Deploy-Monitoring. Wird von `workspace:deploy` (Taskfile.yml) und `feature-promote.sh` aufgerufen. Parameter: `ENV`, `SERVICE` (optional, default: alle). Fuehrt `post_deploy_watch()` fuer jede relevante Deployment aus.

### Geaenderte Dateien

- `scripts/feature-promote.sh`
  - Neue Funktion `post_deploy_watch <cluster> <deployment> <namespace> <context> <health_url>`: 5 Checks im 60s-Abstand. Jeder Check: (1) `kubectl get pods` auf CrashLoopBackOff/Error/OOMKilled pruefen, (2) `curl -fsS <health_url>` auf HTTP 200 pruefen. Zaehler fuer aufeinanderfolgende Healthcheck-Fehler. Bei 3 Fehlern ODER CrashLoopBackOff: `kubectl rollout undo` + `notify_pushover` + return 1.
  - Funktion `observe_prod()` wird refactored: ersetzt die bestehende 5-Probe-Schleife durch `post_deploy_watch()` und behaelt zusaetzlich die Playwright-Smoke-Logik bei (nur bei GREEN des Monitors).
  - Funktion `roll()` bleibt unveraendert (rollout status + undo bei Timeout).
  - Nach jedem `roll prod` Aufruf wird `post_deploy_watch()` aufgerufen.
  - Source `scripts/lib/notify.sh` am Anfang des Scripts.

- `Taskfile.yml`
  - Task `workspace:deploy` (Zeile ~2096, nach dem `kustomize build | kubectl apply` Block): Optionaler Aufruf von `bash scripts/post-deploy-watch.sh "{{.ENV}}"` nach dem Apply, nur fuer `ENV != dev`.
  - Neue Variable `POST_DEPLOY_WATCH` (default: "true") als Gate — kann auf "false" gesetzt werden um den Watch zu ueberspringen.

- `environments/schema.yaml`
  - Zwei neue optionale Secrets: `PUSHOVER_TOKEN` (generate: false, required: false) und `PUSHOVER_USER` (generate: false, required: false).

- `environments/mentolder.yaml` und `environments/korczewski.yaml`
  - Placeholder-Eintraege fuer `PUSHOVER_TOKEN` und `PUSHOVER_USER` (leer, user-provided).

### Nicht geaendert

- `k3d/*.yaml` — Keine Aenderungen an Deployment-Manifesten. Bestehende readinessProbe/livenessProbe werden vom neuen Monitor nicht ersetzt, sondern ergaenzt.
- `scripts/verify-deployment.sh` — Bleibt unveraendert. Der Post-Deploy-Watch ist ein separater Mechanismus.
- `tests/` — Keine neuen Test-Dateien. Die bestehende NFA-02 CrashLoopBackOff-Pruefung bleibt als E2E-Test erhalten.

## Tech-Stack

- Bash 5+ (sourceable Libraries, associative arrays)
- `curl` (Pushover API)
- `kubectl` (Pod-Status, Rollout-Undo)
- `jq` (JSON-Parsing fuer kubectl Output)
- Pushover REST API v1 (`https://api.pushover.net/1/messages.json`)

## Tasks

- [ ] **T1 — Notify-Library erstellen:** `scripts/lib/notify.sh` anlegen. Funktion `notify_pushover <title> <message> [priority]`. Liest `PUSHOVER_TOKEN` und `PUSHOVER_USER` aus Environment. Wenn nicht gesetzt: `echo "⚠ Pushover nicht konfiguriert — Notification uebersprungen"` und return 0. Bei gesetzten Credentials: `curl -fsS --max-time 10 -d "token=${PUSHOVER_TOKEN}&user=${PUSHOVER_USER}&title=${title}&message=${message}&priority=${priority:-0}" https://api.pushover.net/1/messages.json`. Fehler bei curl werden als Warning ausgegeben, nicht als Fehler propagiert (Notify darf den Deploy-Flow nicht blockieren).

- [ ] **T2 — Schema erweitern:** In `environments/schema.yaml` unter `secrets:` zwei neue Eintraege hinzufuegen: `PUSHOVER_TOKEN` (required: false, generate: false, description: "Pushover API token for deploy rollback notifications") und `PUSHOVER_USER` (required: false, generate: false, description: "Pushover user key for deploy rollback notifications").

- [ ] **T3 — Post-Deploy-Watch Function in feature-promote.sh:** Neue Funktion `post_deploy_watch <cluster> <deployment> <namespace> <context> <health_url>` in `scripts/feature-promote.sh` einfuegen (nach der bestehenden `observe_prod()` Funktion, vor der `smoke_one()` Funktion). Logik: (1) `prev_rev` via `kubectl rollout history` erfassen. (2) Loop 5x mit `sleep 60` zwischen den Iterationen. (3) In jeder Iteration: `kubectl get pods --context <ctx> -n <ns> -l app=<deployment> --no-headers` auf `CrashLoopBackOff|Error|OOMKilled` pruefen — bei Match sofort Rollback. (4) `curl -fsS --max-time 20 <health_url>` pruefen — bei Erfolg `fail_count=0`, bei Fehler `fail_count+=1`. (5) Wenn `fail_count >= 3`: Rollback via `kubectl rollout undo deploy/<name> --to-revision=<prev_rev>` (oder ohne `--to-revision` wenn prev_rev leer), dann `notify_pushover "Auto-Rollback <service> <cluster>" "Healthcheck 3x fehlgeschlagen nach Deploy von <image>. Revision <prev_rev> restored."`, dann return 1. (6) Nach Loop-Ende ohne Abbruch: return 0. Source `scripts/lib/notify.sh` am Script-Anfang (nach den bestehenden source-Zeilen).

- [ ] **T4 — observe_prod() refactoren:** Die bestehende `observe_prod()` Funktion in `scripts/feature-promote.sh` (Zeilen ~227-283) so aendern, dass sie intern `post_deploy_watch()` aufruft statt der eigenen 5-Probe-Schleife. Die Playwright-Smoke-Logik (`smoke_one`) wird NACH erfolgreichem `post_deploy_watch()` ausgefuehrt (nicht mehr davor/parallel). Wenn `post_deploy_watch()` return 1 liefert, wird der Playwright-Smoke uebersprungen und der bestehende Rollback-Pfad in `observe_prod()` greift.

- [ ] **T5 — Standalone post-deploy-watch.sh erstellen:** `scripts/post-deploy-watch.sh` anlegen. Standalone-Script das `ENV` als Parameter nimmt. Source `scripts/env-resolve.sh` und `scripts/lib/notify.sh`. Iteriert ueber die relevanten Deployments fuer den gegebenen ENV: `website` (namespace: `WEBSITE_NAMESPACE`), `nextcloud`, `keycloak`, `brett`, und fuer korczewski zusaetzlich `arena-server`. Fuer jedes Deployment: `post_deploy_watch()` mit den korrekten Parametern aufrufen (Context via `env-resolve.sh`, Health-URL basierend auf Deployment: website→`/api/health`, brett→`/healthz`, arena-server→`/healthz`, keycloak→`/health/ready`, nextcloud→`/status.php`). Exit-Code: 1 wenn mindestens ein Watch fehlgeschlagen ist, 0 sonst. Script muss ausfuehrbar sein (`chmod +x`).

- [ ] **T6 — Taskfile.yml workspace:deploy erweitern:** In `Taskfile.yml` im Task `workspace:deploy` (Zeile ~2096, nach dem `kustomize build | kubectl apply --server-side --force-conflicts` Block, vor dem `tests-retention-cronjob` Apply) einen optionalen Block einfuegen: `if [ "{{.ENV}}" != "dev" ] && [ "${POST_DEPLOY_WATCH:-true}" != "false" ]; then bash scripts/post-deploy-watch.sh "{{.ENV}}" || echo "⚠ Post-deploy watch detected issues — check logs"; fi`. Das `|| echo` verhindert dass ein Watch-Fehler den gesamten Deploy-Flow blockiert (der Watch ist informativ + rollt zurueck, aber workspace:deploy soll mit den nachfolgenden Tasks weitermachen).

- [ ] **T7 — Env-Dateien aktualisieren:** In `environments/mentolder.yaml` und `environments/korczewski.yaml` die Placeholder `PUSHOVER_TOKEN: ""` und `PUSHOVER_USER: ""` hinzufuegen (unter den bestehenden user-provided Secrets wie `GHCR_PAT`). In `environments/fleet-mentolder.yaml` und `environments/fleet-korczewski.yaml` ebenfalls hinzufuegen.

- [ ] **T8 — DRY_RUN support:** In `post_deploy_watch()` und `post-deploy-watch.sh` den bestehenden `DRY_RUN` Mechanismus aus `feature-promote.sh` unterstuetzen: Wenn `DRY_RUN=1`, werden kubectl und curl Befehle mit `[dry-run]` Prefix ausgegeben statt ausgefuehrt.

## Verifikation

### Lokal

- `DRY_RUN=1 SERVICE=website TARGET=both bash scripts/feature-promote.sh` — Dry-Run zeigt die post_deploy_watch Aufrufe ohne echte kubectl/curl Befehle.
- `bash scripts/post-deploy-watch.sh dev` — Sollte auf dev entweder skippen oder (wenn kein Pushover konfiguriert) mit Warning durchlaufen.
- `task test:all` — Bestehende Tests muessen gruener bleiben.
- `DRY_RUN=1 bash scripts/post-deploy-watch.sh mentolder` — Dry-Run gegen prod zeigt die erwarteten Checks ohne Seiteneffekte.

- `task freshness:check` — Generierte Artefakte muessen aktuell bleiben.

- [ ] Rollback erfolgt auf das letzte erfolgreiche Image (via `kubectl rollout undo --to-revision`)
- [ ] Pushover-Benachrichtigung wird bei Rollback gesendet
- [ ] Fehlende Pushover-Credentials blockieren den Deploy-Flow nicht
- [ ] Funktioniert fuer beide Brands (mentolder + korczewski)
- [ ] `feature-promote.sh` integriert den Monitor in den bestehenden Promote-Flow
- [ ] `workspace:deploy` ruft den Monitor nach Prod-Deploy auf
- [ ] DRY_RUN=1 unterdrueckt alle Seiteneffekte
