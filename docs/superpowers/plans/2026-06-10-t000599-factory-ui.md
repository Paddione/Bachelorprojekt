## Ziel

`PlanningOffice.svelte` vollständig neu schreiben: Industrial-Loft-Oberfläche mit Drag-Rank-Queue (links, 360 px), Stats-Bar (oben), Detail-Panel (rechts) und vollständiger Mobile-Parität via Bottom-Sheet. Neue Admin-API-Route `/api/admin/planungsbuero` anlegen (Stats + Rank-PATCH). Bestehende `/api/planning-office/*`-Routen bleiben unverändert.

## Tech-Stack

- **Svelte 5 / Svelte 4** (Repo-Standard: `client:load` Hydration in Astro)
- **Vanilla CSS Custom Properties** (kein Tailwind im Svelte-File, Industrial-Loft lokal)
- **HTML5 Drag-and-Drop API** (`draggable`, `dragstart`, `dragover`, `drop`) für Desktop
- **Pointer Events API** (`pointerdown`, `pointermove`, `pointerup`) für Touch-Drag Mobile
- **PostgreSQL `website` DB** via bestehenden `pool` aus `website/src/lib/website-db.ts`
- **`planning-office.ts` DAL** — `listOffice()`, `patchItem()`, `promoteItem()`
- **BATS** für API-Unit-Tests (Fixture-basiert, kein echter DB-Call in Unit-Tests)

## Tasks

- [ ] **Task 1 — API: GET /api/admin/planungsbuero**
  Neue Route `website/src/pages/api/admin/planungsbuero/index.ts`. Ruft `listOffice()` auf, berechnet Stats (`planning` = items.length, `ready` = dorScore === 4, `blocked` = dependsOn.length > 0 && dorScore < 4). Gibt `{ items, stats }` zurück. Auth: `getSession` + `isAdmin`. Export: `prerender = false`.

- [ ] **Task 2 — API: PATCH /api/admin/planungsbuero/[extId]**
  Neue Route `website/src/pages/api/admin/planungsbuero/[extId].ts`. Delegiert an `patchItem(extId, body)`. Felder: `rank`, `readiness`, `effort`, `valueProp`, `dependsOn`. Validierung analog zu `[extId].ts` in `/api/planning-office/`. Gibt `{ ok: true }` bei Erfolg zurück.

- [ ] **Task 3 — BATS: FA-PB-01 bis FA-PB-05**
  Neue Testdatei `tests/unit/planungsbuero-stats.bats`. Tests:
  - FA-PB-01: Stats-Berechnung bei leerer Liste → 0/0/0
  - FA-PB-02: Stats bei 2 planning, 1 ready (dorScore 4), 1 blocked (deps vorhanden, dorScore < 4)
  - FA-PB-03: PATCH-Validierung lehnt ungültigen effort-Wert ab
  - FA-PB-04: Rang-Update via PATCH aktualisiert `planning_rank`
  - FA-PB-05: GET-Response enthält `stats`-Objekt mit korrekten Keys
  Wiring: Eintrag in `tests/runner.sh` unter `local` (offline-sicher, kein DB-Call).

- [ ] **Task 4 — Svelte: Stats Bar**
  Oberer Bereich der Komponente. Reaktive `$: stats` wird aus der API-Response abgeleitet. Markup: `<div class="pb-stats-bar" data-testid="pb-stats-bar"><span>{stats.planning} planning · {stats.ready} ready · {stats.blocked} blocked</span></div>`. CSS: monospace, `#64748b` Farbe, `padding: 8px 16px`, `border-bottom: 1px solid var(--border)`.

- [ ] **Task 5 — Svelte: Queue-Zeilen (Desktop)**
  Linke Spalte. `{#each items as it (it.extId)}` iteriert, jede Zeile ist `<div class="pb-row" draggable="true" data-testid="pb-queue-row-{it.extId}">`. Inhalte: Drag-Handle `☰` (32 px), Rang-Zahl `{String(idx + 1).padStart(2, '0')}`, Priority-Dot 8 px, `external_id` (amber mono), Titel (truncated), 4× Readiness-Square. Selektion: `class:selected={selected?.extId === it.extId}` mit Amber Left-Border + `#1e2736` Hintergrund.

- [ ] **Task 6 — Svelte: Drag-and-Drop Rang-Update**
  `dragstart` speichert `dragSrcExtId`. `dragover` berechnet Zielposition aus `event.clientY` relativ zu Zeilen-Bounding-Boxes, zeigt Drop-Indikator-Linie. `drop` sendet `PATCH /api/admin/planungsbuero/:extId` mit neuem `rank` (Integer-Index der Zielposition) und lädt die Liste neu. Touch-Drag via `pointerdown/pointermove/pointerup` für Mobile (dieselbe Logik, Touch-Target 48 px am Drag-Handle).

- [ ] **Task 7 — Svelte: Detail-Panel (Desktop rechts)**
  `{#if selected}` Block. Markup:
  - `<h2 class="pb-detail-title">{selected.title}</h2>`
  - Textarea für `valueProp` (autosave on blur via PATCH)
  - `<fieldset>` Readiness-Checklist: 4 × `<label><input type="checkbox"> {DOR_LABEL[k]}</label>`, on:change → PATCH + lokales Update
  - Depends-on Tag-Chips: `{#each selected.dependsOn as dep}<span class="pb-chip">{dep}<button on:click={() => removeDep(dep)}>×</button></span>{/each}` + Input für neues Dep (Enter-Submit → PATCH)
  - Effort-Buttons: `klein` / `mittel` / `groß` — class:active bei Match → PATCH on:click
  - Override-Checkbox `data-testid="pb-override"`
  - Promote-Button `data-testid="pb-detail-promote"` — disabled wenn `!override && selected.dorScore < 4`

- [ ] **Task 8 — Svelte: Mobile Bottom-Sheet**
  `$: isMobile` — `window.innerWidth < 768 || viewOverride === 'mobile'`. Bei Mobile: Queue nimmt volle Breite, kein rechtes Panel sichtbar. Tap auf Zeile → `sheetOpen = true`, `sheetItem = it`. Bottom-Sheet: `<div class="pb-sheet" class:open={sheetOpen} data-testid="pb-sheet">`. Enthält denselben Detailbereich wie Desktop-Panel. Schließen: × Button (`data-testid="pb-sheet-close"`) + Swipe-down (`pointerdown` Y-Delta > 80 px → close). CSS: `position: fixed; bottom: 0; left: 0; right: 0; height: 60vh; border-radius: 12px 12px 0 0; transform: translateY(100%); transition: transform 0.25s ease`, `.open { transform: translateY(0) }`.

- [ ] **Task 9 — Svelte: View-Switcher Toggle**
  Dezenter Button oben rechts (⊞ Desktop / ≡ Mobile). `data-testid="pb-view-toggle"`. On:click: liest/setzt `localStorage.planungsbuero_view`, aktualisiert `viewOverride`, re-evaluiert `isMobile`. CSS: `position: absolute; top: 12px; right: 16px; background: none; border: 1px solid var(--border); color: var(--muted); font-size: 0.75rem; padding: 4px 8px; cursor: pointer`.

- [ ] **Task 10 — CSS: Industrial-Loft Custom Properties + Vollständiges Styling**
  Im `<style>`-Block: alle `--*`-Custom-Properties definieren. Sicherstellen dass:
  - Queue-Zeile exakt `min-height: 56px` (Playwright-Test misst `getBoundingClientRect`)
  - Priority-Dot ist `width: 8px; height: 8px; border-radius: 50%`
  - Readiness-Square ist `width: 6px; height: 6px; border-radius: 1px`
  - Monospace-Stellen nutzen `font-family: var(--mono)`
  - Amber Left-Border bei Selektion: `border-left: 3px solid var(--amber)`
  - Responsive: `@media (max-width: 767px)` schaltet Grid auf Single-Column

- [ ] **Task 11 — Verifikation: Playwright E2E Smoke-Test**
  Neuen Test `tests/e2e/planungsbuero-smoke.spec.ts`. Prüft:
  1. Stats Bar sichtbar mit korrektem Format
  2. Erste Queue-Zeile hat `data-testid="pb-queue-row-*"` und ist 56 px hoch
  3. Klick auf Zeile rendert Detail-Panel sichtbar
  4. Promote-Button ist disabled wenn Readiness < 4
  Dieser Test ist für `nightly`-E2E gegen Fleet (nicht `offline`).

- [ ] **Task 12 — Wiring: BATS in test:all**
  `tests/runner.sh`: Neuen Block `planungsbuero-unit` hinzufügen, der `tests/unit/planungsbuero-stats.bats` ausführt. Sicherstellen dass er unter `local` (offline) läuft und `task test:all` grün bleibt.
