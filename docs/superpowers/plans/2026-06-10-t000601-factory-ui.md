## Ziel

Einen neuen "Control Panel"-Tab in `/dev-status` implementieren, der Factory-Betriebsparameter (Kill-Switch, Dry-Run, Slot-Cap, Daily-Cap) browser-seitig steuerbar macht. Ergaenzt durch einen neuen API-Endpunkt `/api/admin/factory-control` (GET/PATCH) und Watchdog-Status-Integration aus dem bestehenden `/api/factory-floor`.

### Datenbankschema

Neue Tabelle `factory_control` (alternativ: Eintraege in `site_settings`):

```sql
CREATE TABLE IF NOT EXISTS factory_control (
  id          SERIAL PRIMARY KEY,
  kill_switch BOOLEAN NOT NULL DEFAULT FALSE,
  dry_run     BOOLEAN NOT NULL DEFAULT FALSE,
  slot_cap    INTEGER NOT NULL DEFAULT 4 CHECK (slot_cap BETWEEN 1 AND 8),
  daily_cap   INTEGER NOT NULL DEFAULT 20 CHECK (daily_cap BETWEEN 1 AND 50),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- Einmalig befuellen (Singleton-Row)
INSERT INTO factory_control DEFAULT VALUES ON CONFLICT DO NOTHING;
```

## Tech-Stack

| Schicht | Technologie |
|---------|-------------|
| Framework | Astro + Svelte (wie bestehende Admin-Seiten) |
| Styling | CSS Custom Properties; kein Tailwind-Utility-Spam; Monospace-Font via CSS-Var |
| API | Astro API Routes (`src/pages/api/admin/*.ts`), SSR-only |
| DB | PostgreSQL 16 via bestehenden `db`-Helper (`website/src/lib/db/index.ts`) |
| Auth | Bestehende Admin-Session-Middleware (`src/middleware/auth.ts` o.ae.) |
| Polling | `setInterval` im Svelte-Lifecycle (`onMount`/`onDestroy`), 30s fuer Watchdog |
| State | Svelte reactive `$state` / Stores; optimistic update + Rollback-Pattern |

## Tasks

- [ ] **T1 — DB-Migration anlegen:** `factory_control`-Tabelle mit Singleton-Row erstellen; Migration in `website/src/lib/db/migrations/` ablegen und lokale DB migrieren.

- [ ] **T2 — API GET `/api/admin/factory-control`:** Singleton-Row auslesen, als JSON zurueckgeben (`killSwitch`, `dryRun`, `slotCap`, `dailyCap`, `updatedAt`); Admin-Auth-Gate (401/403).

- [ ] **T3 — API PATCH `/api/admin/factory-control`:** Validierung der eingehenden Felder (Typen, Bereichsgrenzen); UPDATE Singleton-Row; zurueckgeben des neuen Zustands; Admin-Auth-Gate.

- [ ] **T4 — UI-Primitives implementieren:** `PilotLight.svelte`, `ToggleSwitch.svelte`, `Stepper.svelte`, `SegmentDots.svelte` als isolierte Komponenten mit Props und Emit-Events; Storybook-aehnliche Dev-Ansicht nicht erforderlich, aber Props-Dokumentation als JSDoc.

- [ ] **T5 — Vier Kontrollkarten implementieren:** `KillSwitchCard`, `DryRunCard`, `SlotCapCard`, `DailyCapCard` jeweils mit korrekten Farben, Glow-Shadows, Labels und Stepper-Grenzen gemaess Spec.

- [ ] **T6 — ControlPanel.svelte (Container):** 2x2-Grid-Layout (Desktop) / 1-Spalte-Stack (Mobile < 768px); laedt initialen State per `fetch('/api/admin/factory-control')` beim Mount; leitet Aenderungen per PATCH weiter; implementiert Optimistic-Update + Rollback-Logik; zeigt Fehler-Toast bei API-Fehler.

- [ ] **T7 — StatusStrip.svelte:** Polling alle 30s aus `/api/factory-floor` (`control.watchdogStale`, `control.lastActivity`); zeigt Timestamp formatiert (ISO oder "vor X Minuten") + Pilot-Light (gruen/rot); `clearInterval` bei Destroy.

- [ ] **T8 — Dev-Status-Tab integrieren:** "Control Panel"-Tab in `/dev-status` registrieren; Tab-Selektion in `localStorage` persistieren; ControlPanel-Komponente einbinden; Tab nur fuer Admin-Sessions rendern.

- [ ] **T9 — Mobile-Optimierung validieren:** Viewport-Tests bei 375px und 768px; Touch-Targets manuell pruefen (Toggle >= 64px, Stepper-Buttons >= 44px); CSS-Media-Query einbauen und verifizieren.

- [ ] **T10 — End-to-End-Test (optional, falls Playwright verfuegbar):** Admin-Login → Control-Panel-Tab oeffnen → Kill-Switch toggeln → API-State pruefen → Kill-Switch zuruecksetzen. Als `tests/e2e/factory-control.spec.ts` ablegen.

- [ ] **T11 — Sicherheits-Check:** Manuell verifizieren dass `GET /api/admin/factory-control` mit unauthentifizierter Session 401 zurueckgibt und mit Nicht-Admin-Session 403.
