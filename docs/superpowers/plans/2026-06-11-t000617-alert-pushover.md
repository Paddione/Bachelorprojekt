# Grafana Alert-Regeln + Pushover-Benachrichtigungen Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deploy a single `kube-prometheus-stack`-based monitoring system in a dedicated `monitoring` namespace that observes both brands (mentolder + korczewski) on the fleet cluster, fires 5 mandatory alert rules, and routes notifications to Pushover.

**Architecture:** Helm chart `kube-prometheus-stack` pre-rendered to committed YAML (no Helm operator in-cluster, push-based deploy model preserved). Prometheus scrapes `workspace` + `workspace-korczewski` namespaces. PrometheusRule CR carries the alert rules, AlertmanagerConfig CR carries the Pushover receiver. Grafana authenticates via Keycloak OIDC. Dev (k3d) runs a slimmed stack with local Grafana auth and no Pushover.

**Tech Stack:** Helm (`helm template` pre-render), Kustomize (`k3d/monitoring/` base + `prod/monitoring/` patches), Prometheus/Alertmanager/Grafana CRDs, Keycloak OIDC, Traefik IngressRoute, SealedSecrets, BATS (`promtool`/`amtool`/`kustomize` validation).

---

## Repo conventions this plan follows

- **Base manifests** live in `k3d/`; prod patches in `prod/`; brand overlays in `prod-fleet/mentolder/` + `prod-fleet/korczewski/` which wrap `prod-mentolder`/`prod-korczewski` + `fleet-common`.
- **Never apply `prod/` directly** — it is consumed by the `prod-fleet/*` wrappers.
- **`environments/schema.yaml`** is the authoritative env-var list. Any new `${VAR}` referenced in a manifest must be registered here AND added to the `envsubst` list of every task that renders that manifest (`Taskfile.yml`, not `.yaml`).
- **`scripts/env-resolve.sh` must be sourced, never executed.**
- **`task test:all`** (BATS + `kubectl kustomize` validate + Taskfile dry-run) must stay green.
- **Both brands share the fleet cluster but have separate namespaces** (`workspace` / `workspace-korczewski`) and separate sealed-secrets. Cross-cutting secrets must be sealed for **both** envs.
- Today's date: 2026-06-11.

### Task B2: Write the PrometheusRule with the 5 alerts

**Files:**
- Create: `k3d/monitoring/prometheus-rules.yaml`

- [x] **Step 1: Write the PrometheusRule CR**

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: workspace-alerts
  namespace: monitoring
  labels:
    app: kube-prometheus-stack
    release: monitoring
spec:
  groups:
    - name: workspace.rules
      rules:
        - alert: PodCrashLoopBackOff
          expr: |
            kube_pod_container_status_waiting_reason{reason="CrashLoopBackOff",namespace=~"workspace|workspace-korczewski"} > 0
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "Pod {{ $labels.pod }} in CrashLoopBackOff ({{ $labels.namespace }})"
            description: "Container {{ $labels.container }} crasht wiederholt seit 5 Minuten."
        - alert: HighCPUUsage
          expr: |
            (
              sum by (namespace, pod, container) (
                rate(container_cpu_usage_seconds_total{namespace=~"workspace|workspace-korczewski",container!=""}[5m])
              )
              /
              sum by (namespace, pod, container) (
                kube_pod_container_resource_limits{namespace=~"workspace|workspace-korczewski",resource="cpu"}
              )
            ) > 0.8
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Hohe CPU-Auslastung: {{ $labels.pod }} ({{ $labels.namespace }})"
            description: "CPU-Auslastung > 80% für 5 Minuten: {{ $value | humanizePercentage }}"
        - alert: HighMemoryUsage
          expr: |
            (
              container_memory_working_set_bytes{namespace=~"workspace|workspace-korczewski",container!=""}
              /
              kube_pod_container_resource_limits{namespace=~"workspace|workspace-korczewski",resource="memory"}
            ) > 0.9
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Hohe Memory-Auslastung: {{ $labels.pod }} ({{ $labels.namespace }})"
            description: "Memory > 90% des Limits: {{ $value | humanizePercentage }}"
        - alert: HighDiskUsage
          expr: |
            (kubelet_volume_stats_used_bytes / kubelet_volume_stats_capacity_bytes) > 0.85
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Hohe Disk-Auslastung: PVC {{ $labels.persistentvolumeclaim }}"
            description: "PVC-Auslastung > 85%: {{ $value | humanizePercentage }}"
        - alert: High5xxErrorRate
          expr: |
            (
              sum by (router) (rate(traefik_router_requests_total{code=~"5.."}[5m]))
              /
              sum by (router) (rate(traefik_router_requests_total[5m]))
            ) > 0.01
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Hohe 5xx-Rate auf Router {{ $labels.router }}"
            description: "Mehr als 1% HTTP-5xx-Fehler für 5 Minuten: {{ $value | humanizePercentage }}"
```

```yaml
apiVersion: monitoring.coreos.com/v1alpha1
kind: AlertmanagerConfig
metadata:
  name: pushover
  namespace: monitoring
  labels:
    release: monitoring
spec:
  route:
    groupBy: [alertname, namespace]
    groupWait: 30s
    groupInterval: 5m
    repeatInterval: 4h
    receiver: pushover
    routes:
      - matchers:
          - name: severity
            value: critical
        receiver: pushover
  receivers:
    - name: pushover
      pushoverConfigs:
        - sendResolved: true
          token:
            name: alertmanager-pushover
            key: PUSHOVER_TOKEN
          userKey:
            name: alertmanager-pushover
            key: PUSHOVER_USER
          title: '{{ template "pushover.default.title" . }}'
          message: '{{ template "pushover.default.message" . }}'
          priority: '{{ if eq .CommonLabels.severity "critical" }}1{{ else }}0{{ end }}'
          retry: 30s
          expire: 1h
```

> NOTE: The operator's `AlertmanagerConfig` references the Pushover `token`/`userKey` via `SecretKeySelector` (keys in the `alertmanager-pushover` Secret), NOT inline `${VAR}`. This avoids leaking secrets into committed YAML and avoids an envsubst step. Critical alerts get priority 1 (Pushover retries until acknowledged); warnings get priority 0.

## Rollback Strategy

The monitoring stack lives entirely in its own `monitoring` namespace and is wired into the fleet overlays only via the two added `resources:` lines. Rollback is therefore cleanly scoped:

1. **Revert the fleet wiring (fastest, leaves stack running):**
   Revert the two `resources: ../../prod/monitoring` additions in `prod-fleet/mentolder/kustomization.yaml` + `prod-fleet/korczewski/kustomization.yaml`, re-deploy both brands. Future deploys stop managing monitoring; existing pods keep running until step 2.

2. **Tear down the namespace (full removal):**
   ```bash
   kubectl --context fleet delete namespace monitoring
   ```
   Removes Prometheus/Grafana/Alertmanager/exporters and all PVCs. Longhorn volumes are reclaimed per their reclaim policy.

3. **Revert Traefik metrics (optional):**
   ```bash
   kubectl --context fleet delete -f k3d/monitoring/traefik-metrics.yaml
   ```
   Removes the `metrics` entrypoint; k3s re-renders Traefik without it. (Harmless to leave in place.)

4. **Secrets:** The sealed secrets gain `PUSHOVER_*`/`GRAFANA_*` keys; these are additive and harmless if monitoring is removed. To fully revert, set `PUSHOVER_TOKEN`/`PUSHOVER_USER` back to `required: false` in `schema.yaml`, drop the Grafana secrets, re-seal both envs.

5. **Keycloak:** Delete the `grafana` client from the realm(s) via `kcadm`/Admin UI if removing entirely. Leaving it is harmless (no redirect targets exist after teardown).

6. **Git:** Since each task committed independently on `feature/T000617-alert-pushover`, `git revert` of specific commits (or abandoning the branch / closing the PR unmerged) cleanly removes all changes. No base-manifest (`k3d/kustomization.yaml`) was modified, so the core workspace deploy is unaffected by abandonment.

**Failure-mode notes:**
- If Prometheus crashloops on a bad PVC storageClass, the issue is isolated to `monitoring` — core workspace pods are untouched.
- If the kustomize namespace transformer rewrites the `kube-system` Traefik resources, the build fails offline (caught by `task monitoring:validate` in Task E2/E3 before any deploy).
- If Pushover keys are wrong, alerts simply don't deliver — no impact on cluster workloads.
