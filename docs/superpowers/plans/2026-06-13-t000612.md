# Plan: T000612 — Website Mehrsprachigkeit DE/EN

**Ziel:** Vollständige i18n-Infrastruktur für das Kunden-Frontend der Website aufbauen — Astro built-in i18n mit pathname-Routing, typsichere Locale-Dictionaries, Sprachumschalter, Locale-Detection, hreflang-SEO, und alle statischen UI-Strings aus 22+ Seiten und 10+ Komponenten in DE/EN-Dateien auslagern. Erste Phase: nur mentolder-Brand, Portal und Admin bleiben deutsch.

**Spec:** `docs/superpowers/specs/2026-06-13-t000612-design.md`

**Architektur-Entscheidungen (aus Spec):**
- Astro built-in i18n: `locales: ['de', 'en']`, `routing.prefixDefaultLocale: false` → DE unter `/`, EN unter `/en/`.
- Eigene typsichere Locale-Module (`src/i18n/de.ts`, `en.ts`, `index.ts`) — kein externes Package.
- Middleware für Locale-Detection: Cookie → Accept-Language → Fallback `de`.
- LanguageSwitcher als separate Komponente (Navigation.svelte ist bereits über 500-Zeilen-Limit).
- EN-Pages als dünne Wrapper die dieselbe Komponente mit `locale: 'en'` rendern.
- Brand-Architektur vorbereitet für korczewski, aber nur mentolder implementiert.

## Phase A — i18n-Infrastruktur (Core)

- [x] **A1** — `website/astro.config.mjs` erweitern (26/26 Zeilen, Config-Block).
  ```js
  i18n: {
    defaultLocale: 'de',
    locales: ['de', 'en'],
    routing: {
      prefixDefaultLocale: false,
      redirectToDefaultLocale: false,
    },
  },
  ```

- [x] **A2** — `website/src/i18n/index.ts` erstellen (~50 Zeilen, Limit 600 für .ts).
  - `Locale`-Typ, `locales`-Array, `defaultLocale`.
  - `getDictionary(locale)` Funktion.
  - `t(locale, key)` Funktion mit Fallback auf Key.
  - `getLocaleFromUrl(url)` Hilfsfunktion.
  - `getLocaleFromCookie(cookie)` Hilfsfunktion.

- [x] **A3** — `website/src/i18n/de.ts` erstellen (~200 Zeilen, Limit 600 für .ts).
  - Alle hartcodierten deutschen Strings aus Komponenten und Pages extrahieren.
  - Flaches `Record<string, string>` mit sprechenden Keys (z.B. `'nav.angebote'`, `'hero.title'`, `'footer.kontakt'`).

- [x] **A4** — `website/src/i18n/en.ts` erstellen (~200 Zeilen, Limit 600 für .ts).
  - Identische Keys wie `de.ts`, englische Übersetzungen.
  - Manuell gepflegt (Patrick).

- [x] **A5** — `website/src/middleware.ts` erstellen (~40 Zeilen, Limit 600 für .ts).
  - `onRequest` Hook: liest `locale`-Cookie, dann `Accept-Language`-Header, dann Fallback `de`.
  - Setzt `Astro.locals.locale`.
  - Validiert: nur `'de'` oder `'en'` erlaubt.

- [x] **A6** — Smoke-Test: `cd website && npm run build` — Astro soll EN-Routing akzeptieren, keine Build-Fehler.

## Risiken & Mitigationen

| Risiko | Mitigation |
|--------|-----------|
| Navigation.svelte (704 Zeilen) über Limit — weitere Änderungen könnten CI brechen | LanguageSwitcher als separate Komponente, nur minimale String-Substitution in Navigation |
| BookingForm.svelte (560 Zeilen) bereits über Limit | In Phase D NICHT anfassen — bleibt deutsch (Phase 2) |
| Astro i18n Routing mit bestehenden Custom-Routes kollidiert | `[service].astro` und andere Dynamic Routes testen; ggf. `en/[service].astro` manuell |
| Brand-Config-Änderung (mentolder.ts) bricht bestehende Consumer | Nur additive Änderungen (`{ de, en }` Objekt), bestehende String-Consumer mit `getEffective*()` anpassen |
| EN-Wrapper für `[service].astro` — Slug-Weitergabe | Props-Testing: `Astro.params.service` muss durchgereicht werden |
| Cookie-Domain bei Multi-Brand | Cookie auf Pfad `/` setzen, nicht auf Domain — funktioniert für beide Brands |
