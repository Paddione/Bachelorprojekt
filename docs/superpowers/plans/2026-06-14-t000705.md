# Plan: T000705 — Brett: Board-Templates für Aufstellungs-Szenarien

## Ziel

Coaches sollen häufig verwendete Aufstellungs-Szenarien (Figuren-Positionen, -Farben,
-Beschriftungen) als Board-Templates speichern und beim Board-Start auswählen können.
Templates werden in der DB (neue Tabelle `brett.board_templates`) gespeichert und über
REST-CRUD + einen WS-Ladebefehl genutzt.

## Architektur & Tech-Stack

- **DB:** neue Tabelle `brett.board_templates` (Migration `004_board_templates.sql`)
- **Server:** Express-Router `routes/board-templates.ts` + reine Accessor-Funktion `board-templates.ts`
- **WS:** neuer Admin-Befehl `admin_set_board_template` in `ws-admin-commands.ts`
- **Client:** `board-template-save.ts` (Topbar-UI) + Erweiterung von `lobby.ts` (Picker)
- **Tests:** `node:test`-Unit-Tests (wie alle anderen Brett-Tests)
- **Keine neuen Kubernetes-Manifeste, keine Schema-/Domain-Änderungen** → `shared_changes: false`

## Tasks

### Phase A — Datenbank-Migration

- [ ] **A1** Datei anlegen: `brett/src/server/migrations/004_board_templates.sql`
  - Schema mit `brett.board_templates`-Tabelle (id, brand, name, description, category, state JSONB, is_system, created_by_user, created_at)
  - Index auf `(brand, is_system, created_at DESC)`
  - Drei System-Szenarien per `INSERT … ON CONFLICT DO NOTHING`:
    - "Familiensystem 4 Personen" (4 Figuren, Quadrat, je `adult-average`)
    - "Team-Konflikt" (6 Figuren, 2 Gruppen à 3, Lücke in Mitte)
    - "Innere Anteile" (5 Figuren, Zentrum + 4 Himmelsrichtungen)
  - `is_system = true`, `brand = 'mentolder'` für alle System-Templates (per Migration);
    korczewski-Variante per separatem `INSERT` oder späteres Ticket

### Phase B — Server: Accessor-Modul

- [ ] **B1** Neue Datei: `brett/src/server/board-templates.ts`
  - Interface `BoardTemplate { id, brand, name, description, category, is_system, created_by_user, created_at }` (ohne `state`)
  - Interface `BoardTemplateWithState extends BoardTemplate { state: any }` (für load)
  - `listBoardTemplates(pool, brand): Promise<BoardTemplate[]>` — ORDER BY is_system DESC, created_at DESC
  - `getBoardTemplate(pool, id): Promise<BoardTemplateWithState | null>`
  - `createBoardTemplate(pool, opts: { brand, name, description?, category?, state, created_by_user }): Promise<{ id: string }>` — prüft Soft-Limit 50 Templates pro (brand, user)
  - `deleteBoardTemplate(pool, id, opts: { userId: string; isAdmin: boolean }): Promise<boolean>` — Ownership-Check + is_system-Guard
  - Rein funktional, kein Top-Level-State → kein Import-Zyklus (S2-OK)

### Phase C — Server: REST-Router

- [ ] **C1** Neue Datei: `brett/src/server/routes/board-templates.ts`
  - `boardTemplatesRouter = Router()`
  - `GET /api/board-templates?brand=<brand>` → `listBoardTemplates`, kein Auth-Gate (lesbar für Lobby)
  - `POST /api/board-templates` → `requireSession`, brand aus Session (nie aus Body), `createBoardTemplate`
    - Body: `{ name: string (≤100), description?: string (≤500), category?: string (≤50), state: { figures: Figure[] } }`
    - Validierung: name required, state.figures Array, figures ≤ 200
    - Response 201 `{ id }`
  - `DELETE /api/board-templates/:id` → `requireSession`, ruft `deleteBoardTemplate`
    - Response 204 bei Erfolg, 403 bei Ownership-Fehler, 404 wenn nicht vorhanden

- [ ] **C2** `brett/src/server/index.ts` — `boardTemplatesRouter` registrieren
  - `import { boardTemplatesRouter } from './routes/board-templates';`
  - `app.use(boardTemplatesRouter);` (analog zu snapshotsRouter)
  - Budget: 315 + ~15 = ~330/600 — OK

## Akzeptanzkriterien

1. Coach (Leiter) kann im Board-Topbar "Als Template speichern" klicken, Name eingeben, bestätigen — Template erscheint danach im Lobby-Picker.
2. Leiter kann im Lobby-Screen ein Board-Template auswählen → Board startet mit vorinstallierter Figuren-Konstellation.
3. System-Templates (Familiensystem, Team-Konflikt, Innere Anteile) sind per Migration vorhanden.
4. Löschen eigener Templates möglich; Löschen fremder oder System-Templates abgelehnt (403).
5. Bestehende Snapshot-Templates (`is_template=true`) im Picker bleiben rückwärtskompatibel.
6. Alle Unit-Tests grün; `task test:all` und `task freshness:check` grün.
