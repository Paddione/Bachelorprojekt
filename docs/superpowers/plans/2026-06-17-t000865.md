# Plan: VDA-Triage — Unified Ticket Triage Interface (T000865) — Implementation Plan

**Spec:** `docs/superpowers/specs/2026-06-17-t000865-design.md`
**Parent-Spec:** `docs/superpowers/specs/2026-06-16-vda-unify-design.md`
**Branch:** `feature/t000865`
**Ticket:** T000865

## Ziel

Ein einziger CLI-Befehl `vda.sh ticket triage` (bereich=`ticket`, aktion=`triage`), der ein Ticket im
einheitlichen VDA-Pattern triagiert: View der triage-relevanten Felder → Decision (priority/severity/status,
interaktiv oder per Flag) → Action (SQL-Update + optionaler Triage-Kommentar). Optionaler AI-Vorschlag über
den bestehenden Web-Triage-Pfad (`--suggest`, fail-open). `scripts/lib/ticket-grill.sh` wird **deprecated**
(dünner Shim mit Warnung, **nicht gelöscht**, Rückwärtskompat). Mitgefixt: latenter `NS: unbound variable`
Bug der vda-ticket-Subcommands im Standalone-Aufruf.

## Architektur & Tech-Stack

- **Sprache:** Bash (POSIX-nah, `set -euo pipefail`), kein fzf/gum/dialog, kein neues Dependency.
- **Fundament (existiert bereits, VDA-Unify Phase 1):**
  - `scripts/vda.sh` — Entry-Point.
  - `scripts/lib/vda-core.sh` — `vda_header`/`vda_section`/`vda_choose`/`vda_confirm`/`vda_input`/`vda_error`/`vda_success`/`vda_warn`/`vda_json`.
  - `scripts/vda/ticket.sh` — Ticket-Subcommand-Dispatcher (case-Allowlist + sourcing-Pattern).
  - `scripts/vda/ticket/_ticket-core.sh` — geteilte PG-Helfer `_pgpod`/`_exec_sql` (erwartet `NS/CTX/DB/USER`).
- **Neu:** `scripts/vda/ticket/triage.sh` (< 150 Z), Muster identisch zu `scripts/vda/ticket/update-status.sh`.
- **AI-Vorschlag-Quelle (nur konsumiert):** `website/src/lib/ticket-triage.ts` über HTTP
  `POST /api/admin/tickets/<id>/triage` — URL aus `${TRIAGE_API_URL:-…}`/env, niemals hardcodiert.

## Tasks

### Task 1 — `_ticket-core.sh` Standalone-Härtung (fail-closed Voraussetzung)

- [ ] In `scripts/vda/ticket/_ticket-core.sh` nach dem Header `NS/CTX/DB/USER`-Defaults aus den
      `TICKET_*`-Env-Vars setzen (gleiche Quelle wie `scripts/ticket.sh`):
      ```bash
      : "${NS:=${TICKET_NS:-workspace}}"
      : "${CTX:=${TICKET_CTX:-fleet}}"
      : "${DB:=website}"
      : "${USER:=website}"
      ```
      Additiv, `:=` respektiert vorhandene Werte → `ticket.sh` (setzt sie selbst) bleibt unverändert,
      Standalone-`vda.sh ticket …` failt nicht mehr mit `NS: unbound variable`.
- [ ] Verifizieren: `bash scripts/vda/ticket.sh get --id T000001 2>&1` zeigt **nicht** mehr
      `NS: unbound variable` (es darf an fehlendem Cluster scheitern, aber sauber).

### Task 2 — Subcommand `scripts/vda/ticket/triage.sh` (NEU, < 150 Z)

- [ ] Datei anlegen nach dem Muster von `update-status.sh`:
      - Header-Kommentar + `source "$(dirname "${BASH_SOURCE[0]}")/_ticket-core.sh"` +
        `source "$(dirname "${BASH_SOURCE[0]}")/../../lib/vda-core.sh"`.
      - `main()` mit Flag-Parsing: `--id --priority --severity --status --component --suggest --apply --no-comment -h|--help`.
- [ ] **Validate-before-cluster** (offline-safe, fail-closed, BEVOR `_pgpod`):
      - `--id` fehlt → `vda_error` + exit 2.
      - `--priority`/`--severity`/`--status` (falls gesetzt) gegen Wertebereiche prüfen
        (severity ∈ {critical,major,minor,trivial}; priority normalisieren via low/medium/high/critical→niedrig/mittel/hoch
        bzw. niedrig/mittel/hoch akzeptieren) → ungültig: `vda_error` + exit 2.
      - `--apply` (oder `VDA_NONINTERACTIVE=1` oder kein TTY) und ein Pflicht-Triage-Feld fehlt und kann
        nicht interaktiv erfragt werden → exit 2.
- [ ] **View:** `pod=$(_pgpod)`; `_exec_sql` SELECT `title,type,status,priority,severity,component`
      WHERE `external_id=:'ext_id'`; via `vda_header`/`vda_section` anzeigen. Kein Treffer → `vda_error` + exit 1.
- [ ] **Suggest (optional, fail-open):** bei `--suggest` `curl -fsS -X POST "${TRIAGE_API_URL:-…}/<id>/triage"`;
      Erfolg → priority/severity als Default-Vorauswahl; Fehler/unerreichbar → `vda_warn` + manuell weiter (kein Abbruch).
- [ ] **Decision:** für nicht per Flag gesetzte Felder `vda_choose` (Priority/Severity/Status); im
      Non-Interactive-Mode ausschließlich Flags.
- [ ] **Action:** `_exec_sql` UPDATE `tickets.tickets SET priority=…,severity=…,status=…,component=…`
      WHERE `external_id`; danach (sofern nicht `--no-comment`) `_exec_sql` INSERT in
      `tickets.ticket_comments` (`author_label='triage'`, Body = Zusammenfassung) — Muster aus
      `cmd_add_comment`. `vda_success "Ticket <id> triaged: <prio>/<sev> → <status>"`.
- [ ] Abschluss-Guard `if [[ "${BASH_SOURCE[0]}" == "${0}" ]]; then main "$@"; fi` (wie Sibling-Subcommands).
- [ ] `wc -l scripts/vda/ticket/triage.sh` → bestätigt < 150.

## Risiken & Gegenmaßnahmen

- **`cmd_grill`-Output-Regression:** Deprecation-Warnung könnte das maschinenlesbare stdout verschmutzen →
  strikt nur stderr; bats-Test prüft stdout-Shape (Task 5).
- **`NS`-Default-Härtung ändert ticket.sh-Verhalten:** `:=` respektiert vorhandene Werte; ticket.sh setzt
  `NS/CTX/DB/USER` weiterhin selbst → no-op dort. Regression-Test in Task 5.
- **`--suggest` HTTP nicht erreichbar (CI/offline):** fail-open (`vda_warn` + manuelle Auswahl), niemals
  Abbruch; in Tests wird `--suggest` nicht im offline-Pfad ausgeübt.
- **Validate-before-cluster Reihenfolge:** alle Wertebereichs-Prüfungen VOR `_pgpod` (sonst sind die bats
  nicht offline-safe — vgl. FA-SF Lesson, die bei ticket.sh genau das aufdeckte).
