# Plan: VDA-Slots — Unified Slots & Provider Routing (T000866) — Implementation Plan

**Ticket:** T000866
**Branch:** feature/t000866

## Ziel

Einen neuen Befehlspfad `vda.sh factory slots <aktion> [args]` einführen, der die
bestehende, race-freie Slot-Buchhaltung `scripts/factory/slots.sh` **wrappt** (delegiert,
ersetzt NICHT) und einheitliches Output liefert: **JSON für IPC** (`--json`/`VDA_JSON=1`),
**Plaintext** human-readable (Default). Dies verdrahtet den `factory`-Bereich in den
VDA-Unify-Dispatcher und ist risiko-arm, weil `slots.sh` als SSOT der SQL-Logik unverändert bleibt.

## Architektur

```
vda.sh  (Edit: +case factory)
  └─ scripts/vda/factory.sh        (NEU: Bereichs-Dispatcher factory <aktion>)
       └─ scripts/vda/factory/slots.sh   (NEU: dünner Wrapper)
            └─ delegiert: BRAND=<b> bash scripts/factory/slots.sh <cmd> [args]   (UNVERÄNDERT — SSOT)
            └─ Output via scripts/lib/vda-core.sh  (plaintext | JSON)
```

- `scripts/vda/factory/slots.sh` sourct **nur** `scripts/lib/vda-core.sh` (pure View/Output-Helfer,
  keine DB-Schicht → S2-konform, kein Import-Zyklus) und delegiert die eigentliche
  Slot-Operation per Subprozess an `scripts/factory/slots.sh` (kein Source, kein Zyklus).
- Brand-Auflösung über `--brand <b>` → setzt `BRAND` für den delegierten Aufruf; ohne Flag wird
  vorhandenes `BRAND` durchgereicht. **Kein Default-Brand-Literal im Code** (S3-konform).
- Exit-Codes (`0` ok / `1` claim-failed / `2` usage/error) werden 1:1 von `slots.sh` durchgereicht.

## Tech-Stack

- Bash (set -euo pipefail), nur builtins + `scripts/lib/vda-core.sh`.
- Delegation per Subprozess an `scripts/factory/slots.sh` (PostgreSQL via dessen `factory_psql`).
- Tests: BATS (offline, `FACTORY_DRY_RESOLVE`-basiert — keine Cluster-Abhängigkeit).
- Kein fzf/gum/dialog/python3 (small-model-kompatibel).

## Risiken & Mitigationen

- **Risiko:** `slots.sh`-Output (rohe Zahl / leer / Fehlertext) bricht JSON-Escaping.
  **Mitigation:** `vda_json` escaped via sed; leere `next`-Ausgabe als `value:""` / plaintext `—`.
- **Risiko:** `set -e` im Wrapper killt vor Exit-Code-Durchreichung bei rc=1 (claim-failed).
  **Mitigation:** Delegation mit `|| rc=$?`-Muster, expliziter `exit "$rc"`.
- **Risiko:** Doppelte Funktionalität / SSOT-Drift.
  **Mitigation:** Wrapper enthält **keine** SQL-Logik; alle Slot-Semantik bleibt in `slots.sh`.
- **Risiko:** S4-Orphan, falls neue Skripte nirgends referenziert.
  **Mitigation:** beide via `vda.sh`-Dispatch erreichbar + im neuen BATS-Test + in Help gelistet.
