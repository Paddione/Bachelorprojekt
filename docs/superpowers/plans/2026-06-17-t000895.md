# Implementierungsplan — T000895 Staging-Umgebung (`workspace-staging` auf fleet) — Implementation Plan

**Ticket:** T000895
**Branch:** `feature/t000895`
**Spec:** `docs/superpowers/specs/2026-06-17-t000895-design.md`
**Datum:** 2026-06-17

## Ziel

Eine produktionsnahe Staging-Stufe `workspace-staging` auf dem `fleet`-Cluster bereitstellen:
eigener Namespace, voll isolierte `shared-db`, `*.staging.<domain>`-Wildcard-TLS, eigenes
Kustomize-Overlay `prod-fleet/staging/`, eigene SealedSecrets, push-deploy über neues
`ENV=staging`. Prod-Brands bleiben unberührt.

## Architektur & Tech-Stack

- **Kustomize-Overlay** `prod-fleet/staging/` wrappt die `k3d/`-Base + `fleet-common`-Komponente
  und layert Staging-Patches (Namespace, Domains, Wildcard-Cert, Node-Affinity).
- **ENV-getriebener Push-Deploy**: `environments/staging.yaml` → `scripts/env-resolve.sh` →
  `envsubst` in den bestehenden Tasks. `ENV_OVERLAY = prod-fleet/staging`,
  `WORKSPACE_NAMESPACE = workspace-staging`, `ENV_CONTEXT = fleet`.
- **Isolation**: DB als namespace-lokaler Service `shared-db`; NetworkPolicy
  `netpol-deny-cross-brand` (aus `fleet-common`) mit `${WORKSPACE_NAMESPACE}=workspace-staging`.
- **TLS**: cert-manager `Certificate` `*.staging.<domain>` über den bestehenden
  `letsencrypt-prod`-ClusterIssuer (DNS-01 ipv64). Kein neuer Issuer.
- **Secrets**: plaintext `environments/.secrets/staging.yaml` → `task env:seal ENV=staging` →
  `environments/sealed-secrets/staging.yaml`.

## S1-Zeilenbudget (zu ändernde Dateien)

S1-Ratchet (`scripts/code-quality/check.mjs`) deckt Code-Extensions (`.ts/.js/.sh/.mjs/.astro/…`).
Die hier geänderten/erstellten Dateien sind `.yaml` / `.md` / `.json` — **nicht** im S1-Limit-Set,
daher kein Zeilenbudget-Risiko. Trotzdem dokumentiert:

| Datei | Ist | Baseline | Art | Budget-Hinweis |
|---|---|---|---|---|
| `environments/schema.yaml` | 1062 | nicht-baselined | YAML, nicht S1-limitiert | additiv, OK |
| `Taskfile.yml` | 4722 | nicht-baselined | YAML, nicht S1-limitiert | nur `envsubst`-Listen-Ergänzung |
| `CLAUDE.md` | 233 | nicht-baselined | MD, nicht S1-limitiert | additiv Gotcha-Eintrag |
| `prod-fleet/staging/*.yaml` | neu | neu | YAML, nicht S1-limitiert | neue Overlay-Dateien |
| `environments/staging.yaml` | neu | neu | YAML, nicht S1-limitiert | neue Env-Registry |

Falls ein neues **Skript** (`scripts/*.sh`) nötig wird (Task 9), gilt `.sh`-Limit 500 — neu
angelegte Skripte mit Wachstumsreserve < 400 Zeilen schneiden; sonst Helper-Split.

## S2/S3/S4-Konformität

- **S2 (Import-Zyklen):** keine neuen TS/JS-Module mit Imports geplant → kein Zyklus-Risiko.
  Falls ein BATS-Helper entsteht: pure Funktion ohne Rück-Import.
- **S3 (Hostnamen):** KEINE Brand-Domain-Literale in `prod-fleet/staging/` oder Manifesten.
  Alle Hosts via `${PROD_DOMAIN}` / `*_HOST`-Vars aus `environments/staging.yaml`, per `envsubst`
  gesetzt. Snippets in diesem Plan verwenden Platzhalter `<domain>` / `${PROD_DOMAIN}`, nie
  `mentolder.de`/`korczewski.de`.
- **S4 (Orphans):** jedes neue `prod-fleet/staging/*.yaml` wird in
  `prod-fleet/staging/kustomization.yaml` referenziert. Ein evtl. neues `scripts/*.sh` wird von
  einer Taskfile-Task aufgerufen. `environments/staging.yaml` ist über
  `env-resolve.sh`/`env:validate` erreichbar.

## Tasks

### Task 1 — Env-Registry `environments/staging.yaml` anlegen
- [ ] Datei `environments/staging.yaml` erstellen, analog `environments/fleet-mentolder.yaml`,
      mit:
  - `environment: staging`, `context: fleet`, `domain: staging.<basisdomain>`
  - `env_vars`: `PROD_DOMAIN`, `CLUSTER_ENV: fleet-staging`, `BRAND_NAME: "Staging"`,
    `BRAND_ID: staging`, `WORKSPACE_NAMESPACE: workspace-staging`,
    `TLS_SECRET_NAME: staging-wildcard-tls`, alle `<svc>.staging.<domain>`-Hosts
    (`WEBSITE_HOST`, `KC_DOMAIN`/`AUTH_EXTERNAL_URL`, `NC_DOMAIN`, `VAULT_DOMAIN`, `DOCS_URL`,
    `MAIL_DOMAIN`, `BRETT_DOMAIN`, `MEDIAVIEWER_HOST`, …), SMTP→Mailpit/Sink.
  - `LLM_ENABLED: "false"` (Default; kein eigener GPU-Slot) — oder gleicher `LLM_HOST_IP` falls
    geteilt erwünscht. **Entscheidung: `LLM_ENABLED=false` für Staging** (vermeidet GPU-Last).
  - LiveKit: `LIVEKIT_DOMAIN`/`STREAM_DOMAIN` gesetzt, aber LiveKit-Deployment im Overlay
    deaktiviert (Task 4) — hostNetwork-Slot-Konflikt mit Prod.
  - `overlay: prod-fleet/staging`, `workspace_namespace: workspace-staging`,
    `secrets_ref: sealed-secrets/staging.yaml`.
  - `setup_vars`: Staging-Keycloak-User (eigene Test-Admin-Credentials, `SEALED`).
- [ ] **Keine** Brand-Domain-Literale außerhalb dieser Env-Datei (Env-Datei selbst ist die
      erlaubte SSOT für die Basisdomain).

### Task 2 — Schema-Vars in `environments/schema.yaml` registrieren

      `WORKSPACE_NAMESPACE`, `TLS_SECRET_NAME`, …) werden bereits durch Staging mitbenutzt — nur
      tatsächlich NEUE Namen hinzufügen.
- [ ] Secrets-Sektion: `STAGING_*`-Secrets (DB-Passwörter etc.) als `required` registrieren,
      sofern eigene Namen vergeben werden; bevorzugt die bestehenden generischen Namen
      (`SHARED_DB_PASSWORD`, `KEYCLOAK_DB_PASSWORD`, …) wiederverwenden (sie sind namespace-lokal
      und werden aus `sealed-secrets/staging.yaml` befüllt).
- [ ] `task env:validate ENV=staging` muss nach diesem Task grün sein.
- [ ] **shared_changes:** diese Datei ist geteilt → Frontmatter `shared_changes: true`,
      vor dem Anfassen `bash scripts/agent-msg.sh post "berühre environments/schema.yaml auf feature/t000895"`.

- Spec-Akzeptanzkriterien 1–8 erfüllt.

## Deploy (post-merge, push-based)

```
task env:fetch-cert ENV=staging
task env:seal       ENV=staging        # falls .secrets geändert
task cert:secret -- <ipv64-key> ENV=staging
task workspace:deploy   ENV=staging
task workspace:post-setup ENV=staging
# Verify: Certificate *.staging.<domain> Ready, web.staging.<domain> liefert TLS/HTTP 200/302
```
