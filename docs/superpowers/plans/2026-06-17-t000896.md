# Implementierungsplan — T000896: Alert-Regeln + E-Mail-Benachrichtigung — Implementation Plan

**Ticket:** T000896
**Branch:** feature/t000896
**Spec:** docs/superpowers/specs/2026-06-17-t000896-design.md
**Domains:** infra, security, test

## Ziel

Den **vorhandenen** kube-prometheus-stack so erweitern, dass Pod-Crash-, hohe-CPU-
und Disk-Full-Bedingungen (Pod- **und** Node-Ebene) zuverlässig alarmieren und die
Alarme **zusätzlich per E-Mail** an die pro-Brand konfigurierte `CONTACT_EMAIL`
über die vorhandene SMTP-Config zugestellt werden. Pushover bleibt als zweiter
Kanal erhalten. Ein synthetischer Test-Pfad wird dokumentiert.

## Architektur & Tech-Stack

- **Stack (unverändert):** kube-prometheus-stack (Prometheus Operator,
  node-exporter, kube-state-metrics, Alertmanager) in NS `monitoring`.
- **Regeln:** `PrometheusRule` `workspace-alerts` (`k3d/monitoring/prometheus-rules.yaml`).
- **Routing/Receiver:** `AlertmanagerConfig` (`k3d/monitoring/alertmanager-config.yaml`)
  — neuer `emailConfigs`-Receiver neben dem bestehenden `pushoverConfigs`.
- **Credentials:** neues Secret `alertmanager-smtp` in NS `monitoring`
  (Dev-Platzhalter committet, Prod via SealedSecret je Brand). Empfänger (`to:`)
  und Smarthost via `envsubst` aus `CONTACT_EMAIL`/`SMTP_HOST`/`SMTP_PORT`/`SMTP_FROM`
  → **keine Brand-Domain-Literale im Manifest (S3)**.
- **Deploy:** push-based über das bestehende `prod/monitoring`-Overlay (von
  `prod-fleet/<brand>` konsumiert); kein neuer Deploy-Pfad.

## Tasks

### Task 1 — Alert-Regeln erweitern (Pod-Crash, CPU, Disk auf Node-Ebene)

> **Abgleich mit T000617 (bereits ausgeliefert):** Die fünf bestehenden Alerts sind
> **Pod-/PVC-Level** (`HighCPUUsage` = container vs CPU-Limit, `HighDiskUsage` =
> `kubelet_volume_stats` je PVC, `PodCrashLoopBackOff` = Crashloop-Zustand). Die drei
> neuen Alerts sind bewusst **Node-Level-Komplemente** (node-exporter: Node-CPU-Sättigung,
> Node-Root-Filesystem, Restart-**Rate** statt Crashloop-Zustand) — **kein** Duplikat,
> sondern die vom Ticket geforderte Node-Ebene zusätzlich zur vorhandenen Pod/PVC-Ebene.
> Receiver/Stack/Pushover aus T000617 bleiben unverändert; dieser Plan ergänzt nur
> Node-Alerts + den E-Mail-Receiver.

- [ ] In `k3d/monitoring/prometheus-rules.yaml` in der Gruppe `workspace.rules`
      drei Alerts ergänzen (bestehende fünf unverändert lassen):
  - [ ] `PodRestartSpike` — `rate(kube_pod_container_status_restarts_total{namespace=~"workspace|workspace-korczewski"}[15m]) * 60 > 3` (Restarts/min-Schwelle), `for: 10m`, `severity: warning`; Annotation mit `pod`/`namespace`.
  - [ ] `NodeHighCPUUsage` — node-exporter: `(1 - avg by (instance)(rate(node_cpu_seconds_total{mode="idle"}[5m]))) > 0.9`, `for: 10m`, `severity: warning`; Annotation mit `instance`.
  - [ ] `NodeFilesystemAlmostFull` — node-exporter: `(1 - node_filesystem_avail_bytes{fstype!="",mountpoint="/"} / node_filesystem_size_bytes{fstype!="",mountpoint="/"}) > 0.85`, `for: 10m`, `severity: warning`; Annotation mit `instance`/`mountpoint`.
- [ ] Alle Schwellen als reine Zahlen-Literale, keine Hostnamen. Namespaces als
      Regex `workspace|workspace-korczewski` (bereits etablierte Konvention).
- [ ] Lokal validieren: `yq '.spec' k3d/monitoring/prometheus-rules.yaml > /tmp/r.yaml && promtool check rules /tmp/r.yaml` (falls `promtool`/`yq` lokal vorhanden; sonst CI/bats).

### Task 2 — SMTP-Receiver in Alertmanager verdrahten

- [ ] In `k3d/monitoring/alertmanager-config.yaml`:
  - [ ] Neuen Receiver `email` mit `emailConfigs` ergänzen: `to: ${CONTACT_EMAIL}`,
        `from: ${SMTP_FROM}`, `smarthost: ${SMTP_HOST}:${SMTP_PORT}`,
        `authUsername: ${SMTP_USER}`, `authPassword: { name: alertmanager-smtp, key: SMTP_PASSWORD }`,
        `requireTLS: true`, `sendResolved: true`.
  - [ ] Root-Route so anpassen, dass **jeder** Alert sowohl Pushover **als auch**
        E-Mail erhält: entweder `receiver`-Kaskade oder eine `routes`-Ergänzung
        mit `continue: true` (Pattern im Plan-Review festlegen; Default:
        Default-Receiver `email`, Sub-Route `continue: true` → `pushover` für
        `severity=critical`). E-Mail erhält damit **alle** Severities,
        Pushover-Prio-1 bleibt für `critical`.
- [ ] Die `${...}`-Platzhalter werden zur Deploy-Zeit via `envsubst` ersetzt
      (siehe Task 4); im committeten Manifest stehen **nur** die Variablen-Tokens,
      keine Brand-Domain (S3).

### Task 3 — Secret `alertmanager-smtp` (Dev-Platzhalter + Prod-Sealed)

- [ ] In `prod/monitoring/kustomization.yaml` einen `$patch: delete`-Eintrag für
      `Secret/alertmanager-smtp` ergänzen (analog zum vorhandenen
      `alertmanager-pushover`-Strip).
- [ ] In `environments/.secrets/mentolder.yaml` **und** `.secrets/korczewski.yaml`
      (git-crypt/gitignored — **nicht lesen/echoen ohne expliziten Auftrag**) den

> Hinweis: `SMTP_PASSWORD` existiert bereits in den Brand-`workspace-secrets`. Der
> neue `alertmanager-smtp`-SealedSecret trägt denselben Wert in die `monitoring`-NS,
> da Alertmanager nicht NS-übergreifend lesen kann.

## Offene Punkte für den Execute-Schritt

- Konkrete Routing-Topologie in der `AlertmanagerConfig` (Kaskade vs.
  `continue: true`) im Implementieren final festlegen — Default wie in Task 2.
- `env:seal` pro Brand muss auf der User-Maschine laufen (SealedSecrets-Cert).
- Live-Synthetik-Test ist manuell/Post-Deploy (nicht in CI), pro Brand auf `fleet`.

## Risiken

- **Falsche envsubst-Liste** → `${...}`-Tokens bleiben stehen → Alertmanager lädt
  invalide Config. Mitigation: `monitoring:validate` + bats-Build-Test.
- **Mail-Spam bei Flapping** → konservative `for:`-Fenster (10m) + bestehende
  `repeatInterval: 4h`/`groupInterval` der Route.
- **SMTP-Auth in monitoring-NS** → eigenes SealedSecret nötig (Task 3), sonst
  fällt der E-Mail-Versand still aus.
