# Implementierungsplan — T000897: One-Click Cluster-Reset für dev — Implementation Plan

**Ticket:** T000897
**Branch:** feature/t000897
**Spec:** docs/superpowers/specs/2026-06-17-t000897-design.md

## Ziel

Ein idempotenter Befehl `task dev:reset`, der den lokalen dev-k3d-Cluster komplett neu
aufbaut (delete → create → cert-manager-CRDs → `workspace:deploy ENV=dev` →
`workspace:office:deploy ENV=dev`) und dabei die vorhandenen lokalen
`website`/`brett`/`docs`-Docker-Images per Default re-importiert (kein Rebuild);
`REBUILD=1` erzwingt Neubau. dev-only, Plaintext-Secrets, kein Sealed-Secrets im k3d.

## Architektur & Tech-Stack

- **`task dev:reset`** — dünner Wrapper in `Taskfile.yml`, ruft `scripts/dev-reset.sh`.
- **`scripts/dev-reset.sh`** — Bash-Orchestrierung: Prod-Guard, Docker-Reachability,
  Image-Reuse-vs-Rebuild, Ablauf-Reihenfolge, Idempotenz. Ruft bestehende Tasks
  (`cluster:delete`, `cluster:create`, `workspace:deploy`, `workspace:office:deploy`,
  `*:build:import`) via `task …` auf — keine Logik-Duplikation.
- **`CLUSTER_NAME`** kommt aus der Taskfile-Var (`korczewski`, = `k3d-config.yaml`
  `metadata.name`); das Skript erhält ihn als Env/Arg, hardcodet nichts.
- cert-manager-CRDs werden **CRDs-only** appliziert (kein voller `cert:install`/ACME).

## Risiken & Offene Punkte

- **cert-manager-CRD-Quelle:** Die exakte CRDs-only-Apply-Form (URL der gepinnten
  cert-manager-Version vs. Helm-Flag) beim Implementieren an `cert:install` angleichen,
  damit dev und prod nicht driften. Da dev keinen ACME-Issue macht, reicht CRDs-only.
- **`CLUSTER_NAME`-Konsistenz:** Top-Level-Var ist `korczewski` (= k3d-config). Der Task
  muss exakt diesen Namen an `dev-reset.sh` durchreichen, sonst löscht/erstellt er den
  falschen bzw. keinen Cluster.
- **Image-Fallback-Build** kann lange dauern (mehrere Minuten) — nur wenn ein Image fehlt;
  klar geloggt, damit der Nutzer den Unterschied zum schnellen Re-Import-Pfad versteht.
