# Plan: Dependency-Update-Bot (Renovate) — self-hosted GitHub Action

Spec: `docs/superpowers/specs/2026-06-17-t000898-design.md`

## Ziel

Renovate als self-hosted GitHub Action aktivieren: wöchentliche, gruppierte,
Factory-observable Dependency-Update-PRs über alle Workspaces (npm/pnpm, Docker-Base-Images,
GitHub-Actions-Pins). Automerge nur für `patch` + `devDependencies`. SHA-Pinning der
Actions bleibt erhalten. Die tote `renovate.json` wird durch eine kommentierte `renovate.json5`
ersetzt und endlich von einem Workflow ausgeführt.

## Architektur / Tech-Stack

- **`.github/workflows/renovate.yml`** — `renovatebot/github-action` (SHA-gepinnt), Trigger
  `schedule` (wöchentlich) + `workflow_dispatch`. Auth via Repo-Secret `RENOVATE_TOKEN`
  (App-Installation-Token; App-ID/Key-Variante optional dokumentiert). Liest die Repo-Config.
- **`renovate.json5`** — Konfiguration mit Kommentaren: Manager (npm, kubernetes/dockerfile,
  github-actions), gruppierte Updates, `schedule`, `automerge`-Regeln (patch+devDeps),
  `pinDigests` für Actions, `semanticCommits: enabled`, Ignores, Dependency-Dashboard.
- **`renovate.json` gelöscht** — kein Dual-Config.

### Datei-Budgets (S1)

Beide Dateien sind Config/Manifest (`.yml`, `.json5`) — **nicht** in der S1-Limit-Map
(`gates.yaml` deckt nur Code-Extensions ab: ts/js/svelte/sh/astro/…). Kein Zeilen-Gate.
`renovate.json5` bleibt klein (~80–100 Zeilen, vergleichbar mit der 69-Zeilen-`renovate.json`),
`renovate.yml` ~30–40 Zeilen analog `opencode.yml`. **Kein Modul-Split nötig.**

### Guardrails

- **S3 (Hostnamen):** Keine Brand-Domain-Literale — die Config enthält nur Paket-/Image-Pattern,
  keine `*.mentolder.de`/`*.korczewski.de`-Strings. ✅ trivially erfüllt.
- **S4 (Orphans):** `renovate.json5` wird vom neuen `renovate.yml`-Workflow konsumiert (kein
  Orphan-Skript/-Manifest neu unter `k3d/` oder `scripts/`). Der Workflow ist self-triggernd
  (`schedule`/`workflow_dispatch`), also kein verwaister Einstiegspunkt.
- **S2 (Import-Zyklen):** kein TS/JS-Code, kein Graph betroffen. ✅
- **Supply-Chain:** `renovatebot/github-action` SHA-pinnen (secret-tragende Third-Party-Action),
  Kommentar mit Versions-/Bump-Hinweis wie in `opencode.yml`.

## Tasks

### 1. `renovate.json5` schreiben (Config-Migration)

- [ ] `renovate.json5` neu anlegen mit `$schema`-Verweis und `// Kommentaren` je Regel.
- [ ] Bestehende Inhalte aus `renovate.json` übernehmen: `extends: ['config:recommended']`,
  `minimumReleaseAge: '3 days'`, `ignorePaths` (node_modules, art-library, scripts/coaching,
  scripts/knowledge, tests/e2e, tests/unit), die `kubernetes.fileMatch`-Pfade
  (k3d/prod/prod-mentolder/prod-korczewski), und die Image-Familien-Gruppen
  (livekit/nextcloud/keycloak), `ghcr.io/paddione/*` disabled, `fedirz/faster-whisper-server`
  disabled.
- [ ] `schedule: ['before 9am on monday']` + `timezone: 'Europe/Berlin'`.
- [ ] **Automerge-Regeln präzisieren** (Spec-Policy):
  - `matchManagers:['npm'], matchUpdateTypes:['patch']` → `automerge:true`.
  - `matchManagers:['npm'], matchDepTypes:['devDependencies'], matchUpdateTypes:['minor','patch']`
    → `automerge:true`.
  - `matchManagers:['npm'], matchUpdateTypes:['major']` → `automerge:false`, eigene Gruppe.
  - Production-`dependencies` `minor` → kein automerge (nur PR).
  - `matchManagers:['kubernetes'], matchUpdateTypes:['major']` → `automerge:false`.
  - `matchManagers:['github-actions'], matchUpdateTypes:['major']` → `automerge:false`.
- [ ] `matchManagers:['github-actions']` → `groupName` + `pinDigests:true` (SHA-Pin
  **beibehalten**).
- [ ] Gruppierung pro pnpm-Workspace beibehalten:
  `groupName:'npm minor/patch ({{packageFileDir}})'`.
- [ ] `semanticCommits:'enabled'` setzen (Conventional-Commits-Gate: `chore(deps):`/`fix(deps):`).
- [ ] `dependencyDashboard:true`, `prConcurrentLimit`/`prHourlyLimit` moderat (z.B. 5/2).
- [ ] **Alte `renovate.json` löschen** (`git rm renovate.json`).
- [ ] Validieren: `npx --yes renovate-config-validator renovate.json5` (lokal; muss „valid" melden).

### 2. `.github/workflows/renovate.yml` schreiben

- [ ] Trigger: `schedule: - cron: '0 7 * * 1'` (Mo 07:00 UTC ≈ 08/09 Berlin) **und**
  `workflow_dispatch:` (manueller Lauf nur durch Repo-Mitglieder).
- [ ] `permissions:` minimal (`contents: write`, `pull-requests: write`) — App-Token trägt die
  eigentlichen Rechte; Workflow-`GITHUB_TOKEN` möglichst restriktiv.
- [ ] Job läuft `renovatebot/github-action@<SHA>` (SHA-gepinnt, Kommentar mit Version + Bump-Hinweis
  analog `opencode.yml`).
- [ ] `env`/`with`: `token: ${{ secrets.RENOVATE_TOKEN }}`,
  `configurationFile: renovate.json5`, `RENOVATE_REPOSITORIES` = dieses Repo.
- [ ] `concurrency:` Gruppe, damit sich Läufe nicht überlappen.
- [ ] Kommentar-Header: Verweis auf Spec + dass `RENOVATE_TOKEN` als Repo-Secret gesetzt sein muss
  (Setup-Schritt, nicht im Code).

### 3. Dokumentation / Setup-Hinweis

- [ ] Kurzen Abschnitt in `docs/` (z.B. bestehende CI-/Dependency-Doku oder `CONTRIBUTING`-nahe
  Stelle — **bestehende Datei erweitern**, keine neue Orphan-Doku) ergänzen: GitHub App anlegen,
  `RENOVATE_TOKEN` als Repo-Secret, `workflow_dispatch` für Erst-Lauf. Falls keine passende
  bestehende Datei: Hinweis als Kommentar-Block im Workflow belassen (kein neues `docs/*.md`).
- [ ] Prüfen, ob `CLAUDE.md` (CI/CD-Abschnitt) einen Satz zu Renovate als aktivem Workflow
  braucht — falls ja, **ein Satz** ergänzen, sonst weglassen.

### 4. Verifikation (Pflicht-Gate — finaler Task)

- [ ] `npx --yes renovate-config-validator renovate.json5` → „Configuration valid".
- [ ] YAML-Sanity des Workflows (`yamllint .github/workflows/renovate.yml` falls verfügbar; sonst
  visuelle Prüfung — CI hat keinen yamllint, siehe CLAUDE.md).
- [ ] `task test:changed` — gezielte Tests für geänderte Domains.
- [ ] `task test:all` — vollständiges Offline-Gate (BATS, kustomize-Struktur, Taskfile-Dry-Run,
  Security-Scan inkl. hardcoded-secret-Detection in den neuen Dateien).
- [ ] `task freshness:regenerate` — generierte Artefakte (test-inventory, repo-index) aktualisieren
  und mitcommitten.
- [ ] `task freshness:check` — CI-Äquivalent: Freshness + `quality:check` (S1–S4-Ratchet) +
  Baseline-Key-Count-Assertion. Muss grün sein, keine neue Violation, Baseline wächst nicht.
- [ ] Sicherstellen, dass die alte `renovate.json` wirklich entfernt ist (`git status` zeigt
  Deletion).

## Risiken / Hinweise

- **Erst-Lauf-PR-Flut:** Renovate öffnet beim ersten Lauf evtl. viele PRs. `prConcurrentLimit`
  begrenzt das; den ersten Lauf bewusst per `workflow_dispatch` triggern und beobachten.
- **`RENOVATE_TOKEN` fehlt** → Workflow schlägt fehl (kein stiller No-Op). Setup-Schritt ist
  Voraussetzung, gehört in die PR-Description als manueller Pre-Merge-/Post-Merge-Schritt.
- **Automerge + Branch-Protection:** Auto-Merge greift erst nach grünen Required Checks —
  kein Bypass. `semanticCommits:'enabled'` ist nötig, sonst rotes „Conventional Commits"-Gate.
- **Kein Cluster-Impact:** rein CI/Repo-Änderung, kein Deploy nötig (push-based Prod bleibt
  unberührt).
