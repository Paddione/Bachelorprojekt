# Implementierungsplan — T000901 Autopilot-Qualitäts-Ratchet (DeepSeek-Scout) — Implementation Plan

Spec: `docs/superpowers/specs/2026-06-17-t000901-design.md`
Branch: `feature/t000901`

## Ziel

Zwei additive Bausteine für die Scout-Qualität der Software Factory:
- **A — Hybrid-Scout:** deterministische Discovery in `scout.sh` ausbauen + optionaler DeepSeek-LLM-
  Fallback (eigenes pures Helper-Skript) bei `< N` gefundenen Dateien. Fail-soft, opt-in via Env.
- **B — Drift-Ratchet:** nach Merge `touched_files` (Vorhersage) vs. echter `git diff --name-only`
  (Realität) als Jaccard-Distanz vergleichen, Score in DB loggen, bei Schwellenüberschreitung nur
  Warn-Kommentar. **Kein hartes Gate.**

## Architektur & Tech-Stack

- Bash (`set -uo pipefail`, fail-soft) für `scout.sh`, `scout-llm-fallback.sh`, `scout-drift.sh`.
- Reiner CommonJS-Helper `scout-drift.cjs` (require-bar aus Bats + Skript, keine DB-/Pipeline-Imports
  → **S2**: kein Zyklus).
- Provider-Resolution für DeepSeek ausschließlich über bestehendes `route-provider.sh` (DB-getrieben,
  **keine** Brand-Domain-Literale → **S3**).
- DB: neue Spalten `scout_drift` (NUMERIC) + `scout_drift_at` (TIMESTAMPTZ) auf `tickets.tickets`,
  idempotente Migration. Persistenz/Lesen via `ticket.sh` (analog `set-touched-files`).
- Post-Merge-Hook in bestehendem `.github/workflows/post-merge.yml` (hat bereits Ticket-ID-Extraktion
  + `FLEET_KUBECONFIG` + DB-Zugriff), strikt fail-soft.
