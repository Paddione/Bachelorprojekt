# Plan — T000902: DeepSeek Code-Review-Qualität verbessern — Implementation Plan

**Ticket:** T000902
**Branch:** feature/t000902
**Spec:** docs/superpowers/specs/2026-06-17-t000902-design.md

## Ziel

False positives der DeepSeek-Code-Review im Factory-Pipeline deterministisch reduzieren:
(1) Findings außerhalb der geänderten Diff-Zeilen filtern (Line-Range-Mapping), (2) reine
Stil-Nitpicks killen, (3) vorbestehende Issues (unveränderte Dateien) verwerfen. Dazu
Prompt-Tuning für DeepSeek V3/R1: `confidence`-Feld, expliziter „nur geänderte Zeilen"-
Hard-Constraint, Confidence-Threshold.

## Architektur

- **Neuer pure Helper** `scripts/factory/review-finding-filter.mjs` — importzyklusfrei,
  keine DB-/API-Imports (S2). Reine Funktionen `parseChangedLines`, `isStyleNitpick`,
  `filterFindings`, `formatChangedLinesHint` + optionaler `--cli`-Modus (stdin/args →
  stdout JSON, exit 0 always, fail-open).
- **Konsumenten:** `ci-review.mjs` (CI-Action) importiert und nutzt den Helper inline;
  `pipeline.js` (Workflow-Monolith, im Quality-Gate ignoriert) ruft den `--cli`-Modus über
  einen `agent()`-Bash-Schritt vor `rawBlocking` — keine Logik-Duplikation.
- **Geteilte Prompt-Dateien** (4 Code-Lenses + Coordinator) um `confidence` + Hard-
  Constraint erweitert.

## Tasks

### Task 1 — Pure Helper-Modul + TDD-Tests
- [ ] **TDD zuerst:** `scripts/factory/review-finding-filter.test.mjs` anlegen mit `node:test`,
      dann `node --test scripts/factory/review-finding-filter.test.mjs` laufen lassen
      (expected: fail — das Helper-Modul existiert noch nicht). Tests für:
  - [ ] `parseChangedLines`: einzelner Hunk → korrekte +-Zeilennummern (neue Datei-Nummern).
  - [ ] mehrere Hunks in einer Datei; mehrere Dateien.
  - [ ] gelöschte (`-`) Zeilen erhöhen den new-line-Counter NICHT.
  - [ ] neue Datei (`@@ -0,0 +1,N @@`) → alle Zeilen 1..N.
  - [ ] `filterFindings`: Finding auf +-Zeile → kept; Finding auf Datei nicht im Diff (line gesetzt) → dropped(reason out-of-diff); Finding ohne line → kept (keine Zeilenregel).
  - [ ] `filterFindings`: confidence < threshold → dropped(reason low-confidence); fehlende confidence → kept (fail-open).
  - [ ] `isStyleNitpick`/`filterFindings`: severity low + „naming convention"/„formatting" → dropped(reason style-nitpick); severity high + style-Wort → kept (nicht droppen).
  - [ ] opts-Flags (`requireInDiff:false`, `dropStyleNitpicks:false`, custom `confidenceThreshold`) wirken.
  - [ ] `formatChangedLinesHint`: rendert Ranges kompakt pro Datei.
- [ ] `scripts/factory/review-finding-filter.mjs` implementieren bis Tests grün:
  - [ ] `parseChangedLines(diff)`: über Zeilen iterieren, `diff --git ... b/<path>` → aktueller Pfad, `@@ -a,b +c,d @@` → new-line-Cursor = c, dann je `+`-Zeile push & increment, je Kontextzeile increment, je `-`-Zeile skip. Rückgabe `Map<string, Set<number>>`.
  - [ ] `isStyleNitpick(finding)`: `severity==='low'` && Regex `/\b(naming|format|whitespace|indent|style|rename|typo|cosmetic)\b/i` auf `description`.
  - [ ] `filterFindings(findings, changedLines, opts)`: opts-Defaults `{confidenceThreshold:0.6, requireInDiff:true, dropStyleNitpicks:true}`. Pro Finding Drop-Reason ermitteln (Reihenfolge: out-of-diff → low-confidence → style-nitpick). Rückgabe `{kept, dropped:[{finding,reason}]}`.
  - [ ] `formatChangedLinesHint(changedLines)`: pro Datei sortierte Zeilen zu Ranges (`12-15, 20`).
  - [ ] `--cli`-Modus: `process.argv` `--diff <f> --findings <f> [--threshold <n>]`; liest, filtert, schreibt `{kept,droppedCount}` JSON auf stdout; `try/catch` → bei Fehler Original-Findings unverändert + exit 0 (fail-open).
- [ ] Verify: `node --test scripts/factory/review-finding-filter.test.mjs` grün; `node scripts/factory/review-finding-filter.mjs --cli ...` Smoke gegen Mini-Fixture.
- [ ] `wc -l` beide neuen Dateien — bestätigen ≤300 / ≤250 (sonst Funktion in Hilfsfunktionen splitten, nicht kosmetisch zusammenziehen).
