| Datei | Status | Zweck |
|-------|--------|-------|
| `scripts/vda/release-notes.sh` | NEU | Generator-Skript (generate/publish-github/publish-changelog/help) |

## Ziel

Aus gemergten PRs seit dem letzten Tag strukturierte, user-freundliche Release-Notes
erzeugen (LLM/DeepSeek-gestützt, deterministischer Fallback) und an GitHub-Release-Body +
`CHANGELOG.md` publizieren. Plus: `release-please-config.json` um explizite
`changelog-sections` je Paket erweitern.

## Architektur & Tech-Stack

- **Reines Bash** (passend zum bestehenden `vda.sh`-Pattern), `jq`, `gh` CLI, `git`, `curl`.
- DeepSeek über `DEEPSEEK_API_KEY`/`DEEPSEEK_BASE_URL` — exaktes curl-Muster aus
  `scripts/mishap-categorize.sh` (temperature 0, `--max-time 30`, fail-closed, jq-Extraktion).
- Neuer Subcommand hängt sich in den bestehenden `vda.sh`-Dispatch ein und delegiert an
  `scripts/vda/release-notes.sh` (analog `scripts/vda/ticket.sh`).
- Helper bleiben **pur** (keine DB-Writes; einziger Repo-Seiteneffekt: `publish-changelog`).

- **S2:** Bash, keine TS-Import-Zyklen relevant; Helper bleiben pur.

## Tasks

### Task 1 — `release-please-config.json` um `changelog-sections` erweitern
- [ ] In jedem der vier Pakete (`brett`, `arena-server`, `k3d/docs-content`, `website`) eine
      `changelog-sections`-Liste ergänzen mit der vollständigen Commit-Typ-Map:
      `feat`→Features, `fix`→Bug Fixes, `perf`→Performance, `refactor`→Code Refactoring,
      `docs`→Documentation, `test`→Tests, `build`→Build System, `ci`→CI/CD, `chore`→hidden,
      `revert`→Reverts. (Sektion-Reihenfolge konsistent über alle Pakete.)
- [ ] JSON valide halten (`jq . release-please-config.json` ohne Fehler).
- [ ] Verifizieren, dass `task test:all` Taskfile-Dry-Run + ggf. Manifest-Checks grün bleiben.

### Task 2 — Generator-Skript `scripts/vda/release-notes.sh` (Gerüst + Dispatch)
- [ ] Datei mit `set -euo pipefail`, vda-core sourcen (`source scripts/lib/vda-core.sh` via
      `SCRIPT_DIR/../lib/vda-core.sh`), Subcommand-Dispatch:
      `generate | publish-github | publish-changelog | help`.
- [ ] `help` → exit 0 (listet Subcommands), unknown → `vda_error` + exit 2 (vda-Konvention).
- [ ] Arg-Parsing pro Subcommand (`--since`, `--out`, `--tag`, `--notes-file`, `--dry-run`).

### Task 3 — `generate`: PR-Sammlung + deterministischer Fallback
- [ ] Letzten Tag ermitteln: `git describe --tags --abbrev=0` (override via `--since <tag|date>`).
- [ ] Gemergte PRs seit Tag sammeln via `gh pr list --state merged --search "merged:>=<date>"
      --json number,title,labels,mergedAt` (Datum aus Tag-Commit `git log -1 --format=%cI <tag>`).
- [ ] Fehlt `gh` → degradierter Fallback über `git log <tag>..HEAD --format='%s'` (Commit-Subjects).
- [ ] PRs/Commits nach Conventional-Type bzw. Label gruppieren (feat/fix/perf/…); Markdown-Liste
      mit Abschnittsüberschriften bauen. **Das ist die Grundwahrheit** und immer vorhanden.
- [ ] Output nach stdout oder `--out <file>`.

### Task 4 — `generate`: DeepSeek-Narrative (best-effort, fail-closed)
- [ ] Wenn `DEEPSEEK_API_KEY` gesetzt: PR-Titel-Liste als Prompt an DeepSeek (curl-Muster aus
      `mishap-categorize.sh`, `--max-time 30`, temperature 0), Prompt = "Fasse diese gemergten
      PRs zu einer kurzen, user-freundlichen 'Was ist neu'-Einleitung zusammen".
- [ ] LLM-Antwort als Prosa-Einleitung **über** die deterministische Liste setzen.
- [ ] Fail-closed: kein Key / leere Antwort / curl-Fehler → nur deterministische Liste, exit 0
      (Generator darf nie einen Release blockieren).

### Task 5 — `publish-github` + `publish-changelog`
- [ ] `publish-github [--tag] [--notes-file]`: `gh release edit <tag> --notes-file <f>` bzw.
      `gh release create <tag> --notes-file <f>`; `--dry-run` zeigt nur das Kommando.
- [ ] `publish-changelog [--notes-file]`: neuen Abschnitt **oben** in `CHANGELOG.md` einfügen
      (prepend; Datei anlegen falls fehlend); `--dry-run` zeigt das Diff/den Block ohne Schreiben.
- [ ] Beide best-effort, klare `vda_success`/`vda_warn`-Ausgaben.

### Task 6 — `vda.sh`-Dispatch + Taskfile-Eintrag (Orphan-Vermeidung, S4)
- [ ] In `scripts/vda.sh` einen `release-notes)`-Case ergänzen, der an
      `scripts/vda/release-notes.sh "$@"` exect (analog `ticket)`), plus Zeile in `show_help`.
- [ ] In `Taskfile.yml` Task `release:notes` ergänzen (`cmds: [bash scripts/vda.sh release-notes
      generate {{.CLI_ARGS}}]`, `desc:` für task-oracle-Auffindbarkeit).

### Task 7 — BATS-Smoke-Test (offline, deterministisch)
- [ ] TDD: `tests/unit/vda-release-notes-smoke.bats` **zuerst** schreiben und gegen das noch
      unvollständige Skript ausführen (`bats tests/unit/vda-release-notes-smoke.bats`) —
      **expected: fail** (Skript/Subcommands existieren noch nicht), bevor die Implementierung
      die Tests grün macht.
- [ ] `tests/unit/vda-release-notes-smoke.bats`: `help` exit 0 + listet Subcommands; unknown
      subcommand exit 2.
- [ ] `generate` ohne `DEEPSEEK_API_KEY` mit `gh`/`git`-PATH-Shim (Stub im `$BATS_TMPDIR`,
      **außerhalb des Repo-Trees** anlegen + im teardown aufräumen) → exit 0, Markdown enthält
      mind. eine Abschnittsüberschrift. Kein echter `gh`/curl-Call.
- [ ] `publish-github --dry-run` / `publish-changelog --dry-run` → exit 0, keine Seiteneffekte.
- [ ] Sicherstellen, dass die Datei von der bestehenden `task test:all`-BATS-Selektion erfasst
      wird (gleicher Pfad-Glob wie `vda-ticket-smoke.bats`).

### Task 8 — Doku-Verweis

## Verifikationsstrategie

- Offline-deterministisch über BATS mit PATH-Shims für `gh`/`curl` (kein Netz, kein Cluster).
- Manuelle Sichtprüfung der generierten Notes (ein Lauf gegen den echten letzten Tag, lokal).
- `release-please`-Config-Änderung greift erst beim nächsten main-Push — kein sofortiger
  Seiteneffekt; Taskfile-Dry-Run/JSON-Validität deckt sie im CI ab.
