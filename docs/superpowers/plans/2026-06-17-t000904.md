# Implementierungsplan — T000904 Audit-Log (Wer hat was wann getan) — Implementation Plan

> Spec: `docs/superpowers/specs/2026-06-17-t000904-design.md`
> Branch: `feature/t000904`

## Ziel

Sicherheitsrelevanter Audit-Trail in zwei Säulen: (1) Keycloak Login-/Admin-Event
DB-Logging realm-seitig aktivieren (einheitlich über alle vier Realm-JSONs, 90 Tage
Retention), (2) neue Tabelle `audit.audit_log` + pure Helper `recordAudit()` +
Instrumentierung der kuratierten sicherheitsrelevanten `/api/admin/*`-Endpoints.
**Erfassung + Schema + Verdrahtung only — keine UI** (separates Ticket).

## Architektur & Tech-Stack

- **DB**: shared-db PostgreSQL 16, Datenbank `website`. Neues Schema `audit`
  (`AUTHORIZATION website`), Tabelle `audit.audit_log`. Bootstrap über
  `k3d/website-schema.yaml` (init- **und** ensure-Block, idempotent) + versionierte
  Mirror-Migration `website/src/db/migrations/20260617_create_audit_log.sql`.
- **Backend**: Astro API-Routes (`website/src/pages/api/admin/*`), TypeScript.
  Pool aus `website/src/lib/website-db.ts` (`pool`). Auth via bestehendes
  `getSession`/`isAdmin` (`website/src/lib/auth.ts`).
- **Helper**: `website/src/lib/audit-log.ts` — pures DB-Modul (`recordAudit`,
  `clientIpFromRequest`), nur Import `type { Pool } from 'pg'`. Kein Rück-Import auf
  API-/DB-Schichten (S2).
- **Keycloak**: Realm-JSONs `k3d/realm-workspace-dev.json`,
  `prod/realm-workspace-prod.json`, `prod-mentolder/realm-workspace-mentolder.json`,
  `prod-korczewski/realm-workspace-korczewski.json`.
- **Tests**: Vitest + pg-mem (Muster `website/src/lib/coaching-templates-db.test.ts`).
- **Docs/DSGVO**: neue `docs/audit-log-retention.md`.

  CREATE TABLE IF NOT EXISTS audit.audit_log (

    action      text NOT NULL,
    target_type text,

  CREATE INDEX IF NOT EXISTS audit_log_actor_idx ON audit.audit_log (actor_id, ts DESC);

### Task 3 — Pure Helper `website/src/lib/audit-log.ts` (TDD)
- [ ] **Zuerst** `website/src/lib/audit-log.test.ts` (pg-mem, Muster
      `coaching-templates-db.test.ts`):
  - `recordAudit` schreibt Eintrag mit allen Pflichtfeldern (Read-back).
  - `recordAudit` ist **fail-soft**: bei Insert-Fehler (z.B. fehlende Tabelle)

  - `clientIpFromRequest` parst ersten Hop aus `x-forwarded-for`, gibt `null` bei

- [ ] Test zuerst laufen lassen, Fehlschlag bestätigen (TDD Red):
      `pnpm vitest run website/src/lib/audit-log.test.ts` — **expected: fail**
      (Modul/`recordAudit` existiert noch nicht).
- [ ] Dann `audit-log.ts` implementieren:
  ```ts
  import type { Pool } from 'pg';
  export interface AuditEntry {
    actor_id?: string | null; actor_email?: string | null;
    action: string; target_type?: string | null; target_id?: string | null;
    ip?: string | null; metadata?: unknown;
  }
  export function clientIpFromRequest(request: Request): string | null { /* x-forwarded-for first hop */ }
  export async function recordAudit(pool: Pool, e: AuditEntry): Promise<void> {
    try {
      await pool.query(
        `INSERT INTO audit.audit_log (actor_id, actor_email, action, target_type, target_id, ip, metadata)
         VALUES ($1,$2,$3,$4,$5,$6::inet,$7::jsonb)`,
        [e.actor_id ?? null, e.actor_email ?? null, e.action, e.target_type ?? null,
         e.target_id ?? null, e.ip ?? null, JSON.stringify(e.metadata ?? null)]);
    } catch (err) { console.warn('[audit] recordAudit failed:', err); }
  }
  ```
- [ ] Doku-Kommentar: `metadata` darf **keine** Klartext-Secrets enthalten.
- [ ] `pnpm vitest run website/src/lib/audit-log.test.ts` grün.
