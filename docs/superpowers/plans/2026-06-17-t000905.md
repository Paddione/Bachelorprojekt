# Plan: T000905 — Backup-Status sichtbar in Admin-UI

## Ziel

Read-only Backup-Status-Panel in der Plattform-Admin-UI (`/admin/platform`), das für **beide
Brands** die zwei Backup-Pipelines (`db-backup`, `pvc-backup`) inkl. Filen-Upload-Status mit
Ampel + Zeitstempeln anzeigt. Neuer Endpoint `GET /api/admin/backup-status`, der k8s Jobs/CronJobs
mit `app=db-backup` und `app=pvc-backup` abfragt. Spec:
`docs/superpowers/specs/2026-06-17-t000905-design.md`.

## Architektur

- **Backend:** Astro-API-Route `website/src/pages/api/admin/backup-status.ts`. Muster exakt wie
  `monitoring.ts`/`deployments.ts`/`ops/backup/list.ts`: `getSession`+`isAdmin`-Gate →
  `createK8sClient()` (503 ohne SA-Token) → Namespace aus `BRAND_ID`/`BRAND`
  (`workspace`↔`workspace-korczewski`) → `GET /apis/batch/v1/namespaces/{ns}/jobs?labelSelector=...`
  je Pipeline + `cronjobs/{name}` für `schedule`.
- **Pure Helper (S2-konform, exportiert + unit-getestet):** `derivePipelineStatus(jobs, cronjob,
  now)` und `pickLatest(jobs)` — Eingabe = bereits geparste Job-Objekte, Ausgabe =
  `{ id, light, lastRun, lastSuccessfulUpload, succeeded, failed, active, schedule }`. Reine
  Funktionen, nur Typ-Imports, kein DB-/API-Schicht-Rückimport.
- **Filen-Status:** `job.status.succeeded > 0` ist die SSOT für „verschlüsselter Upload OK" — der
  `filen-upload`-Container macht bei Upload-Fehler `exit 1`, der Job wird also nur grün, wenn
  encrypt **und** upload durchliefen (encrypt-then-upload). `lastSuccessfulUpload` =
  `completionTime` des jüngsten `succeeded`-Jobs. Keine Log-Auswertung im MVP.
- **Frontend:** Neue Svelte-Komponente
  `website/src/components/admin/platform/BackupStatusCard.svelte`, eingebunden in
  `PlatformHub.svelte` (Card, kein neuer Tab → minimaler Diff). Fetcht `/api/admin/backup-status`,
  rendert pro Pipeline Ampel + letzten Lauf + letzten erfolgreichen Upload.
- **RBAC:** `k3d/website-rbac.yaml` `website-monitoring-role` um `batch`
  (`jobs`,`cronjobs`: `get`,`list`) erweitern. Schließt zugleich die bestehende 403-Lücke von
  `ops/backup/list.ts`/`ops/restore.ts`.

## Tech-Stack

Astro API-Route (TS), Svelte (Admin-Komponente), Vitest (Helper-Unit-Tests), Kustomize/RBAC
(YAML). Keine neuen Pakete.

## S3 (Hostnamen)

Keine `*.mentolder.de`/`*.korczewski.de`-Literale. Namespace-Auflösung ausschließlich über
`BRAND_ID`/`BRAND` → `workspace`/`workspace-korczewski` (Muster aus `monitoring.ts`).
Intervall-Schwellen (db ≤ 48h, pvc ≤ 48h) als benannte Konstanten im Helper.

## S4 (Orphans)

`k3d/website-rbac.yaml` ist heute **nicht** in einer `kustomization.yaml` referenziert und wird
ad-hoc appliziert (bekannter „website-ns domain-config gap"). Damit die erweiterte RBAC den Cluster
auch erreicht, wird `website-rbac.yaml` in `prod-fleet/website-mentolder/kustomization.yaml` und
`prod-fleet/website-korczewski/kustomization.yaml` als Resource aufgenommen (Role lebt in ns
`workspace`/`workspace-korczewski`; die Datei selbst pinnt ns `workspace`, daher pro Brand-Overlay
den Namespace berücksichtigen — falls ns-Pinning kollidiert, RoleBinding/Role brand-neutral über
`namespace:`-Direktive des Overlays auflösen). Kein neues `scripts/*.sh`. Keine neuen
k3d-Base-Manifeste → keine `k3d/kustomization.yaml`-Änderung nötig.

## Risiken / offene Punkte

- **ns-Pinning in `website-rbac.yaml`:** Die Datei pinnt Role/RoleBinding auf ns `workspace`. Für
  korczewski muss die Bindung im `workspace-korczewski`-Namespace landen. In der Implementierung
  prüfen, ob der bestehende korczewski-Apply-Pfad das bereits leistet; sonst eine
  brand-neutrale/`namespace:`-overlay-gesteuerte Variante wählen (nicht zwei divergente Kopien).
- **403 vor RBAC-Apply:** Bis die erweiterte Role appliziert ist, liefert der Endpoint pro Pipeline
  grau (403→grau gehandhabt). Nach Merge **explizit deployen** (push-based, kein GitOps): RBAC auf
  beide Brand-Namespaces anwenden.
