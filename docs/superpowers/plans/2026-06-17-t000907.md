# Implementierungsplan — T000907 Ordnerstruktur-Templates für neue Projekte

**Ticket:** T000907
**Branch:** feature/t000907
**Spec:** docs/superpowers/specs/2026-06-17-t000907-design.md

## Ziel

Konfigurierbare Ordnerstruktur-Templates (Admin-CRUD + mitgeliefertes Default-Set), die bei
Projektanlage wählbar sind und automatisch per WebDAV `MKCOL` im Nextcloud-Projektordner
(`Projects/${projectId}/<ordner>`) angelegt werden. Brand-isoliert, DB-gespeichert
(`folder_templates`, JSONB-Struktur).

## Architektur & Tech-Stack

- **DB:** PostgreSQL (website-DB), neue Tabelle `public.folder_templates`, manuelle idempotente
  SQL-Migration nach Repo-Konvention (`website/src/db/migrations/*.sql`, kein Auto-Runner).
- **Backend:** Astro API-Routen (`website/src/pages/api/admin/...`), TypeScript. Neues pures
  DB-/Validierungs-Modul `website/src/lib/folder-templates-db.ts`. Wiederverwendung von
  `ensureFolder` aus `nextcloud-files.ts` für die WebDAV-Anlage.
- **Frontend:** Astro-Admin-Seiten (`.astro`), Settings-Tab. Create-Form-Partial-Extraktion,
  um das Zeilenbudget von `projekte.astro` (Budget 0) einzuhalten.
- **Tests:** Vitest für pure Validatoren + Default-Konstante.

## Tasks

### Task 1 — DB-Migration `folder_templates` + Seed
- [ ] `website/src/db/migrations/20260617_create_folder_templates.sql` anlegen:
  - Header-Kommentar mit manueller Apply-Anleitung (analog `20260607_create_generation_jobs.sql`).
  - `CREATE TABLE IF NOT EXISTS public.folder_templates (...)` gemäß Spec §4
    (`id`, `brand`, `name`, `structure JSONB`, `is_default`, `created_at`, `updated_at`).
  - `CREATE UNIQUE INDEX IF NOT EXISTS` auf `(brand, name)`.
  - Partieller Unique-Index `... ON folder_templates (brand) WHERE is_default`.
  - `GRANT ALL PRIVILEGES ON public.folder_templates TO website;`.
  - Idempotenter Seed: `INSERT ... SELECT ... WHERE NOT EXISTS` des Default-Templates „Standard"
    (`is_default = true`, 5 Ordner) **pro im System bekannter Brand** — als Parameter/`unnest`
    über eine kleine Brand-Liste ODER nur für den aktuellen `:brand` (Apply-Anleitung dokumentiert
    den Aufruf je Brand). Keine Brand-Domain-Literale, nur Brand-IDs (`mentolder`/`korczewski`).
- [ ] Migration zweimal lokal anwenden (dev k3d shared-db) → kein Fehler; Default existiert
      je Brand genau einmal (`SELECT brand,count(*) ... WHERE is_default GROUP BY brand`).
- [ ] Apply-Schritt in `docs/` Migrations-/Runbook-Stelle notieren, damit die Datei nicht als
      Orphan gilt (S4) und post-merge auf beide Brand-DBs angewandt wird.

### Task 2 — Pures Modul `folder-templates-db.ts` (TDD)
- [ ] **Test zuerst:** `website/src/lib/folder-templates-db.test.ts`:
  - `DEFAULT_FOLDERS` = `['01_Vertrag','02_Rechnungen','03_Dokumente','04_Assets','05_Kommunikation']`
    (Reihenfolge geprüft).
  - `validateStructure`: akzeptiert gültige Liste; lehnt ab bei `..`, `/..`, führendem `/`,
    leerem String, > 50 Ordnern, Segment > 100 Zeichen, unerlaubten Zeichen.
- [ ] `folder-templates-db.ts` implementieren:
  - Konstante `DEFAULT_FOLDERS`, `MAX_FOLDERS = 50`, `FOLDER_PATH_RE = /^[A-Za-z0-9 _.\/-]+$/`.
  - **Pure** `validateStructure(folders: unknown): { ok: boolean; error?: string; folders?: string[] }`
    — kein DB-/Nextcloud-Import (S2/Unit-testbar).
  - DB-Funktionen (nutzen vorhandenen `pool` aus `website-db`/`platform-db`-Muster, **per Import
    des bestehenden Pools, nicht via Erweiterung von website-db.ts**):
    `listTemplates(brand)`, `getTemplate(brand,id)`, `getDefaultTemplate(brand)`,
    `createTemplate({brand,name,folders,isDefault})`, `updateTemplate(brand,id,{...})`,
    `deleteTemplate(brand,id)`. Beim Setzen `is_default=true` vorher andere Defaults des Brands
    auf `false` setzen (Transaktion).
  - `materializeTemplate(projectId, folders)`: für jeden Ordner `ensureFolder(\`Projects/${projectId}/${folder}\`)`
    aufrufen (aus `nextcloud-files.ts` importiert); Fehler sammeln, `{ created, failed }` zurückgeben.
