# Plan: Auto-Deploy bei Merge [T000932] — Implementation Plan

> Spec: `docs/superpowers/specs/2026-06-17-t000932-design.md`
> Branch: `feature/t000932`

## Ziel

Ein **unified post-merge Deploy-Workflow**: bei Merge nach `main` wird das Ticket auf
`awaiting_deploy` gesetzt; berührt der Merge Manifeste, fährt ein `workspace:deploy`-Fan-out über
**beide Brands** (mentolder + korczewski) auf fleet; nach erfolgreichem Deploy → `done`.
Website/Brett/Docs behalten ihre eigenen `build-*.yml`. Push-based Modell bleibt (kein Reconciler).

## Architektur & Tech-Stack

- **GitHub Actions** (`.github/workflows/post-merge.yml`) — erweitert um Manifest-Deploy + finalen
  `done`-Übergang. Trigger bleibt `push: branches: [main]`.
- **Pure Bash-Helper** `scripts/changed-manifests.sh` — Pfad-Allowlist-Erkennung
  (`git diff --name-only <prev>..<head>` ⋂ Manifest-Pfade), Exit 0 (Manifest berührt) / 1 (nicht),
  Begründung auf stdout. Keine DB-/API-/Netz-Aufrufe (S2: pure, kein Import-Zyklus). Liegt im
  `scripts/`-Root analog zu `scripts/ci-dummy-secrets.sh` (S4: wird von der Workflow-YAML **und**
  vom BATS-Test referenziert → kein Orphan).
- **Status-Lebenszyklus** über vorhandenes `scripts/ticket.sh update-status`
  (`awaiting_deploy` → `done`); Logik-Referenz: `scripts/factory/deploy-transition.mjs`
  (push-based ⇒ `awaiting_deploy`). Kein neuer State, kein neues Skript dafür.
- **Deploy** via `task workspace:deploy ENV=mentolder` + `ENV=korczewski`
  (entspricht `workspace:deploy:all-prods`, im Runner mit explizit gesetztem `FLEET_KUBECONFIG`/Kontext).
- **Tests**: BATS für den Helper, eingehängt in `task test:unit` (→ `task test:all`).

## S2 / S3 / S4 Gates

- **S2**: `scripts/changed-manifests.sh` ist pure (nur `git`/Stringvergleich), kein Import auf
  DB-/API-Layer, kein Zyklus.
- **S3**: **keine** Brand-Domain-Literale (`*.mentolder.de`/`*.korczewski.de`) in Skript oder
  Workflow-Snippets. Brand-Auswahl über `ENV=mentolder|korczewski`-Schleife; Kontext/Domain löst
  `scripts/env-resolve.sh` + `configmap-domains.yaml` auf (bestehendes Verhalten von
  `workspace:deploy`). Die Manifest-Pfad-Allowlist enthält **Verzeichnis**namen
  (`k3d/`, `prod/`, `prod-fleet/`, `prod-mentolder/`, `prod-korczewski/`, `environments/`), keine Hostnamen.
- **S4**: `scripts/changed-manifests.sh` wird von `post-merge.yml` **und**
  `tests/unit/changed-manifests.bats` referenziert; die `.bats`-Datei wird in `task test:unit`
  eingehängt → keine Orphan-Manifeste/-Skripte.

## Tasks

### Task 1 — Pre-Flight & Baseline-Recompute
- [ ] `git pull --rebase origin main` (sauberer Stand).
- [ ] Für jede zu ändernde Datei `wc -l` + Baseline-Lookup (siehe S1-Tabelle) erneut ausführen und
      Budget bestätigen.
- [ ] `task test:all` einmal grün laufen lassen (Baseline für „vorher grün").
- [ ] Prüfen, ob `gates.yaml` ein `.yml`-S1-Limit kennt; falls ja, Budget für `post-merge.yml`
      gegen dieses Limit notieren.

### Task 2 — Pure Manifest-Erkennungs-Helper (TDD)
- [ ] **Test zuerst**: `tests/unit/changed-manifests.bats` anlegen:
  - [ ] Hit: Diff enthält `k3d/foo.yaml` → Exit 0 + Begründung nennt die Manifest-Pfade.
  - [ ] Hit: Diff enthält `prod-fleet/mentolder/kustomization.yaml` → Exit 0.
  - [ ] Hit: Diff enthält `environments/mentolder.yaml` → Exit 0.
  - [ ] Miss: Diff nur `docs/x.md` + `tests/unit/y.bats` → Exit 1.
  - [ ] Miss: Diff nur `website/src/pages/index.astro` → Exit 1 (Website hat eigenen Pfad).
  - [ ] Edge: leerer Diff → Exit 1.
  - [ ] (Setup über ein temporäres Git-Repo in `$BATS_TMPDIR`/`mktemp -d`, **nicht** im Tree;
        teardown räumt auf — Memory `feedback_bats_temp_outside_tree`.)
  - [ ] Test jetzt laufen lassen, solange `scripts/changed-manifests.sh` noch nicht existiert
        (`task test:unit:changed-manifests`) — **expected: fail** (TDD: rot vor grün).
- [ ] **Dann** `scripts/changed-manifests.sh` implementieren:
  - [ ] Args: `<base_ref> <head_ref>` (default `HEAD~1 HEAD`).
  - [ ] `git diff --name-only "$base".."$head"` gegen Allowlist-Präfixe
        (`k3d/ prod/ prod-fleet/ prod-mentolder/ prod-korczewski/ environments/`) matchen.
  - [ ] Treffer → die matchenden Pfade auf stdout, Exit 0; sonst „no manifest changes", Exit 1.
  - [ ] `set -euo pipefail`, robuste Quoting, kein Brand-Domain-Literal (S3).
- [ ] `chmod +x scripts/changed-manifests.sh`.
- [ ] `wc -l scripts/changed-manifests.sh` → unter ~80 % von 500 bestätigen.

### Task 3 — Bats in test:all einhängen (S4)
- [ ] Neuen internen Task `test:unit:changed-manifests` in `Taskfile.yml` ergänzen
      (Muster wie `test:unit:scripts`: `./tests/unit/lib/bats-core/bin/bats tests/unit/changed-manifests.bats`).
- [ ] In `test:unit` als Step eintragen (Reihenfolge analog bestehender `test:unit:*`-Liste).
- [ ] `task test:unit:changed-manifests` grün.

### Task 4 — post-merge.yml zum unified Deploy-Workflow erweitern
- [ ] Job 1 `mark-awaiting` (aus bestehendem `close-ticket` umgebaut): Ticket-ID extrahieren,
      `update-status --status awaiting_deploy` (statt `done`). Kein Ticket → Exit 0. Non-fatal bei
      DB-Hiccup (bestehendes Verhalten beibehalten).
- [ ] Job 2 `deploy-manifests` (`needs: mark-awaiting`):
  - [ ] `actions/checkout@v4` mit `fetch-depth: 2` (für Diff gegen Vorgänger-Commit).
  - [ ] Schritt „detect": `bash scripts/changed-manifests.sh` → Output `manifests_changed=true|false`
        in `$GITHUB_OUTPUT` (Helper-Exit 0/1 auswerten; **nicht** den Job hart faillen lassen,
        sondern Flag setzen).
  - [ ] Folgeschritte nur `if: manifests_changed == 'true'`:
        kubectl/kustomize installieren (Snippet wie `build-website.yml`), `FLEET_KUBECONFIG`
        base64-dekodieren nach `~/.kube/config`, `arduino/setup-task` (oder vorhandenes
        `task`-Setup) bereitstellen, dann `task workspace:deploy ENV=mentolder` **und**
        `ENV=korczewski` (explizit, fail-closed).
  - [ ] **Finaler Status-Step** (`if: success()`): Ticket → `done` via `update-status`. Läuft sowohl
        wenn deploy lief als auch wenn `manifests_changed=false` (dann gibt es keinen Cluster-Change →
        direkt `done`). Bei Deploy-Fehler bleibt das Ticket `awaiting_deploy` (Step wird übersprungen,
        Job rot — AK4).
  - [ ] `permissions: contents: read` (+ ggf. `actions: read`), keine unnötigen Scopes.
- [ ] Sicherstellen, dass `website/**`, `brett/**`, reine `docs/**`-Merges **kein**
      `manifests_changed=true` ergeben (Allowlist greift nicht) → kein Doppel-Deploy (AK5).
- [ ] Kein Brand-Domain-Literal im YAML (S3) — Domains kommen aus `env-resolve.sh`/ConfigMap.

### Task 5 — Final Verification (Pflicht-Gate)
- [ ] `task test:changed` (gezielte Tests der geänderten Domains, BATS inkl. neuem
      `test:unit:changed-manifests`, Manifest-Struktur, Taskfile-Dry-Run).
- [ ] `task test:inventory` regenerieren und `website/src/data/test-inventory.json` mitcommitten
      (Test-Datei hinzugefügt → CI-Inventory-Check sonst rot).
- [ ] `task freshness:regenerate` (generierte Artefakte: test-inventory, repo-index, …).
- [ ] `task freshness:check` (CI-Äquivalent: Freshness + `quality:check` S1–S4-Ratchet + Baseline-Assertion).
- [ ] `act` bzw. YAML-Lint des Workflows lokal (optional, kein CI-Gate) — mind. `yamllint`
      auf `post-merge.yml`.
- [ ] Sicherstellen: **keine** neuen Einträge in `docs/code-quality/baseline.json` (Key-Count-Assertion).
