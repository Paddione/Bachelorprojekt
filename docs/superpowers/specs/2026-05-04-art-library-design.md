# Art Library — Design Spec

**Date:** 2026-05-04
**Author:** brainstorm session (Patrick + Claude)
**Status:** Approved for implementation planning

---

## Problem

The Korczewski tabletop board webapp (Brett, `brett.localhost`) currently uses four abstract Three.js token shapes (`pawn / triangle / square / diamond`). A high-fidelity art handoff exists at `c:/Users/PatrickKorczewski/Downloads/Assets_korczewski/design_handoff_artlibrary` containing four character figurines, six props, six terrain swatches, and brand logos in 2D SVG form. Two needs follow:

1. Brett's tokens should become real characters instead of abstract shapes.
2. Admins should be able to view the cluster's installed art catalog (designer reference, palette swatches, asset existence verification) from the dashboard admin menu.

The bridge problem: Brett is **3D** (Three.js procedural meshes); the handoff is **2D** (SVG portraits + figurines). The library also needs to be a single source of truth that both Brett and the dashboard consume.

## Goals

- Replace Brett's abstract token row with character figurines, preserving existing rotation/facing UX.
- Add a read-only "Art Library" tab to the dashboard admin menu that catalogs every asset, filterable by kind and tags, with palette swatches and SVG download.
- Ship one library that both apps consume, deployed cluster-natively as a Kubernetes ConfigMap.
- Stay brand-scoped: the Korczewski art set deploys to the Korczewski environment; other environments fall back to existing behavior.

## Non-goals (v1)

- Runtime palette swaps / per-tenant skinning. Each character ships with a baked default palette.
- Asset upload / edit / delete from the admin UI. Library is git-managed.
- Bilingual coverage beyond DE / EN. Names are functional labels, not narrative.
- Automatic snapshot migration. Existing Brett saves with legacy `pawn/triangle/square/diamond` types continue to render.
- Mentolder art set. Scaffolding supports it; content is out of scope for v1.

## Architecture

```
art-library/                                  ← repo dir, source of truth
├── manifest.schema.json                      ← JSON Schema for any set's manifest
├── README.md                                 ← layout + how to add a set
└── sets/
    └── korczewski/                           ← one brand set
        ├── manifest.json
        ├── tokens.css                        ← color/type/spacing/animations
        ├── characters/                       ← 4 × {portrait.svg, figurine.svg}
        ├── props/                            ← 6 × icon.svg
        ├── terrain/                          ← 6 × swatch.svg
        ├── logos/                            ← 5 × svg
        └── CREDITS.md                        ← original handoff names + bios (designer reference only)
```

**Kustomize wiring is per-overlay, not a shared component.** Each overlay that wants the library declares its own `configMapGenerator` pointing at the chosen set:

| Overlay | Wires up the ConfigMap? | Set used |
|---|---|---|
| `k3d/` (dev base) | yes | `art-library/sets/korczewski/` |
| `prod-korczewski/` | yes (via overlay's own `configMapGenerator`) | `art-library/sets/korczewski/` |
| `prod-mentolder/` | no | — (admin tab self-degrades to empty-state) |

A future mentolder set would be wired by adding a `configMapGenerator` to `prod-mentolder/kustomization.yaml`. No shared Kustomize component needed; the path stays explicit per environment.

At runtime:

- One `ConfigMap art-library` in the `workspace` namespace, generated by Kustomize from the active set's directory.
- Brett pod and dashboard pod both mount it read-only at the static directory their server serves (e.g. `brett/public/art-library`, `dashboard/web/public/art-library`).
- Both apps `fetch('/art-library/manifest.json')` once on init.
- 404 path: Brett falls back to legacy abstract shapes; dashboard art tab shows empty-state copy.

The original handoff names (Elara/Korrin/Vex/Brann) and bios are preserved only in `CREDITS.md` for designer reference. Runtime never displays them.

### JSON Schema highlights

- `id` — unique, kebab-case slug, stable.
- `kind` — enum `character | prop | terrain | logo`.
- `name_de` — required. `name_en` — optional.
- `tags` — array of strings, lowercase, hyphen-separated.
- `files` — object whose values are paths relative to the manifest, all of which must exist on disk.
- Characters require `files.portrait` and `files.figurine`; props require `files.icon`; terrain requires `files.swatch`; logos require `files.svg`.

CI validates the manifest against the schema and asserts every referenced file exists.

## JSX → SVG conversion

The handoff's `characters.jsx` builds each SVG inline with `{p}` palette interpolation. We render once per character with the canonical palette baked in and save static `.svg` files. Recoloring is **not** runtime-dynamic in v1 (palette stored as metadata only). Conversion is a one-time prep step performed during initial PR construction; runtime never touches JSX.

## Brett figurine integration

**Goal:** keep the base disc + direction arrow exactly as-is; replace only the upper body mesh with a flat figurine sprite that always faces the camera.

### Render path

```js
const ART = await fetch('/art-library/manifest.json').then(r => r.json()).catch(() => null);
const characterIds = ART ? new Set(ART.assets.filter(a => a.kind === 'character').map(a => a.id)) : new Set();
const characterTextures = new Map();   // id → THREE.CanvasTexture

async function loadCharacterTexture(id) {
  if (characterTextures.has(id)) return characterTextures.get(id);
  const meta = ART.assets.find(a => a.id === id);
  const svgText = await fetch('/art-library/' + meta.files.figurine).then(r => r.text());
  const img = await svgToImage(svgText);
  const c = document.createElement('canvas'); c.width = 240; c.height = 400;
  c.getContext('2d').drawImage(img, 0, 0, 240, 400);
  const tex = new THREE.CanvasTexture(c);
  tex.minFilter = THREE.LinearFilter;
  characterTextures.set(id, tex);
  return tex;
}

function buildFigure(type, color) {
  const group = new THREE.Group();

  if (characterIds.has(type)) {
    const sprite = new THREE.Sprite(new THREE.SpriteMaterial({
      map: characterTextures.get(type),
      transparent: true,
      depthTest: true
    }));
    sprite.scale.set(2.0, 3.4, 1);
    sprite.position.y = 1.7;
    group.add(sprite);
  } else {
    /* legacy pawn/triangle/square/diamond meshes — kept verbatim */
  }

  // Existing base disc + direction arrow (unchanged)
  const baseMat = new THREE.MeshStandardMaterial({
    color: hex2col(color).multiplyScalar(0.45), roughness: 0.7
  });
  const base = new THREE.Mesh(new THREE.CylinderGeometry(0.78, 0.78, 0.18, 32), baseMat);
  base.position.y = 0.09; base.castShadow = true;
  group.add(base, makeDirectionArrow(color));

  return group;
}
```

### Token picker UI

- Replace the four `data-type="pawn|triangle|square|diamond"` buttons with `data-type="figure-01|figure-02|figure-03|figure-04"`.
- Each button renders the *figurine* SVG (scaled to fit ~64×96).
- **No text label.** ARIA label uses the manifest `name_de` for screen readers; sighted users see only the figurine.
- The existing color picker still tints the base disc.
- Save / restore wire formats use the new IDs; legacy IDs continue to be accepted on read.

### Boot order

1. Brett init fetches `/art-library/manifest.json`.
2. If manifest available: pre-load all four character textures, swap the picker row to figurines.
3. If manifest missing (404): keep the existing legacy picker; nothing else changes.

## Dashboard admin viewer

### Tab registration

A new `art` entry in `state.tabs` (managed in `dashboard/web/public/app.js`), gated by the same `realm-admin` role check that gates the existing `argocd` tab. Tab order and visibility configurable through the existing settings panel.

### Page layout

Single-page list, grouped by `kind`. Filter chips (kind), tag multiselect, live text filter on `name_de` / `name_en` / `id`. Cards render the asset's primary file (`portrait` for characters, `icon` for props, `swatch` for terrain, `svg` for logos) plus a small two-digit index (`01`, `02`, …).

### Detail panel

Slides in from the right when a card is clicked. Shows:

- The portrait/figurine/icon at full size.
- `id`, `kind`, `tags`.
- Palette swatches (where present): hex chip + copy-to-clipboard button.
- Download links for every entry in `files.*`.

No role text, no bio, no narrative anchor. The detail panel is for designers and admins verifying assets exist and grabbing palettes — names are de-emphasized.

### i18n keys

`tab_art`, `art_search_ph`, `art_kind_all`, `art_kind_character`, `art_kind_prop`, `art_kind_terrain`, `art_kind_logo`, `art_palette`, `art_download`, `art_no_assets`, `art_copied`, `art_tags`. Two locales: `de` (default), `en`. Asset display uses `name_de` / `name_en` per active locale.

### Loading & errors

- Skeleton cards while manifest fetches.
- `/art-library/manifest.json` 404 → render single empty-state ("No art library configured for this environment — see `docs/art-library.md`").
- Manifest schema validation failure on the client → render an error banner with the schema path and value, plus the same empty-state below.

## Cluster scope and deployment

The art set is brand-specific (the K mark, German names, copper-lime palette are Korczewski's voice).

- `art-library/sets/korczewski/` ships in v1.
- Dev (k3d/) overlay bundles the Korczewski set by default for local iteration.
- `prod-korczewski/` overlay generates the ConfigMap from the Korczewski set.
- `prod-mentolder/` overlay does **not** generate the ConfigMap. Brett and the dashboard tab self-degrade as described above.
- Future: a mentolder set could be added under `art-library/sets/mentolder/` and wired into `prod-mentolder/`. Out of scope for v1.

The dashboard admin host is already gated by oauth2-proxy → Keycloak. The `art` tab visibility is gated by the same `realm-admin` role check used by other admin-only tabs.

## Backwards compatibility

- Brett snapshot format accepts both legacy types (`pawn / triangle / square / diamond`) and new types (`figure-01..04`).
- Existing saved sessions render via the legacy code path; users can replace tokens manually if they want characters.
- No DB migration. No forced cutover.

## Testing

| Dashboard render | `/admin/art-library` loads, one card per asset, click → detail panel matches id | Playwright |
| Brett render | click figure-01 button, place token, asserts mesh group has a `THREE.Sprite` child | Playwright |
| Empty state | env without ConfigMap renders empty-state copy in dashboard | Playwright |
| Visual regression | render every asset, manual diff against handoff `Portfolio.html` | manual, on PR |
