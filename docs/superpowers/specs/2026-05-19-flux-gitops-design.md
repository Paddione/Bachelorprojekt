## Summary

Implement Flux CD as a pull-based GitOps operator on both production clusters
(`mentolder` and `korczewski`). Flux replaces the manual `task workspace:deploy`
workflow with automatic reconciliation and adds Image Automation so that image
tag updates are committed back to Git by Flux rather than driven imperatively by
CI.

The existing Kustomize overlay structure (`k3d/` → `prod/` → `prod-mentolder/`
/ `prod-korczewski/`) is kept intact. A new `flux/` top-level directory wraps
it with Flux CRDs. The `task workspace:deploy` command is preserved as an
emergency fallback.

---

## Architecture

### Why pull-based matters here

`mentolder` has 6 home-LAN worker nodes behind NAT (`192.168.100.*`). These
cannot accept inbound connections from GitHub webhooks. Pull-based GitOps
(every cluster's Flux operator reaches outbound to GitHub) eliminates the need
for reverse proxies or tunnels — the NAT asymmetry becomes irrelevant.

## envsubst Migration

### The problem

146 occurrences of `${VAR}` in the prod overlay YAML files. Flux's Kustomize
Controller does not run bash `envsubst` — it uses `postBuild.substituteFrom`
with `$(VAR)` syntax instead.

### The solution

All `${VAR}` placeholders in `prod/`, `prod-mentolder/`, `prod-korczewski/`
are renamed to `$(VAR)`. The affected variables are:

- `$(PROD_DOMAIN)`
- `$(WORKSPACE_NAMESPACE)`
- `$(WEBSITE_NAMESPACE)`
- `$(DEV_DOMAIN)`
- `$(BRAND_NAME)`
- `$(BRAND_ID)`

### Taskfile fallback compatibility

`task workspace:deploy` gets a preprocessing step that converts `$(VAR)` back to
`${VAR}` before piping to envsubst:

```bash
# In the workspace:deploy task, before the kubectl apply -k step:
kustomize build prod-mentolder \
  | sed 's/\$(\([^)]*\))/\${\1}/g' \
  | envsubst "$PROD_DOMAIN $WORKSPACE_NAMESPACE $WEBSITE_NAMESPACE $DEV_DOMAIN $BRAND_NAME $BRAND_ID" \
  | kubectl apply -f -
```

This keeps the Taskfile as a fully functional emergency fallback without
maintaining two sets of manifests.

## Image Automation

### Tag format

All images receive two tags on each CI build:

- `:latest` — retained for compatibility and local dev
- `:sha-YYYYMMDD-HHmmss-<shortsha>` — e.g. `sha-20260519-143025-a1b2c3`

The timestamp prefix makes tags alphabetically sortable by recency.

### Images tracked

| Image | Registry path | Policy name | Cluster |
|---|---|---|---|
| mentolder website | `ghcr.io/paddione/mentolder-website` | `mentolder-website` | mentolder |
| korczewski website | `ghcr.io/paddione/korczewski-website` | `korczewski-website` | korczewski |
| Brett | `ghcr.io/paddione/workspace-brett` | `brett` | both |
| Docs | `ghcr.io/paddione/workspace-docs` | `docs` | both |

### ImageRepository + ImagePolicy (example)

```yaml
# flux/images/mentolder-website.yaml
---
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageRepository
metadata:
  name: mentolder-website
  namespace: flux-system
spec:
  image: ghcr.io/paddione/mentolder-website
  interval: 5m
  secretRef:
    name: ghcr-pull-secret
---
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata:
  name: mentolder-website
  namespace: flux-system
spec:
  imageRepositoryRef:
    name: mentolder-website
  filterTags:
    pattern: '^sha-[0-9]+-[0-9]+-[a-f0-9]+'
  policy:
    alphabetical:
      order: asc
```

### ImageUpdateAutomation

```yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageUpdateAutomation
metadata:
  name: flux-image-updates
  namespace: flux-system
spec:
  interval: 5m
  sourceRef:
    kind: GitRepository
    name: flux-system
  git:
    checkout:
      ref:
        branch: main
    commit:
      author:
        name: flux-bot
        email: flux@mentolder.de
      messageTemplate: 'chore(flux): update {{range .Updated.Images}}{{.}} {{end}}[skip ci]'
    push:
      branch: main
  update:
    strategy: Setters
    path: ./k3d
```

### Manifest marker

Flux uses inline YAML comments to locate which line to update:

```yaml
# k3d/website.yaml
image: ghcr.io/paddione/mentolder-website:sha-20260519-143025-a1b2c3 # {"$imagepolicy": "flux-system:mentolder-website"}
```

Flux replaces the tag portion on every new policy match and commits to `main`.

## Bootstrap Process

Run once per cluster after the Sealed Secrets controller and cert-manager are
already installed (they must exist before Flux reconciles the workspace overlay).

```bash
# Install flux CLI locally (once)
brew install fluxcd/tap/flux  # or curl -s https://fluxcd.io/install.sh | sudo bash

# Bootstrap mentolder
flux bootstrap github \
  --context=mentolder \
  --owner=Paddione \
  --repository=Bachelorprojekt \
  --branch=main \
  --path=flux/clusters/mentolder \
  --personal \
  --components-extra=image-reflector-controller,image-automation-controller

# Bootstrap korczewski
flux bootstrap github \
  --context=korczewski \
  --owner=Paddione \
  --repository=Bachelorprojekt \
  --branch=main \
  --path=flux/clusters/korczewski \
  --personal \
  --components-extra=image-reflector-controller,image-automation-controller
```

`flux bootstrap` creates a GitHub deploy key with write access (needed for
Image Automation commits), installs Flux system pods in `flux-system`, and
pushes the initial `flux-system/` manifests to this repo.

### GHCR pull secret (create after bootstrap)

```bash
flux create secret oci ghcr-pull-secret \
  --url=ghcr.io \
  --username=Paddione \
  --password=<ghcr-readonly-pat> \
  --context=mentolder

flux create secret oci ghcr-pull-secret \
  --url=ghcr.io \
  --username=Paddione \
  --password=<ghcr-readonly-pat> \
  --context=korczewski
```

Store the PAT as a SealedSecret so it survives cluster resets.

## SealedSecrets Compatibility

No conflict. Flux applies `SealedSecret` CRD objects from Git. The Sealed
Secrets controller creates the decrypted `Secret` objects with an owner
reference to the `SealedSecret`. Since Flux never created those plain Secrets,
`prune: true` will not delete them.

Existing SealedSecrets in `environments/sealed-secrets/` are applied by
`task workspace:deploy` during initial cluster bring-up, before Flux is
bootstrapped. After bootstrap, Flux reconciles them on every sync.

## End-to-End Flow Summary

### Scenario A — Application code change

```
git push → main (website/**)
  → GitHub Actions: docker build + push :sha-YYYYMMDD-HHmmss-abc + :latest
  → Flux Image Reflector polls ghcr.io (every 5m)
  → New tag detected → ImagePolicy selects it
  → Flux Image Automation: commits updated tag to k3d/website.yaml [skip ci]
  → Flux Source Controller: detects new commit
  → Flux Kustomize Controller (mentolder + korczewski): applies overlay
  ✓ Both clusters live ~10 minutes after push
```

### Scenario B — Manifest change (Ingress, ConfigMap, resource limits)

```
git push → main (k3d/** or prod-*/**)
  → No CI build needed
  → Flux Source Controller detects commit (every 1m)
  → Flux Kustomize Controller applies change
  ✓ Both clusters live ~6 minutes after push
```

### Scenario C — Drift correction

```
kubectl edit deployment/keycloak -n workspace  (manual change)
  → Flux Kustomize Controller detects drift at next sync interval
  → Reverts to Git state automatically
  ✓ Cluster is Git-conformant again, no manual intervention needed
```

- Multi-tenancy / RBAC within Flux — single-tenant deployment is sufficient
- Flux Helm Controller — no Helm releases in this repo, not needed
- Progressive delivery (Flagger/canary) — not required for this stack
