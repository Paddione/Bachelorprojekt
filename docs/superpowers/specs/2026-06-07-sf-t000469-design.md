# Design: Brett — Notizen & Statements pro Figur (Slice 5)

**Ticket:** T000469
**Spec File:** `docs/superpowers/specs/2026-06-07-sf-t000469-design.md`
**Timestamp:** 2026-06-07
**Branch (suggested):** `feature/brett-figure-notes`
**Brand/target:** `brett.mentolder.de` (coaching mode)
**Complexity:** medium → adversarial review section included.

---

## Goal

Give each figure on the constellation board a small **text payload** so a coach (or
permitted participant) can capture what a figure *says* or *represents* during a
session:

- **Note** — a private/working annotation about the figure (free text, e.g. "fühlt
  sich ausgeschlossen").
- **Statement** — the figure's spoken sentence in the constellation ("Ich stehe hier
  ganz alleine.").

Both are per-figure, edited in the existing **figure panel** (`fig-panel.ts`),
synced live to every client, and **persisted** across reload / last-leave-and-rejoin
exactly like `label` and `appearance` already are. This is "Slice 5" — it builds
directly on the coaching-product session layer (Slices 1–4: presence, locks, roles,
phases, ownership) and adds no new transport, no new gate, no new schema table.

Out of scope: rich text / markdown, multi-line history/threads, per-note authorship
trails, export to PDF, AI summarisation. (Called out explicitly so the slice stays
small; see Decisions D4/D5.)

---

## Architectural Decision

**Notes and statements are two new optional string fields on the `Figure` type
(`note?`, `statement?`), carried over the *existing* `update` mutation
(`{ type:'update', id, changes:{ note } | { statement } }`) — the identical path
`label` already uses.** No new message type, no new `RELAY_TYPES` entry, no new
`canMutate` branch, no new persistence sentinel.

This is the right call because the codebase has already paid the full cost of a
generic, server-authoritative, gated, relayed, persisted per-figure field — that is
exactly what `label` is. `applyMutation`'s `update` case already does a shallow merge
of `msg.changes` into the stored figure (stripping `id`/`ownerId`), `buildStateFromMutations`
already round-trips arbitrary figure fields, `seedFigureMapFromState` already re-seeds
the whole figure object, the `update` type is already in `RELAY_TYPES` and already in
the `canMutate` allow-list with the correct leiter / stellvertreter-owns-figure /
beobachter-deny matrix. Adding `note`/`statement` as plain fields inherits *all* of
that for free and is impossible to get wrong at the transport/gate layer because we
are not touching the transport or the gate. The only net-new code is (a) two type
fields, (b) two text inputs in the panel + their `input` handlers, (c) snapshot/add
re-render of the text into the panel, and (d) a `figure-note.test.ts` proving the
round-trip — mirroring `figure-label.test.ts` one-for-one.

## Architecture

### Data model (`brett/src/types/state.ts`)

Two new optional fields on `interface Figure`, alongside `label`:

```ts
export interface Figure {
  // …existing…
  label?: string;
  /** Working annotation about the figure (coaching note). Free text, ≤ 500 chars. */
  note?: string;
  /** The figure's spoken sentence in the constellation. Free text, ≤ 280 chars. */
  statement?: string;
  // …existing…
}
```

`messages.ts` needs **no** change: the `update` message already types `changes` as
`Partial<Figure> & { appearance? }`, so `{ note }` / `{ statement }` are already
valid payloads once the `Figure` fields exist.

### Server (`brett/src/server/figures.ts`)

**No code change required** for the happy path — `applyMutation`'s existing `update`
case shallow-merges `safeChanges` (after stripping `id`/`ownerId`) into the stored
figure, so `note`/`statement` persist automatically; `add` already spreads the whole
`safeFigData`, so a figure created with a note keeps it. `buildStateFromMutations`
and `seedFigureMapFromState` emit/re-seed the full figure object, so the round-trip
is covered.

**One hardening addition (server-authoritative validation):** add a length clamp in
the `update`/`add` path so a malicious/oversized payload can't bloat room state.
Implemented as a small pure helper `clampFigureText(changes)` called inside the
`update` and `add` cases (truncate `note` to 500, `statement` to 280, coerce
non-string → drop the field). This mirrors the spirit of the existing
appearance-validation and 200-figure-cap guards. Keep it in `figures.ts` so it is
unit-testable without a socket.

### Gate (`brett/src/server/permissions.ts`, `ws-handler.ts`)

**No change.** `note`/`statement` are part of an `update`, so `canMutate` already
enforces:
- **leiter** → may set note/statement on any figure,
- **stellvertreter** → only on a figure they own (`figureOwnerId === playerId`),
- **beobachter / anon** → denied (write rejected, no broadcast),
- **legacy free-board** (no session code, no roles) → permitted (REG-1 bypass), same
  as label today.

This is a deliberate inheritance, not an oversight — see Adversarial Challenge 1.

### Relay / sync (`ws-handler.ts`)

**No change.** `update` is already in `RELAY_TYPES`; the gated relay broadcasts the
`update` to all room clients, who apply it in `ws-client.ts`.

### Client state apply (`brett/src/client/ws-client.ts`)

The runtime `STATE.figures[*]` are THREE objects with metadata, not the raw server
figure. Today the `update` case applies `preset` / `boneOverrides` / `appearance`
but **ignores unknown fields**. Add:

```ts
// inside case 'update':
if (typeof c.note === 'string') fig.note = c.note;
if (typeof c.statement === 'string') fig.statement = c.statement;
// if this figure is the open panel's selection, refresh the inputs live
if (STATE.selectedId === fig.id) syncPanelToSelection(fig.id);
```

And in the `snapshot` and `add` cases, copy `note`/`statement` onto the local fig
(mirroring how `_serverPossessor` is copied) so a late-joiner / reload sees them.

### UI (`brett/public/index.html`, `brett/src/client/ui/fig-panel.ts`)

- **`index.html`** — add two controls to the existing figure panel, directly under
  the `#fig-label-input` row:
  - `<textarea id="fig-statement-input">` ("Statement der Figur")
  - `<textarea id="fig-note-input">` ("Notiz")
  Styled with the existing panel CSS; `maxlength` set to 280 / 500 to mirror the
  server clamp (defence-in-depth, not the source of truth).
- **`fig-panel.ts`**:
  - `syncPanelToSelection(id)` — also set `statementInput.value = fig.statement || ''`
    and `noteInput.value = fig.note || ''` (and clear them in the "NEUE FIGUR" branch).
  - `initFigPanel()` — add two `input` listeners mirroring the existing label
    listener: on input, write `fig.note`/`fig.statement` and call
    `sendUpdate(fig, { note })` / `sendUpdate(fig, { statement })`.
  - Debounce the `sendUpdate` (e.g. 250 ms trailing) so a long typed statement does
    not emit one mutation per keystroke. (`label` is un-debounced today; a textarea
    warrants it. Keep the debounce client-side only.)

### Tests (`brett/test/figure-note.test.ts`)

Mirror `figure-label.test.ts` (node:test, no socket):
1. `note` rides along on `add` and persists through `buildStateFromMutations`.
2. `note` updates via `update.changes`; `statement` likewise; setting one does not
   clobber the other or `label`/`appearance` (shallow-merge proof).
3. `clampFigureText` truncates over-length note/statement and drops non-string.
4. (gate) a `beobachter` `update` carrying `note` is denied by `canMutate`; a
   `stellvertreter` may set it only on an owned figure — reuse the `assign-role` /
   `figure-owner` test fixtures.
5. (round-trip) `seedFigureMapFromState` re-seeds a figure's `note`/`statement` after
   a simulated DB round-trip.

## Adversarial Review

**Reviewer stance:** "Try to refute this design." Treat reuse-of-`update` as guilty
until proven innocent — the repo's Default-Deny gate and persistence round-trip are
exactly where this class of feature has silently broken before.

**Verdict:** `approved` (with two required guards folded into the design above:
server-side `clampFigureText` and a `figure-note` *gate* test, not just a round-trip
test).

### Challenge 1: "Reusing `update` silently lets the wrong people write notes."
- **Claim**: Riding `update` inherits the correct permission matrix for free.
- **Counter-argument**: `update` is a coarse type. A `stellvertreter` may legitimately
  `move`/recolor a figure they own — but is it intended that they can also overwrite
  the **coach's** note/statement on that same figure? If a participant owns the figure
  representing them, they could rewrite the coach's working annotation.
- **Resolution**: For Slice 5 this is **acceptable and intended**: a statement is the
  figure's *own* spoken sentence, so the person embodying/owning the figure authoring
  it is correct; the coach (leiter) can always override. The note is a shared working
  field, not a private coach-only field — consistent with the slice's "no per-field
  ACL, no authorship" scope (D4/D5). If a coach-private note is needed later, that is a
  *new* `figure_note` type with its own gate branch — explicitly deferred. The gate
  test (test 4) pins the current behaviour so any future tightening is a conscious change.

### Challenge 2: "Unbounded free text bloats room state and the DB blob."
- **Claim**: Notes are short.
- **Counter-argument**: Nothing stops a client sending a 5 MB string in
  `changes.note`. Room state is held in memory and persisted as a JSON blob via
  `buildStateFromMutations`; 200 figures × multi-KB notes is real memory + a slow
  snapshot broadcast, and there is **no** existing length guard on `label` either —
  so "label does it" is not a safety argument.
- **Resolution**: Added server-authoritative `clampFigureText` (note ≤ 500,
  statement ≤ 280, non-string dropped) in the `add`/`update` path — the client
  `maxlength` is only defence-in-depth. This is the one piece of net-new *server*
  logic the slice carries, and it is pure/unit-tested. (Optionally backfill the same
  clamp onto `label`; out of scope but noted.)

### Challenge 3: "The open figure panel won't refresh when a *remote* edit arrives."
- **Claim**: `ws-client.ts` applying `c.note`/`c.statement` is enough.
- **Counter-argument**: If coach A has figure X selected with the note panel open and
  participant B edits X's statement, A's panel `<textarea>` still shows the stale
  value — the local `fig.statement` updates but the DOM input does not.
- **Resolution**: The design calls `syncPanelToSelection(fig.id)` from the `update`
  apply path **only when `STATE.selectedId === fig.id`**, refreshing the inputs. Edge
  case: if A is *mid-typing* into the same field when the remote edit lands,
  re-syncing would clobber A's cursor — so the refresh must **skip the focused
  element** (`if (document.activeElement !== statementInput) statementInput.value = …`).
  This is folded into `syncPanelToSelection`. Last-write-wins is accepted (D4).

### Challenge 4: "A new field that isn't in RELAY/persist will be silently dropped — this repo has that exact footgun."
- **Claim**: No transport/persist change is needed.
- **Counter-argument**: The CLAUDE.md gotchas and the `figures.ts` comments document
  multiple cases where a field/sentinel was *dropped on DB round-trip* because a
  `seedFigureMapFromState` branch was missing (optik PD-1, lobby_settings B3). Is
  `note`/`statement` the same trap?
- **Resolution**: No — those were **sentinel** objects (`__optik__`, `__roles__`)
  stored under synthetic keys that need explicit re-seed branches. `note`/`statement`
  are **plain fields on a real figure object**, and `seedFigureMapFromState` re-seeds
  the *entire* figure (`map.set(fig.id, fig)`), so they round-trip with the figure
  automatically. Test 5 (round-trip) and test 1 (add-persist) prove this rather than
  assume it — which is the correct response to a known footgun: verify, don't trust.

### Challenge 5: "Per-keystroke `sendUpdate` from a textarea floods the relay."
- **Claim**: Reuse label's input handler.
- **Counter-argument**: `label` is a single-line input; a multi-line statement typed
  at speed could emit dozens of `update` mutations/second, each gated + relayed +
  merged + (eventually) persisted across the room.
- **Resolution**: Client-side 250 ms trailing debounce on the note/statement senders
  (design §UI). Server stays stateless about debouncing; last value wins. This is a
  pure client optimisation with no protocol impact.

## Migration / Rollback Plan

- **Forward**:
  1. Land the TS changes (types, `figures.ts` clamp, `ws-client.ts` apply, `fig-panel.ts`
     UI) + `index.html` markup + `figure-note.test.ts`.
  2. `npm test` in `brett/` (node:test) green, including the new gate + round-trip tests.
  3. brett typecheck gate (CI, live since #1375) must pass — the two new `Figure`
     fields and DOM handles must be correctly typed.
  4. Build + import/push the brett `:latest` image and roll out via `task feature:brett`
     (per CLAUDE.md: brett uses `:latest` intentionally, rebuilt each release).
- **Backward**: Fully additive and forward/backward compatible.
  - Old clients ignore `note`/`statement` in snapshots/updates (unknown fields are
    dropped); new server tolerates their absence (fields are optional).
  - Rollback = redeploy the previous brett image. Persisted notes remain in the room
    blob harmlessly (an old server just won't render or clamp them). No schema
    migration, no destructive change, so no data-loss risk on either direction.
  - No DB DDL: notes live inside the existing room-state JSON blob, so there is
    nothing to migrate or revert at the database layer.
