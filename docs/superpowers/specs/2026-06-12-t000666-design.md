# Spec: CI API-Auth-Coverage-Gate (T000666)

**Ticket:** T000666  
**Branch:** feature/t000666  
**Date:** 2026-06-12  
**Status:** Draft

---

## Problem

`docs/generated/api-surface.md` classifies ~373 website API endpoints by auth level (admin/session/public), but this classification is documentation-only — no CI gate enforces it. PR #T000660 demonstrated that unprotected endpoints (Brett) can slip through unnoticed. The current generator (`scripts/build-api-map.mjs`) defaults to `'public'` for any endpoint that lacks recognisable auth patterns, silently treating misclassified or genuinely unprotected routes the same way.

Three failure modes that the gate must close:
1. **New endpoint, no auth, not allowlisted** — developer adds a route without auth; CI is silent.
2. **Regression: auth weakened** — admin/session route loses its auth guard during refactor; classification drops to public/unclassified; CI is silent.
3. **Unclassified by generator** — internal or cron-secret endpoints whose patterns the generator does not yet recognise; classified as public even though they are protected.

---

## Auth Categories

The gate recognises five auth levels. Everything else is `unclassified`.

| Level | Pattern (code heuristics) | Examples |
|-------|--------------------------|---------|
| `admin` | `isAdmin(`, `requireAdmin`, `role.*admin` | `/api/admin/**` |
| `session` | `getSession(`, `requireAuth`, `session = await`, `verifyToken`, `checkAuth` | `/api/portal/**`, `/api/arena/**` |
| `internal` | `INTERNAL_API_TOKEN`, `x-internal-token`, `/api/internal/` path prefix | `/api/internal/tickets/notify-close` |
| `cron` | `CRON_SECRET`, `Bearer ${CRON_SECRET}` | `/api/cron/scheduled-publish` |
| `public` | No auth guard detected | `/api/health`, `/api/contact`, etc. |
| `unclassified` | Generator could not determine auth (fallthrough, ambiguous pattern) | (should not exist after hardening) |

**Key rule:** `unclassified` always breaks the gate. `public` breaks the gate unless the endpoint appears in the allowlist with a `reason`.

## Allowlist File

`website/api-public-allowlist.json` — committed to the repo, reviewable in PRs.

Schema per entry:

```json
{
  "path": "/api/contact",
  "methods": ["POST"],
  "reason": "Public contact form — no auth required by design."
}
```

Fields:
- `path` (required): exact URL path as in `api-map.json`, e.g. `/api/health`
- `methods` (required): array of HTTP methods covered
- `reason` (required): free-text explanation; must be non-empty (enforced)

Baseline entries use: `"reason": "baseline — not individually audited"`.  
Exceptions confirmed as protected use their correct auth category instead (see Findings below).

## Generator Hardening (build-api-map.mjs)

The generator is extended to recognise `internal` and `cron` patterns before the `public` fallthrough. The new fallthrough produces `unclassified` instead of `public`, so truly ambiguous files are visible as gate failures.

Detection order (first match wins):

1. `admin` — existing patterns
2. `session` — existing patterns (rename `auth` → `session` in output)
3. `internal` — `INTERNAL_API_TOKEN` OR `x-internal-token` OR path prefix `/api/internal/`
4. `cron` — `CRON_SECRET` OR `Bearer ${CRON_SECRET}`
5. `public` — explicit annotation `@auth public` OR matches the allowlist (generator does not consult the allowlist at build time; gate does)
6. `unclassified` — none of the above

> Note: The generator **does not consult the allowlist**. The gate script reads both `api-map.json` and the allowlist at check time.

Endpoints that appeared as `public` in the pre-existing `api-map.json` but are actually protected:

## Gate Behaviour

### Script: `scripts/api-auth-check.mjs`

- Reads `docs/generated/api-map.json`
- Reads `website/api-public-allowlist.json`
- Loads `ALLOWLIST_PATH` env override (optional, for testing)
- For each endpoint:
  - `unclassified` → always fail
  - `public` and NOT in allowlist → fail (coverage gap)
  - `public` and in allowlist → pass (explicit approval)
  - `admin` / `session` / `internal` / `cron` → pass
- Regression mode: if an endpoint was previously `admin` or `session` (looked up from `api-map.json` on `origin/main`) and is now `public` or `unclassified`, fail — unless the allowlist was updated in the same commit

### Exit codes

| Exit | Meaning |
|------|---------|
| 0 | All endpoints classified; all `public` endpoints allowlisted |
| 1 | One or more `unclassified` endpoints, or `public` without allowlist entry |

### CI integration

New task `test:api-auth` in `Taskfile.yml`:
```yaml
test:api-auth:
  desc: "API auth coverage gate — every endpoint must be classified; public endpoints need an allowlist entry"
  cmds:
    - node scripts/api-auth-check.mjs
```

Added as a dependency of `test:all`.

BATS wrapper `tests/unit/api-auth-gate.bats` — thin offline test that exercises the script against fixture files (no live cluster required).

---

## Regression Gate

In CI (`.github/workflows/ci.yml`) the gate additionally runs a regression check:

1. Dump `api-map.json` from `origin/main` into a temp file
2. Run `api-auth-check.mjs --regression --main-map <path>` against the PR's current `api-map.json`
3. Fail if any endpoint weakened its auth classification without a matching allowlist update

This is handled inside `api-auth-check.mjs` via `--regression` flag; the BATS tests cover the regression path with fixtures.

## Non-Goals

- Not a runtime firewall — the gate is a static analysis check only
- Not a full security audit — individual endpoint code review is out of scope (handled by follow-up tickets)
- Does not replace existing session/admin middleware — purely additive CI gate
