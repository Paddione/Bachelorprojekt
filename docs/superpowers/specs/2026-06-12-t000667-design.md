# Spec: Knowledge-Graph v2 — Ingress/Secret/ConfigMap-Kanten + Blast-Radius-Report

**Ticket:** T000667  
**Datum:** 2026-06-12  
**Status:** draft  
**Branch:** feature/t000667

---

## Problemstellung

`scripts/build-graph.mjs` extrahiert bisher nur `env`- und `command`-Kanten. Resultat: 14 von 72 Knoten sind isoliert (`mcp-browser`, `mcp-github`, `sish`, `docs`, `whisper`, `sealed-secrets-controller`, `coredns`, `livekit-redis`, `nextcloud-redis`, `shared-db-dev`, `shared-db-staging`, `einvoice-sidecar`, `monitoring-kube-state-metrics`, `monitoring-operator`), obwohl sie über Ingress/IngressRoute-Ketten, Secret-Mounts und Service-Selectoren real verbunden sind.

Außerdem fehlt ein operativ nutzbarer Blast-Radius-Report: Welche Services brechen, wenn `shared-db` ausfällt? Diese Information existiert im Graphen implizit, ist aber nirgends sichtbar.

---

## Ziele

1. **Extractor erweitern** — vier neue Kantentypen: `ingress`, `secret`, `selector`, `configmap` (letzterer bereits partiell vorhanden, aber ohne `kind`-Feld).
2. **Kanten-Schema** — additives `kind`-Feld (`env|command|ingress|secret|configmap|selector|annotation|initContainer`) auf allen Kanten. Bestehende `{from, to, via}` bleiben unverändert.
3. **Blast-Radius-Report** — `docs/generated/blast-radius.md`: pro Service die transitiven Abhängigen (downstream) plus In-Degree (upstream).
4. **Freshness-Integration** — Report in `freshness:regenerate` + CI-Drift-Check aufnehmen.

## Kantentypen (Design-Entscheidungen)

### 1. Ingress-Kette: Traefik → Service → Workload

**Vanilla `Ingress`** (bereits partiell implementiert, aber ohne `kind`-Feld):
```yaml
spec.rules[].http.paths[].backend.service.name → Kante (traefik → <serviceName>, kind: ingress)
```

**Traefik `IngressRoute`** (CRD, `traefik.io/v1alpha1`):
```yaml
spec.routes[].services[].name → Kante (traefik → <serviceName>, kind: ingress)
```

**Service-Selector → Workload** (neue Kette):
- Alle `Service`-Objekte sammeln: `spec.selector` → label-map.
- Alle Workload-Pods prüfen: wenn `template.metadata.labels` die Selector-Labels enthält → Kante (`<serviceName>` → `<workloadName>`, kind: `selector`).
- Damit entsteht die vollständige Kette: `traefik → <svc> → <deployment>`.

### 2. Secret-Kanten (Shared-Secret-Pattern)

**Entscheidung:** Secrets sind KEINE Knoten. Kante nur wenn ≥ 2 Workloads dasselbe Secret referenzieren (`via: secret:<name>`, `kind: secret`).

**Quellen für Secret-Referenzen in einem Pod:**
- `spec.volumes[].secret.secretName`
- `spec.containers[].env[].valueFrom.secretKeyRef.name`
- `spec.containers[].envFrom[].secretRef.name`
- `spec.imagePullSecrets[].name` (ignorieren — nur Registry-Credentials)

**Algorithmus:**
1. Pass 1: Für jede Workload alle referenzierten Secret-Namen sammeln → `Map<secretName, Set<workloadId>>`.
2. Pass 2: Für jedes Secret mit ≥ 2 Workloads: alle Workload-Paare als Kanten ausgeben (`workloadA → workloadB`, `via: secret:<name>`, `kind: secret`).
3. Symmetrisch (A→B und B→A), dedupliziert über den bestehenden `edgeSet`.

### 3. ConfigMap-Kanten (bereits vorhanden, `kind` fehlt)

Bestehende Logik bleibt. `kind: configmap` wird additiv hinzugefügt. `via` bleibt `configmap:<name>`.

### 4. Kanten-Schema: `kind`-Feld

Alle Kanten erhalten ein `kind`-Feld:

| `kind`        | `via`-Muster              | Beschreibung |
|---------------|---------------------------|--------------|
| `env`         | `env`                     | ENV-Variable referenziert Service-Name |
| `command`     | `command`                 | Container-CMD/Args referenziert Service |
| `initContainer` | `initContainer:wait`    | initContainer wartet auf Service |
| `annotation`  | `annotation`              | `graph.bachelorprojekt.de/depends-on` |
| `ingress`     | `ingress`                 | Traefik-Route (Ingress oder IngressRoute) |
| `selector`    | `selector`                | Service.spec.selector matcht Workload-Labels |
| `secret`      | `secret:<secretName>`     | Shared-Secret (≥ 2 Workloads) |
| `configmap`   | `configmap:<cmName>`      | ConfigMap-Ref mit Service-Hostname-Wert |

**Abwärtskompatibilität:** `{from, to, via}` werden nicht verändert. `kind` ist ein neues Feld. Bestehende Konsumenten von `{from, to, via}` (z.B. `build-graph-docs.mjs`, `k3d/docs-content-built/architecture/`) brechen nicht.

**Pre-Implementation-Check:** `grep -r '"from"\|\.from\b\|\.to\b\|\.via\b' scripts/build-graph-docs.mjs` ausführen und alle Konsumenten prüfen bevor `kind` hinzugefügt wird.

## Blast-Radius-Report

**Output:** `docs/generated/blast-radius.md`

**Methode:** Transitive Closure (BFS/DFS) auf dem Graphen, **downstream** = wer hängt von X ab (wer ruft X als `to` auf).

**Algorithmus:**
1. Baue Reverse-Adjazenzliste: für jeden Knoten, wer referenziert ihn als `to`?
2. BFS vom Startknoten: finde alle direkten + transitiven Abhängigen.
3. Sortiere Services nach Anzahl transitiver Abhängiger (absteigend) = Blast-Radius-Ranking.

**Report-Format:**
```markdown
# Blast-Radius-Report

Generated: <ISO-Timestamp>

## Ranking (nach transitivem Blast-Radius)

| Rank | Service | Direkte Abhängige | Transitive Abhängige | In-Degree (upstream) |
|------|---------|-------------------|---------------------|---------------------|
| 1    | shared-db | 8 | 23 | 0 |
| 2    | keycloak  | 5 | 18 | 1 |
...

## Details pro Service

### shared-db
**Direkte Abhängige:** nextcloud, tracking, docuseal, ...
**Transitive Abhängige (vollständig):** nextcloud, tracking, ...
**Upstream (wer stellt shared-db bereit):** — (kein Upstream)
```

**Einschränkungen:**
- Nur Knotentypen `Deployment`, `StatefulSet`, `CronJob` im Report (Service-Pseudo-Knoten wie `traefik` nur wenn sie Workload-Typ haben).
- Zirkuläre Referenzen (Kanten A→B + B→A) werden erkannt und protokolliert, aber nicht als Fehler behandelt (Cycle-Marker im Report).

## Freshness-Integration

### `freshness:regenerate` (Taskfile.yml)
`blast-radius.md` wird nach `graph:build` generiert. Neuer Sub-Task `graph:blast-radius`:
```yaml
graph:blast-radius:
  desc: "Generiere Blast-Radius-Report nach docs/generated/blast-radius.md"
  cmds:
    - node scripts/build-blast-radius.mjs
```

`graph:build` ruft `graph:blast-radius` auf (oder inline in `build-graph.mjs`).

### `freshness:check` / `freshness:graph-check`
`docs/generated/blast-radius.md` in den Staleness-Check aufnehmen (wie `api-surface.md`):
- Text-Diff (kein `jq`, da Markdown).
- `generatedAt`-Zeile vor Diff herausfiltern (sed/grep).

**PR-#1501-Muster beachten:** Regenerate + Commit im selben Schritt (nicht zwei getrennte Commits). In `freshness:graph-check` das neue File erst generieren, dann diff-checken.

## Nicht in Scope

- Secret-Inhalte lesen oder entschlüsseln.
- SealedSecret als eigene Knoten (nur wenn als `secretRef` referenziert und ≥ 2 Consumer).
- Laufzeitdaten (kein `kubectl` in Build-Scripts).
- Kanten zwischen Namespaces über NetworkPolicy-Regeln.
- Blast-Radius-Report als Website-Seite (nur Markdown, kein HTML).
