# Spec: Admin Live-Architektur-Graph mit Pod-Status-Overlay (T000668)

**Datum:** 2026-06-12
**Ticket:** T000668
**Branch:** feature/t000668
**Status:** staged
**Depends-on:** T000667 (Graph v2, optional — funktioniert mit heutigem graph.json)

---

## Problemstellung

`docs/generated/graph.json` (72 Knoten, 128 Kanten) beschreibt die vollständige
Architektur der Plattform — Deployments, StatefulSets, CronJobs, Services, Namespaces
und ihre Abhängigkeiten. Die Admin-Cluster-APIs (`/api/admin/cluster/pods-list`,
`/api/admin/cluster/warnings`, `/api/admin/cluster/logs`) liefern Live-Pod-Daten.

Es gibt bislang **keine Ansicht, die beides verbindet**. Der Administrator sieht entweder
die statische Architektur (im Docs-System) oder den Live-Cluster-Status (im Platform
Hub / Logs-Tab) — nie beides gleichzeitig. Fehler wie ein CrashLoop in einem abhängigen
Service sind ohne Kontextualisierung im Graphen schwer zu lokalisieren.

---

## Lösung: `/admin/architektur` — Live-Architektur-Graph

Eine neue Admin-Seite (`/admin/architektur`), die:

1. Den Architektur-Graphen interaktiv als SVG mit D3-Force-Layout rendert
2. Per Live-Polling (10 s, pausiert bei verborgenem Tab) den Pod-Status überlagert
3. Namespace-Gruppierung mit deterministisch geseedetem Layout
4. Klick auf einen Knoten öffnet ein Detail-Panel (Pod-Status, Warnings, Logs-Link)
5. BRAND-aware: Namespace-Platzhalter werden serverseitig aufgelöst

---

## Architektur

### Neuer API-Endpoint: `/api/admin/cluster/graph`

- Admin-geschützt (Keycloak Session, `isAdmin()`)
- Liest `docs/generated/graph.json` via `fs.readFile` zur Request-Zeit
- Löst Namespace-Platzhalter serverseitig auf:
  - `${WEBSITE_NAMESPACE}` → `website` (mentolder) oder `website-korczewski` (korczewski)
  - `${WORKSPACE_NAMESPACE}` → `workspace` (mentolder) oder `workspace-korczewski` (korczewski)
  - `$STAGING_NS` → `workspace-dev` (beide Brands, kein Alias)
- Gibt das aufgelöste `{nodes, edges, generatedAt}` als JSON zurück
- Factory-observable und skriptbar (kein Build-Time-Bundling)

### Frontend: Astro-Seite + Svelte-Komponente

```
website/src/pages/admin/architektur.astro          ← Astro-Seite, SSR, Admin-Guard
website/src/components/admin/ArchitekturGraph.svelte ← Haupt-Komponente
website/src/components/admin/graph/
  ├── GraphCanvas.svelte                            ← D3-Force-SVG-Canvas
  ├── GraphNode.svelte                              ← einzelner Knoten mit Status-Ring
  ├── GraphLegend.svelte                            ← Farb-Legende
  └── NodeDetailPanel.svelte                        ← Klick-Detail-Panel
```

### Sidebar-Eintrag

Neuer Eintrag in `AdminLayout.astro` unter Gruppe "Kontrollzentrum":

## Rendering: D3-Force + SVG in Svelte

### Layout

- **d3-force** mit deterministisch geseedetem Zufallsgenerator (`d3.randomLcg(42)`)
- Namespace-Gruppierung: Knoten einer Namespace-Gruppe werden durch eine zusätzliche
  Cluster-Force (`d3.forceCollide` + benutzerdefinierter Centering-Force) zusammengehalten
- Namespace-Hintergrund-Rechtecke als SVG `<rect>`-Elemente (leicht transparenter Hintergrund)
- Kanten als `<line>`-Elemente; Kantenlabels (kind) als optionale `<text>`-Elemente
  (sichtbar ab Zoom-Level > 1.2 — Folgeschritt aus T000667, nicht hart verkoppelt)
- Pan & Zoom via `d3.zoom`

### Status-Overlay (Ring um jeden Knoten)

| Status | Farbe | Bedingung |
|--------|-------|-----------|
| Grün | `#22c55e` | Alle Pods ready, 0 Restarts in letzter Minute |
| Gelb | `#eab308` | Degraded: restarts > 0 oder ready < desired |
| Rot | `#ef4444` | Kein Pod ready, CrashLoop, oder Phase Failed |
| Grau | `#6b7280` | CronJob / kein Matching (kein Running-Pod erwartet) |
| Unbekannt | `#374151` (Umriss) | Kein Match via Label oder Präfix gefunden |

### Pod → Node-Matching

Priorität:
1. Label `app=<node.id>` oder `app=<node.name>` am Pod
2. Präfix-Match: Pod-Name beginnt mit `<node.name>-`
3. Kein Match → Knoten erhält "Unbekannt"-Ring; nicht matchende Pods erscheinen
   in einem Hinweis-Panel "Unzugeordnete Pods" (Anzahl + Namen)

### Detail-Panel (rechts, 320 px)

Öffnet sich bei Klick auf einen Knoten. Enthält:
- Knotenname, Typ (Deployment / CronJob / …), Namespace
- Pod-Liste mit Status-Ampel, Restart-Zähler, Phase
- Letzte Warnings (gefiltert nach `involvedObject.name` enthält Pod-Namen)
- Schaltfläche "Logs anzeigen" → `/admin/platform?tab=logs&pod=<pod-name>&ns=<ns>`

## BRAND-aware Namespace-Auflösung

Die Funktion `resolveNamespace(placeholder, brand)` im neuen API-Endpoint:

| Platzhalter | mentolder | korczewski |
|-------------|-----------|------------|
| `${WEBSITE_NAMESPACE}` | `website` | `website-korczewski` |
| `${WORKSPACE_NAMESPACE}` | `workspace` | `workspace-korczewski` |
| `$STAGING_NS` | `workspace-dev` | `workspace-dev` |

Brand wird via `process.env.BRAND_ID ?? process.env.BRAND ?? 'mentolder'` bestimmt.
Der graph-Endpoint gibt im Response-Body zusätzlich `resolvedBrand` zurück, damit
der Client weiß, welche Namespaces zu pollen sind.

---

## Pods-List-Polling: Multi-Namespace

Der Client pollt alle relevanten Namespaces (Rückgabe aus graph-API) sequenziell oder
via `Promise.all`. Die bekannten Namespaces nach Auflösung:

- `workspace` / `workspace-korczewski` (je nach Brand)
- `website` / `website-korczewski`
- `workspace-office` (Collabora/CoTURN — read-only, kein Write)
- `coturn`
- `kube-system` (Traefik)
- `sealed-secrets`
- `monitoring`
- `workspace-dev` (Staging, wenn vorhanden)

Namespaces, die die K8s-API mit 403 ablehnen, werden still übersprungen (Namespace außer
Reichweite des Service-Account).

---

## Polling-Strategie

- `setInterval` mit 10 000 ms
- Pause wenn `document.visibilityState === 'hidden'` (via `visibilitychange`-Event)
- Automatisches Resume bei `visibilitychange` → visible
- Loading-Indicator: Spinner-Ring um das letzte aktualisierte Zeitstempel-Label

---

## Fehlerbehandlung

- Graph-API nicht erreichbar → leere Leinwand + Fehlermeldung "Graph konnte nicht geladen werden"
- Pods-API nicht erreichbar → Graph rendert ohne Overlay, gelber Hinweis "Live-Status nicht verfügbar"
- Service-Account fehlt → 503 von pods-list wird angezeigt (kein Absturz)
- Einzelner Namespace 403 → wird übersprungen, restliche Namespaces werden gezeigt
