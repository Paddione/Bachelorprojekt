# Spec: Website — Mehrsprachigkeit DE/EN (Kunden-Frontend)

**Ticket:** T000612  
**Datum:** 2026-06-13  
**Status:** draft  
**Branch:** feature/t000612

---

## Problemstellung

Die Website ist ausschließlich deutsch. Alle UI-Strings (Buttons, Labels, Navigation, Footer, Cookie-Banner, Formular-Texte, Fehlermeldungen) sind hartcodiert in `.astro`- und `.svelte`-Dateien sowie in den Brand-Config-Dateien (`mentolder.ts`, `korczewski.ts`). Es existiert keinerlei i18n-Infrastruktur: keine Locale-Dateien, keine Übersetzungsfunktion, kein Sprachumschalter, keine i18n-Packages.

Die Astro-Config enthält nur einen Single-Locale-Stub (`defaultLocale: 'de', locales: ['de']`).

---

## Ziele

1. **i18n-Infrastruktur aufbauen** — Astro's built-in i18n mit `routingStrategy: 'pathname'` aktivieren. Locale-Dictionary-System einführen (kein externes Package, eigene typsichere Lösung).
2. **Sprachumschalter** — DE/EN-Toggle im Header (Navigation). Sprache wird per Cookie persistiert + `Accept-Language`-Header bei Erstbesuch erkannt.
3. **Statische UI-Strings extrahieren** — Alle hartcodierten deutschen Texte in Komponenten, Layouts und Pages in Locale-Dateien auslagern.
4. **EN-Übersetzungen** — Manuell gepflegte englische Texte (Patrick). Erste Phase: nur mentolder-Brand.
5. **Beide Brands vorbereiten** — Architektur von Anfang an für mentolder + korczewski ausgelegt, aber nur mentolder implementiert. Korczewski wird separates Folge-Ticket.
6. **SEO** — `hreflang`-Tags, `lang`-Attribut dynamisch, Canonical-URLs korrekt.

---

## Nicht in Scope

- **DB-Content zweisprachig** — `site_settings`, `pageContent`, `service_config` bleiben deutsch. Nur statische UI-Strings.
- **Admin-UI** — Admin-Seiten bleiben deutsch.
- **Korczewski-Implementierung** — Architektur vorbereitet, aber nicht aktiviert.
- **Automatische Übersetzung** — Keine KI-Übersetzung, alle EN-Texte manuell.
- **Portal-Seiten** — Kunden-Portal (`/portal/*`) bleibt deutsch (Phase 1).
- **E-Mail-Templates** — Bleiben deutsch (Phase 1).

## Architektur-Entscheidungen

### 1. Routing: Astro built-in i18n mit `pathname`

```js
// astro.config.mjs
export default defineConfig({
  i18n: {
    defaultLocale: 'de',
    locales: ['de', 'en'],
    routing: {
      prefixDefaultLocale: false,  // /... für DE, /en/... für EN
      redirectToDefaultLocale: false,
    },
  },
});
```

**Konsequenz:** Deutsche Seiten bleiben unter `/kontakt`, `/leistungen`, etc. Englische Seiten unter `/en/kontakt`, `/en/leistungen`, etc. Keine Subdomain, kein Query-Param.

**Dateistruktur:**
```
src/pages/
  index.astro              → DE (default)
  kontakt.astro            → DE
  en/
    index.astro            → EN (re-use DE component with EN locale)
    kontakt.astro          → EN
    [...alle DE-Pages als EN-Variante]
```

Jede EN-Page ist eine dünne Wrapper-Datei, die dieselbe Komponente wie die DE-Page rendert, aber mit `locale: 'en'` im Context.

### 2. Locale-Dictionary: Typsichere TS-Module

Kein externes i18n-Package. Eigene Lösung:

```
src/i18n/
  de.ts          → alle deutschen Strings (flaches Objekt)
  en.ts          → alle englischen Strings (gleiche Keys)
  index.ts       → getLocale(), t() Funktion, Locale-Typ
```

**Format:**
```ts
// src/i18n/de.ts
export default {
  'nav.angebote': 'Angebote',
  'nav.ueber-mich': 'Über mich',
  'nav.kontakt': 'Kontakt',
  'hero.title': 'Menschen, Prozesse und Technik',
  'hero.cta-primary': 'Kostenloses Erstgespräch',
  'hero.cta-secondary': 'Angebote ansehen',
  'footer.kontakt': 'Kontakt',
  'footer.rechtliches': 'Rechtliches',
  'cookie.banner-text': 'Diese Website verwendet Cookies...',
  'cookie.accept': 'Akzeptieren',
  'cookie.reject': 'Ablehnen',
  // ... etc
} satisfies Record<string, string>;
```

```ts
// src/i18n/index.ts
import de from './de';
import en from './en';

export type Locale = 'de' | 'en';
export const locales: Locale[] = ['de', 'en'];
export const defaultLocale: Locale = 'de';

const dictionaries = { de, en } as const;

export function getDictionary(locale: Locale) {
  return dictionaries[locale] ?? dictionaries.de;
}

export function t(locale: Locale, key: string): string {
  const dict = getDictionary(locale);
  return dict[key as keyof typeof dict] ?? key;
}
```

### 3. Locale-Detection

**Reihenfolge:**
1. Cookie `locale` (vom Sprachumschalter gesetzt)
2. `Accept-Language` Header (bei Erstbesuch)
3. Fallback: `de`

**Implementierung:** Middleware (`src/middleware.ts`) liest Cookie/Header und setzt `Astro.locals.locale`.

### 4. Sprachumschalter

Separate Komponente `LanguageSwitcher.svelte` (nicht in Navigation.svelte integrieren — Navigation ist bereits über 500 Zeilen).

```svelte
<!-- src/components/LanguageSwitcher.svelte -->
<script lang="ts">
  import { getLocaleFromUrl } from '../i18n';
  // ... DE/EN Toggle, setzt Cookie + navigiert zu /en/... oder /...
</script>
```

Eingebaut in `Navigation.svelte` als neuer Props-Slot oder direkt neben dem Mobile-Menü.

### 5. Brand-Architektur (beide Brands vorbereiten)

Locale-Keys sind brand-unabhängig. Die Dictionary-Struktur ist identisch für mentolder und korczewski. Wenn korczewski später EN bekommt, werden nur die EN-Texte in `en.ts` angepasst (keine Architektur-Änderung).

Brand-spezifische Strings (z.B. Firmenname, Slogan) kommen aus den bestehenden Brand-Config-Dateien (`mentolder.ts`, `korczewski.ts`) und werden bei Bedarf um EN-Varianten erweitert.
