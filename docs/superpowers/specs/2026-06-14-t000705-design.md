# Spec: Brett — Board-Templates für Aufstellungs-Szenarien

**Ticket:** T000705
**Datum:** 2026-06-14
**Status:** draft
**Branch:** feature/t000705

---

## Problemstellung

Coaches wiederholen häufig dieselben Aufstellungs-Szenarien (Familiensystem, Team-Konflikt, Innere Anteile etc.). Derzeit gibt es keine Möglichkeit, eine Figuren-Konstellation (Positionen, Farben, Beschriftungen) als wiederverwendbares Template zu speichern und beim Board-Start auszuwählen. Coaches müssen Szenarien jede Sitzung manuell neu aufstellen.

Das Brett-System hat bereits zwei Konzepte:

1. **Coaching-Templates** (`brett.coaching_templates`) — Schritt-Listen für den Ablauf einer Sitzung (Texte/Phasen), keine Figuren-Daten.
2. **Snapshots** (`brett_snapshots`) mit `is_template=true` — Figuren-Zustands-Abzüge, bereits als "curated templates" nutzbar. `admin_set_template` lädt einen Snapshot-State in den Raum.

Das fehlende Glied ist die **Coach-seitige Verwaltung von Board-Templates**: Coaches können derzeit keine eigenen Templates aus dem laufenden Board heraus speichern, und die Template-Auswahl im Lobby-Screen ist auf die bestehende Snapshot-API begrenzt — ohne Board-spezifische Metadaten (Szenario-Typ, Teilnehmer-Anzahl-Hinweis).

---

## Ziele

1. **Coach-Self-Service-Speichern** — Im aktiven Board oder über die Lobby kann der Leiter den aktuellen Figuren-Stand als Board-Template benennen und speichern. Kein Admin-Zugriff nötig.
2. **Template-Auswahl beim Board-Start** — Im Lobby-Screen kann der Leiter vor dem Start aus einer Liste eigener und system-kuratierter Templates auswählen. Das Board wird mit den Template-Figuren vorbelegt.
3. **Template-Metadaten** — Name, optionaler Beschreibungstext, Szenario-Kategorie (Freitext-Tag), Erstellungsdatum. Kein Figuren-Editor — das Template entsteht ausschließlich aus einem Live-Board-Stand.
4. **DB-Persistierung** — Neue Tabelle `brett.board_templates` (oder Nutzung der bestehenden `brett_snapshots`-Infrastruktur mit erweiterter Sichtbarkeit, siehe Architektur-Entscheidung).
5. **Lösch-Funktion** — Coach kann eigene Templates löschen. System-Templates sind schreibgeschützt.

---

## Nicht in Scope

- **Template-Editor** (Drag-and-Drop ohne Live-Board) — Templates entstehen immer aus einem bestehenden Board.
- **Template-Sharing zwischen Coaches / Brands** — Templates sind brand-scoped, nicht öffentlich oder nutzer-übergreifend für Phase 1.
- **Bild-Preview** — Kein Screenshot oder 3D-Thumbnail.
- **Kollaboratives Bearbeiten** — Templates sind read-only nach der Erstellung.
- **Import/Export** (JSON-Datei) — Phase 2.
- **Template-Versionierung** — Kein Update bestehender Templates, nur Create + Delete.

---

## Architektur-Entscheidungen

### A1 — Eigene Tabelle `brett.board_templates` statt Erweiterung von `brett_snapshots`

**Begründung:** `brett_snapshots` ist raumgebunden (`room_token`) und anonym (`customer_id`). Board-Templates sind coach-authored (`created_by_user`), brand-scoped und haben keinen Raum-Bezug. Eine eigene Tabelle vermeidet das Aufweichen des bestehenden Snapshot-Schemas und erlaubt Template-spezifische Felder (Kategorie, Systemflag) ohne Rückwärtskompatibilitätsprobleme.

**Alternative verworfen:** `is_template=true` auf `brett_snapshots` ist bereits für admin-kuratierte Vorlagen belegt; Coach-Templates würden sich ohne klare Eigentümerschaft darunter mischen.

### A2 — State-Format: JSON-Blob identisch zu Snapshots (`{ figures: Figure[] }`)

Das Template speichert denselben State-Blob wie ein Snapshot. Der bestehende `seedFiguresFromTemplate`-Mechanismus (D6) kann unverändert für Board-Templates genutzt werden. Keine neue Deserialisierung nötig.

### A3 — REST-API, keine WebSocket-Befehle für CRUD

Template-Erstellen/-Löschen/-Listen sind seltene, nicht-echtzeit-kritische Operationen. REST-Endpunkte passen in das bestehende Express-Router-Muster (analog `routes/snapshots.ts`). Nur das **Laden** eines Templates in den Raum bleibt ein WS-Admin-Befehl (`admin_set_board_template`, erweitertes `admin_set_template`).

### A4 — coach-owned (kein Admin-Gate für Create)

Im Gegensatz zu `brett_snapshots is_template=true` (admin-only) darf jeder authentifizierte `leiter` eigene Board-Templates anlegen. Admin-Gate gilt nur für `is_system=true`-Einträge (vorinstallierte Szenarien). Delete ist auf `created_by_user = current_user` begrenzt (plus Admin-Override).

### A5 — Systemvorgaben per Migration

Die folgenden drei Szenarien werden als `is_system=true`-Templates per SQL-Migration eingeliefert:
- **Familiensystem 4 Personen** — 4 Figuren in Quadrat-Anordnung
- **Team-Konflikt** — 2 Gruppen à 3 Figuren mit Lücke in der Mitte
- **Innere Anteile** — 5 Figuren (Zentrum + 4 Richtungen)

### A6 — UI: "Als Template speichern"-Button in der Board-Topbar

Parallel zu "Snapshot speichern" (bestehend) erscheint für Leiter ein neuer "Template speichern"-Button. Ein Inline-Formular (Name + optionale Kategorie) wird eingeblendet; nach Speichern erscheint eine Bestätigungs-Meldung.

### A7 — Lobby-Auswahl: erweiterter Template-Picker

Der bestehende `<select>` in der Lobby (`onSetTemplate`) wird zu einer kombinierten Liste aus:
1. Board-Templates (coach-owned + system) vom neuen `/api/board-templates`-Endpunkt
2. Bestehende Snapshots mit `is_template=true` (weiterhin unterstützt, rückwärtskompatibel)

Ein neuer `admin_set_board_template`-WS-Befehl (oder eine Erweiterung von `admin_set_template` mit `source: 'board-template'`) lädt den Template-State.

## Datenmodell

```sql
CREATE TABLE IF NOT EXISTS brett.board_templates (
  id             TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
  brand          TEXT NOT NULL,
  name           TEXT NOT NULL,
  description    TEXT,
  category       TEXT,                       -- Freitext-Tag (z.B. 'Familiensystem')
  state          JSONB NOT NULL,             -- { figures: Figure[] }
  is_system      BOOLEAN NOT NULL DEFAULT false,
  created_by_user TEXT,                      -- OIDC sub; NULL for system templates
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS board_templates_brand_idx
  ON brett.board_templates (brand, is_system, created_at DESC);
```

## API

### `GET /api/board-templates`
Query-Parameter: `brand` (required)
Response: `BoardTemplate[]` (ohne `state`-Blob).

### `POST /api/board-templates`
Body: `{ name, description?, category?, state }` (state = aktueller Board-State, erhalten via `GET /api/snapshots` oder direkt vom Client nach board-export)
Auth: requireSession (leiter)
Response: `{ id }` (201)

### `DELETE /api/board-templates/:id`
Auth: requireSession; nur Creator oder Admin darf löschen; System-Templates sind unantastbar.
Response: 204

### WS-Befehl: `admin_set_board_template`
```ts
{ type: 'admin_set_board_template'; boardTemplateId: string }
```
Server lädt `brett.board_templates WHERE id = $1`, ruft `applyTemplateToRoom` auf.
Ist bereits abwärtskompatibel mit `admin_set_template` (bleibt für Snapshot-basierte Templates).

---

## Client-Flow

```
Leiter im Board → [Template speichern]
  → Inline-Formular (name, category)
  → POST /api/board-templates (state = aktuelles Board via board-state-export helper)
  → Bestätigung: "Template '...' gespeichert"

Leiter in Lobby → [Vorlage wählen]
  → GET /api/board-templates?brand=<brand>
  → Kombinierte Liste: System-Templates + eigene Templates
  → Wahl → send { type: 'admin_set_board_template', boardTemplateId }
  → Server: seed figures → broadcast snapshot
  → Board zeigt Template-Figuren
```

## Sicherheit & Guardrails

- **Brand-Isolation:** `brand`-Feld wird server-seitig aus dem OIDC-Session-Kontext gesetzt — nie aus dem Request-Body.
- **Ownership-Check bei Delete:** `created_by_user = session.userId OR session.isAdmin`.
- **State-Validierung:** Bevor ein Template-State via `applyTemplateToRoom` angewendet wird, wird `state.figures` als Array geprüft (wie bei Snapshots).
- **Max-Templates:** Soft-Limit von 50 Templates pro `(brand, created_by_user)`-Paar, um DB-Aufblähung zu begrenzen.
