# Spec: Brett Zuschauer-Modus (T000706)

**Ticket:** T000706  
**Datum:** 2026-06-14  
**Status:** draft  
**Bereich:** brett

---

## Ziel

Supervisoren, Ausbilder und Beobachter sollen einem Brett-Board live zuschauen können, ohne Figuren
verschieben oder Board-Zustände ändern zu können. Zuschauer empfangen alle Board-Events in Echtzeit
über die bestehende WebSocket-Verbindung, haben jedoch keinerlei Schreibrechte. Der Zugang erfolgt
über einen separaten, widerrufbaren Zuschauer-Link (analog dem bestehenden Share-Token-System).

---

## Kontext & Abgrenzung

### Bestehende Mechanismen

Das Brett-Backend hat bereits:

- **Rollen:** `leiter | stellvertreter | beobachter | gast` (types/state.ts)
- **`beobachter`-Rolle:** bereits read-only in `canMutate` — darf nur `request_state_snapshot` und
  `figure_possess`/`figure_release` (letztere sind Sonderrechte für verkörperte Figuren)
- **Share-Token-System (T000608):** `brett_share_tokens`-Tabelle, `/share/:token`-Route, `ws._isGuest`-Flag
- **`ws._isGuest`:** verhindert bereits ALLE Mutationen außer `request_state_snapshot` (in `gateMutation`)

### Problem / Gap

Der bestehende Share-Token (`/share/:token`) macht den Nutzer zu einem anonymen **Gast** (`ws._isGuest = true`),
der:
- die statische `share.html`-Seite erhält (kein volles UI)
- keine Teilnehmerliste im Topbar sieht
- anonym (`currentUser.userId = 'anon'`) erscheint
- keinen Zuschauer-Badge in der Teilnehmerliste bekommt
- keinerlei Benutzeridentität trägt

Für den **Zuschauer-Modus** brauchen wir eine dedizierte UX:
1. Ein eigener Token-Typ (`zuschauer`) mit Typ-Flag in der DB
2. Der Zuschauer öffnet die vollständige Board-UI (nicht `share.html`)
3. Alle Aktionselemente (Drag, Fig-Panel-Buttons, Undo/Redo, Import) sind deaktiviert/ausgeblendet
4. Der Zuschauer erscheint in der Teilnehmerliste mit Badge „Zuschauer"
5. Kamera-Navigation (FreeFly, POV-View) bleibt erlaubt (read-only)
6. Link-Erzeugung durch Leiter oder Admin im bestehenden Share-UI

## Anforderungen

### Funktionale Anforderungen

| ID | Anforderung |
|----|-------------|
| FA-1 | Leiter und Admin können einen Zuschauer-Link für einen Raum erzeugen (POST /api/rooms/:token/zuschauer-share) |
| FA-2 | Zuschauer-Link öffnet vollständige Board-UI (board.html), nicht share.html |
| FA-3 | WebSocket-Verbindung mit `zuschauer_token`-Parameter → Server setzt `ws._isZuschauer = true` |
| FA-4 | `ws._isZuschauer = true` blockiert alle Mutationen auf Server-Seite (kein Schreiben) |
| FA-5 | Zuschauer empfängt vollständigen Board-Snapshot + alle Live-Broadcasts |
| FA-6 | Zuschauer erscheint in Teilnehmerliste mit Rolle `zuschauer` + visuell unterscheidbarem Badge |
| FA-7 | Alle schreibenden UI-Elemente (Drag & Drop, Fig-Panel-Mutationsbuttons, Undo/Redo, Import) sind deaktiviert |
| FA-8 | Kamera-Navigation (FreeFly, POV-Kamera, Zoom/Orbit) bleibt für Zuschauer erlaubt |
| FA-9 | Zuschauer-Token kann vom Leiter/Admin widerrufen werden (DELETE .../zuschauer-share/:token) |
| FA-10 | Zuschauer-Link ist von Share-Link klar unterscheidbar (URL-Pfad `/zuschauer/:token`) |

### Nicht-funktionale Anforderungen

| ID | Anforderung |
|----|-------------|
| NFA-1 | Zuschauer erhalten niemals eine `forbidden`-Fehlermeldung — schreibende Eingaben werden clientseitig bereits blockiert |
| NFA-2 | Server-Seite: fail-closed; `ws._isZuschauer` wird nur gesetzt nach erfolgreichem DB-Lookup |
| NFA-3 | Bestehende `gast`-/Share-Token-Logik (T000608) bleibt unverändert |
| NFA-4 | Keine Brand-Hostnamen in Code oder Manifesten (S3-Gate) |
| NFA-5 | Keine Zeilenüberschreitungen: ws-connection.ts (379/600), share-tokens.ts (55/600), share-routes.ts (44/600) — alle unkritisch |

## Datenmodell

### Migration 004 — Zuschauer-Token

```sql
ALTER TABLE brett_share_tokens
  ADD COLUMN IF NOT EXISTS token_type TEXT NOT NULL DEFAULT 'share';
-- Bestehende Tokens bleiben 'share', neue Zuschauer-Tokens werden 'zuschauer'.
```

Alternative: eigene Tabelle (nicht bevorzugt — teilt dasselbe Widerruf-/Ablaufmuster).

Token-Typ wird beim Resolve mitgeprüft: ein `zuschauer`-Token setzt `ws._isZuschauer = true`
statt `ws._isGuest = true`.

## Sicherheitsmodell

```
                    ┌──────────────────────────────────┐
                    │  WebSocket-Verbindung              │
                    │  ?zuschauer_token=<tok>            │
                    └────────────┬─────────────────────┘
                                 │
                    ┌────────────▼─────────────────────┐
                    │  DB-Lookup: resolveZuschauerToken  │
                    │  → roomToken + type = 'zuschauer'  │
                    │  ungültig/disabled → close(4403)   │
                    └────────────┬─────────────────────┘
                                 │
                    ┌────────────▼─────────────────────┐
                    │  ws._isZuschauer = true            │
                    │  ws._shareRoom = roomToken         │
                    └────────────┬─────────────────────┘
                                 │
                    ┌────────────▼─────────────────────┐
                    │  gateMutation:                     │
                    │  if (ws._isZuschauer)              │
                    │    return msgType ===              │
                    │      'request_state_snapshot'      │
                    └──────────────────────────────────┘
```

Zuschauer dürfen:
- `request_state_snapshot` (passiv, no-op)
- Kamera-Events (rein clientseitig, nie ans WS gesendet)

Zuschauer dürfen nicht:
- Jede andere WS-Mutation (add, move, delete, figure_lock, admin_*)
- figure_possess / figure_release (anders als `beobachter`)

## Client-UX

### Zuschauer-Modus aktivieren

`board-boot.ts` liest `?zuschauer_token=<tok>` aus der URL.  
Wenn vorhanden: `window.__brettIsZuschauer = true`.

### Deaktivierte Elemente

| Element | Maßnahme |
|---------|----------|
| Figur-Drag (mannequin.ts) | `sendMove` no-op wenn `__brettIsZuschauer` |
| Fig-Panel-Buttons (fig-panel.ts) | Schaltflächen `disabled` |
| Undo/Redo (undo-redo-ui.ts) | Buttons `disabled` |
| Import-UI (import.ts) | Button `disabled` |
| Figure-Lock (ws-client.ts) | `sendFigureLock` no-op |
| Figure-Possess/Release | no-op |
| Topbar-Invite | ausgeblendet (Zuschauer kann nicht einladen) |

### Teilnehmerliste

Zuschauer werden mit Rolle `zuschauer` und Badge `👁 Zuschauer` in der Roster-UI
(`topbar-participants.ts`) angezeigt. Die Rolle wird serverseitig ins `presence_join`-Paket
eingetragen (`role: 'zuschauer'`).

### Zuschauer-Link-UI

Im bestehenden Share-Panel (`topbar-share.ts`) einen neuen Abschnitt „Zuschauer-Link" mit
Erzeugen/Kopieren/Widerrufen-Buttons ergänzen.

## API-Erweiterungen

```
POST   /api/rooms/:roomToken/zuschauer-share
       → { token, url }   (Leiter oder Admin)

GET    /api/rooms/:roomToken/zuschauer-shares
       → { tokens: [...] }

DELETE /api/rooms/:roomToken/zuschauer-share/:token
       → { disabled: true }

GET    /api/zuschauer/:token
       → { valid: true, roomToken }   (öffentlich, kein Auth)

GET    /zuschauer/:token
       → zuschauer.html (board.html-Klon mit Zuschauer-Init)
```
