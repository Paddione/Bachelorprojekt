# Spec — T000788: Cockpit Feature-Suggest — kritische Bugs (SSRF, blocking LLM, N+1)

## Kontext

PR #1706 (`feat(cockpit): add feature suggestion manager with AI-powered roll [T000784]`) wurde
am **2026-06-15 gemerged** und ist **LIVE in `main`**. Das AI Code Review hat vor dem Merge
`CHANGES_REQUESTED` gesetzt; der Merge erfolgte trotzdem. Damit sind sechs sicherheits- und
stabilitätsrelevante Defekte produktiv. Dies ist ein **Post-Merge-Bugfix** (kein „vor Merge"),
der die sechs Befunde gezielt, einzeln und testgetrieben (TDD: failing test first) korrigiert.

Betroffene Oberfläche: der Cockpit-Feature-Suggest-Pfad unter `website/src/`:
- `website/src/pages/api/admin/cockpit/suggest.ts` — LLM-Roll-Endpoint (Provider, API-Key, Parse).
- `website/src/lib/tickets/suggest-prompt.ts` — reiner Prompt-/Parse-Helper.
- `website/src/lib/tickets/cockpit-db.ts` — `getPortfolio()` (DB-Layer, LIMIT-Befund).
- `website/src/components/admin/CockpitSidebar.svelte` — `handleApply`/`handleReset` (N+1).
- `website/src/pages/api/admin/cockpit/feature-action.ts` — Einzel-Mutation (Batch-Gegenstück).

## Befunde (Ist-Zustand, code-verifiziert)

### B1 — Blockierender synchroner LLM-Call ohne Timeout
`suggest.ts` ruft `client.chat.completions.create({...})` **ohne Timeout/Abbruch** auf. Ein
hängender Upstream (DeepSeek/Anthropic) hält den Request offen, bis der Server-seitige Default
greift; unter Concurrency staut sich der Astro/Node-Event-Loop (Connection-Starvation). Es gibt
keinen `AbortController`, kein Timeout-Argument am OpenAI-Client.

### B2 — N+1 Portfolio-Refetch in `handleApply`/`handleReset`
In `CockpitSidebar.svelte` iterieren `handleApply()` und `handleReset()` über
`allFeatures.filter(f => f.nextStep)` und rufen für **jedes** Feature `onFeatureAction(...)` auf.
`onFeatureAction` ist in `Cockpit.svelte` `featureAction()` — jeder Aufruf feuert **einen**
`POST /feature-action` **und danach** `loadPortfolio()`. Bei N markierten Features ergibt das
**N sequentielle Mutations-POSTs + N volle Portfolio-Refetches** (`onMutated?.()` danach erneut).
Das ist eine sequenzielle await-in-for-of-Schleife, die linear mit der Feature-Zahl skaliert.

### B3 — SSRF-Risiko durch freie Provider-Auswahl vom Client
`suggest.ts` übernimmt `body.provider` ungeprüft vom Client. Aktuell verzweigt der Code nur für
`'deepseek'` auf einen festen Endpoint und fällt sonst auf `undefined` (OpenAI-Default) zurück —
aber es existiert **keine server-seitige Allowlist**. Wächst der Endpoint-Mapping-Code (z. B.
client-gelieferter `baseURL`/Endpoint), kann ein Client den Upstream auf eine interne Adresse
zeigen lassen (Server-Side Request Forgery). Die Provider-/Endpoint-Auswahl muss **server-seitig**
über eine feste Allowlist erlaubter Provider-IDs erfolgen; unbekannte IDs werden abgelehnt.

### B4 — Ungefilterter Parse / Fehlerpfad auf LLM-Output
`parseSuggestions()` hat zwar bereits `try/catch` um `JSON.parse`, **aber**: der Endpoint
unterscheidet einen leeren/strukturell unparsbaren Output nicht von einem echten Modellfehler —
beides wird als HTTP 500 zurückgegeben (`'AI response could not be parsed'`). Zudem gibt es keine
**Schema-Validierung** auf Item-Ebene jenseits von `featureId`. Befund: härtere, schemavalidierte
Parse-Funktion und ein **deterministisches** Verhalten (leeres `suggestions: []` mit 200 statt
500) bei „Modell lieferte keine verwertbaren Vorschläge", damit ein gutartig leerer Output den
Endpoint nicht als Fehler markiert.

### B5 — Hardcoded `'not-required'` API-Key-Fallback ermöglicht stille 401
`suggest.ts` setzt `apiKey = provider === 'deepseek' ? (process.env.DEEPSEEK_API_KEY ?? 'not-required') : 'not-required'`.
Fehlt der Key, wird der String `'not-required'` gesendet → der Upstream antwortet mit 401, was als
generischer Fehler durchschlägt. Das soll **fail-closed** werden: fehlt der für den (erlaubten)
Provider erforderliche Key, antwortet der Endpoint sofort mit einem klaren
„Provider nicht konfiguriert"-Fehler (HTTP 503/500), **bevor** ein Upstream-Call passiert.

### B6 — `getPortfolio()` ohne LIMIT
`getPortfolio()` (und die Synthetik-Helfer `getLeafTickets`/`fetchLeafRollup`) laden über
`SELECT … FROM tickets.tickets WHERE brand = $1 …` **ohne `LIMIT`**. Bei unbegrenzt wachsendem
Ticket-Bestand lädt der Endpoint alle Zeilen in den Node-Heap und in den LLM-Prompt. Es braucht
eine **defensive Obergrenze** (LIMIT) auf der Feature-/Projekt-Query (und auf der Leaf-Ticket-
Query), mit definiertem, stabilem Sortier-/Trunkierungsverhalten.

## Ziel / Nicht-Ziel

**Ziel:** Die sechs Befunde fail-closed, getestet und ohne Verhaltensregression für den
Happy-Path beheben. Jeder Befund = ein Task mit zuerst geschriebenem, fehlschlagendem Test.

**Nicht-Ziel:** Funktionale Erweiterung der Suggest-Feature (keine neuen Provider, keine neue UI),
keine DB-Migration, keine Änderung am LLM-Prompt-Inhalt, keine geteilten Registry-Dateien
(`configmap-domains.yaml`, `schema.yaml`, `kustomization.yaml`).

## Akzeptanzkriterien

- **AK-B1:** Der Upstream-Call in `suggest.ts` läuft mit Timeout (~10 s, `AbortController` bzw.
  OpenAI-`timeout`-Option). Ein Timeout liefert einen definierten Fehler (HTTP 504/500), ohne den
  Request unbegrenzt offen zu halten. Test: gemockter Client, der nie auflöst → Endpoint kehrt
  innerhalb der Timeout-Grenze mit Fehler zurück.
- **AK-B2:** `handleApply`/`handleReset` lösen genau **einen** Backend-Roundtrip (Batch) und
  **einen** Portfolio-Refetch aus, unabhängig von der Anzahl markierter Features. Test: N=3
  markierte Features → genau 1 Mutations-Aufruf + 1 Refetch (Spy-Count).
- **AK-B3:** Ein `provider`, der nicht in der server-seitigen Allowlist steht, wird mit HTTP 400
  („invalid provider") abgelehnt, **ohne** Upstream-Call. Test: `provider:'evil'` → 400.
- **AK-B4:** Strukturell unparsbarer/leerer Modelloutput führt zu `200 { suggestions: [] }`
  (nicht 500); Items, die das Schema verletzen, werden verworfen. Test: Mock liefert Prosa ohne
  JSON → 200 + leere Liste.
- **AK-B5:** Fehlt der erforderliche API-Key des erlaubten Providers, antwortet der Endpoint
  fail-closed mit klarem Fehler **vor** dem Upstream-Call; `'not-required'` wird nie als Key
  gesendet. Test: `DEEPSEEK_API_KEY` unset → definierter „nicht konfiguriert"-Fehler, Client wird
  nie konstruiert/aufgerufen.
- **AK-B6:** `getPortfolio()` wendet ein server-seitiges `LIMIT` (Konstante) auf die
  Feature-/Projekt-Query und die Leaf-Query an. Test: pg-mem mit > LIMIT Zeilen → höchstens LIMIT
  Container/Leaves im Ergebnis.

## Qualitäts-Gates (S1–S4)

- **S1 (Zeilenbudget):** Siehe Plan — pro Datei Ist + Baseline ermittelt. `cockpit-db.ts` (377)
  und `CockpitSidebar.svelte` (379) liegen nahe an ihren Limits (600 `.ts` / 500 `.svelte`) — die
  Änderungen sind klein, neue Logik (Batch-Apply, Provider-Allowlist) wird in **pure Helper-Module
  bzw. das bestehende `cockpit-db.ts`** gelegt; bei Überschreiten von ~80 % der wirksamen Schwelle
  wird extrahiert statt verdichtet.
- **S2 (Import-Zyklen):** Neue Helfer (Provider-Allowlist, Timeout-Konstante) als pure Module ohne
  Rück-Import auf DB-/API-Schichten.
- **S3 (Hostnamen):** Keine Brand-Domain-Literale; Provider-Endpoints sind externe API-URLs
  (deepseek.com/anthropic), keine `*.mentolder.de`/`*.korczewski.de`.
- **S4 (Orphans):** Kein neues `k3d/*.yaml` und kein neues `scripts/*.sh` — falls ein neues
  TS-Helper-Modul entsteht, wird es vom Endpoint importiert (kein Orphan-Risiko für TS).
