# Spec: VDA-Unify — Restgap-Schließung (T000864)

**Datum:** 2026-06-17
**Status:** design
**Kanonische Spec:** `docs/superpowers/specs/2026-06-16-vda-unify-design.md` (Vollkonzept VDA-Unify — diese Spec setzt sie voraus und grenzt T000864 auf die noch offenen Lücken ein.)

## 1. Kontext

VDA-Unify (kanonische Spec) bündelt 11+ interaktive Factory-Skripte unter einem Single Entry-Point `scripts/vda.sh` mit gemeinsamer Helfer-Library `scripts/lib/vda-core.sh`. Der Großteil ist auf `main` bereits umgesetzt: Core-Library (alle `vda_*`-Helfer), Entry-Point + Dispatch, die Subcommands `oracle`/`frontmatter`/`brainstorm`/`factory-prep`/`backup`, ein teilweiser `ticket`-Dispatcher (5 von 24 Subcommands extrahiert), `lib/promote-phases.sh`, sowie Deprecation-Wrapper für `task-oracle.sh`/`t.sh`/`plan-frontmatter-hook.sh`/`brainstorm-bridge.sh`. Tests: `tests/unit/vda-core.bats`, `tests/unit/vda-ticket-smoke.bats`.

## 2. Problem (verbleibende Lücken)

1. **`vda.sh promote` ist nicht lauffähig.** `scripts/vda.sh` delegiert den `promote)`-Arm via `exec` auf `scripts/vda/promote.sh`, **diese Datei fehlt** → jeder Aufruf bricht mit „No such file or directory" ab. Spec §4.2 verlangt das Subcommand.
2. **`vda.sh ticket` ist unvollständig.** Der Dispatcher kennt nur `create|get|update-status|enqueue|stage-plan`. Die übrigen **19** Subcommands (`add-comment`, `add-pr-link`, `grill`, `archive-plan`, `get-attachments`, `set-touched-files`, `set-pipeline-slot`, `release-slot`, `touch`, `retry-count`, `factory-control`, `dryrun-mark`, `dryrun-check`, `feature-flag`, `phase`, `inject`, `get-injections`, `plan-meta`, `lastenheft`) sind nur über den Monolithen `scripts/ticket.sh` erreichbar. Spec §4.5 / §8 verlangt vollständige Erreichbarkeit über die `vda.sh ticket`-Fassade.

## 3. Ziel

`vda.sh promote` und `vda.sh ticket <jeder-subcommand>` funktionieren vollständig und **verhaltensidentisch** zu den bestehenden Skripten — ohne Regression an den Factory-Aufrufpfaden (`dispatcher.js`, `wakeup.sh`, `auto-enqueue.sh`, `watchdog.sh`, `openspec.sh`, `batch-workflow-gen.sh`), die `ticket.sh` direkt nutzen.

## 4. Lösungsansatz

### 4.1 `scripts/vda/promote.sh` (NEU) — Pass-Through-Wrapper

Analog `scripts/vda/backup.sh`. Übersetzt `[service]` + `--target <wert>` in die von `scripts/feature-promote.sh` erwarteten Env-Vars `SERVICE`/`TARGET` und delegiert per `exec`. Keine zweite Promotion-Implementierung (Vermeidung von Duplikation/S2-Risiko); `feature-promote.sh` bleibt SSOT und unverändert. Ohne Argumente: ohne Env-Vars delegieren → `feature-promote.sh` promptet selbst (Spec §4.2 Phase 1/2). Ziel < 40 Zeilen.

### 4.2 `scripts/vda/ticket.sh` — Vollständige Fassade

`case`-Arm um einen **Fallback** ergänzen: extrahierte Subcommands laufen weiter über `scripts/vda/ticket/*.sh`; **alle übrigen** delegieren transparent via `exec bash scripts/ticket.sh <subcommand> "$@"` (Env-Vars wie `BRAND` bleiben erhalten — kein Var-Stripping). `show_help` um die durchgereichten Subcommands ergänzen. Physische Restextraktion bleibt der kanonischen Spec-Phase 2 (separate PRs) überlassen — T000864 stellt nur Fassaden-Vollständigkeit her (null Regressions-Risiko an Factory-Pfaden). Ziel < 80 Zeilen.

### 4.3 Tests

`tests/unit/vda-ticket-smoke.bats` (bestehend, erweitern) deckt offline/non-interactive ab: `ticket help` listet beide Klassen, durchgereichte Subcommands erreichen `ticket.sh` ohne vda-Dispatcher-„unknown"-Fehler (über `--help`/Validierungs-Pfad vor `_pgpod`, analog FA-SF-Tests). Promote-Smoke: `vda.sh promote --help` Exit 0; ungültiger Service → kontrollierter Fehler statt „No such file".

## 5. Quality Gates

| Gate | Status |
|------|--------|
| S1 | `promote.sh` neu < 40 Z (Limit 500); `ticket.sh` 32→~52 Z (Limit 500). Keine gebaselinete Datei wächst. |
| S2 | Keine neuen Source-Kanten/Zyklen — Delegation via `exec bash`. |
| S3 | Keine Brand-Domain-Literale; `mentolder`/`korczewski` nur als Ziel-Bezeichner durchgereicht, Domains via `env-resolve.sh` in `feature-promote.sh`. |
| S4 | `promote.sh` von `vda.sh` referenziert (kein Orphan); keine neuen Manifeste. |

## 6. Erfolgskriterien

- `vda.sh promote website --target mentolder` bricht nicht mehr mit „No such file or directory" ab.
- `vda.sh ticket <jeder der 24 Subcommands>` erreichbar und funktionsidentisch zu `ticket.sh <subcommand>`.
- `task test:all` + `task freshness:check` (S1–S4) grün; Baseline-Key-Anzahl unverändert.

## 7. Out of Scope

Physische Extraktion der 19 Monolith-Subcommands (Spec-Phase 2); Umbau von `feature-promote.sh`/`backup-restore.sh`/`ticket.sh` zu Wrappern (Spec-Phase 3); Verkleinerung von `oracle.sh`; Änderungen an `pipeline.js`/`dispatcher.js`/`guards.sh`.
