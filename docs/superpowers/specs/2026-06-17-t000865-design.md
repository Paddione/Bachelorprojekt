# Spec: VDA-Triage — Unified Ticket Triage Interface

**Datum:** 2026-06-17
**Status:** design
**Parent:** VDA-Unify (`docs/superpowers/specs/2026-06-16-vda-unify-design.md`)

## 1. Kontext & Einordnung

Dies ist ein **Sub-Interface von VDA-Unify**. VDA-Unify führt einen einheitlichen
`vda.sh <bereich> <aktion>` Entry-Point ein, der auf `scripts/lib/vda-core.sh` aufbaut
(View-/Decision-/Action-Helfer, **kein fzf/gum/dialog**, fail-closed, jeder Subcommand < 150 Zeilen).

Der Entry-Point `scripts/vda.sh`, die Core-Library `scripts/lib/vda-core.sh`, der Ticket-Dispatcher
`scripts/vda/ticket.sh` und die geteilten PG-Helfer `scripts/vda/ticket/_ticket-core.sh` **existieren
bereits** (VDA-Unify Phase 1). Dieses Ticket fügt **einen** Subcommand hinzu:

```
vda.sh ticket triage     (bereich=ticket, aktion=triage)
```

### Heutiger Zustand (Ist)

Es gibt **zwei separate, unverbundene Triage-/Grilling-Pfade**:

1. **Web-AI-Triage** — `website/src/lib/ticket-triage.ts` (`autoTriage`/`runTriage`) + API
   `website/src/pages/api/admin/tickets/[id]/triage.ts`. LLM schlägt `priority`/`severity`/`component`
   vor und schreibt einen Timeline-Kommentar. Nur über die Admin-UI / API erreichbar — **kein CLI-Pfad**.
2. **CLI-Grilling** — `scripts/lib/ticket-grill.sh` (218 Z), gesourct von `scripts/ticket.sh` als
   `cmd_grill`. Schreibt Grilling-Q/A in `tickets.tickets.grilling_answers`. Eigenes UX, kein
   VDA-Pattern, nicht über `vda.sh` erreichbar.

**Problem:** Wer ein Ticket von der CLI/von einem Agenten aus triagieren will (priority/severity setzen,
Status `triage`→`planning`/`backlog` schieben, AI-Vorschlag holen), hat keinen einheitlichen Befehl. Die
CLI kann nur `ticket update-status` (roher Status-Set) — nichts Triage-spezifisches, kein VDA-View, kein
fail-closed Eingabe-Layer.

### Entscheidungen (final, aus dem Ticket — nicht hinterfragt)

| Dimension | Entscheidung |
|-----------|-------------|
| Command-Pfad | `vda.sh ticket triage` — bereich=`ticket`, aktion=`triage` |
| Fundament | baut auf `scripts/lib/vda-core.sh` auf (View/Choice/Confirm/Input-Helfer) |
| Dependencies | **kein fzf/gum/dialog** — nur bash-builtins + kubectl + psql (jq optional) |
| Größe | Subcommand < 150 Zeilen |
| Verhalten | deterministisch, **fail-closed** (Pflichtfelder fehlen → exit 2; kein DB-Pod → exit 1) |
| Deprecation | `scripts/lib/ticket-grill.sh` wird **deprecated** und delegiert als **dünner Shim** an den neuen Command — **NICHT löschen** (Rückwärtskompatibilität) |

## 2. Ziel

Ein einziger CLI-Befehl, der die Triage eines Tickets als sauberes VDA durchführt:

```
vda.sh ticket triage --id <ext-id> [--priority …] [--severity …] [--status …] [--suggest] [--apply]
```

- **View:** zeigt die triage-relevanten Felder des Tickets (title, type, status, priority, severity,
  component) strukturiert via `vda_header`/`vda_section`.
- **Decision:** interaktiv `vda_choose` für priority/severity/status, wenn nicht per Flag vorgegeben;
  im Non-Interactive-Mode (`VDA_NONINTERACTIVE=1` / kein TTY / `--apply`) ausschließlich Flags, sonst Fail.
- **Action:** schreibt `priority`/`severity`/`status` per `_exec_sql` (gleiches Pattern wie
  `update-status.sh`) und postet optional einen Triage-Timeline-Kommentar (author `triage`).
- **`--suggest`:** holt einen **AI-Vorschlag** über den bestehenden Web-Triage-Pfad (HTTP-POST an
  `/api/admin/tickets/<id>/triage`, falls erreichbar) und zeigt ihn als Default-Vorauswahl an — niemals
  blind angewendet (fail-open: AI nicht erreichbar → manuelle Auswahl, kein Abbruch).

## 3. Architektur

```
vda.sh ticket triage
  │  (Dispatch in scripts/vda/ticket.sh — case-Allowlist erweitern um "triage")
  ▼
scripts/vda/ticket/triage.sh          (NEU, < 150 Z)
  ├─ source _ticket-core.sh           (geteilte PG-Helfer: _pgpod, _exec_sql)
  ├─ source ../../lib/vda-core.sh      (View/Choice/Confirm/Input)
  │
  ├─ View    → SELECT title,type,status,priority,severity,component … via _exec_sql
  ├─ Suggest → optional: curl POST /api/admin/tickets/<id>/triage  (fail-open)
  ├─ Decide  → vda_choose priority/severity/status  (oder Flags / non-interactive)
  └─ Action  → UPDATE tickets.tickets SET priority,severity,status …
              + optional INSERT tickets.ticket_comments (author_label='triage')

scripts/lib/ticket-grill.sh           (DEPRECATED SHIM)
  └─ cmd_grill bleibt erhalten (Rückwärtskompat), gibt Deprecation-Warnung aus,
     delegiert NICHT die Grilling-Logik, sondern weist auf `vda.sh ticket triage` /
     `vda.sh ticket grill` hin. Datei NICHT löschen.
```

### Latenter Bug, der hier mitgefixt wird (fail-closed Pflicht)

Die bestehenden `scripts/vda/ticket/*.sh` referenzieren `$NS`/`$CTX`/`$DB`/`$USER` ungebunden — sie
funktionieren nur, wenn von `scripts/ticket.sh` gesourct (das setzt die Vars). Standalone via
`vda.sh ticket get` failt `_ticket-core.sh` mit `NS: unbound variable` (verifiziert 2026-06-17):

```
$ bash scripts/vda/ticket.sh get --id T000001
.../_ticket-core.sh: line 8: NS: unbound variable
```

Damit `vda.sh ticket triage` standalone fail-**closed** statt fail-**ugly** ist, setzt
`_ticket-core.sh` Defaults aus den `TICKET_*`-Env-Vars (gleiche Quelle wie `ticket.sh`):

```bash
: "${NS:=${TICKET_NS:-workspace}}"
: "${CTX:=${TICKET_CTX:-fleet}}"
: "${DB:=website}"
: "${USER:=website}"
```

Das ist eine 4-Zeilen-Härtung im geteilten Helfer, additiv und rückwärtskompatibel (ticket.sh setzt
diese Vars weiterhin selbst; `:=` respektiert vorhandene Werte).

## 4. Command-Oberfläche

```
vda.sh ticket triage --id <ext-id> [flags]

Pflicht:
  --id <ext-id>          Ticket external_id (z.B. T000865)

Triage-Felder (alle optional; fehlende → interaktiv erfragt, sofern TTY & nicht --apply):
  --priority <p>         niedrig | mittel | hoch            (DB-Wertebereich, vgl. PRIORITY_MAP)
  --severity <s>         critical | major | minor | trivial (VALID_SEVERITIES)
  --status <st>          triage | planning | backlog | …    (bestehender Status-Wertebereich)
  --component <text>     freier Component-Name

Modus:
  --suggest              AI-Vorschlag holen (HTTP, fail-open) und als Default anzeigen
  --apply                Non-interaktiv: nur Flags verwenden, KEIN Prompt; fehlende Pflicht → exit 2
  --no-comment           keinen Triage-Timeline-Kommentar posten
  -h | --help            Hilfe
```

**Wertebereiche** (Quelle: `website/src/lib/ticket-triage.ts`):
- `severity` ∈ `{critical, major, minor, trivial}` (`VALID_SEVERITIES`).
- `priority` (DB) ∈ `{niedrig, mittel, hoch}` — die AI mappt `low/medium/high/critical → niedrig/mittel/hoch`
  (`PRIORITY_MAP`); der CLI-Command akzeptiert sowohl die DB-Werte als auch die AI-Eingangswerte und
  normalisiert über dieselbe Map.
- `status`: bestehender Ticket-Status-Wertebereich (u.a. `triage`, `planning`, `plan_staged`,
  `backlog`, `done`, `archived`).

**Validierung (fail-closed):** unbekannter `--priority`/`--severity`/`--status`-Wert → `vda_error` + exit 2,
**bevor** ein DB-Pod gesucht wird (gleiche Reihenfolge wie `ticket.sh` validate-before-`_pgpod`, vgl.
FA-SF Lesson „validation tests not offline-safe").

## 6. Deprecation-Shim für `ticket-grill.sh`

`scripts/lib/ticket-grill.sh` bleibt erhalten (von `scripts/ticket.sh` gesourct → `cmd_grill`). Wir
fügen am Anfang von `cmd_grill` eine **einmalige Deprecation-Warnung** (stderr) hinzu, die auf den
neuen unified Pfad verweist:

```bash
cmd_grill() {
  vda_warn "ticket-grill.sh/cmd_grill is deprecated. Use: vda.sh ticket triage --id <id>" >&2 2>/dev/null \
    || echo "⚠ ticket-grill.sh/cmd_grill is deprecated. Use: vda.sh ticket triage --id <id>" >&2
  # … bestehende Grilling-Logik unverändert …
}
```

**Wichtig:**
- Die Grilling-**Logik** (Q/A in `grilling_answers`) wird **nicht** entfernt — sie ist eine andere
  Capability als Triage (priority/severity). Triage und Grilling sind komplementär.
- Die Datei wird **nicht gelöscht** (Rückwärtskompat; `tests/unit/ticket-grill.bats` bleibt grün).
- Die Warnung darf den `cmd_grill`-Output (TSV/JSON für GrillingAnswersPanel) **nicht** verschmutzen →
  ausschließlich auf **stderr**.

## 7. Quality Gates (S1–S4)

| Gate | Status |
|------|--------|
| S1 (Zeilen) | `scripts/vda/ticket/triage.sh` NEU, Ziel < 150 Z (Limit `.sh`=500, non-baselined → Budget groß). `_ticket-core.sh` 20→~24 Z. `scripts/vda/ticket.sh` 32→~34 Z. `ticket-grill.sh` 218→~221 Z (Limit 500). Alle non-baselined, weit unter Limit. |
| S2 (Zyklen) | `triage.sh` sourct nur `_ticket-core.sh` (pure PG-Helfer) + `vda-core.sh` (pure View-Helfer). Keine Rück-Imports, kein neuer Zyklus. |
| S3 (Brand-Literale) | **Keine** `*.mentolder.de`/`*.korczewski.de`-Literale. Die `--suggest`-HTTP-URL kommt aus `${TRIAGE_API_URL:-…}` / env-resolve, nie hardcodiert. |
| S4 (Orphans) | `triage.sh` wird vom Dispatcher `vda/ticket.sh` (case-Allowlist + help) referenziert → kein Orphan. Neue bats werden in `Taskfile.yml`/`test:all` erreichbar gemacht. |

## 9. Erfolgskriterien

- `vda.sh ticket triage --id T000865 --priority hoch --severity major --status planning --apply` setzt die
  Felder in `tickets.tickets` und postet (sofern nicht `--no-comment`) einen `triage`-Kommentar.
- `VDA_NONINTERACTIVE=1 vda.sh ticket triage --id … --apply` ohne Pflicht-Flags → exit 2, kein Prompt.
- `vda.sh ticket triage --id … --priority bogus` → exit 2 mit `vda_error`, **vor** kubectl/psql.
- `vda.sh ticket help` listet `triage`.
- `bash scripts/ticket.sh grill …` funktioniert unverändert, gibt aber Deprecation-Warnung auf stderr.
- `scripts/vda/ticket.sh get --id …` failt **nicht** mehr mit `NS: unbound variable`.
- `task test:all`, `task freshness:regenerate`, `task freshness:check` grün; bestehende
  `ticket-grill.bats`/`vda-ticket-smoke.bats`/FA-SF-Tests grün.

## 10. Out of Scope

- Keine Änderung an `website/src/lib/ticket-triage.ts` oder der Admin-UI/API — nur **konsumiert** über `--suggest`.
- Keine Migration der übrigen `ticket.sh`-Subcommands (das ist VDA-Unify Phase 2, separate Tickets).
- Kein Entfernen der Grilling-Capability oder von `ticket-grill.sh` (nur Deprecation-Shim).
- Kein neues DB-Schema, keine neuen Spalten (priority/severity/status/component existieren bereits).
- Kein fzf/gum/dialog, keine neue Dependency.
