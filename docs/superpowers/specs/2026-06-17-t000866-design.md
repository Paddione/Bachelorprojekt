# Spec: VDA-Slots — Unified Slots & Provider Routing

**Datum:** 2026-06-17
**Status:** design
**Parent:** VDA-Unify (`docs/superpowers/specs/2026-06-16-vda-unify-design.md`)

## 1. Kontext & Einordnung

Dies ist ein **Sub-Interface von VDA-Unify** (T000866). Das VDA-Unify-Dachvorhaben
führt einen Single-Entry-Point `scripts/vda.sh` ein, der alle interaktiven Factory-Skripte
als Subcommands bereitstellt (siehe Parent-Spec). VDA-Unify hat bereits ein Skelett geliefert:
`scripts/vda.sh` (Entry-Point + flacher Dispatch), `scripts/lib/vda-core.sh` (View/Decision/Action-Helfer)
und mehrere Subcommands unter `scripts/vda/`.

Diese Spec ergänzt den **`factory`-Bereich** des Dispatchers und den **`slots`-Aktion** darin.
Bislang adressiert `vda.sh` nur flache Top-Level-Commands (`oracle`, `promote`, `ticket`, …).
Die Slot-Buchhaltung des Dispatchers lebt in `scripts/factory/slots.sh` (per-Brand Slot-Accounting,
race-free claim/release über PostgreSQL). Es gibt aktuell **keinen einheitlichen VDA-Einstieg**
in die Slot-Verwaltung — Agenten/Operatoren müssen `BRAND=<brand> bash scripts/factory/slots.sh <cmd>`
direkt aufrufen, mit eigenem Output-Format (mal plaintext-Zahl, mal JSON-Error).

## 2. Problem

1. **Kein VDA-Einstieg in Slots:** `slots.sh` ist nur direkt erreichbar, nicht über `vda.sh`.
   Damit fehlt der Slot-Verwaltung das einheitliche VDA-Pattern (View/Decision/Action) und
   das konsistente Output-Format (JSON für IPC, Plaintext human-readable).
2. **Kein `factory`-Bereich im Dispatcher:** `vda.sh` kennt nur flache Commands. Das in der
   Parent-Spec vorgesehene `vda.sh <bereich> <aktion>`-Muster (z.B. `vda.sh factory slots`)
   ist für den `factory`-Bereich noch nicht verdrahtet.
3. **Output-Inkonsistenz:** `slots.sh count`/`next` geben rohe Zahlen, `claim` echot `n`,
   Fehler kommen mal als plaintext (`claim failed …`), mal als JSON (`{"error":…}`). Für
   IPC-Konsumenten (Dispatcher-JS) wäre ein einheitliches, optional-JSON-Format wünschenswert.

## 3. Ziel

Ein neuer Befehlspfad **`vda.sh factory slots <aktion> [args]`** (Bereich = `factory`,
Aktion = `slots`), der `scripts/factory/slots.sh` **wrappt** (delegiert, ersetzt NICHT) und
einheitliches Output liefert: **JSON für IPC** (`--json` oder `VDA_JSON=1`), **Plaintext**
für human-readable (Default).

### Architektur-Entscheidungen (final, aus Ticket-Kontext)

| Dimension | Entscheidung |
|-----------|-------------|
| Command-Pfad | `vda.sh factory slots <count\|next\|claim\|release> [args]` |
| Bereich/Aktion | `bereich=factory`, `aktion=slots` (zweistufiges Dispatch-Muster der Parent-Spec) |
| Shared Library | nutzt `scripts/lib/vda-core.sh` (View-/Output-Helfer) |
| UI-Dependency | **KEIN fzf** — nur bash-builtins + vda-core (small-model-kompatibel) |
| Wrapping | **WRAPPT `scripts/factory/slots.sh`** — delegiert, ersetzt NICHT. `slots.sh` bleibt SSOT für die race-freie SQL-Logik (risiko-arm). |
| Output | Einheitlich: **JSON für IPC** (`--json`/`VDA_JSON=1`), **Plaintext** human-readable (Default). |
| Risiko | risiko-arm — keine Änderung an `slots.sh`-Semantik, nur ein dünner Adapter davor. |

### Warum Wrappen statt Ersetzen

`slots.sh` enthält die **race-freie** Claim-Logik (`UPDATE … WHERE pipeline_slot IS NULL …
RETURNING …`), die vom FA-SF-23-Test als atomar abgesichert ist und vom Dispatcher
produktiv genutzt wird. Diese Logik in `vda.sh` neu zu implementieren wäre risikoreich und
würde die SSOT duplizieren. Der Wrapper ist ein dünner Adapter, der nur Eingabe normalisiert,
`slots.sh` aufruft und die Ausgabe in das einheitliche VDA-Format bringt — `slots.sh` bleibt
unverändert die einzige Quelle der Slot-Semantik.

## 4. Befehls-Kontrakt

```
vda.sh factory slots count   [--brand <b>] [--json]   # belegte Slots dieser Brand
vda.sh factory slots next    [--brand <b>] [--json]   # niedrigster freier Slot 1..N, leer wenn voll
vda.sh factory slots claim   <ext_id> <n> [--brand <b>] [--json]   # atomar, echot n bei Erfolg
vda.sh factory slots release <ext_id> [--brand <b>] [--json]
vda.sh factory slots help
```

- **Brand-Auflösung:** `--brand <b>` setzt `BRAND=<b>` für den delegierten `slots.sh`-Aufruf.
  Ohne `--brand` wird die bestehende `BRAND`-Env durchgereicht (kein Default-Brand-Literal
  im Code — S3-konform). Fehlt `BRAND` ganz, schlägt `slots.sh` mit seinem eigenen Fehler fehl
  (durchgereicht).
- **Exit-Codes (durchgereicht von slots.sh):** `0` ok, `1` claim-failed, `2` usage/error.
- **Output-Modi:**
  - **Plaintext (Default):** human-readable über `vda-core`-Helfer
    (`count` → `vda_section "Belegte Slots" "<n>"`; `next` → `vda_section "Nächster freier Slot" "<n|—>"`;
    `claim`/`release` → `vda_success`).
  - **JSON (`--json` oder `VDA_JSON=1`):** einheitliches IPC-Format via `vda_json`, z.B.
    `{"action":"count","brand":"mentolder","value":"2"}`,
    `{"action":"claim","brand":"mentolder","ext_id":"T000866","slot":"1","ok":"true"}`,
    `{"action":"claim","ok":"false","error":"claim failed (already slotted …)"}`.

## 5. Dispatch-Architektur

```
vda.sh
  └─ case factory)              # NEU: factory-Bereich
       └─ exec scripts/vda/factory.sh "<aktion>" "$@"
            └─ case slots)      # aktion=slots
                 └─ exec scripts/vda/factory/slots.sh "$@"   # der Wrapper
                      └─ delegiert: BRAND=<b> bash scripts/factory/slots.sh <cmd> [args]
                      └─ formatiert Output (plaintext | JSON via vda-core)
```

Zwei neue Dateien:
1. `scripts/vda/factory.sh` — Bereichs-Dispatcher (`factory <aktion>`), routet `slots` →
   `scripts/vda/factory/slots.sh`. Schlank, erweiterbar für künftige factory-Aktionen.
2. `scripts/vda/factory/slots.sh` — der dünne Wrapper um `scripts/factory/slots.sh`.

Ein Edit:
3. `scripts/vda.sh` — neuer `case factory)`-Zweig + Help-Zeile.

## 6. Datenfluss (Dispatcher-IPC-Beispiel)

```
Dispatcher / Agent
  └─ vda.sh factory slots next --brand mentolder --json
       └─ scripts/vda/factory.sh slots next --brand mentolder --json
            └─ scripts/vda/factory/slots.sh next --brand mentolder --json
                 └─ BRAND=mentolder bash scripts/factory/slots.sh next   →  "2"
                 └─ vda_json action=next brand=mentolder value=2
                 └─ stdout: {"action":"next","brand":"mentolder","value":"2"}
```

## 7. Quality Gates

| Gate | Check |
|------|-------|
| S1 (File Size) | `vda.sh`-Edit klein (Ist 71, Limit `.sh`=500). `factory.sh` + `factory/slots.sh` jeweils < 80 Z (neu, Limit 500). vda-core unverändert. |
| S2 (Import Cycles) | `factory/slots.sh` sourct nur `vda-core.sh` (pure Helfer, keine DB-Schicht) + delegiert via Subprozess an `slots.sh` (kein Source-Zyklus). |
| S3 (No Brand Literals) | Kein `*.mentolder.de`/`*.korczewski.de`-Literal; `--brand` reicht Wert durch, kein Default-Brand-String im Code. |
| S4 (No Orphans) | `factory.sh` + `factory/slots.sh` sind über `vda.sh`-Dispatch erreichbar + von neuem BATS-Test referenziert + in `vda.sh help`/`factory help` gelistet. |
| Small-Model | Kein fzf/gum/dialog. Wrapper nutzt nur bash-builtins + vda-core. |

## 8. Erfolgskriterien

- `vda.sh factory slots count --brand mentolder` → gibt belegte Slot-Zahl (plaintext) aus,
  identisch zur `slots.sh count`-Semantik.
- `vda.sh factory slots next --brand mentolder --json` → `{"action":"next","brand":"mentolder","value":"<n>"}`.
- `vda.sh factory slots claim <ext> <n> --json` bei Erfolg `…"ok":"true","slot":"<n>"`, bei
  bereits belegtem Ticket Exit 1 + `…"ok":"false"…` (delegierte slots.sh-Semantik unverändert).
- `vda.sh factory slots bogus` → Exit 2 (durchgereicht von slots.sh).
- `vda.sh factory bogus` → Exit 2 (factory-Bereichs-Dispatcher).
- `scripts/factory/slots.sh` bleibt **unverändert** und weiterhin direkt aufrufbar (Dispatcher-JS
  unberührt); FA-SF-23 bleibt grün.
- Neuer Offline-BATS-Test `tests/unit/vda-factory-slots.bats` deckt Dispatch + Output-Format
  (FACTORY_DRY_RESOLVE-basiert, keine Cluster-Abhängigkeit) ab und ist grün.

## 9. Out of Scope

- `scripts/factory/slots.sh` selbst (SSOT, bleibt unverändert).
- Dispatcher-JS (`scripts/factory/*.js`) — ruft weiterhin `slots.sh` direkt; kein Umbau in dieser PR.
- Weitere `factory`-Aktionen (z.B. `factory prep`, `factory watchdog`) — nur `slots` in dieser PR;
  `factory.sh` ist aber so geschnitten, dass künftige Aktionen leicht hinzugefügt werden können.
- Deprecation/Aliasing von `slots.sh` — bleibt direkt erreichbar.
