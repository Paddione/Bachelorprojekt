# T000895 — Staging-Umgebung (`dev.*`-Subdomains im fleet-Cluster)

**Ticket:** T000895
**Status:** spec
**Datum:** 2026-06-17
**Branch:** `feature/t000895`

## 1. Problem / Motivation

Änderungen gehen heute direkt von der lokalen k3d-Iteration auf die produktiven Brands
(`mentolder` / `korczewski`) auf dem `fleet`-Cluster. Es fehlt eine **produktionsnahe
Staging-Stufe** auf demselben Cluster, in der Manifest-, OIDC-, TLS- und Schema-Änderungen
unter realen fleet-Bedingungen (echte cert-manager-DNS-01-Zertifikate, echter Traefik-Edge,
echte SealedSecrets, fleet-Netzwerk) verifiziert werden, **bevor** sie eine der beiden Brands
berühren.

Lokales k3d testet Funktionalität, aber nicht die fleet-spezifischen Footguns
(Wildcard-Cert-Issuance, wg-fleet-Pod-Routing, NetworkPolicy-Egress, SealedSecrets-Decrypt,
push-deploy-`envsubst`). Genau diese Klasse von Fehlern hat in der Vergangenheit wiederholt
erst in Prod zugeschlagen (frische-Cluster-Deploy-Bugs, OIDC-Secret-fehlt, Wildcard-Cert
won't issue).

## 2. Entscheidung (final, vom User bestätigt)

- **Eigener Namespace `workspace-staging`** auf dem bestehenden `fleet`-Cluster.
- **Eigene, voll isolierte `shared-db`-Instanz** in diesem Namespace (NICHT geteilt mit
  `workspace` / `workspace-korczewski`). Begründung: DSGVO/Datentrennung — Staging-Daten
  und Prod-Daten dürfen sich physisch nicht vermischen. Da alle Services die DB über den
  **Service-Namen `shared-db`** (namespace-lokal) ansprechen, resolved derselbe Name in
  `workspace-staging` automatisch auf das eigene DB-Pod — kein Hostname-Rewrite in den
  Service-Manifesten nötig.
- **`dev.*`-Subdomains** als Staging-Hostnamen (Schema: `*.staging.<brand-domain>`, z.B.
  `web.staging.<domain>`, `auth.staging.<domain>`, `files.staging.<domain>`). Realisiert über
  den bestehenden Traefik-IngressRoute/Ingress-Mechanismus + **cert-manager WILDCARD-TLS**
  (`*.staging.<domain>`) via DNS-01 (ipv64), genau wie die Prod-Wildcards.
- **Eigenes Kustomize-Overlay `prod-fleet/staging/`** — wrappt die `k3d/`-Base (analog zur
  Art, wie `prod-fleet/<brand>` die Brand-Overlays wrappt) und layert Staging-Patches:
  Namespace `workspace-staging`, Staging-Domains, eigene Wildcard-Certificate-Ressource,
  fleet-Node-Affinity-Repoints. **Kein** geteilter cluster-scoped Singleton-Konflikt
  (ClusterIssuer/IngressClass werden via `fleet-common`-Komponente per `$patch: delete`
  entfernt, da `prod-fleet/platform` sie einmalig besitzt).
- **Eigene SealedSecrets** (`environments/sealed-secrets/staging.yaml`), erzeugt aus
  plaintext `environments/.secrets/staging.yaml` über `task env:seal ENV=staging`.
- **Push-Deploy über neues `ENV=staging`** (`environments/staging.yaml`), gefahren über die
  bestehenden ENV-getriebenen Tasks (`workspace:deploy`, `env:seal`, `env:validate`, …).
- **Höchster Footprint ist akzeptiert** — Staging spiegelt den vollen Service-Stack
  produktionsnah (kein abgespeckter Subset).

## 3. Scope

### In Scope
- `environments/staging.yaml` — vollständige Env-Registry (Domains, Kontext `fleet`,
  Namespace `workspace-staging`, SMTP/Mail-Sink, alle `*.staging.<domain>`-Hosts).
- `prod-fleet/staging/kustomization.yaml` — neues Overlay, wrappt `../../k3d` (bzw. eine
  passende Base) + `fleet-common`-Komponente + Staging-Patches.
- Staging-Wildcard-`Certificate` (`*.staging.<domain>`) + ein `TLS_SECRET_NAME` für Staging.
- `environments/.secrets/staging.yaml` (gitignored) + `environments/sealed-secrets/staging.yaml`
  (committed, verschlüsselt).
- Keycloak-Realm für Staging (`realm-workspace-staging.json` analog Prod-Realms) mit
  Redirect-URIs auf die `*.staging.<domain>`-Hosts.
- Schema-Registrierung neuer Vars in `environments/schema.yaml` und der `envsubst`-Listen
  in `Taskfile.yml` für alle Tasks, die Staging-Manifeste bauen.
- Doku: kurzer Runbook-Abschnitt „Staging deployen/verifizieren" + CLAUDE.md-Gotcha-Eintrag.
- Test: BATS-Coverage für `env-resolve`/`env:validate` gegen `ENV=staging` (Erweiterung
  bestehender Tests, keine neue Datei wenn vermeidbar).

### Out of Scope
- Kein eigener separater Cluster (Staging lebt auf `fleet`).
- Keine automatische Prod→Staging-Datensynchronisation (Staging-DB startet leer/geseedet;
  optionaler Refresh-Job ist ein Folge-Ticket).
- Keine Änderung am Prod-Deploy-Verhalten der beiden Brands.
- Kein GitOps-Reconciler (bleibt push-based).
- Kein eigener GPU-/LLM-Host (Staging zeigt auf denselben `LLM_HOST_IP` oder hat
  `LLM_ENABLED=false` — Entscheidung im Plan).

## 4. Architektur

```
fleet-Cluster (pk-hetzner-4/6/8 + gekko-2/3/4)
├── ns workspace            → mentolder brand (prod)        [unverändert]
├── ns workspace-korczewski → korczewski brand (prod)       [unverändert]
└── ns workspace-staging    → STAGING (neu)
    ├── shared-db (eigenes Pod+PVC, isoliert)
    ├── keycloak (realm-workspace-staging)
    ├── nextcloud / vaultwarden / docs / website-schema / …
    ├── Certificate  *.staging.<domain>  (DNS-01 ipv64)
    └── Traefik IngressRoutes  *.staging.<domain>

prod-fleet/staging/                  ← neues Overlay (ENV_OVERLAY)
  resources: ../../k3d  (+ ggf. prod/ gemeinsame Patches)
  components: ../components/fleet-common
  patches: namespace=workspace-staging, staging-domains,
           staging-wildcard-cert, fleet node-affinity repoints

prod-fleet/platform/                 ← unverändert: besitzt ClusterIssuer/IngressClass
                                        einmalig (Staging nutzt denselben ClusterIssuer)
```

### Namensauflösung / Isolation
- Services sprechen die DB als `shared-db` (kurzer Service-Name) an → in `workspace-staging`
  resolved das auf das **staging-eigene** Pod. Keine Cross-Namespace-Connection.
- Die `netpol-deny-cross-brand`-NetworkPolicy aus `fleet-common` greift mit
  `${WORKSPACE_NAMESPACE}`=`workspace-staging` und isoliert Staging analog zu den Brands.
- DNS-01-Wildcard `*.staging.<domain>` wird über denselben `letsencrypt-prod`-ClusterIssuer
  ausgestellt (ipv64-DNS-Solver) — kein neuer Issuer nötig.

### Domain-/Hostnamen (S3-konform)
Alle Hosts kommen aus `environments/staging.yaml` (`*_DOMAIN`/`*_HOST`-Vars) und werden via
`envsubst` in die Manifeste gesetzt — **keine** Brand-Domain-Literale in Manifesten/Snippets.
Schema: jeder Prod-Host `<svc>.<domain>` bekommt das Staging-Pendant `<svc>.staging.<domain>`.

## 5. Constraints / Gotchas (aus CLAUDE.md)

- **Fresh-namespace Deploy-Reihenfolge:** SealedSecrets-Controller existiert bereits auf fleet;
  Reihenfolge für Staging: `env:fetch-cert` → `env:seal ENV=staging` → `cert:secret` (ipv64-Key
  auch in `workspace-staging`) → `workspace:deploy ENV=staging`.
- **`prod/`-Base nie direkt anwenden** — Staging-Overlay wrappt die Base und reused ggf.
  gemeinsame `prod/`-Patches; der `$patch: delete` auf `workspace-secrets` muss erhalten bleiben,
  damit SealedSecrets überleben.
- **`envsubst`-Var-Listen sind pro Task hardcodiert** — jede neue `${VAR}` in einem
  Staging-Manifest MUSS in `environments/schema.yaml` UND der `envsubst`-Liste jeder bauenden
  Task registriert werden.
- **`env-resolve.sh` wird gesourct, nie ausgeführt.**
- **`WORKSPACE_NAMESPACE` durchziehen** — alle ENV-sensiblen Tasks/Skripte nutzen
  `${WORKSPACE_NAMESPACE:-workspace}`; Staging setzt ihn auf `workspace-staging`.
- **LiveKit hostNetwork/DNS-Pin** — Staging-LiveKit braucht entweder einen eigenen Pin oder
  wird (höchster Footprint, aber LiveKit-Pin ist node-exklusiv) auf `LLM_ENABLED`-Analogie
  deaktiviert/auf einen freien Node gepinnt. Entscheidung im Plan (Default: LiveKit auf
  Staging deaktivieren, da nur ein hostNetwork-Slot pro Node frei ist — Prod belegt pk-4/pk-6).

## 6. Akzeptanzkriterien

1. `task env:validate ENV=staging` ist grün (alle Schema-Vars erfüllt).
2. `kustomize build prod-fleet/staging | envsubst` rendert valide (server-side dry-run grün),
   alle Ressourcen tragen `namespace: workspace-staging`.
3. Es gibt KEINE Brand-Domain-Literale in `prod-fleet/staging/` (S3-Gate grün).
4. `prod-fleet/staging/kustomization.yaml` referenziert seine neuen Manifeste; keine
   Orphan-Manifeste/-Skripte (S4-Gate grün).
5. SealedSecret `environments/sealed-secrets/staging.yaml` existiert und entschlüsselt unter
   dem fleet-Controller.
6. Nach Deploy: `workspace-staging` läuft mit eigener `shared-db`, Wildcard-Cert
   `*.staging.<domain>` ist `Ready`, mindestens `web.staging.<domain>` + `auth.staging.<domain>`
   liefern HTTP 200/302 über echtes TLS.
7. Prod-Brands (`workspace`, `workspace-korczewski`) bleiben unverändert (keine
   Pod-Restarts/Diffs durch die Staging-Einführung).
8. `task test:all`, `task freshness:regenerate`, `task freshness:check` grün; S1–S4-Gates grün.

## 7. Risiken

| Risiko | Mitigation |
|---|---|
| Cross-namespace DB-Leak | Service-Name `shared-db` ist namespace-lokal; NetworkPolicy `deny-cross-brand` greift |
| Wildcard-Cert issuet nicht (DNS-01) | Gleicher bewährter `letsencrypt-prod`-Issuer; `*.staging`-DNS-Eintrag im ipv64-Panel anlegen |
| LiveKit hostNetwork-Slot-Konflikt | Staging-LiveKit deaktivieren oder dedizierten Node pinnen (Plan-Entscheidung) |
| Ressourcen-Footprint auf fleet | Akzeptiert (User); ggf. niedrigere replicas/requests im Staging-Patch |
| `envsubst`-Var vergessen | Plan-Task erzwingt Schema+Taskfile-Registrierung + `env:validate` |
