# T000897 — One-Click Cluster-Reset für dev

**Ticket:** T000897
**Status:** spec
**Datum:** 2026-06-17
**Pfad:** feature
**Branch:** feature/t000897

## Problem

Ein vollständiger Neuaufbau des lokalen dev-k3d-Clusters ist heute ein mehrstufiger,
fehleranfälliger Handlauf: `task cluster:delete` → `task cluster:create` →
`task cert:install ENV=dev` → `task workspace:deploy ENV=dev` →
`task workspace:office:deploy ENV=dev`. Jeder Schritt muss einzeln, in der richtigen
Reihenfolge, mit dem korrekten `ENV=dev` aufgerufen werden. Vergisst man einen Schritt
oder die Reihenfolge, kommt der Stack nicht sauber hoch (z. B. cert-manager-CRDs fehlen,
bevor ein Manifest sie referenziert; `workspace-office` wird vor `workspace` deployt).

Beim Reset gehen außerdem die in den k3d-Knoten importierten Docker-Images
(`website`, `brett`, `docs`) verloren — der frische Cluster hat sie nicht mehr. Ohne
Re-Import schlagen die Pods mit `ErrImageNeverPull` fehl (k3d zieht `:latest`-Images
mit `imagePullPolicy: Never` nicht aus einer Registry). Ein voller Rebuild dieser drei
Images dauert mehrere Minuten und ist beim reinen „Cluster zurücksetzen" meist
unnötig — die vorhandenen lokalen Docker-Images sind aktuell genug.

## Ziel

Ein einziger, idempotenter Befehl `task dev:reset`, der den lokalen dev-k3d-Cluster
komplett neu aufbaut — Cluster löschen, neu erstellen, cert-manager-CRDs installieren,
Workspace deployen, Office-Stack deployen — und dabei die vorhandenen lokalen
`website`/`brett`/`docs`-Docker-Images per Default **wiederverwendet** (re-import statt
rebuild). Ein Flag `REBUILD=1` erzwingt den Neubau dieser Images.

## Nicht-Ziele

- **Kein** Sealed-Secrets-Flow. dev nutzt die Plaintext-Secrets aus `k3d/secrets.yaml`
  bzw. `environments/dev.yaml`; im lokalen k3d wird **kein** Sealed-Secrets-Controller
  installiert. Der dokumentierte Sealed-Secrets-Bring-up (CLAUDE.md „Cluster reset /
  fresh cluster bring-up order") gilt ausschließlich für Prod-Cluster, nicht für dev.
- **Kein** echter ACME-/Wildcard-Cert-Issue in dev. cert-manager-CRDs werden installiert
  (damit Manifeste, die `cert-manager.io`-Ressourcen referenzieren, applybar bleiben),
  aber es wird kein `cert:secret` und kein DNS-01-Issue erzwungen — dev nutzt
  Traefik-Default-Self-Signed bzw. plain HTTP auf `localhost`.
- **Kein** Prod-Reset. Der Task ist hart auf `ENV=dev` / den dev-k3d-Cluster gepinnt und
  verweigert die Ausführung gegen einen nicht-dev-Kontext.
- **Kein** Talk/Recording/Transcriber/MCP-Post-Setup. `dev:reset` bringt den Kern-Stack
  (Workspace + Office) hoch; optionale Post-Setup-Schritte bleiben separat
  (`task workspace:up` / `task workspace:setup` decken das weiterhin ab).

## Entscheidungen (final)

| Thema | Entscheidung |
|---|---|
| Einstiegspunkt | Neuer Task `dev:reset` in `Taskfile.yml` |
| Reihenfolge | `cluster:delete` → `cluster:create` → cert-manager-CRDs → `workspace:deploy ENV=dev` → `workspace:office:deploy ENV=dev` |
| Secrets | dev-only, **Plaintext** aus `k3d/secrets.yaml`/`environments/dev.yaml`; **kein** Sealed-Secrets im lokalen k3d |
| Image-Strategie (Default) | Vorhandene lokale Docker-Images `website`/`brett`/`docs` **re-importieren** (kein Rebuild) |
| Image-Strategie (`REBUILD=1`) | Images über die bestehenden `*:build:import`-Tasks neu bauen + importieren |
| Idempotenz | Ein Befehl, beliebig oft wiederholbar; löscht/neu-erstellt sauber, kein Doppel-Cluster |
| Bestätigung | `CONFIRM=yes` oder `--yes` umgeht den interaktiven Prompt für Skript-/CI-Nutzung |

## Architektur

`task dev:reset` ist ein **dünner Taskfile-Wrapper** um ein neues Helper-Skript
`scripts/dev-reset.sh`. Die Orchestrierungs-Logik (Reachability-Guard, Image-Reuse-vs-Rebuild,
Ablauf-Reihenfolge, Idempotenz) lebt im Skript — damit bleibt sie BATS-testbar und das
Taskfile wächst nur minimal.

```
task dev:reset [REBUILD=1] [CONFIRM=yes]
   └─ scripts/dev-reset.sh
        1. Guard: ENV=dev pinnen; CLUSTER_NAME aus Taskfile-Var; abbrechen wenn aktiver
           kube-context ein Prod-Kontext ist (fleet/…)
        2. Bestätigung (übersprungen bei CONFIRM=yes/--yes)
        3. Image-Snapshot: vor dem Löschen prüfen, welche der lokalen Docker-Images
           ghcr.io/paddione/{workspace-website,workspace-brett,workspace-docs}:latest
           existieren (für späteren Re-Import)
        4. task cluster:delete  (idempotent: k3d cluster delete … || true)
        5. task cluster:create
        6. cert-manager-CRDs installieren (CRDs-only Apply — kein voller cert:install,
           da dev keinen ACME-Issue braucht)
        7. REBUILD=1 → task website:build:import / brett:build / docs:build:import (ENV=dev)
           sonst → vorhandene Docker-Images per `k3d image import` re-importieren
        8. task workspace:deploy ENV=dev
        9. task workspace:office:deploy ENV=dev
       10. Abschluss-Hinweis (URLs, Pod-Status-Check)
```

### Image-Reuse-Detail

- Default: für jedes der drei Images prüft das Skript `docker image inspect <img>:latest`.
  - existiert → `k3d image import <img>:latest -c $CLUSTER_NAME`
  - existiert **nicht** → genau dieses eine Image über seinen `*:build:import`-Task bauen
    (Fallback, damit der Stack auch nach einem `docker image prune` hochkommt) und eine
    Warnung ausgeben.
- `REBUILD=1`: alle drei Images über die bestehenden Build-Tasks neu bauen + importieren.

Damit ist „re-import vorhandener Images, kein Rebuild" der Default und `REBUILD=1` der
explizite Override — exakt wie in den finalen Ticket-Entscheidungen festgelegt.

### Guards & Idempotenz

- **Prod-Schutz:** Bricht ab, wenn der aktive `kubectl config current-context` ein
  bekannter Prod-Kontext ist (`fleet`) oder nicht der k3d-dev-Kontext — verhindert ein
  versehentliches Löschen eines Prod-Clusters. `cluster:delete`/`cluster:create` arbeiten
  ohnehin nur über `k3d` gegen `$CLUSTER_NAME`, aber der Guard ist die zweite Sicherung.
- **Docker-Reachability:** Bricht früh ab, wenn `docker info` fehlschlägt.
- **Idempotenz:** `cluster:delete` ist mit `… || true` no-op-sicher, wenn kein Cluster
  existiert; `cluster:create` baut frisch. Wiederholtes `task dev:reset` führt immer zum
  selben Endzustand.

## Betroffene Dateien

| Datei | Art | Anmerkung |
|---|---|---|
| `Taskfile.yml` | ändern | neuer Task `dev:reset` (dünner Wrapper, ~15 Zeilen) |
| `scripts/dev-reset.sh` | neu | Orchestrierungs-Skript (~120–160 Zeilen, Limit 500) |
| `tests/unit/scripts/dev-reset.test.sh` | neu | BATS-Tests (Guard, Image-Reuse-Logik, Reihenfolge — alles offline/mockbar) |
| `docs/dev-stack/README.md` o. ä. | ändern (optional) | Doku-Zeile „One-Click-Reset: `task dev:reset`" |

Keine Änderung an `k3d/configmap-domains.yaml`, `environments/schema.yaml` oder
`k3d/kustomization.yaml` — der Task orchestriert nur bestehende Tasks/Manifeste, fügt
keine neuen `${VAR}`-Referenzen, Domains oder Manifeste hinzu.

## Quality-Gates (S1–S4)

- **S1 (Zeilen):** `Taskfile.yml` ist `.yml` → kein S1-Limit. `scripts/dev-reset.sh` neu,
  `.sh`-Limit 500, Zielgröße ~160 → reichlich Reserve. `*.test.sh` neu, ebenfalls weit
  unter Limit.
- **S2 (Import-Zyklen):** reines Bash, kein TS-Graph betroffen.
- **S3 (Hostnamen):** keine Brand-Domain-Literale im Skript; alle Hosts kommen aus
  `localhost` (dev) bzw. werden gar nicht referenziert. `CLUSTER_NAME` aus Taskfile-Var.
- **S4 (Orphans):** `scripts/dev-reset.sh` wird vom neuen `dev:reset`-Task aufgerufen
  (referenziert) und in der Doku erwähnt → kein Orphan. `*.test.sh` wird in
  `task test:unit:scripts` (bzw. eigenes `test:unit:*`-Target) eingehängt.

## Verifikation

- `task dev:reset CONFIRM=yes` baut den Cluster real neu (manueller Smoke; benötigt
  Docker + k3d) → alle Kern-Pods `Running`, `http://localhost:8080` erreichbar.
- `task dev:reset REBUILD=1 CONFIRM=yes` baut Images neu.
- Offline-Gate: `task test:all`, `task freshness:regenerate`, `task freshness:check`
  (im finalen Plan-Task).
