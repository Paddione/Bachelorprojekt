# Dependency-Update-Bot (Renovate) — self-hosted GitHub Action

## Problem / Warum

Es gibt im Repo bereits eine `renovate.json` (config:recommended), aber **keinen Workflow,
der Renovate ausführt**. Das heißt: die Konfiguration ist tot — es laufen keine
Dependency-Update-PRs. Updates über alle Workspaces (zwei pnpm-Workspaces, Docker-Base-Images,
GitHub-Actions-Pins) müssen heute manuell per `update-dependencies`-Skill durchgezogen werden.

Wir wollen automatisierte, gruppierte, planbare Dependency-Update-PRs — **agentic-first und
Factory-observable**: als GitHub Action (`renovatebot/github-action`), nicht als In-Cluster-CronJob.
Ein Workflow im Repo ist über die GitHub-API beobachtbar (Run-Status, PR-Liste), belegt keine
Cluster-Ressourcen und braucht keinen Zugriff auf Kubeconfig/Secrets der Fleet.

## Entscheidungen (final, agentic-first)

1. **Renovate als GitHub Action**, nicht als In-Cluster-CronJob.
   - Neue Datei `.github/workflows/renovate.yml`, self-hosted via `renovatebot/github-action`.
   - Vorteil: Factory-observable (Workflow-Runs + PRs über GitHub-API), keine Cluster-Last,
     kein zusätzlicher Secret-Pfad in die Fleet.
2. **Auth über dedizierte GitHub App** (App-ID + Private Key oder PAT als Fallback).
   - Token wird als Repo-Secret `RENOVATE_TOKEN` bereitgestellt (plus optional App-ID/Key,
     siehe Sicherheits-Abschnitt). Das Repo nutzt bereits App-Auth-Muster (siehe `opencode.yml`).
3. **`renovate.json5`** ersetzt die bestehende `renovate.json`.
   - JSON5 erlaubt Kommentare → Begründung jeder Regel inline (Wartbarkeit).
   - Inhaltlich erweitert: gruppierte Updates, **wöchentlicher** Schedule, **automerge nur für
     `patch` + `devDependencies`**, beide pnpm-Workspaces (`website`, `arena-server`/`brett`),
     Docker-Base-Images und GitHub-Actions-Pins. **SHA-Pinning bleibt erhalten** (Actions
     `pinDigests: true`).
4. Die `renovate.json` wird **gelöscht** und durch `renovate.json5` ersetzt (kein Dual-Config —
     Renovate würde sonst nur eine der beiden lesen, Verwirrung).

## Scope der Manager (was Renovate anfasst)

| Manager | Pfade | Update-Strategie |
|---|---|---|
| `npm` (pnpm) | `website/`, `arena-server/`, `brett/`, root `package.json` u.a. | gruppiert pro Workspace-Dir; automerge nur patch + devDeps |
| `dockerfile` / `kubernetes` | `k3d/*.yaml`, `prod*/*.yaml` Base-Images | gruppiert pro Image-Familie (livekit/nextcloud/keycloak), major nie automerge |
| `github-actions` | `.github/workflows/*.yml` | `pinDigests: true` (SHA-Pin **beibehalten**), gruppiert |

**Ignoriert** bleiben (wie heute): `node_modules`, `art-library/**`, vendored/experimentelle
Trees (`scripts/coaching`, `scripts/knowledge`, `tests/e2e`, `tests/unit`), eigene
`ghcr.io/paddione/*`-Images (per `:latest` rebuild, kein Renovate-Bump) und
`fedirz/faster-whisper-server` (manuell gepinnt).

## Automerge-Policy (präzise)

Automerge **nur** wenn **beide** Bedingungen gelten:
- Update-Typ ist `patch`, **oder** das Paket ist eine `devDependency` (jeder nicht-major Typ), **und**
- `minimumReleaseAge` (z.B. „3 days") ist erfüllt (kein same-day-Merge frischer Releases).

**Nie automerge:** `major` (alle Manager), Production-`dependencies` mit `minor`/`major`,
Docker-`major`, GitHub-Actions-`major`. Diese öffnen nur einen PR zur Review.

> Automerge respektiert die bestehende Branch-Protection (Required Checks: Offline Tests,
> Security Scan, Brett TypeScript, Vitest, Conventional Commits). Ein automerge-fähiger PR
> wird trotzdem erst grün gemergt — Renovate setzt nur das Auto-Merge-Flag.

## Schedule

- **Wöchentlich**, z.B. `before 9am on monday` (Europe/Berlin via `timezone`).
- `prConcurrentLimit` / `prHourlyLimit` moderat setzen, um PR-Flut zu vermeiden.
- Dependency-Dashboard aktiviert (ein Issue als Übersicht aller pendenten Updates).

## Sicherheit

- Workflow ist **nicht** durch beliebige User triggerbar: `schedule` + `workflow_dispatch`
  (nur Repo-Mitglieder können `workflow_dispatch` auslösen). Kein `issue_comment`-Trigger.
- `renovatebot/github-action` wird **SHA-gepinnt** referenziert (gleiche Supply-Chain-Politik
  wie `opencode.yml`: secret-tragende Third-Party-Action nie `@latest`).
- `RENOVATE_TOKEN` (bzw. App-Credentials) sind Repo-Secrets, nie im Klartext im Workflow.
- Conventional-Commits-Gate: Renovate-Commits müssen `chore(deps): …` / `fix(deps): …`
  matchen → `commitMessagePrefix`/`semanticCommits: enabled` setzen, sonst rotes
  „Conventional Commits"-Required-Check und kein Merge.

## Nicht-Ziele

- Kein In-Cluster-Renovate, kein Mend-Cloud-App (self-hosted Action).
- Keine Änderung an der Branch-Protection-Konfiguration.
- Kein automatisches Mergen von Production-`dependencies`-minor/major.
- Keine Migration von pnpm-Versionen oder Node-Engine-Bumps in diesem Ticket.

## Abnahme-Kriterien

1. `.github/workflows/renovate.yml` existiert, SHA-gepinnt, Trigger `schedule` + `workflow_dispatch`.
2. `renovate.json5` ersetzt `renovate.json` (Letztere gelöscht), valide gegen
   `renovate-schema.json`, mit Kommentaren je Regel.
3. Manager npm/docker(kubernetes)/github-actions konfiguriert; Ignores wie oben.
4. Automerge greift **nur** für patch + devDeps; major/prod-minor öffnen Review-PRs.
5. GitHub-Actions-Updates behalten `pinDigests: true` (SHA-Pin).
6. `semanticCommits: enabled` → Renovate-PRs sind conventional-commit-konform.
7. `task test:all` + `task freshness:check` grün; keine neue S1–S4-Violation.
