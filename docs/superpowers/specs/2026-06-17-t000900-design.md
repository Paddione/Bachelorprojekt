# T000900 — Log-Aggregation (Loki + Grafana) — Design-Spec

**Ticket:** T000900
**Status:** plan_staged
**Datum:** 2026-06-17
**Branch:** feature/t000900

## 1. Problem & Ziel

Der fleet-Cluster hat **keine zentrale Log-Aggregation**. Logs leben heute nur
flüchtig in den einzelnen Pods (`kubectl logs`), gehen bei Pod-Restart verloren und
sind nicht cluster-weit durchsuchbar. Bei Incident-Triage (siehe `incident-response`-Skill)
muss man Node für Node und Pod für Pod manuell graben.

**Ziel:** Eine zentrale, durchsuchbare Log-Plattform über **alle fleet-Nodes** und
**beide Brands** (mentolder + korczewski), als Read-UI integriert in das **bereits
existierende Grafana** (kube-prometheus-stack in der `monitoring`-Namespace).

## 2. Bestätigte Entscheidungen (User-final)

| Aspekt | Entscheidung |
|---|---|
| Storage-Backend | **Filesystem auf longhorn-PVC** — KEIN MinIO/S3 |
| Retention | **14 Tage** |
| Brand-Scope | **UNIFIED** über beide Brands — kein per-Brand-Split |
| Loki-Deployment | **Single-binary / monolithic mode** (SingleBinary, replicas=1) |
| Log-Collection | **promtail DaemonSet** über alle fleet-Nodes |
| Admin-Zugang | Über das **bestehende Grafana** (OIDC/SSO + Traefik-Ingress in prod) |

## 3. Bestehender Kontext (wiederverwenden, NICHT neu bauen)

Der Cluster hat bereits einen **vollständigen Observability-Stack** in `k3d/monitoring/`,
über `helm template` vorgerendert und committet:

- **Grafana** (`monitoring-grafana`, kube-prometheus-stack) — die Read-UI existiert schon.
  - Datasource-Discovery läuft über einen **`k8s-sidecar`**: jede ConfigMap mit Label
    `grafana_datasource: "1"` in der `monitoring`-Namespace wird automatisch als Grafana-
    Datasource provisioniert (`LABEL=grafana_datasource`, `METHOD=WATCH`, Reload via API).
  - **→ Loki muss NICHT in Grafana-Deployment eingreifen.** Eine gelabelte Datasource-
    ConfigMap genügt; der Sidecar lädt sie ohne Grafana-Neustart.
- **prod-Overlay** unter `prod/monitoring/` patcht dev→prod (storage-class local-path→longhorn,
  Ingress, OIDC, Resource-Limits). Loki folgt exakt diesem Muster.
- **Render-Task** `monitoring:render` rendert kube-prometheus-stack aus den Helm-Values.
  Loki bekommt einen analogen Render-Task.
- **NetworkPolicy** `monitoring-allow-egress` erlaubt bereits all-egress aus `monitoring`
  (promtail-DaemonSet-Pods laufen aber in `monitoring`-ns → siehe §5 Ingress-Regel).

## 4. Architektur

```
 fleet-Node 1        fleet-Node 2        fleet-Node 3   ... (alle Nodes)
 ┌──────────┐        ┌──────────┐        ┌──────────┐
 │ promtail │        │ promtail │        │ promtail │   DaemonSet (1 pro Node)
 │ (tail    │        │          │        │          │   liest /var/log/pods/*
 │  /var/log│        │          │        │          │   beider Brands (alle ns)
 └────┬─────┘        └────┬─────┘        └────┬─────┘
      │   push (HTTP /loki/api/v1/push)       │
      └──────────────┬───────────────────────┘
                     ▼
            ┌──────────────────┐
            │  Loki (single-   │   Deployment replicas=1, monolithic
            │  binary,         │   Service loki:3100
            │  filesystem TSDB)│   PVC (longhorn in prod, local-path in dev)
            └────────┬─────────┘   Retention 14d (compactor delete)
                     │ query (LogQL)
                     ▼
            ┌──────────────────┐
            │  Grafana (besteht)│  Datasource "Loki" via gelabelte ConfigMap
            │  + Explore / Logs │  → SSO/Traefik (prod), grafana.localhost (dev)
            └──────────────────┘
```

Alles in der **`monitoring`-Namespace** — keine neue Namespace, kein Brand-Split.
promtail erfasst Logs **aller** Namespaces (`workspace`, `workspace-korczewski`,
`website`, `website-korczewski`, `monitoring`, `cert-manager`, …) auf jedem Node, also
unified über beide Brands.

## 5. Komponenten im Detail

### 5.1 Loki (monolithic / SingleBinary)
- Helm-Chart `grafana/loki` (gerendert + committet, analog kube-prometheus-stack).
- `deploymentMode: SingleBinary`, `singleBinary.replicas: 1`.
- **Storage:** `storage.type: filesystem`, `schemaConfig` mit `store: tsdb`,
  `object_store: filesystem`. Persistenz auf PVC (`/var/loki`).
- **Retention 14d:** `limits_config.retention_period: 336h` + `compactor` mit
  `retention_enabled: true`, `delete_request_store: filesystem`.
- **Auth aus** (`auth_enabled: false`) — Single-Tenant, intern, durch NetworkPolicy +
  Grafana-SSO geschützt. Kein Multi-Tenancy / kein per-Brand-Tenant.
- Ressourcen-Limits konservativ (Plan §Tasks): requests cpu 100m/mem 256Mi.
- PVC dev `local-path` 5Gi; prod via storage-class-patch → `longhorn` (Größe in Plan
  festgelegt, z.B. 20Gi für 14d unified).

### 5.2 promtail (DaemonSet)
- Helm-Chart `grafana/promtail` (gerendert + committet) ODER als Teil des loki-stack —
  Plan entscheidet pro Render-Stabilität für **separates `grafana/promtail`-Chart**
  (loki-stack-Chart ist deprecated).
- DaemonSet → ein Pod pro fleet-Node, mountet `/var/log/pods` + `/var/lib/docker/containers`
  read-only, tailt alle Container-Logs.
- Pipeline: Kubernetes-Service-Discovery, Labels `namespace`, `pod`, `container`, `node`,
  `app`. **Brand-Label** abgeleitet aus der Namespace (`workspace*` → mentolder vs
  `*-korczewski` → korczewski) als optionales Query-Label — aber **ein** Loki, kein Split.
- Tolerations für Control-Plane-Nodes (pk-hetzner-4/6/8 sind CP) → promtail läuft auch dort.
- Push-Ziel: `http://loki.monitoring.svc:3100/loki/api/v1/push`.

### 5.3 Grafana-Datasource (ConfigMap, kein Grafana-Eingriff)
- ConfigMap `loki-grafana-datasource` in `monitoring`, Label `grafana_datasource: "1"`.
- Inhalt: Datasource `name: Loki`, `type: loki`, `url: http://loki.monitoring.svc:3100`,
  `access: proxy`, `isDefault: false`.
- Der bestehende Grafana-`k8s-sidecar` lädt sie automatisch (WATCH) → in Grafana unter
  **Explore** und einem optionalen **Logs-Dashboard** durchsuchbar.

### 5.4 NetworkPolicy
- promtail + Loki laufen in `monitoring`; bestehende `monitoring-allow-egress` (all-egress)
  und Ingress-Policy decken intra-ns ab. **Prüfen/ergänzen:** Loki-Service muss intra-ns
  von promtail + Grafana erreichbar sein (gleicher ns → durch `podSelector: {}`-Ingress
  bereits erlaubt). Keine cross-ns-Öffnung nötig (promtail liest Logs vom Node-Filesystem,
  nicht über die Pod-Netzwerke der Ziel-ns).

## 6. Storage-Sizing (14d unified)

Filesystem-TSDB, beide Brands, 14d. Annahme grob: aktuelle Pod-Anzahl ~52 (26+26) +
System-Pods. Loki komprimiert stark (chunks gzip). 20Gi longhorn-PVC ist großzügig
budgetiert; Plan dokumentiert die Annahme und macht die Größe per Patch leicht anpassbar.
Compactor löscht > 336h hart.

## 7. Deploy-Modell

Push-based (kein GitOps auf fleet). Loki-Manifeste landen in `k3d/monitoring/` (dev-Base)
und werden über `prod/monitoring/` für prod gepatcht. Deploy für beide Brands über das
bestehende monitoring-Kustomize (Loki ist **nicht** brand-spezifisch — eine Instanz in der
cluster-weiten `monitoring`-ns deckt beide Brands ab).

**Validation:** `task monitoring:validate` (build dev + beide prod-fleet-Overlays).

## 8. Out of Scope

- Kein MinIO/S3 (explizit ausgeschlossen).
- Kein Multi-Tenant-Loki / per-Brand-Loki-Split.
- Keine Alerting-Regeln auf Logs (Loki Ruler) — Phase 2 falls gewünscht.
- Keine Log-basierte Metrics/Recording-Rules.
- Kein Eingriff in das Grafana-Deployment selbst (nur additive Datasource-ConfigMap).

## 9. Risiken & Mitigation

| Risiko | Mitigation |
|---|---|
| promtail tailt Node-Logs → braucht hostPath + ggf. privilegierte SC | `monitoring`-ns ist bereits `pod-security: privileged` (siehe namespace.yaml) |
| Riesiger gerenderter YAML-Blob (wie kube-prometheus-stack 4.9MB) bläht Repo | loki+promtail sind klein; trotzdem als eigene Dateien rendern, nicht in den Prometheus-Blob |
| Loki PVC voll → Ingestion-Stop | Retention 14d + Compactor + großzügige PVC; monitoring-egress erlaubt Alertmanager bereits |
| longhorn nur in prod, dev = local-path | storage-class-patch im prod-Overlay (exakt wie Prometheus/Grafana heute) |
| CP-Nodes ohne promtail (Taints) | Tolerations im DaemonSet |
