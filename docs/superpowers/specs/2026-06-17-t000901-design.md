# T000901 — Autopilot-Qualitäts-Ratchet (DeepSeek-Scout verbessern)

## Kontext / Problem

Der Software-Factory-Autopilot (`scripts/factory/pipeline.js`) startet jeden Ticket-Build mit
einem **deterministischen Scout** (`scripts/factory/scout.sh`), der per grep/find die voraussichtlich
betroffenen Dateien (`touched_files`) ermittelt, die Komplexität klassifiziert und Risiko-Bereiche
taggt. Diese `touched_files`-Liste steuert nachgelagert die Modell-Wahl (Provisioning),
Conflict-Checks und den Diff-Budget-Guard.

Zwei Schwächen bestehen heute:

1. **Scout-Coverage ist dünn.** Die drei Discovery-Strategien (Keyword-grep, Filename-Pattern,
   Infra-Scan) finden bei vielen Tickets zu wenige oder gar keine Dateien. Ist `touched_files` leer,
   biased der Scout auf `complexity=medium` (Fallback), aber die nachgelagerten Stufen arbeiten dann
   praktisch blind. Es gibt **keinen LLM-gestützten Fallback** — der frühere DeepSeek-Scout wurde
   durch den deterministischen ersetzt (siehe `docs/superpowers/specs/2026-06-10-factory-scout-deterministic.md`).
2. **Es gibt keine Erfolgsmessung.** Niemand prüft nach dem Merge, **wie gut** die Scout-Vorhersage
   war: Welche Dateien hat der gemergte PR tatsächlich angefasst (`git diff --name-only`) gegenüber
   den vom Scout vorhergesagten `touched_files`? Ohne diese Rückkopplung lässt sich die Scout-Qualität
   weder beobachten noch verbessern.

Heutiger Quality-Check (`scripts/factory/scout-quality-check.cjs`) ist nur ein **Pre-Build-Gate**
(`SCOUT_WEAK`, wenn `touched_files` leer / Spec zu kurz). Es gibt keine **Post-Merge-Bewertung**
und keinen sich verschärfenden Ratchet.

## Ziel

Ein **Qualitäts-Ratchet** für den Scout, bestehend aus zwei unabhängigen, additiven Bausteinen:

- **Baustein A — Hybrid-Scout (besseres `touched_files`):** Die deterministischen Regex-/Pattern-
  Strategien in `scout.sh` ausbauen **und** einen optionalen DeepSeek-LLM-Call als **Fallback**
  ergänzen, der nur greift, wenn die deterministische Discovery weniger als `N` Dateien findet.
- **Baustein B — Drift-Ratchet (Erfolgsmessung):** Nach jedem Merge die vom Scout vorhergesagten
  `touched_files` gegen den echten `git diff --name-only` des PR vergleichen, einen **Drift-Score**
  berechnen und in der DB loggen. **Kein hartes Gate** zunächst — nur Logging plus eine Ratchet-
  Schwelle, die sich über die Zeit verschärft (Beobachtung first, Enforcement später).

## Nicht-Ziele

- Kein hartes CI-Gate / Build-Block aufgrund schlechter Scout-Qualität in dieser Iteration.
- Kein Ersatz des deterministischen Scouts durch einen reinen LLM-Scout — der LLM-Call ist nur
  **Fallback**, der deterministische Pfad bleibt primär und fail-soft.
- Keine Änderung an der Modell-Routing-/Provider-Logik (`route-provider.sh`, `provision.js`) selbst —
  der Hybrid-Scout nutzt die bestehende Provider-Resolution.
- Keine Brand-spezifische Drift-Auswertung in dieser Iteration (ein Score pro Ticket genügt).

## Entscheidungen (final)

- **Hybrid-Scout** = deterministische Patterns in `scout.sh` ausbauen UND optionaler LLM-Call
  (DeepSeek) als Fallback bei `< N` touched_files. Der LLM-Call lebt in einem **eigenen, puren
  Helper-Skript** (`scripts/factory/scout-llm-fallback.sh`), nicht inline in `scout.sh` — so bleibt
  `scout.sh` unter dem Zeilenlimit und der LLM-Pfad bleibt einzeln testbar/abschaltbar.
- **Schwelle `N`** ist via Env konfigurierbar (`SCOUT_LLM_MIN_FILES`, Default klein, z.B. `2`).
  Der Fallback ist **opt-in über Provider-Verfügbarkeit**: Ist kein DeepSeek-Provider auflösbar
  (`route-provider.sh factory-scout cheap` liefert nichts) oder `SCOUT_LLM_ENABLED=false`, wird der
  Fallback übersprungen — der deterministische Output (auch leer) bleibt gültig. **Fail-soft.**
- **Quality-Ratchet** = nach Merge Scout-`touched_files` vs. echter `git diff --name-only` des PR
  vergleichen, **Drift-Score** (0.0–1.0, niedriger = bessere Vorhersage) in der DB loggen.
- **Speicherort des Scores:** neue Spalte `scout_drift` (NUMERIC, nullable) auf `tickets.tickets`
  (konsistent mit `touched_files`, das dort bereits liegt) plus optional `scout_drift_at` (TIMESTAMP).
  Idempotente Migration unter `scripts/migrations/`.
- **KEIN hartes Gate zunächst** — nur Logging. Eine **Ratchet-Schwelle** (`SCOUT_DRIFT_THRESHOLD`,
  Default großzügig, z.B. `0.9`) wird gespeichert/dokumentiert und kann über die Zeit gesenkt werden;
  bei Überschreitung **nur ein Warn-Kommentar** am Ticket (kein Block, kein Status-Wechsel).

## Drift-Score-Definition

Gegeben:
- `P` = Menge der vom Scout vorhergesagten Dateien (`tickets.touched_files`, relativiert auf Repo-Root).
- `A` = Menge der tatsächlich im PR geänderten Dateien (`git diff --name-only <base>..<merge>`),
  gefiltert auf Code/Manifest-Pfade (Doku/generierte Artefakte ausgenommen — `docs/generated/**`,
  `docs/code-quality/repo-index.json`, `website/src/data/test-inventory.json`, Plan/Spec-Markdown,
  damit der Score nicht durch Plan-Commits verfälscht wird).

Drift-Score (Jaccard-Distanz auf der Dateimenge):

```
drift = 1 - |P ∩ A| / |P ∪ A|      (für |P ∪ A| > 0; sonst drift = 0)
```

- `drift = 0.0` → perfekte Vorhersage (P == A).
- `drift = 1.0` → komplette Fehlvorhersage (kein Overlap).
- `P` leer, `A` nicht-leer → `drift = 1.0` (Scout hat nichts gefunden, PR hat aber geändert).
- Beide leer → `drift = 0.0` (z.B. reiner Doku-PR; nichts zu bewerten).

Der Score ist eine **Metrik**, kein Gate. Er macht Scout-Qualität pro Ticket beobachtbar und liefert
die Datengrundlage, um `SCOUT_LLM_MIN_FILES` und die deterministischen Patterns iterativ zu tunen.

## Komponenten & Datenfluss

```
┌─────────────────────── Build (pipeline.js) ───────────────────────┐
│ scout.sh (deterministic discovery)                                │
│   └─ if |touched_files| < SCOUT_LLM_MIN_FILES && LLM enabled:      │
│        scout-llm-fallback.sh (DeepSeek via route-provider.sh)      │
│        → merge LLM-vorgeschlagene Pfade in touched_files (union)   │
│   └─ emit SCOUT_SCHEMA JSON                                        │
│ pipeline.js → ticket.sh set-touched-files (persist P in DB)       │
└───────────────────────────────────────────────────────────────────┘
                              │  (PR merged to main, T###### im Commit)
                              ▼
┌─────────────────── Post-Merge (post-merge.yml) ───────────────────┐
│ scout-drift.sh --ticket <id> --base <sha> --head <sha>            │
│   ├─ P  = ticket.sh get → touched_files                          │
│   ├─ A  = git diff --name-only base..head | filter-noise          │
│   ├─ drift = jaccard_distance(P, A)                               │
│   ├─ ticket.sh set-scout-drift --id <id> --drift <d>  (persist)   │
│   └─ if drift > SCOUT_DRIFT_THRESHOLD: ticket.sh add-comment (warn)│
└───────────────────────────────────────────────────────────────────┘
```

## Risiken & Mitigation

- **LLM-Fallback macht Builds langsam/teuer.** → Nur bei `< N` Dateien, Timeout (z.B. 20s),
  fail-soft (Fehler/Timeout → deterministisches Ergebnis behalten). Standardmäßig opt-in via Env.
- **DeepSeek-Halluzination (nicht-existente Pfade).** → LLM-Output gegen `test -e` im Repo filtern,
  bevor in `touched_files` gemergt wird; nur reale Dateien übernehmen.
- **Drift-Score durch Plan/Doku-Commits verfälscht.** → Noise-Filter in `scout-drift.cjs` schließt
  Plan-/Spec-Markdown, generierte Artefakte und Test-Inventar aus.
- **Post-Merge-Step bricht den Workflow.** → Wie `close-ticket` strikt **fail-soft**
  (`|| echo WARNING`), Drift-Logging darf nie den Push-to-main-Workflow rot machen.
- **`tickets.tickets` ist eine geteilte Tabelle (beide Brands).** Migration ist `ADD COLUMN IF NOT
   EXISTS` (idempotent) und muss auf **beide** Brand-DBs angewandt werden (post-merge dokumentiert).

## Akzeptanzkriterien

1. `scout.sh` ruft `scout-llm-fallback.sh` auf, wenn deterministische Discovery `< SCOUT_LLM_MIN_FILES`
   Dateien findet **und** ein Provider auflösbar ist; sonst Skip. Output bleibt SCOUT_SCHEMA-konform.
2. Ist kein DeepSeek-Provider verfügbar oder `SCOUT_LLM_ENABLED=false`, liefert `scout.sh` exakt das
   bisherige deterministische Ergebnis (kein Crash, Exit 0).
3. LLM-vorgeschlagene Pfade werden **nur** übernommen, wenn die Datei real existiert; Union mit dem
   deterministischen Set, dedupliziert, weiterhin auf 30 Dateien gekappt.
4. `scout-drift.cjs` berechnet die Jaccard-Distanz korrekt (Unit-Tests: P==A→0, disjunkt→1, P leer &
   A nicht-leer→1, beide leer→0, Teilüberlappung→korrekter Bruch) und filtert Noise-Pfade.
5. `scout-drift.sh` liest `P` aus dem Ticket, berechnet `A` aus `git diff --name-only base..head`,
   persistiert den Score via `ticket.sh set-scout-drift` und postet bei `drift > SCOUT_DRIFT_THRESHOLD`
   einen Warn-Kommentar — **ohne** Status-Wechsel/Block.
6. Migration `2026-06-17-scout-drift.sql` ist idempotent (`IF NOT EXISTS`) und legt `scout_drift`
   (+ `scout_drift_at`) auf `tickets.tickets` an.
7. `post-merge.yml` ruft den Drift-Step fail-soft nach `close-ticket` auf.
8. `task test:changed`, `task freshness:regenerate`, `task freshness:check` grün; neue Bats in
   `task test:all` (test:factory) erfasst; `test-inventory.json` regeneriert.
