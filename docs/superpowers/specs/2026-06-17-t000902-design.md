# T000902 — DeepSeek Code-Review-Qualität verbessern

**Status:** design
**Ticket:** T000902
**Datum:** 2026-06-17
**Branch:** feature/t000902

## Problem

Die DeepSeek-gestützte Code-Review im Factory-Pipeline (`ci-review.mjs` als CI-Action
`ai-review.yml` und die Verify-Phase in `pipeline.js`) produziert zu viele **false
positives**. Die Lens-Agenten (bug/security/pattern/perf) bekommen den **gesamten
gefilterten Diff** als Text, melden aber regelmäßig Findings, die:

1. **außerhalb der tatsächlich geänderten Zeilen** liegen (Kontextzeilen, benachbarter
   unveränderter Code, oder gar erfundene Zeilennummern),
2. **reine Stil-Nitpicks** sind (Naming, Formatierung — explizit als „NOT to flag"
   deklariert, aber DeepSeek V3/R1 hält sich nicht zuverlässig daran),
3. **bereits vorbestehende Issues** betreffen (Code, den der Diff nur als Kontext zeigt,
   nicht hinzufügt/ändert).

Aktuell gibt es **keine deterministische Nachfilterung**: `parseJson` zieht die Findings
aus der Modell-Antwort, der Coordinator soll (nur im `full`-Tier, nur per Prompt)
„speculative findings, nitpicks, and any finding pointing at code the diff does not change"
droppen — aber das ist ein weicher LLM-Schritt ohne Garantie, im `lite`/`trivial`-Tier
fehlt er ganz, und DeepSeek-Modelle befolgen die Negativ-Constraints schwächer als Opus.

## Entscheidungen (final, aus Ticket-Kontext)

1. **Line-Range-Mapping aus `git diff`:** Aus dem Unified-Diff die Menge der tatsächlich
   **hinzugefügten/geänderten** Zeilennummern pro Datei (neue Datei-Zeilennummern, also
   `+`-Zeilen aus den `@@`-Hunks) extrahieren. Findings, deren `file:line` nicht in dieser
   Menge liegt, werden **deterministisch verworfen** (False-Positive-Klasse 1 + 3).
2. **Stil-Nitpick-Killer:** Findings mit `severity: low` UND einer als Stil/Nitpick
   klassifizierten Beschreibung (Heuristik über Schlüsselwort-Liste + niedrige Confidence)
   werden verworfen (Klasse 2).
3. **Prompt-Tuning DeepSeek V3/R1:**
   - Strukturiertes Output-Schema um `confidence` (0.0–1.0) erweitern, zusätzlich zu
     `severity`/`file`/`line`.
   - Expliziter, an den Anfang gezogener **„nur geänderte Zeilen"-Constraint** mit der
     konkreten Zeilen-Liste pro Datei (aus dem Mapping), nicht nur als Prosa.
   - **Confidence-Threshold:** Findings unter einem Schwellwert (Default `0.6`) werden
     verworfen.

## Architektur

### Neuer pure Helper: `scripts/factory/review-finding-filter.mjs`

Ein **importzyklus-freies, DB-/API-loses** ES-Modul mit reinen Funktionen. Wird von beiden
Konsumenten (`ci-review.mjs` + Verify-Phase) genutzt — Single Source of Truth für die
Filter-Logik.

Exportierte Funktionen:

```
parseChangedLines(unifiedDiff) -> Map<filePath, Set<number>>
  // Parst @@-Hunks, sammelt NEUE-Datei-Zeilennummern aller +-Zeilen pro b/<path>.

isStyleNitpick(finding) -> boolean
  // Heuristik: severity 'low' + Beschreibung matcht Nitpick-Regex
  // (naming|formatting|whitespace|indentation|style|rename|typo-in-comment …).

filterFindings(findings, changedLines, opts) -> { kept, dropped }
  // opts: { confidenceThreshold = 0.6, requireInDiff = true, dropStyleNitpicks = true }
  // Verwirft Findings die: nicht in changedLines[file] liegen (wenn line gesetzt),
  // unter confidenceThreshold liegen (wenn confidence gesetzt), oder Style-Nitpick sind.
  // dropped enthält {finding, reason} für Observability.

formatChangedLinesHint(changedLines) -> string
  // Rendert die Zeilen-Liste pro Datei kompakt (Ranges) für den Prompt-Constraint.
```

**Designentscheidungen:**
- `requireInDiff` greift nur, wenn das Finding eine `line` trägt **und** die Datei im
  Diff vorkommt. Findings **ohne** Zeilennummer (z.B. agents-md-Staleness, dateiglobale
  Hinweise) werden nicht über die Zeilenregel verworfen — sonst killt man legitime
  Architektur-Findings. Findings mit Zeile zu einer Datei, die gar nicht im Diff ist,
  werden verworfen (vorbestehender Code in unverändertem File).
- **Toleranz ±0:** exakte Zeilen-Mitgliedschaft. (Keine Fuzz-Toleranz — sonst sickern
  Kontextzeilen-Findings wieder ein. Begründung: das Mapping enthält bereits **alle**
  `+`-Zeilen, und ein echter Bug auf einer geänderten Zeile trägt diese Zeilennummer.)
- Confidence-Threshold und Style-Drop sind **per opts abschaltbar** (Tests + zukünftige
  Kalibrierung).

### Konsumenten-Verdrahtung

**`ci-review.mjs`:**
- Nach `parseChangedLines(diff)` einmal berechnen.
- Den `formatChangedLinesHint`-String + Confidence-Anforderung in den User-Content jeder
  Lens injizieren (vor dem Diff).
- Nach jedem Lens-Lauf bzw. vor `coordinate`/`fallbackVerdict` die Findings durch
  `filterFindings` schicken. `fallbackVerdict` und `renderBody` arbeiten dann nur noch auf
  gefilterten Findings. Verworfene Findings werden gezählt und in den Review-Body als
  `<sub>` Transparenz-Zeile geschrieben (z.B. „3 low-confidence/out-of-diff findings
  suppressed").
- Env-Override `CI_REVIEW_CONFIDENCE_THRESHOLD` (Default `0.6`) für Kalibrierung ohne
  Code-Change.

**`pipeline.js` (Verify-Phase):** Da `pipeline.js` ein monolithisches Workflow-Skript ist
(im Quality-Gate explizit ignoriert) und Findings über `agent()`-Aufrufe einsammelt, wird
die Filterung **nicht** inline in JS dupliziert. Stattdessen:
- Der bestehende `rawBlocking`-Schritt (`reviews.flatMap(...).filter(severity high/crit)`)
  bekommt einen vorgeschalteten Aufruf eines kleinen **CLI-Wrappers** des Helpers
  (`review-finding-filter.mjs` mit `--cli`-Modus): nimmt `--diff <file> --findings <file>`
  und gibt gefilterte Findings als JSON auf stdout. So bleibt die Logik in einem Modul
  und `pipeline.js` ändert sich minimal (ein zusätzlicher `agent()`/`bash`-Schritt, der
  den Wrapper aufruft).
- Die Lens-Prompt-Constraints (Confidence + „nur geänderte Zeilen") wirken automatisch,
  weil sie in den geteilten Prompt-Dateien stehen.

### Prompt-Änderungen (geteilte Dateien)

In allen vier Code-Lens-Prompts (`review-bug-hunter`, `review-security-auditor`,
`review-pattern-enforcer`, `review-perf-reviewer`):
- Output-Schema um `"confidence": 0.0–1.0` erweitern (Pflichtfeld pro Finding).
- Einen **„HARD CONSTRAINT"-Block ganz oben** ergänzen: nur Zeilen melden, die im Diff
  als `+` markiert sind; Findings auf unverändertem/Kontext-Code sind verboten; Stil/
  Naming/Formatierung niemals melden; bei Unsicherheit niedrige Confidence vergeben statt
  zu raten.

`review-coordinator.prompt.md`: dokumentieren, dass die deterministische Filterung dem
Coordinator **vorgelagert** ist (er sieht nur noch gefilterte Findings) — der weiche
Reasonableness-Filter bleibt als zweite Verteidigungslinie.

## Nicht-Ziele

- Keine Änderung am Verdict-Mapping/Branch-Protection (Review bleibt advisory).
- Kein Modellwechsel weg von DeepSeek.
- Keine Fuzz-Toleranz beim Line-Mapping (bewusst exakt).
- `agents-md`-Lens-Findings bleiben unberührt von der Zeilenregel (dateiglobal).

## Risiken & Gegenmaßnahmen

| Risiko | Gegenmaßnahme |
|--------|---------------|
| Echter Bug auf einer geänderten Zeile wird verworfen, weil das Modell die falsche Zeilennummer nennt | Mapping enthält **alle** +-Zeilen der Datei; nur Findings zu komplett unveränderten Dateien fallen weg. Prompt zwingt Modell auf +-Zeilen. |
| Confidence-Feld wird vom Modell nicht geliefert | `filterFindings` behandelt fehlende `confidence` als „nicht filtern" (kein Drop) — fail-open, kein Verlust legitimer Findings. |
| Nitpick-Heuristik verwirft echtes Low-Severity-Finding | Nur `severity: low` + Regex-Match; Schwelle konservativ; per opts abschaltbar; dropped-Liste geloggt. |
| `pipeline.js`-CLI-Wrapper bricht Workflow-Lauf | Wrapper exit 0 always, fail-open: bei Parse-Fehler unveränderte Findings durchreichen. |

## Erfolgskriterien

- Unit-Tests (`review-finding-filter.test.mjs`) decken ab: Hunk-Parsing (mehrere Hunks,
  gelöschte Zeilen, neue Datei), Out-of-diff-Drop, Confidence-Drop, Style-Nitpick-Drop,
  fail-open bei fehlenden Feldern, dropped-Reason-Korrektheit.
- `ci-review.mjs` injiziert den Zeilen-Hint und reportet die suppressed-Count.
- Prompts enthalten Confidence-Feld + Hard-Constraint.
- `task test:all` / `task freshness:check` grün.
