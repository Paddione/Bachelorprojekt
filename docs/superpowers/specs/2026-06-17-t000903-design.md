# T000903 — Release-Notes-Generator (aus PRs)

## Problem / Motivation

Der Release-Prozess läuft heute über `release-please` (config: `release-please-config.json`,
manifest: `.release-please-manifest.json`, Workflow: `.github/workflows/release-please.yml`).
Das erzeugt pro Paket (`brett`, `arena-server`, `website`, `docs`) ein `CHANGELOG.md` aus
Conventional-Commit-Typen — aber:

1. Die Changelog-Abschnitte sind nur die `release-please`-Defaults (nicht alle Commit-Typen
   tauchen auf; keine konsistente Sektion-Reihenfolge für unser Projekt).
2. Es gibt **keine** user-freundliche, narrative Release-Note. `release-please` listet rohe
   Commit-Subjects — gut für Entwickler, schlecht für eine "Was ist neu"-Kommunikation.

Ziel: Ein Generator, der aus den **gemergten PRs seit dem letzten Tag** strukturierte,
les­bare Release-Notes baut (LLM/DeepSeek-gestützt, best-effort mit deterministischem
Fallback) und sie an zwei Ziele publiziert: **GitHub Release-Body** und **`CHANGELOG.md`**.

## Entscheidungen (final, agentic-first)

- **Zwei komplementäre Bausteine**, nicht entweder/oder:
  1. **`release-please-config.json` erweitern** — explizite `changelog-sections` (Commit-Typ →
     Abschnittstitel) je Paket, damit `release-please` selbst vollständigere, konsistent
     sortierte Changelogs erzeugt (deterministisch, kein LLM).
  2. **Zusätzlicher Generator** als `vda.sh release-notes` (delegiert an
     `scripts/vda/release-notes.sh`), der aus gemergten PRs seit dem letzten Tag strukturierte,
     user-freundliche Notes baut (LLM/DeepSeek-gestützt).
- **Datenquelle des Generators:** gemergte PRs seit dem letzten Tag, via `gh` CLI
  (`gh pr list --search "merged:>=<date>"` bzw. `gh api` mit Compare-Range). PR-Titel + Body +
  Labels sind reichhaltiger als rohe Commit-Subjects.
- **LLM:** DeepSeek über `DEEPSEEK_API_KEY` / `DEEPSEEK_BASE_URL`, exakt das curl-Muster aus
  `scripts/mishap-categorize.sh` (temperature 0, `--max-time`, jq-Extraktion, Fail-closed →
  deterministischer Fallback wenn Key fehlt oder API down). **Kein Cloud-Zwang** — ohne Key
  baut der Generator eine deterministische Liste aus PR-Titeln gruppiert nach Label/Typ.
- **Publish-Ziele (v1):** (a) GitHub Release-Body (`gh release create`/`gh release edit
  --notes-file`), (b) `CHANGELOG.md` (Prepend des neuen Abschnitts). **Kein Website-Publish**
  in v1 (Timeline/`v_timeline` bleibt unberührt).
- **Best-effort, fail-open:** Generator-Fehler dürfen keinen Release blockieren — exit 0 bei
  LLM-Ausfall, deterministischer Fallback greift.

## Scope

### In Scope (v1)
- `release-please-config.json`: pro Paket `changelog-sections` ergänzen (vollständige
  Commit-Typ-Map: feat→Features, fix→Bug Fixes, perf, refactor, docs, chore[hidden], etc.).
- Neues Skript `scripts/vda/release-notes.sh` mit Subcommands:
  - `generate [--since <tag|date>] [--out <file>]` — sammelt gemergte PRs, baut Notes
    (LLM oder Fallback), schreibt nach stdout oder `--out`.
  - `publish-github [--tag <tag>] [--notes-file <file>] [--dry-run]` — setzt Release-Body.
  - `publish-changelog [--notes-file <file>] [--dry-run]` — prepend in `CHANGELOG.md`.
  - `help` / unknown → exit 2 (vda-Konvention).
- `vda.sh`-Dispatch um `release-notes)` erweitern.
- Taskfile-Eintrag `release:notes` (ruft `vda.sh release-notes generate`) — verhindert
  Orphan-Skript (S4) und macht es task-oracle-auffindbar.
- BATS-Smoke-Test `tests/unit/vda-release-notes-smoke.bats` (offline: help exit 0, unknown
  subcommand exit 2, generate ohne `gh`/`DEEPSEEK_API_KEY` → deterministischer Fallback ohne
  Crash; `gh`-Aufrufe gemockt via PATH-Shim).
- Doku-Zeile in der Release-/CHANGELOG-Beschreibung (CLAUDE.md CI/CD-Abschnitt-naher Verweis
  oder `docs/`-Notiz), damit der Generator-Command auffindbar ist.

### Out of Scope (v1)
- Website-/Timeline-Publish, `v_timeline`-Einträge.
- Automatischer Aufruf im `release-please.yml`-Workflow (kann in einem Follow-up verdrahtet
  werden; v1 ist manuell/per-task aufrufbar). Wir halten es bewusst entkoppelt, damit ein
  LLM-Ausfall den Release-PR nicht beeinflusst.
- Mehrsprachige Notes / per-Brand-Notes.

## Architektur

```
vda.sh release-notes ──► scripts/vda/release-notes.sh
                               │
        ┌──────────────────────┼───────────────────────────┐
   generate                publish-github            publish-changelog
        │                       │                           │
  gh pr list (merged          gh release edit          prepend CHANGELOG.md
  since last tag) ─┐           --notes-file
                   ▼
        DeepSeek (best-effort, curl-Muster
        aus mishap-categorize.sh)
                   │  (kein Key / Fehler)
                   ▼
        deterministischer Fallback:
        PRs gruppiert nach Label/Conventional-Type
```

- Reine Bash-Helper, keine Import-Zyklen (S2 n/a für Bash, aber Helper bleiben pur — keine
  DB-Schreibzugriffe; Generator ist read-only gegenüber dem Repo außer beim expl/expliziten
  `publish-changelog`).
- Keine Brand-Domain-Literale (S3) — der Generator kennt keine Hostnamen; `gh` nutzt
  `GH_REPO`/Default-Remote.
- Datums-/Tag-Ermittlung: letzter Tag via `git describe --tags --abbrev=0` (bzw.
  paket-spezifisch `git tag --list 'website-v*' | sort -V | tail -1`); `--since` überschreibt.

## Akzeptanzkriterien

- **AK-1:** `release-please-config.json` enthält pro Paket eine `changelog-sections`-Liste;
  `task openspec:validate`/JSON-Schema bleibt valide; `release-please` dry-run akzeptiert die
  Config (CI Taskfile-Dry-Run grün).
- **AK-2:** `bash scripts/vda.sh release-notes help` → exit 0, listet `generate`,
  `publish-github`, `publish-changelog`.
- **AK-3:** `vda.sh release-notes generate` ohne `DEEPSEEK_API_KEY` produziert deterministische,
  nach Typ/Label gruppierte Markdown-Notes aus den gemergten PRs (kein Crash, exit 0).
- **AK-4:** Mit gesetztem `DEEPSEEK_API_KEY` ruft `generate` DeepSeek (gleiches curl-Muster,
  `--max-time`, fail-closed) und bettet die narrative Zusammenfassung über die deterministische
  Liste ein; LLM-Ausfall → Fallback, exit 0.
- **AK-5:** `publish-github --dry-run` und `publish-changelog --dry-run` zeigen die Aktion ohne
  Seiteneffekt; ohne `--dry-run` setzt `publish-github` den Release-Body und `publish-changelog`
  prepended den Abschnitt in `CHANGELOG.md`.
- **AK-6:** `tests/unit/vda-release-notes-smoke.bats` grün (offline, `gh`/curl gemockt);
  in `task test:all` über die bestehende BATS-Selektion erfasst; `test-inventory.json`
  regeneriert.
- **AK-7:** Skript ist Orphan-frei (S4): von `Taskfile.yml` (`release:notes`) und von
  `vda.sh`-Dispatch referenziert.

## Risiken / Trade-offs

- **LLM-Qualität (DeepSeek):** narrative Zusammenfassung kann ungenau sein → deshalb immer die
  deterministische PR-Liste als Grundwahrheit darunter; LLM nur als ergänzende Prosa.
- **`gh`-Verfügbarkeit:** im CI vorhanden; lokal Voraussetzung. Fehlt `gh`, fällt `generate`
  auf `git log`-basierte Commit-Subjects zurück (degradiert, aber kein Crash).
- **`changelog-sections`-Drift:** Erweiterung der Config ändert künftige `release-please`-PRs;
  bewusst — gewünschtes Verhalten.
