# T000904 — Audit-Log (Wer hat was wann getan)

## Problemstellung

Es gibt aktuell keinen durchgängigen, abfragbaren Audit-Trail für
sicherheitsrelevante Aktionen auf der Plattform. Zwei Lücken:

1. **Keycloak-Events** werden im Dev-Realm (`k3d/realm-workspace-dev.json`) und im
   Base-Prod-Realm (`prod/realm-workspace-prod.json`) gar nicht erfasst. Die
   Brand-Realms (`prod-mentolder` / `prod-korczewski`) haben zwar bereits
   `eventsEnabled`/`adminEventsEnabled`, schreiben aber nur via `jboss-logging` in
   den Container-Log und persistieren die Events ausschließlich in den
   Keycloak-eigenen DB-Tabellen mit inkonsistenter Retention.
2. **Admin-Backend-Aktionen** (`/api/admin/*`, mutierend) hinterlassen keinen
   einheitlichen, einheitlich-strukturierten Audit-Eintrag. Die bestehende
   Tabelle `public.admin_actions` ist ein **operativer Task-Lifecycle-Log**
   (Cluster-Operationen, `in_progress`→`success`/`failed`, Concurrency-Guard) —
   kein Sicherheits-Audit-Log und nicht für „wer hat welche sicherheitsrelevante
   Aktion ausgeführt" gedacht.

## Ziel

Ein **erfassender** Sicherheits-Audit-Trail in zwei Säulen:

- **Keycloak**: Login- und Admin-Events realm-seitig aktivieren (DB-Persistenz in
  Keycloak) — einheitlich über **alle** Realm-JSONs (dev + base-prod + beide
  Brands), mit dokumentierter Retention.
- **Admin-Backend**: neue Tabelle `audit.audit_log` plus ein schlanker,
  pure-Helper `recordAudit()`. Die **sicherheitsrelevanten** mutierenden Endpoints
  werden instrumentiert (kuratierte Auswahl, nicht alle 215 mutierenden Routes).

Out-of-scope (explizit, separates Ticket): die **Read-only Admin-UI-Ansicht** des
Audit-Logs. Dieser Plan liefert nur **Erfassung + Schema + Helper + Verdrahtung
ausgewählter Endpoints**.

## Nicht-Ziele / Abgrenzung

- Keine UI/Anzeige-Seite (`/admin/audit` o.ä.) — späteres Ticket.
- Kein Re-Use / keine Erweiterung von `public.admin_actions` — bewusst getrennte
  Verantwortlichkeit (operativer Task-Lifecycle vs. Security-Audit).
- Kein Loggen nicht-sicherheitsrelevanter Lese-/CRUD-Operationen (DSGVO:
  Datensparsamkeit). Nur sicherheitsrelevante Aktionen.
- Keine externe SIEM-Anbindung.

## Entscheidungen (final)

1. **Keycloak DB-Event-Logging**: `eventsEnabled: true`,
   `adminEventsEnabled: true`, `adminEventsDetailsEnabled: true`,
   `eventsListeners: ["jboss-logging"]`, `eventsExpiration: 7776000` (**90 Tage**,
   = `90*24*3600`) in **allen vier** Realm-JSONs. Brand-Realms haben bereits
   `eventsEnabled` etc., aber `eventsExpiration: 2592000` (30 Tage) — wird auf den
   einheitlichen 90-Tage-Wert angeglichen.
2. **Neue Tabelle `audit.audit_log`** (eigenes Schema `audit`, nicht `public`):
   `id bigserial PK`, `actor_id text` (Keycloak `sub`), `actor_email text`,
   `action text NOT NULL`, `target_type text`, `target_id text`, `ip inet`,
   `ts timestamptz NOT NULL DEFAULT now()`, `metadata jsonb`. Indizes auf
   `(ts DESC)` und `(actor_id, ts DESC)` und `(action, ts DESC)`.
3. **Helper `recordAudit(pool, entry)`** als **pure DB-Modul**
   (`website/src/lib/audit-log.ts`) — nur Insert, kein Lifecycle, keine
   Rück-Imports auf API-Schichten. Fail-soft: ein Insert-Fehler darf die
   eigentliche Admin-Aktion **nicht** brechen (try/catch + `console.warn`), aber
   wird geloggt.
4. **IP-Extraktion** aus dem Request (Reverse-Proxy via Traefik →
   `x-forwarded-for` first hop) in einem kleinen pure-Helper
   `clientIpFromRequest(request)`.
5. **DSGVO/Retention**: nur sicherheitsrelevante Aktionen; Retention **90 Tage**,
   in `docs/` dokumentiert (siehe Plan). Keycloak-Events laufen über
   `eventsExpiration` ab; `audit.audit_log` bekommt eine dokumentierte
   Pruning-Strategie (Cleanup via idempotentem `DELETE ... WHERE ts < now() -
   interval '90 days'` — als ensure-Schema-Step oder dokumentierte manuelle
   Query; **kein** neuer CronJob in diesem Scope).

## Erfassungsumfang (kuratierte Endpoint-Auswahl)

Sicherheitsrelevant = Änderungen an Identität, Rechten, Secrets, Deployments,
Kundendaten-Löschung, Factory-Steuerung. Instrumentiert werden u.a.:

- `api/admin/customers*` (Anlage/Löschung von Kunden)
- `api/admin/bugs/*` mutierend (reopen/resolve/archive) — security-adjacent
- `api/admin/factory-control.ts` (Steuerung der Software-Factory)
- `api/admin/deployments.ts` (Deploy-Trigger)
- `api/admin/brand-starter.ts` (Brand-Provisionierung)

Die konkrete Endpunktliste wird im Plan als Tasks fixiert. Nicht-mutierende
List-/Read-Endpoints werden **nicht** instrumentiert.

## Architektur

```
Request (Cookie-Session)
  → getSession()/isAdmin()  (bestehend, website/src/lib/auth.ts)
  → Handler führt Mutation aus
  → recordAudit(pool, {actor_id, actor_email, action, target_type, target_id, ip, metadata})
        └─ INSERT INTO audit.audit_log (fail-soft)

Keycloak (Realm-Setting eventsEnabled) → persistiert Login-/Admin-Events in KC-DB
```

- DB-Pool: bestehender `pool` aus `website/src/lib/website-db.ts` (kein neuer
  Pool).
- Schema-Bootstrap: `audit`-Schema + Tabelle werden in **beide** Blöcke von
  `k3d/website-schema.yaml` eingefügt (`init-meetings-schema.sh` für frische DB,
  `ensure-meetings-schema.sh` für bestehende DB) — exakt nach dem Muster des
  `bugs`-Schemas (`CREATE SCHEMA IF NOT EXISTS audit AUTHORIZATION website` +
  Tabelle + GRANTs). Zusätzlich eine versionierte Migration
  `website/src/db/migrations/20260617_create_audit_log.sql` als SSOT-Mirror
  (konsistent mit dem `20260525_admin_actions.sql`-Muster).

## Datenfluss & Sicherheit

- `actor_id`/`actor_email` kommen aus der verifizierten OIDC-Session — nie aus dem
  Request-Body (kein Spoofing).
- `ip` aus `x-forwarded-for` (Traefik-Hop), best-effort, kein Fail bei fehlendem
  Header.
- `metadata` enthält **keine** Klartext-Secrets/Passwörter (DSGVO + Security):
  Helper-Doku schreibt vor, nur Identifier/Statuswerte abzulegen.

## Akzeptanzkriterien

1. `audit.audit_log` existiert nach frischem Bootstrap (init-Block) **und** nach
   Neustart auf bestehender DB (ensure-Block), idempotent.
2. `recordAudit()` schreibt einen Eintrag mit allen Pflichtfeldern; ein
   DB-Fehler bricht die Admin-Aktion **nicht** (fail-soft, Unit-Test beweist
   beides).
3. Alle vier Realm-JSONs haben `eventsEnabled`/`adminEventsEnabled` und
   `eventsExpiration: 7776000`.
4. Mindestens die kuratierten Endpoints rufen `recordAudit()` bei Erfolg auf.
5. Retention/DSGVO ist in `docs/` dokumentiert (90 Tage, Umfang, Pruning).
6. `task test:all`, `task freshness:regenerate`, `task freshness:check` grün;
   `task workspace:validate` grün (Manifest-Änderung).

## Risiken / Footguns

- **S1-Zeilenbudget** `k3d/website-schema.yaml` (1892 Zeilen, nicht-baselined,
  YAML — kein S1-Limit für `.yaml`, aber Wachstum bleibt klein): nur ~15 Zeilen ×2
  Blöcke.
- **Doppel-Block-Drift**: init- und ensure-Block müssen denselben DDL haben →
  beide Stellen gleich patchen (bekannter Footgun in dieser Datei).
- **GRANT-Reihenfolge**: `audit`-Schema mit `AUTHORIZATION website`, Tabelle
  `OWNER website` — wie `bugs`-Schema, damit der `website`-Role INSERT darf.
- **Cross-Brand**: Schema-Änderungen müssen in **beiden** Namespaces deployt
  werden (`workspace` + `workspace-korczewski`) — über `task feature:*`-Umbrella.
- **Keine** Brand-Domain-Literale in Code/Manifest-Snippets (S3).
