# T000905 — Backup-Status sichtbar in Admin-UI

**Ticket:** T000905
**Typ:** feature
**Datum:** 2026-06-17
**Branch:** feature/t000905

## Problem

Der Backup-Status der Plattform (verschlüsselte PostgreSQL-Dumps **und** PVC-Snapshots der
Nextcloud-Daten, jeweils mit Filen-Remote-Upload) ist heute nur per `kubectl get jobs -l app=db-backup`
bzw. `-l app=pvc-backup` und Log-Inspektion sichtbar. Ein Admin kann in der Plattform-UI **nicht**
auf einen Blick erkennen, ob das letzte Backup erfolgreich war, wann es lief und ob der
verschlüsselte Filen-Upload durchlief. Das ist ein operatives Risiko: ein stillschweigend
fehlgeschlagenes Backup fällt erst im Ernstfall (Restore) auf.

## Ziel

Read-only Backup-Status-Panel in der bestehenden Plattform-Admin-UI (`/admin/platform`), das für
**beide Brands** (mentolder = ns `workspace`, korczewski = ns `workspace-korczewski`) den Status der
beiden Backup-Pipelines anzeigt:

- **db-backup** (CronJob `db-backup`, `app=db-backup` Job-Pods) — tägliche DB-Dumps + Filen-Upload.
- **pvc-backup** (CronJob `pvc-backup`, `app=pvc-backup` Job-Pods) — Nextcloud/Vaultwarden/DocuSeal
  PVC-Snapshots + Filen-Upload.

Pro Pipeline: Ampel (grün/gelb/rot/grau), letzter Zeitstempel (`completionTime` bzw. `startTime`),
Erfolg/Fehler des letzten Jobs sowie der **Filen-Upload-Status** des letzten erfolgreichen Laufs.

## Entscheidungen (final, aus Ticket-Zusatzkontext)

1. **Backend-Endpoint** `GET /api/admin/backup-status` fragt k8s Jobs/CronJobs mit
   `app=db-backup` UND `app=pvc-backup` ab. Pro Selektor: jüngster Job → Status,
   `completionTime`, Erfolg/Fehler.
2. **Filen-Upload-Status:** Da der Filen-Upload als zweiter Container (`filen-upload`) **im selben
   Job** läuft und bei Upload-Fehler `exit 1` macht (→ Job `failed`), ist `job.status.succeeded > 0`
   bereits das verlässliche „verschlüsselter Upload erfolgreich"-Signal (encrypt-then-upload, der
   Job wird nur grün, wenn beide Container durchliefen). Der **letzte erfolgreiche** Upload-Zeitstempel
   = `completionTime` des jüngsten `succeeded`-Jobs je Selektor. Eine Log-/Annotation-Auswertung ist
   für das MVP **nicht** nötig — der Job-Exit-Code ist die SSOT. (Begründung im Plan dokumentiert.)
3. **Admin-UI-Panel** mit Ampel pro Pipeline + letzter Zeitstempel + letzter erfolgreicher
   Upload-Zeitstempel. **Read-only**, kein Trigger/Restore in diesem Ticket (das existiert separat
   unter `ops/backup/trigger.ts` / `ops/restore.ts`).
4. **Beide Brands** — Namespace wird wie in `monitoring.ts`/`deployments.ts` aus
   `BRAND_ID`/`BRAND` abgeleitet (`workspace` ↔ `workspace-korczewski`).

## Ampel-Semantik

Pro Pipeline (db-backup, pvc-backup), ausgewertet über die jüngsten Jobs des jeweiligen Selektors:

| Zustand | Bedingung |
|---------|-----------|
| 🟢 grün | jüngster abgeschlossener Job `succeeded` UND innerhalb des erwarteten Intervalls (db: ≤ 48h, pvc: ≤ 48h seit `completionTime`) |
| 🟡 gelb | jüngster Job `succeeded`, aber `completionTime` älter als das erwartete Intervall (stale) ODER ein Job läuft gerade (`active`) ohne jüngeren Erfolg |
| 🔴 rot | jüngster abgeschlossener Job `failed` (kein jüngerer Erfolg) |
| ⚪ grau | keine Jobs gefunden / RBAC-/Token-Fehler / kein Status verfügbar |

Die Intervall-Schwellen sind reine Helper-Konstanten (keine Brand-Domain-Literale).

## Architektur & Datenfluss

```
Admin-Browser
   │  GET /api/admin/backup-status   (Cookie-Session, isAdmin-Gate)
   ▼
website-Pod (Astro API-Route)
   │  createK8sClient()  → SA-Token /var/run/secrets/...
   │  GET /apis/batch/v1/namespaces/{ns}/jobs?labelSelector=app=db-backup
   │  GET /apis/batch/v1/namespaces/{ns}/jobs?labelSelector=app=pvc-backup
   │  GET /apis/batch/v1/namespaces/{ns}/cronjobs/{db-backup,pvc-backup}  (schedule/suspend)
   ▼
k8s API (RBAC: website SA braucht batch/jobs,cronjobs get+list)
```

Die **reine** Status-Ableitung (Job-Liste → `{pipeline, light, lastRun, lastSuccessfulUpload,
succeeded, failed, active}`) wird als pure Helper-Funktion `derivePipelineStatus(jobs, cronjob, now)`
aus der Route exportiert und unit-getestet (Muster wie `cluster/graph.ts` →
`buildStatusMap`/`resolveGraph`). Kein Import-Zyklus (pure Funktion, nur Typen).

## RBAC-Lücke (wichtig)

`k3d/website-rbac.yaml` (`website-monitoring-role`, ns `workspace`) grantet aktuell nur
`pods`, `events`, `deployments`. Es gibt **keinen** `batch`-Zugriff. Der bereits existierende
`ops/backup/list.ts` (liest `app=db-backup` Jobs) und `ops/restore.ts` (liest Jobs/CronJobs)
würden in Prod also 403 bekommen — diese Lücke wird mit diesem Ticket geschlossen, indem die Role
um `batch` (`jobs`, `cronjobs`: `get`, `list`) erweitert wird. Das ist eine **Role** (namespaced),
also pro Brand-Namespace via RoleBinding wirksam — die Role lebt in ns `workspace`; für
korczewski existiert die analoge Bindung über den Fleet-Overlay (gleiche `website-rbac.yaml`-Basis,
RoleBinding-Subject = website-SA). Keine ClusterRole nötig (Jobs sind namespaced).

## UI-Verortung

Neues Panel in der bestehenden Plattform-Hub-Struktur (`PlatformHub.svelte`). Option A (gewählt):
eine neue Sektion/Card im **Health-/Integrität**-Bereich bzw. ein eigenes leichtgewichtiges
`BackupStatusCard.svelte`, eingebunden als Card im `health`-Tab-Bereich oder als eigene Card im
Hub. Da `PlatformHub.svelte` mit 96 Zeilen unter Budget liegt, wird eine eigene Komponente
`BackupStatusCard.svelte` erstellt und im Hub referenziert (keine Tab-Explosion, minimaler Diff).

## Out of Scope

- Kein Auslösen/Triggern von Backups (read-only).
- Kein Restore-Flow (existiert separat).
- Keine Log-Volltext-Anzeige (nur abgeleiteter Status + Zeitstempel; Log-Drill-down kann ein
  Folge-Ticket sein).
- Keine Alerting-/Notification-Integration.
