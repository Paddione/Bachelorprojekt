# T000906 — Direkter Datei-Share-Link im Chat (Nextcloud → Chat)

**Status:** design
**Ticket:** T000906
**Branch:** `feature/t000906`
**Domains:** website (Astro/Svelte messaging + Nextcloud OCS integration)

## 1. Problem / Ziel

Im plattform-eigenen Chat (Astro-Website, „ChatRoom"-System) soll ein Nutzer — **Admin
oder Kunde** — einen Nextcloud-Datei-Share-Link **direkt aus dem Chat heraus** erzeugen
und als Chat-Nachricht einfügen können, ohne den Chat zu verlassen, in Nextcloud zu
navigieren, manuell zu teilen und den Link zurück zu kopieren.

Heute existieren bereits:
- Ein Chat-Room-System (`chat_messages`, `chat_rooms`) mit REST-Endpoints unter
  `/api/portal/rooms/[id]/messages` (Kunde) und einem Admin-Pendant, gerendert über
  `ChatRoomPanel.svelte` (Reply-Bar mit Textarea + Senden-Button).
- WebDAV-Helfer in `website/src/lib/nextcloud-files.ts` (`listFiles`, `getFileUrl`,
  `getClientFolderPath`, …), die mit **Nextcloud-Admin-Credentials** (`NEXTCLOUD_ADMIN_USER`/
  `NEXTCLOUD_ADMIN_PASS`) gegen `remote.php/dav/files/<admin>/…` sprechen.
- Keycloak→Nextcloud-User-Mapping über OIDC: `session.preferred_username` (Keycloak-Username)
  == Nextcloud-Username (NC provisioniert den User per OIDC `preferred_username`).

Es fehlt: das Erzeugen eines **Share-Links** (intern oder öffentlich) über die Nextcloud
**OCS Sharing-API** und das Einfügen dieses Links als Chat-Nachricht.

## 2. Finale Entscheidungen (aus Ticket-Kontext)

| Frage | Entscheidung |
|---|---|
| Wer darf teilen? | **Admin UND Kunde.** Die Nextcloud-Share-Permissions des jeweiligen Users werden respektiert — **kein Rechte-Bypass**. |
| Dateiauswahl v1 | **Such-/Pfad-Eingabe** (Text). **KEINE** WebDAV-Tree-Navigation (Folge-Ticket). |
| Link-Erzeugung | Nextcloud **OCS Share-API** (`ocs/v2.php/apps/files_sharing/api/v1/shares`) erzeugt internen oder öffentlichen Link. |
| Einfügen | Der erzeugte Link wird als **Chat-Nachricht** eingefügt (normale `chat_messages`-Zeile, kein neuer Nachrichtentyp). |
| User-Mapping | Keycloak-User → Nextcloud-User über OIDC `sub`/`preferred_username`. |
| Mandantentrennung | **Brand-isoliert** — jede Brand spricht ihren eigenen Nextcloud-Endpoint (`NEXTCLOUD_URL` / `NEXTCLOUD_EXTERNAL_URL` aus der env-Resolution), keine Cross-Brand-Pfade. |

## 3. „Kein Rechte-Bypass" — Berechtigungsmodell (zentrale Design-Entscheidung)

Die bestehenden WebDAV-Helfer laufen mit **Admin-Credentials**. Würde der Share-Endpoint
naiv „jede Datei, die der Admin sieht" für jeden Chat-User teilbar machen, wäre das ein
massiver Rechte-Bypass (ein Kunde könnte fremde Ordner teilen).

**v1-Regel (fail-closed, serverseitig erzwungen):**

- **Kunde:** darf ausschließlich Pfade **unterhalb seines eigenen Client-Ordners**
  teilen — `getClientFolderPath(session.preferred_username)` → `Clients/<username>/…`.
  Jeder Pfad außerhalb wird mit `403` abgelehnt. Das ist exakt der Bereich, den der Kunde
  in Nextcloud ohnehin besitzt (sein OIDC-provisionierter Eigenbereich), daher
  permissions-äquivalent — kein Bypass.
- **Admin:** darf jeden Pfad teilen (Admins haben in Nextcloud Vollzugriff; kein zusätzlicher
  Gewinn gegenüber ihrem NC-Login).
- Der serverseitige **Pfad-Guard** (Normalisierung + Präfix-Check + bestehender
  `../`-Reject aus `davUrl`) ist die einzige Vertrauensgrenze. Die Client-UI ist nur
  Komfort, nie Sicherheitsgrenze.

> Trade-off / Scope-Grenze: Echte per-User-OCS-Impersonation (Share **im Namen** des
> Kunden via dessen NC-App-Passwort) ist v1 **nicht** vorgesehen — sie erfordert einen
> App-Passwort-Provisioning-Flow, der nicht existiert. Der Pfad-Präfix-Guard liefert das
> gleiche Sicherheitsniveau (Kunde kann nur Eigenes teilen) ohne neuen Secret-Flow. Owner
> des Shares ist technisch der Admin-Account; der geteilte Pfad bleibt aber strikt der
> Eigenbereich des Kunden. Dieser Trade-off ist in der Spec dokumentiert und im
> Folge-Ticket (Tree-Navigation) auf echte Impersonation erweiterbar.

## 4. Scope

### In-Scope (v1)
1. Pures Helper-Modul `website/src/lib/nextcloud-shares.ts` mit:
   - `createShareLink({ path, shareType, permissions, password?, expireDate? })` →
     OCS-`POST` gegen `ocs/v2.php/apps/files_sharing/api/v1/shares`, liefert
     `{ url, token, shareType }`.
   - `findFilesByName(scopePath, query)` → dünne Wrapper-Suche über den vorhandenen
     `listFiles` (Pfad-präfix-beschränkt) für die Such-/Pfad-Eingabe (keine Rekursion v1;
     listet die im Scope-Ordner liegenden Treffer).
   - `assertPathAllowed(path, session)` → reiner Guard (Admin → frei; Kunde → muss unter
     `Clients/<preferred_username>/` liegen), wirft bei Verletzung.
   - Reines Modul: nur `fetch` + `node:path` + Re-Use von `nextcloud-files.ts`-Helfern;
     **kein** Import aus DB-/API-Schichten (S2).
2. Neuer Server-Endpoint `POST /api/portal/rooms/[id]/share` (Kunde) und
   `POST /api/admin/rooms/[id]/share` (Admin) — bzw. ein gemeinsamer Endpoint, der über
   `isAdmin(session)` verzweigt — der:
   - Session + Room-Membership prüft (Re-Use der vorhandenen Guards),
   - `assertPathAllowed` erzwingt,
   - via `createShareLink` den Link erzeugt,
   - **und** den Link als `chat_messages`-Zeile über `addRoomMessage` einfügt
     (eine einzige atomare Server-Aktion → der Link erscheint sofort als Nachricht).
3. UI in `ChatRoomPanel.svelte`: ein „Datei teilen"-Button (📎/Link-Icon) neben der
   Reply-Bar öffnet eine kleine Inline-Eingabe (Pfad/Name-Suche + Auswahl intern/öffentlich
   + optional Ablaufdatum/Passwort für öffentliche Links), POSTet an den Share-Endpoint,
   die zurückkommende Nachricht wird wie eine normale Chat-Nachricht angehängt.
4. Render: Link in der Chat-Nachricht klickbar darstellen (der Nachrichtentext enthält den
   Link; minimaler Linkify oder Klartext-URL — kein neuer Message-Typ nötig).
5. Env-Var-Registrierung: `NEXTCLOUD_URL` / `NEXTCLOUD_EXTERNAL_URL` werden bereits genutzt;
   ggf. fehlende OCS-relevante Vars in `schema.yaml` + `env.d.ts` ergänzen (kein neues Secret,
   Admin-Creds existieren bereits).
6. Unit-Tests für `nextcloud-shares.ts` (Guard-Logik, OCS-Body-Aufbau, Pfad-Reject) mit
   gemocktem `fetch`.

### Out-of-Scope (Folge-Tickets)
- WebDAV-Tree-Navigation / Dateibrowser-UI.
- Echte per-User-OCS-Impersonation via NC-App-Passwort.
- Inline-Vorschau (Thumbnails) geteilter Dateien im Chat.
- Share-Verwaltung (Widerruf/Liste bestehender Shares) aus dem Chat.
- Rekursive Volltextsuche über den gesamten NC-Tree.

## 5. Architektur / Datenfluss

```
ChatRoomPanel.svelte  ──POST {path, shareType, ...}──▶  /api/.../rooms/[id]/share
   (Share-Inline-Form)                                        │
                                                              │ 1. getSession + Room-Membership-Guard
                                                              │ 2. assertPathAllowed(path, session)   ← Rechte-Grenze
                                                              │ 3. nextcloud-shares.createShareLink() ──OCS POST──▶ Nextcloud
                                                              │ 4. addRoomMessage({ body: "<msg> <url>" })
                                                              ▼
                                            201 { message }  ──▶  Panel hängt Nachricht an (wie normaler Send)
```

- **Brand-Isolation:** `NEXTCLOUD_URL` (intern) / `NEXTCLOUD_EXTERNAL_URL` (Link-Host) kommen
  pro Brand aus `env-resolve.sh`/ConfigMap — **keine** Brand-Domain-Literale im Code (S3).
- **Pure Module:** `nextcloud-shares.ts` hat keine Rück-Importe auf DB/API → keine S2-Zyklen.
- Der Endpoint ist die einzige Stelle, die `nextcloud-shares` **und** `messaging-db`
  zusammenführt (Komposition in der API-Schicht, nicht im Helper).

## 6. Nextcloud OCS Share-API (Referenz)

`POST {NEXTCLOUD_URL}/ocs/v2.php/apps/files_sharing/api/v1/shares`
Header: `Authorization: Basic <admin>`, `OCS-APIRequest: true`, `Accept: application/json`.
Body (form-urlencoded):
- `path` — Pfad relativ zum NC-Root des Owners.
- `shareType` — `3` = öffentlicher Link, `0`/`1` = User/Group (v1: 3 für öffentlich,
  intern als Variante dokumentiert).
- `permissions` — `1` = read (Default für Link-Share).
- optional `password`, `expireDate` (`YYYY-MM-DD`).
Antwort (JSON): `ocs.data.url` (öffentlicher Link), `ocs.data.token`. Der `url`-Host wird auf
`NEXTCLOUD_EXTERNAL_URL` normalisiert, falls NC den internen Host zurückgibt.

## 7. Sicherheit / DSGVO

- **Rechte-Grenze rein serverseitig** (Pfad-Präfix-Guard, fail-closed) — siehe §3.
- Öffentliche Links: optional Passwort + Ablaufdatum anbieten; Default-Permissions = read-only.
- Keine Brand-übergreifenden Pfade (jede Brand eigener NC-Endpoint).
- Kein Logging von Share-Tokens im Klartext über das nötige Maß hinaus.
- Pfad-Traversal bleibt durch den bestehenden `../`-Reject in `nextcloud-files.davUrl`-Logik
  abgedeckt, plus zusätzlichem Präfix-Check.

## 8. Akzeptanzkriterien

- [ ] Kunde kann im Chat einen Pfad **unter seinem Client-Ordner** als Link teilen; Link
      erscheint sofort als Chat-Nachricht und ist im Browser öffenbar.
- [ ] Kunde, der einen Pfad **außerhalb** seines Ordners angibt, erhält `403`, keine Nachricht.
- [ ] Admin kann beliebige Pfade teilen.
- [ ] Öffentlicher Link kann optional mit Passwort + Ablaufdatum erzeugt werden.
- [ ] Der Share-Link-Host ist der **externe** Brand-Host (`NEXTCLOUD_EXTERNAL_URL`), nicht
      der cluster-interne.
- [ ] Brand-Isolation: mentolder-Chat teilt nur mentolder-NC, korczewski nur korczewski-NC.
- [ ] Unit-Tests für Guard + OCS-Body grün; `task test:all` + `task freshness:check` grün.
- [ ] Keine S1/S2/S3/S4-Verletzung (Zeilenbudgets eingehalten, keine Domain-Literale,
      pures Helper-Modul, keine Orphan-Skripte/Manifeste).
