# T000907 — Ordnerstruktur-Templates für neue Projekte

**Status:** design
**Ticket:** T000907
**Datum:** 2026-06-17
**Domäne:** website (Admin), Nextcloud (WebDAV)

## 1. Problem & Ziel

Bei der Anlage eines neuen Projekts (`type='project'` Ticket) wird im Nextcloud aktuell
**kein** strukturierter Ablageordner erzeugt. Erst wenn der erste Anhang hochgeladen wird,
legt `attachments/upload.ts` lazily `Projects/${projectId}` an (flach, ohne Unterstruktur).
Teams müssen die Ablage-Konvention (Vertrag, Rechnungen, Dokumente, …) für jedes Projekt
manuell von Hand nachbauen — fehleranfällig und inkonsistent.

**Ziel:** Vordefinierte, **konfigurierbare** Ordnerstruktur-Templates, die bei Projektanlage
optional ausgewählt und automatisch per WebDAV (`MKCOL`) im Projektordner angelegt werden.
Ein mitgeliefertes Default-Set deckt den Standard-Workflow ab; Admins können eigene Templates
über eine CRUD-Oberfläche pflegen. Brand-isoliert.

## 2. Entscheidung (final, vom User bestätigt)

- **Konfigurierbar:** Admin-UI zum Pflegen der Templates (Create/Read/Update/Delete).
- **Mitgeliefertes Default-Set** (Seed): Default-Template „Standard" mit den Ordnern
  - `01_Vertrag`
  - `02_Rechnungen`
  - `03_Dokumente`
  - `04_Assets`
  - `05_Kommunikation`
- **Bei Projektanlage Template wählbar** (Dropdown im „Neues Projekt"-Dialog; Default
  vorausgewählt, „— keine Struktur —" möglich).
- **Speicherung in DB** (website-Schema), Tabelle `folder_templates`:
  - `name TEXT`, `structure JSONB`, plus `brand`, Default-Flag, Timestamps.
- **Anlage in Nextcloud via WebDAV `MKCOL`** beim Projekt-Setup (idempotent, nutzt
  vorhandenen `ensureFolder`-Helper aus `nextcloud-files.ts`).
- **Brand-isoliert:** Templates sind pro `brand` gefiltert; Default-Seed pro Brand.

## 4. Datenmodell

Tabelle `public.folder_templates` (website-DB):

| Spalte | Typ | Notizen |
|---|---|---|
| `id` | `UUID PRIMARY KEY DEFAULT gen_random_uuid()` | |
| `brand` | `TEXT NOT NULL` | Brand-Isolation |
| `name` | `TEXT NOT NULL` | Anzeigename (z. B. „Standard") |
| `structure` | `JSONB NOT NULL` | geordnete Liste relativer Ordnerpfade |
| `is_default` | `BOOLEAN NOT NULL DEFAULT false` | je Brand höchstens eines vorausgewählt |
| `created_at` | `TIMESTAMPTZ DEFAULT now()` | |
| `updated_at` | `TIMESTAMPTZ DEFAULT now()` | |

- `UNIQUE (brand, name)` — keine doppelten Template-Namen je Brand.
- Partieller Unique-Index `UNIQUE (brand) WHERE is_default` — höchstens **ein** Default je Brand.

**`structure` JSONB-Form** (bewusst flach + simpel, erlaubt aber verschachtelte Pfade über
`/`-Segmente):

```json
{ "folders": ["01_Vertrag", "02_Rechnungen", "03_Dokumente", "04_Assets", "05_Kommunikation"] }
```

Jeder Eintrag ist ein **relativer** Pfad unterhalb von `Projects/${projectId}/`. Erlaubte
Zeichen werden validiert (keine `..`, kein führender `/`, Whitelist `[A-Za-z0-9 _./-]`,
Länge je Segment ≤ 100, max. 50 Ordner je Template) — verhindert Path-Traversal beim MKCOL.

## 5. Funktionale Anforderungen

- **FA-1:** Migration legt `folder_templates` an (idempotent) und seedet pro Brand das
  Default-Template „Standard" mit dem 5-Ordner-Set (`is_default = true`).
- **FA-2:** Pures Validierungs-/Helfer-Modul `folder-templates-db.ts`:
  - `FOLDER_PATH_RE`, `validateStructure(input)` (pure, kein DB-Import) → `{ ok, error?, folders? }`.
  - `listTemplates(brand)`, `getTemplate(brand, id)`, `getDefaultTemplate(brand)`,
    `createTemplate(...)`, `updateTemplate(...)`, `deleteTemplate(...)`.
  - `materializeTemplate(projectId, folders)` → erzeugt via `ensureFolder` die Ordner unter
    `Projects/${projectId}/<folder>` (jeder Pfad einzeln, idempotent, sammelt Fehler).
- **FA-3:** Im „Neues Projekt"-Dialog steht ein Dropdown „Ordnerstruktur" mit allen Templates
  des aktuellen Brands; das Default-Template ist vorausgewählt, plus Option „— keine Struktur —".
- **FA-4:** `projekte/create.ts` nimmt `folderTemplateId` entgegen; nach erfolgreichem
  `createProject` wird (falls Template gewählt) `materializeTemplate` aufgerufen. Fehler bei der
  Ordner-Anlage werden geloggt und als nicht-blockierende Warnung in den Redirect gehängt
  (`?saved=1&foldersWarn=1`) — das Projekt bleibt angelegt (Nextcloud-Erreichbarkeit ist nicht
  Voraussetzung für die Projektanlage).
- **FA-5:** Admin-CRUD-Seite `/admin/einstellungen/ordner-templates`:
  - Liste aller Templates (Name, Ordner-Vorschau, Default-Badge).
  - Anlegen/Bearbeiten (Name, Ordnerliste als Textarea, eine Zeile = ein Ordner, Default-Checkbox).
  - Löschen (Default-Template kann nicht gelöscht werden, solange es das einzige ist).
  - Neuer Tab in `AdminEinstellungenTabs.astro`.
- **FA-6:** Alle API-Routen sind admin-gegated (`getSession` + `isAdmin`, 403 sonst), analog
  zum bestehenden `projekte/create.ts`. Brand wird serverseitig aus `process.env.BRAND`/Form
  aufgelöst — Templates eines fremden Brands sind nicht editier-/sichtbar.

## 6. Nicht-funktionale Anforderungen

- **NFA-1 (S1):** Keine bestehende baselined-Datei wächst. `projekte.astro` (Budget 0) wird durch
  Extraktion des Create-Form-Partials **netto verkleinert**; neue Module bleiben unter ihren
  Extension-Limits mit Wachstumsreserve.
- **NFA-2 (S2):** `folder-templates-db.ts` ist als Modul mit pure-Validatoren strukturiert; die
  Validatoren (`validateStructure`, `FOLDER_PATH_RE`) importieren **nichts** aus DB-/API-Schichten,
  damit sie ohne `pg`-Pool unit-testbar sind (kein Import-Zyklus).
- **NFA-3 (S3):** Keine Brand-Domain-Literale (`*.mentolder.de` / `*.korczewski.de`) in Code.
  Brand kommt aus `process.env.BRAND`; Nextcloud-URL aus `NEXTCLOUD_URL`-Env.
- **NFA-4 (S4):** Neue Migration/Helper sind referenziert (Migration in der Migrations-Doku/Runbook,
  Module von Seiten/Tests importiert). Keine neuen `k3d/*.yaml`/`scripts/*.sh` nötig.
- **NFA-5 (Sicherheit):** Path-Traversal-Schutz in `validateStructure` + erneute `davUrl`-Prüfung
  in `ensureFolder` (Defense-in-Depth). MKCOL ist idempotent (405 = existiert → ok).
- **NFA-6 (DSGVO/On-Prem):** Keine externen Dienste; alle Ordner liegen im selbst-gehosteten
  Nextcloud. Keine PII in Template-Definitionen.

## 7. Out of Scope

- Rückwirkende Strukturanlage für **bestehende** Projekte (separates Ticket möglich).
- Hochladen/Seedung von Default-**Dateien** (nur Ordner).
- Pro-Kunde-/pro-Subprojekt-Templates (nur auf Projektebene).
- Verschachtelte Template-Vererbung / Template-Versionierung.

## 8. Verifikation (Akzeptanz)

- Migration zweimal anwendbar ohne Fehler; Default-Template existiert je Brand genau einmal.
- Unit-Tests (Vitest) für `validateStructure` (gültig / `..` / führender `/` / leer / zu viele /
  zu lange Segmente) und für die Default-Konstante (5 Ordner in korrekter Reihenfolge).
- Manueller/integrierter Test: Projekt mit „Standard"-Template anlegen → `Projects/<id>/01_Vertrag`
  … `05_Kommunikation` existieren im Nextcloud (PROPFIND).
- Admin-CRUD: Template anlegen, bearbeiten, als Default setzen, löschen — jeweils brand-gefiltert.
- `task test:changed`, `task freshness:regenerate`, `task freshness:check` grün; S1–S4-Ratchet
  ohne Verschlechterung.
