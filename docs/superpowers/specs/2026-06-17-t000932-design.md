# Design: Auto-Deploy bei Merge [T000932]

## Problem

Das Repo ist **push-based**: ein Merge nach `main` wendet Manifest-Änderungen **nicht** automatisch
auf den Fleet-Cluster an (kein GitOps-Reconciler / kein Flux/Argo auf fleet). Nach jedem Merge muss
ein Mensch manuell `task workspace:deploy ENV=mentolder` **und** `ENV=korczewski` ausführen, damit
`k3d/`-, `prod/`-, `prod-fleet/`- oder `environments/`-Änderungen produktiv werden.

Folgen dieser manuellen Lücke:
- **Drift**: `main` und der Live-Cluster laufen auseinander; gemergte Manifest-Fixes sind „grün",
  aber nie ausgerollt.
- **Vergessene Brand**: typischer Fehler ist, nur eine der beiden Brands zu deployen.
- **Kein Status-Signal**: Tickets springen direkt von „gemergt" auf `done`, obwohl der Cluster die
  Änderung noch nicht hat — die `awaiting_deploy`-Lane des Cockpits (merge ≠ prod) wird umgangen.

Heute existiert nur `post-merge.yml`, das ausschließlich das Ticket auf `done` setzt (kein Deploy),
sowie die komponentenspezifischen `build-website*.yml` / `build-brett.yml` / `build-docs.yml`, die
ihre eigenen Images bauen und ausrollen. **Manifest-Änderungen haben keinen Auto-Deploy-Pfad.**

## Ziel

Ein **unified post-merge Deploy-Workflow**, der bei Merge nach `main`:
1. das Ticket aus dem Merge-Commit auf `awaiting_deploy` setzt (statt sofort `done`),
2. bei **Manifest-Änderungen** einen `workspace:deploy`-Fan-out über **beide Brands**
   (mentolder + korczewski) auf dem Fleet-Cluster ausführt,
3. nach erfolgreichem Deploy das Ticket auf `done` setzt,
4. das push-based Modell respektiert (explizit, kein In-Cluster-Reconciler) und die
   komponentenspezifischen `build-*.yml` (Website/Brett/Docs mit eigenem Digest/`:latest`-Rollout)
   **unangetastet** lässt.

## Nicht-Ziele

- **Kein** GitOps-Reconciler auf fleet einführen (Flux/Argo bleiben draußen).
- **Kein** Ablösen der `build-website*.yml` / `build-brett.yml` / `build-docs.yml` — diese behalten
  ihren eigenen Build- und Rollout-Pfad (Image-Digest bzw. `:latest`, eigener `rollout restart`).
- **Kein** Deploy von Office/CoTURN (separater `workspace:office:deploy`-Pfad, bleibt manuell).
- **Kein** sealed-secrets-Reseal oder Cluster-Bring-up — der Workflow setzt einen lauffähigen
  Cluster mit gültigen Secrets voraus (wie heute der Website-Build).

## Architektur-Entscheidungen (final)

| Entscheidung | Wahl | Begründung |
|---|---|---|
| Granularität | **Ein** unified Workflow (erweitert `post-merge.yml`) statt per-Komponente | Eine Quelle der Wahrheit für „merge → deploy"; keine N parallelen Manifest-Deployer. |
| Status-Lebenszyklus | Merge → `awaiting_deploy` → (deploy ok) → `done` | Nutzt die **vorhandene** `awaiting_deploy`-Lane (Cockpit „merge ≠ prod"); kein neuer State. |
| Manifest-Trigger | nur bei Änderung von `k3d/`, `prod/`, `prod-fleet/`, `prod-mentolder/`, `prod-korczewski/`, `environments/` | Reiner Doku-/Test-Merge soll keinen Cluster-Deploy auslösen. |
| Brand-Fan-out | `workspace:deploy` für mentolder **und** korczewski | Beide Brands leben auf demselben fleet-Cluster, aber als getrennte Deployments/Namespaces. |
| Website/Brett/Docs | unverändert über eigene `build-*.yml` | Diese haben Image-Builds + eigenen Rollout; der Manifest-Deploy würde nur ihre `:latest`/Digest-Pins doppeln. |
| Modell | push-based, **explizit** im Workflow | kein Reconciler; der Workflow *ist* der Push. |

## Lösungsskizze

### Workflow-Form

`post-merge.yml` wird zu einem mehrstufigen Workflow mit zwei (sequenziellen) Jobs:

1. **`mark-awaiting`** — extrahiert `T######` aus dem Merge-Commit (wie heute) und setzt das Ticket
   per `scripts/ticket.sh update-status --status awaiting_deploy`. Kein Ticket im Commit → no-op
   (Exit 0). Idempotent / non-fatal bei DB-Hiccup (wie heute).

2. **`deploy-manifests`** — läuft **nur**, wenn der Merge-Commit Manifest-Pfade berührt
   (Pfad-Erkennung deterministisch in einem **pure Bash-Helper** `scripts/ci/changed-manifests.sh`,
   der `git diff --name-only` gegen den Vorgänger-Commit gegen eine Pfad-Allowlist matcht und
   Exit 0/1 + eine Begründung liefert — keine DB-/API-Imports, testbar via BATS). Bei Treffer:
   kubectl/kustomize installieren (wie `build-website.yml`), `FLEET_KUBECONFIG` einrichten, dann
   `task workspace:deploy ENV=mentolder` **und** `ENV=korczewski` (entspricht
   `workspace:deploy:all-prods`, hier aber GitHub-Actions-tauglich mit gesetztem Kontext).
   Bei Erfolg: Ticket → `done`. Bei Fehlschlag: Workflow rot, Ticket bleibt `awaiting_deploy`
   (Cockpit zeigt den Stau).

### Status-Übergangslogik

Wiederverwendung des bestehenden reinen Helpers `scripts/factory/deploy-transition.mjs`
(`decideDeployTransition`) bzw. seiner Logik als Referenz: push-based ⇒ `awaiting_deploy`, erst nach
erfolgreichem expliziten Deploy ⇒ `done`. Der Workflow kodiert genau diesen Lebenszyklus.

### Idempotenz & Sicherheit

- `update-status` auf bereits-`done`/`awaiting_deploy` ist ein No-op (idempotent).
- Deploy-Schritt **fail-closed**: schlägt `workspace:deploy` für eine Brand fehl, wird das Ticket
  **nicht** auf `done` gesetzt (bleibt `awaiting_deploy`), Workflow ist rot.
- Kein Reseal, kein `$patch: delete`-Eingriff — `workspace:deploy` wendet die committeten
  SealedSecrets + Overlay an (bestehendes Verhalten).

## Betroffene Dateien (Übersicht)

| Datei | Art | Zweck |
|---|---|---|
| `.github/workflows/post-merge.yml` | ändern | `awaiting_deploy` setzen + Manifest-Deploy-Job + finaler `done`-Übergang |
| `scripts/ci/changed-manifests.sh` | neu | pure Pfad-Allowlist-Erkennung „berührt der Merge Manifeste?" |
| `tests/unit/changed-manifests.bats` *(oder bestehende CI-Test-Datei erweitern)* | neu/ändern | BATS für den Helper (Allowlist-Hit/Miss, Doku-only-Merge) |

## Akzeptanzkriterien

- **AK1**: Merge eines reinen Manifest-Fixes (`k3d/*.yaml`) mit `T######` im Commit → Ticket geht
  `awaiting_deploy` → `workspace:deploy` läuft für **beide** Brands → Ticket `done`.
- **AK2**: Merge ohne Manifest-Änderung (nur `docs/`, `tests/`) → Ticket `awaiting_deploy` → kein
  Manifest-Deploy-Job → Ticket wird (mangels Cluster-Änderung) direkt auf `done` gesetzt.
- **AK3**: Merge ohne `T######` im Commit → kein Status-Update, kein Fehler (Exit 0).
- **AK4**: Schlägt `workspace:deploy` für eine Brand fehl → Workflow rot, Ticket bleibt
  `awaiting_deploy` (nicht `done`).
- **AK5**: Website-/Brett-/Docs-Merges lösen weiterhin ihre eigenen `build-*.yml` aus; der unified
  Workflow startet **keinen** doppelten Manifest-Deploy für diese (Pfad-Allowlist schließt
  `website/**`, `brett/**`, reine `docs/**` aus, sofern sie nicht zugleich Manifeste berühren).
- **AK6**: `scripts/ci/changed-manifests.sh` ist ein pure Helper (keine DB-/API-Imports, kein
  Brand-Domain-Literal) mit BATS-Abdeckung.
- **AK7**: CI grün — `task test:all`, `task freshness:regenerate`, `task freshness:check`.

## Risiken & Trade-offs

- **Deploy-Dauer / Cluster-Last**: jeder Manifest-Merge triggert einen Full-`workspace:deploy` über
  beide Brands. Akzeptabel — `workspace:deploy` ist apply-idempotent; Frequenz von Manifest-Merges
  ist niedrig. Verfeinerung (nur geänderte Ressourcen) ist out-of-scope.
- **Reihenfolge gegen `build-*.yml`**: läuft ein Website-Merge, der *auch* ein Manifest berührt,
  laufen beide Pfade — unkritisch, da `set image`/`rollout` der `build-*.yml` autoritativ für die
  App-Pods bleibt und `workspace:deploy` die übrigen Ressourcen idempotent appliziert.
- **Kontext-Fehlzieher**: GitHub-Runner muss `ENV=…` + fleet-Kontext explizit setzen (CLAUDE.md:
  „`ENV=` ist immer explizit"). Wird im Workflow hart gesetzt, nicht aus dem aktiven Kontext
  abgeleitet.
