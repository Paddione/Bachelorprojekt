# T000933 — Ticket-Auto-Triage

## Problem / Motivation

Neu eingehende Support- und Feature-Tickets landen unklassifiziert im Backlog
(`status` in `backlog`/`intake`). Bisher muss ein Mensch jedes Ticket manuell
sichten und `type`, `priority`, `severity`, betroffene `areas`/`component` sowie
einen sinnvollen Bearbeiter (`assignee`) bestimmen, bevor es ins Planungsbüro
oder an die Software Factory übergeben werden kann. Das ist Reibung am Anfang der
Pipeline und der einzige Pipeline-Schritt vor `dev-flow-plan`, der noch komplett
manuell ist.

Ziel: Eingehende, noch nicht triagierte Tickets werden **automatisch von der KI
(DeepSeek)** vorklassifiziert. Die Vorschläge werden in die Ticket-Zeile
geschrieben (`grilling_meta` + `triaged_at`), damit sie im Planungsbüro sichtbar
sind. Ein Mensch bestätigt oder korrigiert die Vorschläge — **kein Auto-Apply
ohne Review**.

## Ziele

1. KI-gestützte Vorklassifizierung neuer Tickets: `type`, `priority`, `severity`,
   `areas[]`, `component`, plus ein vorgeschlagener `assignee`.
2. Läuft als **Factory-Cron/Poll-Schritt** (kein `LISTEN/NOTIFY`) analog zu
   `auto-enqueue.sh` — wird vor dem Dispatcher-Tick in `wakeup.sh` aufgerufen.
3. **Idempotent**: verarbeitet nur Tickets ohne `triaged_at` (untriagiert) in den
   Eingangs-Status (`backlog`/`intake`). Bereits triagierte Tickets werden
   übersprungen.
4. Schreibt Vorschläge nach `grilling_meta` (JSONB, Schlüssel `triage`) und setzt
   `triaged_at = now()`. Setzt **keine** harten Felder (`type`/`priority`/…)
   direkt — diese bleiben dem menschlichen Review im Planungsbüro vorbehalten.
5. Mensch bestätigt im Planungsbüro: die Triage-Vorschläge sind dort sichtbar und
   per Knopfdruck in die echten Spalten übernehmbar (oder verwerfbar).

## Nicht-Ziele

- **Kein** automatisches Setzen von `type`/`priority`/`severity`/`component` ohne
  Review (kein Auto-Apply).
- **Kein** `LISTEN/NOTIFY` oder DB-Trigger — bewusst cron-poll wie der Autopilot.
- Keine Änderung am bestehenden Grilling-/Readiness-Flow (T000730); Triage ist ein
  vorgelagerter, unabhängiger Schritt.
- Kein neuer Cluster-Workload, kein neues Deployment — der Poll-Schritt läuft im
  bestehenden WSL-Autopilot-Tick (`factory.timer` → `wakeup.sh`).

## Architektur

```
factory.timer (systemd, WSL)
  └─ wakeup.sh  (jeder Tick, pro Brand mentolder|korczewski)
       ├─ auto-enqueue.sh                 (bestehend: plan_staged → backlog)
       ├─ auto-triage.sh   ← NEU          (untriagiert → KI-Vorschlag in grilling_meta)
       └─ dispatcher.js                   (bestehend)
```

### Datenfluss `auto-triage.sh`

1. `factory_resolve` (Kontext + Namespace pro Brand) — wie `auto-enqueue.sh`.
2. Untriagierte Eingangs-Tickets abfragen:
   ```sql
   SELECT external_id, title, description, type
   FROM tickets.tickets
   WHERE status IN ('backlog','intake')
     AND triaged_at IS NULL
   ORDER BY created_at ASC
   LIMIT :batch;   -- Batch-Cap gegen Budget-Spikes
   ```
3. Pro Ticket: Prompt an DeepSeek über den bestehenden Provider-Pfad
   (`route-provider.sh <source=triage> <tier=flash>` → DeepSeek-Modell) bauen.
   Der Prompt enthält Titel + Beschreibung + die **erlaubten Enum-Werte** (siehe
   unten) und fordert striktes JSON.
4. Antwort parsen + **fail-closed validieren** (`jq`): nur erlaubte Enum-Werte,
   `areas` ⊆ bekannte Bereiche, `assignee` ⊆ bekannte Bearbeiter-Liste. Bei
   ungültigem/leerem JSON: Ticket überspringen (kein `triaged_at` setzen → nächster
   Tick versucht es erneut), Fehler nicht fatal.
5. Validierten Vorschlag schreiben:
   ```sql
   UPDATE tickets.tickets
   SET grilling_meta = COALESCE(grilling_meta,'{}'::jsonb)
                       || jsonb_build_object('triage', :suggestion::jsonb),
       triaged_at = now()
   WHERE external_id = :ext AND triaged_at IS NULL;
   ```
   Der `triaged_at IS NULL`-Guard im `WHERE` macht den Write **idempotent** auch
   bei paralleler Verarbeitung (kein Doppel-Triage).
6. OTel-Metrik (`factory.triage.count`) emittieren, Logzeile mit `[auto-triage:<brand>]`.

### `grilling_meta.triage`-Form (Vorschlag, kein Apply)

```json
{
  "triage": {
    "type": "feature",
    "priority": "P2",
    "severity": "minor",
    "areas": ["website", "tickets"],
    "component": "planungsbuero",
    "assignee_suggested": "patrick",
    "rationale": "kurze Begründung der KI",
    "model": "deepseek-...",
    "at": "2026-06-17T..."
  }
}
```

### Enum-Quellen (SSOT, keine Literale im Skript hartkodieren)

- `type` ∈ `{bug, feature, task, project}` (bestehende Ticket-Typen).
- `severity` ∈ `{critical, major, minor, trivial}` (bestehendes Enum).
- `priority` ∈ den bestehenden `priority`-Werten der `tickets`-Tabelle.
- `areas`/`component`/`assignee_suggested`: aus einer **versionierten
  Enum-Datei** `scripts/factory/triage-enums.json` gelesen (nicht im Bash-Skript
  inline) — so bleibt die Liste pflegbar und der `assignee`-Vorschlag an eine
  bekannte, kuratierte Menge gebunden (keine Halluzination freier Namen).

## Planungsbüro-Integration (Review-UI)

- Die Planungsbüro-API (`website/src/pages/api/admin/planungsbuero/index.ts` /
  `[extId].ts`) liefert `grillingMeta` bereits aus (`admin.ts` selektiert
  `grilling_meta`). Die Triage-Vorschläge sind damit ohne Schema-Änderung lesbar.
- `planungsbuero.astro` zeigt den `triage`-Block als „KI-Vorschlag"-Badge mit den
  fünf Feldern + `rationale`. Zwei Aktionen:
  - **Übernehmen**: schreibt `type`/`priority`/`severity`/`areas`/`component` aus
    dem Vorschlag in die echten Spalten (PATCH auf `[extId].ts`).
  - **Verwerfen**: löscht `grilling_meta.triage` (Vorschlag weg, `triaged_at`
    bleibt → kein Re-Triage).
- `assignee_suggested` wird nur angezeigt (Zuweisung bleibt manuell — Assignee ist
  prozessual, kein KI-Auto-Set).

## Idempotenz & Sicherheit

- **Idempotent** über `triaged_at IS NULL` (Query-Filter + UPDATE-Guard).
- **Fail-closed**: ungültige KI-Antwort → kein Write, Ticket bleibt untriagiert,
  Retry im nächsten Tick. Best-effort im `wakeup.sh`-Aufruf (`|| true`), damit ein
  Triage-Fehler den Dispatcher-Tick nie blockiert (gleiches Muster wie auto-enqueue).
- **Budget**: `LIMIT :batch` (z.B. 5) pro Brand pro Tick gegen DeepSeek-Kostenspikes.
- **Kein Auto-Apply**: harte Felder werden nie ohne Review gesetzt.
- **Offline-Test-Shortcut**: `FACTORY_DRY_RESOLVE` kurzschließt Cluster-Zugriff
  (wie `auto-enqueue.sh`), damit BATS offline laufen.

## Migration

Alle benötigten Spalten (`component`, `areas`, `triaged_at`, `grilling_meta`)
existieren bereits laut `website/src/lib/tickets-db.ts` (idempotentes
`ensureSchema`). Falls auf einer Brand-DB noch nicht vorhanden, deckt eine
idempotente `ADD COLUMN IF NOT EXISTS`-Migration die Lücke ab (auf **beide**
Brand-DBs anzuwenden). Es werden **keine** Enum-Werte oder Trigger geändert.

## Tests

- BATS (`scripts/factory/auto-triage.test.bats` bzw. erweitern eines bestehenden
  Factory-BATS): Enum-Validierung (gültig/ungültig), Idempotenz-Skip bei
  gesetztem `triaged_at`, fail-closed bei Müll-JSON, `FACTORY_DRY_RESOLVE`-Pfad.
- Vitest: Planungsbüro-API gibt `triage`-Block aus; PATCH „Übernehmen" mappt
  Felder korrekt; „Verwerfen" entfernt nur `grilling_meta.triage`.
- `node --check` auf neue/​geänderte `.mjs`/`.js`; `node --test` falls Helper als
  ESM-Modul.

## Risiken / Trade-offs

- **DeepSeek-Klassifikationsqualität** (Scout-Schwäche dokumentiert): mitigiert
  durch menschliches Review (kein Auto-Apply) + fail-closed Validierung. Der Wert
  liegt in der Vorsortierung, nicht in autonomer Entscheidung.
- **Poll-Latenz**: ein Ticket wird erst im nächsten Tick triagiert — akzeptabel,
  da Triage nicht zeitkritisch ist (gleiche Begründung wie cron-poll-Autopilot).
- **Enum-Drift**: `triage-enums.json` muss bei neuen Bereichen/Bearbeitern
  gepflegt werden; bewusst versioniert statt aus der DB gelesen, damit der
  Vorschlagsraum kuratiert bleibt.
