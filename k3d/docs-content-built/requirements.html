Funktionale Anforderungen (FA)

ID Name Beschreibung Erfüllungskriterien Testfall
FA-01 Messaging (Echtzeit) Echtzeit-Kommunikation über WebSocket-Protokoll. Unterstützt 1:1-Direktnachrichten, Gruppen-Direktnachrichten sowie kanalbasierte Kommunikation. Nachrichten können mit Markdown, Emoji-Reaktionen und Thread-Antworten versehen werden. Persistente Speicherung in PostgreSQL. 1) Nachricht erscheint beim Empfänger in unter 500 ms; 2) 1:1-DM, Gruppen-DM und Channel-Nachrichten möglich; 3) Threads/Antworten unterstützt; 4) Nachrichten bleiben nach Container-Neustart erhalten T1: Sende DM → Nachricht erscheint; T2: Gruppen-DM → alle sehen Nachricht; T3: Channel-Nachricht → alle Mitglieder benachrichtigt
FA-02 Kanäle / Workspaces Öffentliche und private Kanäle. Öffentliche Kanäle für alle Mitglieder sichtbar und beitretbar; private Kanäle nur für eingeladene Mitglieder. 1) Öffentliche Kanäle für alle sichtbar; 2) Private Kanäle nur für eingeladene; 3) Kanal-Namen konfigurierbar T1: Öffentlicher Kanal → User tritt ohne Einladung bei; T2: Privater Kanal → User sieht ihn nicht
FA-03 Videokonferenzen Vollständig self-hosted Video- und Audiokonferenzen via Nextcloud Talk mit High Performance Backend (HPB). Mindestens 10 gleichzeitige Teilnehmer. Gäste können per Link beitreten. 1) Mindestens 10 gleichzeitige Teilnehmer; 2) Kamera- und Mikrofon-Übertragung; 3) Bildschirmfreigabe möglich; 4) Kein externer WebRTC-Dienst; 5) Gast-Beitritt per Link T1: Talk-Konversation → Video und Audio übertragen; T2: HPB Signaling-Server erreichbar; T3: Gast-Beitritt ohne Login möglich
FA-04 Dateiablage Datei-Upload in Nextcloud. Alle gängigen Dateitypen. Persistente Speicherung auf Kubernetes-Volumes. Dateien per Share-Link teilbar. 1) Upload aller gängigen Dateitypen; 2) Mindestens 10 GB Speicher; 3) Dateien per Link teilbar; 4) Persistent nach Neustart T1: 100 MB ZIP hochladen; T2: Nextcloud-Quota > 10 GB; T3: Link-Download funktioniert
FA-05 Nutzerverwaltung Vollständiger Lebenszyklus von Nutzerkonten. Rollenmodell: Admin, Nutzer. Benutzerverwaltung über Keycloak. CSV/LDIF-Import via import-users.sh. SSO-Login via Keycloak OIDC. 1) Admin kann User anlegen, bearbeiten, deaktivieren; 2) Rollen Admin, User verfügbar; 3) Benutzerverwaltung in Keycloak; 4) CSV-Import möglich; 5) SSO-Login via Keycloak T1: Admin legt User an → User kann sich einloggen; T2: User in Keycloak angelegt → SSO funktioniert
FA-06 Benachrichtigungen Browser-Push-Benachrichtigungen. Granulare Konfiguration. Do-Not-Disturb-Modus. 1) Web-Push bei neuer Nachricht; 2) Benachrichtigungen pro Kanal stummschaltbar; 3) DND-Zeitfenster konfigurierbar T1: DND aktivieren → keine Push in gesetztem Zeitraum
FA-07 Suche Integrierte Volltextsuche über Nachrichten und Dateinamen (PostgreSQL FTS). Filteroptionen. Suche respektiert Zugriffsrechte. 1) Volltextsuche über eigene Nachrichten; 2) Dateisuche nach Dateiname; 3) Suche in unter 2 s T1: Nachricht mit 'Testbegriff' → Suche findet Ergebnis; T2: Dateiname suchen → Datei erscheint
FA-08 Workspace-spezifisch Anpassbare Status-Texte (Homeoffice, Beschäftigt, Abwesend) mit automatischer Ablaufzeit. 1) Status für andere User sichtbar; 2) Benutzerdefinierter Status-Text möglich T1: Status setzen → anderer User sieht Status
FA-10 Kundenanfragen-Kontaktformular Website + Admin-Inbox: Kontaktformular auf /kontakt sendet Anfragen direkt in die Admin-Inbox. 1) Website-Deployment läuft; 2) Kontaktformular auf /kontakt funktionsfähig; 3) Nachricht erscheint in Admin-Inbox T1: Website-Pod läuft; T2: Kontaktformular ausfüllen → Nachricht in Inbox
FA-11 Kunden-Portal Infrastruktur und Keycloak-Integration für den Kunden-Portalbereich der Website. 1) Keycloak workspace-Realm erreichbar; 2) KC_DOMAIN in domain-config gesetzt T1: Keycloak läuft; T2: KEYCLOAK_ADMIN_PASSWORD gesetzt
FA-12 Claude Code AI Assistant KI-Assistent für Cluster- und Service-Management. MCP-Server stellen Kubernetes-, Datenbank-, Nextcloud- und Keycloak-Zugriff bereit. ForwardAuth-Proxy sichert MCP-Endpunkte. 1) MCP-Server-Pods (core, apps, auth) running; 2) ForwardAuth validiert Bearer-Tokens; 3) ForwardAuth lehnt Anfragen ohne Token ab T1: MCP-Pods running; T2: Anfrage ohne Token → 401; T3: Gültiges Token → 200
FA-13 Dokumentation Docs-Service-Infrastruktur mit Docsify und oauth2-proxy-Absicherung. 1) Docs-Service erreichbar; 2) oauth2-proxy schützt Docs via Keycloak T1: Docs-Pod running; T2: Docs erreichbar
FA-14 User Registration Flow Registrierungsseite /registrieren auf der Website. Formular erfasst Daten und löst Benachrichtigung aus. 1) /registrieren Seite erreichbar; 2) Formular validiert Pflichtfelder; 3) Bestätigungs-E-Mail in Mailpit sichtbar T1: /registrieren → 200; T2: POST /api/register ohne Daten → 400; T3: POST mit Daten → 200
FA-15 OIDC Website Login Website-Login über Keycloak OIDC (Authorization Code Flow). Session in HTTP-only Cookie. 1) /api/auth/login gibt 302 zu Keycloak zurück; 2) /api/auth/me gibt User-Info zurück; 3) /api/auth/logout löscht Session; 4) Keycloak-Realm hat "website"-Client T1: GET /api/auth/login → 302; T2: GET /api/auth/me ohne Session → {authenticated:false}
FA-16 Calendar Booking Website liest Admin-Kalender via Nextcloud CalDAV, zeigt freie Termine. Konfigurierbare Arbeitszeiten (Mo–Fr 9–17 Uhr), Slot-Dauer 60 min. 1) GET /api/calendar/slots liefert JSON-Array; 2) Slots nur an Arbeitstagen Mo–Fr; 3) Keine Wochenend-Slots; 4) POST /api/booking möglich T1: GET /api/calendar/slots → JSON; T2: Keine Wochenend-Slots; T3: POST /api/booking → 200
FA-17 Meeting Lifecycle Bei Terminbestätigung: Talk-Raum erstellt, Gast per E-Mail eingeladen, Kalender eintrag mit Meeting-Link, Erinnerung 10 min vorher. CronJob meeting-reminders triggert /api/reminders/process. 1) Talk-Raum via OCS-API erstellt; 2) Erinnerungs-E-Mail 10 min vorher; 3) POST /api/reminders/process verarbeitet Erinnerungen T1: POST /api/reminders/process → {sent,pending}; T2: CronJob meeting-reminders in K8s definiert
FA-18 Meeting Transcription Audio-Upload via POST /api/meeting/transcribe → faster-whisper (Medium-Modell, CPU). Rückgabe: Text + Zeitstempel. 1) faster-whisper Deployment läuft; 2) POST /api/meeting/transcribe akzeptiert Audio; 3) Transkript enthält Zeitstempel T1: Whisper-Pod running; T2: Whisper /health erreichbar
FA-20 Meeting Finalization Pipeline POST /api/meeting/finalize erstellt Profil-Dokument und Session-Dokument, postet Zusammenfassung. 1) POST /api/meeting/finalize verarbeitet Meeting-Daten; 2) Zusammenfassung gepostet T1: POST /api/meeting/finalize ohne Daten → 400; T2: Mit Daten → 200
FA-21 Service Catalog /leistungen-Seite zeigt Dienstleistungen mit Preisen. Jeder Service verlinkt zu /termin. 1) /leistungen zeigt Services; 2) Jeder Service hat "Termin buchen"-Link T1: /leistungen → 200
FA-23 Vaultwarden Passwort-Manager Self-hosted Passwort-Manager mit Keycloak SSO-Integration und shared PostgreSQL. Seed-Job befüllt Tresor mit Produktions-Secret-Templates. 1) Vaultwarden-Deployment läuft; 2) Erreichbar unter vault.{domain}; 3) SSO-Login via Keycloak; 4) Seed-Job erfolgreich T1: Vaultwarden-Pod running; T2: vault.{domain} erreichbar; T3: Seed-Job abgeschlossen
FA-24 Kollaboratives Whiteboard Nextcloud Whiteboard für visuelle Zusammenarbeit. Eigenständiges Deployment. Mehrere Nutzer können gleichzeitig zeichnen. 1) Whiteboard-Deployment läuft; 2) Erreichbar unter board.{domain}; 3) Nextcloud-Integration aktiv T1: Whiteboard-Pod running; T2: board.{domain} erreichbar
FA-25 Mailpit E-Mail-Server Mailpit als self-hosted E-Mail-Server. Fängt alle ausgehenden E-Mails ab (SMTP). Web-UI zur E-Mail-Inspektion. 1) Mailpit läuft; 2) Web-UI unter mail.{domain}; 3) SMTP-Port 1025 intern erreichbar T1: Mailpit-Pod running; T2: mail.{domain} erreichbar; T3: SMTP-Verbindung möglich
FA-26 Bug Report Form Website bug report endpoint. Formular sendet Tickets an Admin-Panel. 1) /api/bug-report Endpunkt erreichbar; 2) Ticket in Admin-Panel sichtbar T1: POST /api/bug-report → 200

Sicherheitsanforderungen (SA)

ID Name Beschreibung Erfüllungskriterien Testfall
SA-01 Transportverschlüsselung Alle externen Verbindungen über TLS 1.3 (Fallback TLS 1.2). TLS-Terminierung am Ingress-Controller. Automatische Zertifikatsverwaltung via Let's Encrypt. HSTS aktiviert. 1) Alle HTTP-Verbindungen auf HTTPS umgeleitet; 2) TLS 1.3 oder TLS 1.2 mit starken Cipher-Suites; 3) HSTS-Header vorhanden; 4) Zertifikat gültig T1: HTTP → HTTPS Redirect; T2: HSTS-Header in Response; T3: Zertifikat gültig
SA-02 Authentifizierung Primäre Authentifizierung über Benutzername/Passwort mit Rate-Limiting. Optionale 2FA via TOTP. Passwort-Reset über verifizierte E-Mail. 1) Login nur mit korrektem Passwort; 2) Rate-Limiting: Sperrung nach Fehlversuchen; 3) 2FA via TOTP konfigurierbar T1: Login mit gültigen Credentials → Erfolg; T2: Ungültiges Passwort → Fehler; T3: 10 Fehlversuche → Konto gesperrt
SA-03 Passwörter Passwörter mit bcrypt (Kostenfaktor >= 12) gehasht. Policy: Mindestlänge 12 Zeichen, Groß-/Kleinbuchstaben, Zahlen und Sonderzeichen. 1) Passwörter als bcrypt-Hash gespeichert; 2) Mindestlänge 12 Zeichen; 3) Kein Klartext in Logs T1: Passwort-Policy in Keycloak konfiguriert; T2: Schwaches Passwort → abgelehnt
SA-04 Session-Timeout Sessions nach 30 Minuten Inaktivität automatisch invalidiert. JWT-Tokens mit kurzer Gültigkeitsdauer. 1) Session-Timeout nach 30 Minuten; 2) Access Token Gültigkeitsdauer konfiguriert; 3) Logout invalidiert alle Sessions T1: Session-Timeout in Keycloak konfiguriert; T2: Logout → alle Sessions ungültig
SA-05 Audit-Log Unveränderliches Audit-Log für sicherheitsrelevante Ereignisse. Aufbewahrung mindestens 90 Tage. 1) Login-Ereignisse protokolliert; 2) Audit-Log enthält Zeitstempel, User-ID, Aktion; 3) Aufbewahrung mindestens 90 Tage T1: Login-Event in Keycloak Audit-Log; T2: Fehlgeschlagener Login protokolliert
SA-06 RBAC Rollenbasierte Zugriffskontrolle. Rollen: Admin, Member. Prinzip der minimalen Rechtevergabe. 1) Rollen Admin und User mit unterschiedlichen Rechten; 2) Neue User erhalten standardmäßig Member-Rechte T1: Admin-Rolle → volles Zugriffsrecht; T2: Member → kein Admin-Panel-Zugriff
SA-07 Backup Automatisierte tägliche Backups aller kritischen Daten. Backup-Rotation: 30 Tage. 1) Tägliches Backup per CronJob; 2) Backup-Rotation: 30 Tage; 3) Restore erfolgreich verifiziert T1: CronJob db-backup aktiv; T2: Backup-Datei vorhanden
SA-08 Single Sign-On (SSO) Zentrales Identity-Management über Keycloak OIDC. Alle Dienste (Nextcloud, Vaultwarden, Website, Docs) authentifizieren über denselben Keycloak-Realm. 1) Keycloak OIDC-Clients für Nextcloud, Vaultwarden und Website konfiguriert; 2) Redirect-URIs korrekt; 3) Nextcloud SSO-Login leitet zu Keycloak weiter; 4) Talk erbt Nextcloud-OIDC-Session T1: Nextcloud OIDC-Client in Keycloak konfiguriert; T2: SSO-Login Redirect zu Keycloak; T3: Cross-Service-SSO funktioniert
SA-10 MCP-Endpunkt-Absicherung ForwardAuth-Proxy schützt alle MCP-Server-Endpunkte. Zugriff nur mit gültigem Bearer-Token (aus mcp-tokens Secret). Unauthentifizierte Anfragen erhalten HTTP 401. 1) ForwardAuth-Proxy deployed und running; 2) Anfragen ohne Token → 401; 3) Anfragen mit gültigem Token → 200; 4) mcp-tokens Secret vorhanden T1: ForwardAuth-Proxy-Pod running; T2: Anfrage ohne Token → 401; T3: Gültiges Token → 200

Nicht-funktionale Anforderungen (NFA)

ID Name Beschreibung Erfüllungskriterien Testfall
NFA-01 Datenschutz Alle personenbezogenen Daten ausschließlich auf Servern in Deutschland. Keine Übermittlung an Drittanbieter außerhalb der EU. DSGVO-Löschkonzept implementiert. 1) Alle Daten nur auf eigenem Server; 2) Kein Telemetrie-Datenabfluss; 3) DSGVO-Auskunftspflicht: User-Daten-Export möglich; 4) Recht auf Löschung: Account vollständig löschbar T1: Keine Verbindungen zu externen Cloud-Diensten; T2: Container-Images aus DE/EU Registries
NFA-02 Performance UI-Ladezeit unter 2 s. API-Response unter 1 s für 95 % der Requests. 1) Seitenaufruf unter 2 s; 2) API-Response unter 1 s für 95 % T1: Seitenaufruf messen; T2: API-Antwortzeit prüfen
NFA-03 Verfügbarkeit Systemverfügbarkeit >= 99,5 % im Jahresdurchschnitt. Kubernetes-basiertes Auto-Restart via Liveness/Readiness-Probes. 1) Liveness- und Readiness-Probes konfiguriert; 2) Rolling Updates ohne Downtime; 3) Pod-Neustart nach Crash automatisch T1: Liveness-Probes für alle Deployments konfiguriert; T2: Pod-Neustart automatisch
NFA-04 Skalierbarkeit Horizontale Skalierung durch Kubernetes. Getesteter Bereich: 10–500 gleichzeitige Nutzer. 1) Stabil mit 10 gleichzeitigen Usern; 2) RollingUpdate-Strategie aktiv; 3) Resource Requests/Limits gesetzt T1: 10 gleichzeitige User → alle Requests erfolgreich; T2: RollingUpdate-Strategie aktiv
NFA-05 Usability UI vollständig auf Deutsch lokalisiert. Einarbeitungszeit für Grundfunktionen maximal 30 Minuten. 1) Nextcloud-UI auf Deutsch; 2) Login-Seite lädt in unter 5 s T1: Nextcloud Default-Sprache Deutsch; T2: Login-Seite erreichbar und unter 5 s
NFA-06 Wartbarkeit Gesamtes System als Kubernetes-Container. Updates via Rolling-Deployment. Konfiguration versioniert (GitOps). 1) Alle Services via kubectl/Kustomize steuerbar; 2) Konfiguration vollständig in Git; 3) Log-Zugriff via kubectl logs T1: kubectl rollout restart → Rolling Update ohne Fehler; T2: Liveness-Probes konfiguriert
NFA-07 Lizenz Alle Softwarekomponenten unter anerkannten Open-Source-Lizenzen (MIT, AGPL-3.0, MPL-2.0, Apache 2.0, PostgreSQL License). 1) Nextcloud (AGPL-3.0); 2) Keycloak (Apache 2.0); 3) PostgreSQL (PostgreSQL License); 4) Keine kommerziellen Lizenzen T1: Keine proprietären Container-Images; T2: Alle Images haben stabile Release-Tags
NFA-08 Produktions-Deployment Deployment auf Hetzner-Server mit k3s. Eigene Domain. Wildcard-TLS via Let's Encrypt (cert-manager + DNS-01). 1) k3s-Cluster auf Hetzner erreichbar; 2) Alle Services unter *.{domain} erreichbar; 3) Wildcard-TLS-Zertifikat gültig T1: kubectl get nodes → Ready; T2: TLS-Zertifikat gültig; T3: cert-manager Pod running
NFA-09 Statisches DNS Beide Produktionscluster (korczewski, mentolder) nutzen statische IPs — DNS-Einträge werden einmalig in ipv64.net gesetzt, kein DDNS-CronJob notwendig. DNS-Eintrag zeigt auf korrekte statische IP T1: A-Record in ipv64.net stimmt mit Server-IP überein
