Backup Check

End-to-End Audit des Backup-Systems: Setup verifizieren → Live-Backup auslösen → Dateien prüfen → Sicherer Restore-Test → Fehler beheben.

Auslöser: Audit, Restore-Test, oder Backup-System-Diagnose
⚠️
Immer auf beiden Clustern: mentolder + korczewski prüfen, außer der Nutzer gibt explizit einen Cluster an. Jeder Cluster hat eigene shared-db und Backups.

Nicht im Backup: Nextcloud File PVC, VW Attachments, LiveKit Recordings.

1 Setup Audit CronJob, PVC, Passphrase

Prüfe CronJob Name und Schedule (erwartet: db-backup, 0 2 * * *):

kubectl get cronjob -n <ns> --context <ctx>
ℹ️
Name-Drift: Falls backup-postgres existiert, mit task workspace:deploy auf den neuen Namen db-backup migrieren.

Prüfe BACKUP_PASSPHRASE Stärke (≥ 32 bytes):

kubectl get secret workspace-secrets -n <ns> -o jsonpath='{.data.BACKUP_PASSPHRASE}' | base64 -d | wc -c
2 Live Backup Trigger On-demand Backup auslösen 
bash scripts/backup-restore.sh trigger --context mentolder
bash scripts/backup-restore.sh trigger --context korczewski --namespace workspace-korczewski

Erfolg wenn Log zeigt: ✓ Backup complete

3 Dateien verifizieren Verschlüsselung prüfen

Spawn Busybox Pod gegen backup-pvc:

kubectl run backup-verify-$$ -n <ns> --restart=Never --image=busybox --overrides='...'

Alle 5 .dump.enc Dateien müssen vorhanden und > 1 KB sein.

4 Sicherer Restore-Test In Temp-DB (NIEMALS Prod)
🚫
NIEMALS eine Produktions-Datenbank für einen Test droppen. Nutze immer website_restore_test.

Erstelle einen temporären Restore-Job, der website.dump.enc in eine neue Test-DB einspielt und danach die Tabellen zählt.

Restore succeeded. Table count:
SELECT count(*) FROM information_schema.tables...

Notfall-Restore Runbook

# 1. Betroffenen Service stoppen
kubectl scale deployment/<svc> --replicas=0
# 2. Restore ausführen
task workspace:restore -- <db> <timestamp> ENV=<env> -y
# 3. Service starten
kubectl scale deployment/<svc> --replicas=1
