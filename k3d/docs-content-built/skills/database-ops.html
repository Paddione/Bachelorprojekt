Database Schema & Backups

Writing database schema migrations, applying DDL changes across independent clusters, re-granting service permissions, triggering backups, and running safe restore verification.

Auslöser: Schema adjustments required, permission fixes, or weekly backup audits.
ℹ️
Independent DBs: Both mentolder and korczewski host stand-alone PostgreSQL databases. You must apply migrations to both clusters separately.
1 Write Migration Idempotency & Transactions

Save script under scripts/datamodel/. Use IF NOT EXISTS and wrap commands in transactions:

BEGIN;
CREATE TABLE IF NOT EXISTS coaching.sessions (...);
COMMIT;
2 Apply & Grant Run DDL Updates

Apply to dev, then prod clusters. Connect as superuser for schemas owned by postgres:

kubectl exec -i "$PGPOD" -- psql -U postgres < migration.sql
task workspace:fix-tickets-grants ENV=mentolder
3 Trigger Backup Verify Dump Files

Ensure the backup CronJob is active. Trigger a manual backup to verify passphrase encryption:

bash scripts/backup-restore.sh trigger --context mentolder
4 Safe Restore Test No Prod Data Loss

Decrypt and load dump files into a temporary test database (e.g. website_restore_test) to verify data integrity. Drop the test DB afterwards.

Troubleshooting

SymptomLösung
"must be owner of table"Do not use website role. Connect as postgres superuser to run DDL.
Trigger: cronjob not foundDrift: CronJob is named backup-postgres. Re-apply backup-cronjob.yaml.
pg_restore fails during testEnsure target test database is fresh and completely empty.
