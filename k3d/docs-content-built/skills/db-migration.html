Database Migration

Sichere Schema-Änderungen über beide Produktions-Cluster hinweg: Tabellen, Spalten, Indizes und Rollenberechtigungen.

Auslöser: Neue Features mit DB-Abhängigkeit, Refactoring oder Rollen-Updates
🚫
Cluster-Isolation: mentolder und korczewski sind unabhängig. Migrationen müssen immer explizit auf beiden angewendet werden.
1 Migration schreiben SQL in scripts/datamodel/

Nutze IF NOT EXISTS und Transaktionen für Idempotenz:

BEGIN;
CREATE TABLE IF NOT EXISTS coaching.sessions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id TEXT NOT NULL
);
COMMIT;
2 Test & Apply Dev → Mentolder → Korczewski

1. Test auf Dev:

task workspace:psql ENV=dev -- website < scripts/datamodel/<file>.sql

2. Backup & Apply Mentolder:

task workspace:backup
task workspace:psql ENV=mentolder -- website < scripts/datamodel/<file>.sql

3. Apply Korczewski:

task workspace:psql ENV=korczewski -- website < scripts/datamodel/<file>.sql
3 Berechtigungen fixen Grants für Service-Rollen

Nach Erstellung neuer Schemas/Tabellen müssen Rollen wie website oder brett oft explizit berechtigt werden:

task workspace:fix-tickets-grants ENV=mentolder
task workspace:fix-tickets-grants ENV=korczewski

Manueller Grant:

GRANT SELECT, INSERT ON <schema>.<table> TO website;
4 Abschluss Diagramm & Commit

ER-Diagramm aktualisieren und mit SQL-Datei committen:

task db:diagram
git add scripts/datamodel/<file>.sql docs/db-diagram.png

Common Patterns

  • 1
    Spalte mit Default hinzufügen 
    ALTER TABLE coaching.chunks ADD COLUMN IF NOT EXISTS reviewed_at TIMESTAMPTZ;
  • 2
    Schema mit Default-Privilegien 
    CREATE SCHEMA IF NOT EXISTS analytics;
ALTER DEFAULT PRIVILEGES IN SCHEMA analytics GRANT ALL ON TABLES TO website;
