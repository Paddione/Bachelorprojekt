dev.mentolder.de Stack Operations

Betrieb und Reparatur des Dev-k3d-Clusters auf k3s-1: SSH-Zugang, task dev:* Kommandos, Dev-DB-Refresh, Tunnel-Publishing und Keycloak /dev-access Probleme.

Auslöser: Dev-Stack reagiert nicht, 403-Loop, Cluster-Neubau oder Tunnel-Setup nötig
⚠️
task dev:cluster:create muss auf k3s-1 ausgeführt werden. Das Ausführen an anderer Stelle schlägt fehl, weil Docker dort nicht verfügbar ist.
ℹ️
Dev sieht Prod-Daten. Der dev-db-refresh-CronJob (03:30 UTC) löscht und recreated website, bugs, bachelorprojekt in shared-db-dev aus dem neuesten Prod-Snapshot. Keine Prod-Rituale gegen die Dev-DB schreiben.
1 Cluster prüfen Existiert er noch? 
task dev:cluster:status
2 Cluster anlegen Via k3s-1 SSH-Wrapper 
task dev:cluster:create
3 Stack deployen Website + Brett + Workspace 
task dev:deploy
4 Firewall öffnen DEV_SSH_ALLOWLIST aus mentolder.yaml 
task dev:firewall:open

Häufige Day-2 Kommandos

task dev:cluster:status          # Pod-Status in workspace-dev
task dev:redeploy:website        # Website rebuild + rollout
task dev:redeploy:brett          # Brett rebuild + rollout
task dev:db:refresh              # Manueller Restore neuester Prod-Snapshot → shared-db-dev
task dev:logs -- <svc>           # Dev-Pod-Logs tailen
task dev:psql                    # psql in shared-db-dev
task dev:tunnel -- <name> <port> # localhost:<port> als https://<name>.dev.mentolder.de publizieren

Zugriffsvoraussetzungen

Keycloak /dev-access Gruppe

workspace-dev erzwingt /dev-access-Gruppenmitgliedschaft am oauth2-proxy. Ohne diese loopt der Browser auf 403.

# KC Admin UI: web.mentolder.de/keycloak/admin → Realm workspace → Groups → /dev-access
# User dort hinzufügen, dann erneut versuchen

SSH-Zugang (Brainstorm-Tunnel)

  • Eigene CIDR in DEV_SSH_ALLOWLIST (environments/mentolder.yaml) → dann task dev:firewall:open
  • Eigener Public Key in DEV_SISH_AUTHORIZED_KEYS → dann task brainstorm:materialise-keys

Port-Mappings (load-bearing)

PortZweck
127.0.0.1:18080k3d HTTP LB (prod Traefik → workspace-dev)

Diese kommen aus k3d-config.yaml. Cluster ohne task dev:cluster:create recreaten verliert diese Mappings.

Secret-Handling
