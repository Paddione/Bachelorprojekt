Infra hetzner-node

Hetzner Node Management

Geführtes Provisioning und Reset von Hetzner Cloud Servern: Key-Management, Cloud-Config Generierung, Rescue-Mode Reinstall und Cluster-Join.

Auslöser: Neuer Cluster-Node benötigt oder bestehender Node kompromittiert/defekt
ℹ️
Persistente Identität: WireGuard-Keys werden in .secrets/<env>.yaml gespeichert. Dies ermöglicht Resets, ohne dass Peers aktualisiert werden müssen.
1 Input & Key Selection Modus, Rolle & WG-Keys

Bestimme Modus (New vs. Reset) und Rolle (CP Init, CP Join, Worker). Prüfe vorhandene WG-Keys:

grep "WG_MESH" environments/.secrets/mentolder.yaml

Falls neu, generiere Keys und trage sie in wireguard/wg-mesh-nodes.yaml ein.

2 Infrastruktur-Setup Hetzner Console / CLI

Variante A: Neuer Server (Cloud-Init)

hcloud server create --user-data-from-file /tmp/cloud-init.yaml

Variante B: Reset (Rescue Mode)

ssh root@<IP> "/root/.oldroot/nfs/install/installimage -a -c /tmp/config"
3 Node-Konfiguration Setup-Script ausführen

Führe das generierte Setup-Script auf dem frischen OS aus (SSH-Härtung, Pakete, k3s):

ssh root@<NODE_IP> bash -s < /tmp/setup-<nodename>.sh
⚠️
Nach dem Script ist Root-Login deaktiviert. Nutze patrick@ für weiteren Zugriff.
4 Cluster-Integration Verify & Label

Prüfe den Node-Status und setze die Rollen-Labels:

kubectl get nodes --context <env>
kubectl label node <name> node-role.kubernetes.io/worker=""
