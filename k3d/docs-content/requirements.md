## Sicherheitsanforderungen (SA)

| ID | Name | Beschreibung | Erfüllungskriterien | Testfall |
|----|------|-------------|--------------------|---------| 
| SA-01 | Transportverschlüsselung | Alle externen Verbindungen über TLS 1.3 (Fallback TLS 1.2). TLS-Terminierung am Ingress-Controller. Automatische Zertifikatsverwaltung via Let's Encrypt. HSTS aktiviert. | 1) Alle HTTP-Verbindungen auf HTTPS umgeleitet; 2) TLS 1.3 oder TLS 1.2 mit starken Cipher-Suites; 3) HSTS-Header vorhanden; 4) Zertifikat gültig | T1: HTTP → HTTPS Redirect; T2: HSTS-Header in Response; T3: Zertifikat gültig |
| SA-02 | Authentifizierung | Primäre Authentifizierung über Benutzername/Passwort mit Rate-Limiting. Optionale 2FA via TOTP. Passwort-Reset über verifizierte E-Mail. | 1) Login nur mit korrektem Passwort; 2) Rate-Limiting: Sperrung nach Fehlversuchen; 3) 2FA via TOTP konfigurierbar | T1: Login mit gültigen Credentials → Erfolg; T2: Ungültiges Passwort → Fehler; T3: 10 Fehlversuche → Konto gesperrt |
| SA-03 | Passwörter | Passwörter mit bcrypt (Kostenfaktor >= 12) gehasht. Policy: Mindestlänge 12 Zeichen, Groß-/Kleinbuchstaben, Zahlen und Sonderzeichen. | 1) Passwörter als bcrypt-Hash gespeichert; 2) Mindestlänge 12 Zeichen; 3) Kein Klartext in Logs | T1: Passwort-Policy in Keycloak konfiguriert; T2: Schwaches Passwort → abgelehnt |
| SA-04 | Session-Timeout | Sessions nach 30 Minuten Inaktivität automatisch invalidiert. JWT-Tokens mit kurzer Gültigkeitsdauer. | 1) Session-Timeout nach 30 Minuten; 2) Access Token Gültigkeitsdauer konfiguriert; 3) Logout invalidiert alle Sessions | T1: Session-Timeout in Keycloak konfiguriert; T2: Logout → alle Sessions ungültig |
| SA-05 | Audit-Log | Unveränderliches Audit-Log für sicherheitsrelevante Ereignisse. Aufbewahrung mindestens 90 Tage. | 1) Login-Ereignisse protokolliert; 2) Audit-Log enthält Zeitstempel, User-ID, Aktion; 3) Aufbewahrung mindestens 90 Tage | T1: Login-Event in Keycloak Audit-Log; T2: Fehlgeschlagener Login protokolliert |
| SA-06 | RBAC | Rollenbasierte Zugriffskontrolle. Rollen: Admin, Member. Prinzip der minimalen Rechtevergabe. | 1) Rollen Admin und User mit unterschiedlichen Rechten; 2) Neue User erhalten standardmäßig Member-Rechte | T1: Admin-Rolle → volles Zugriffsrecht; T2: Member → kein Admin-Panel-Zugriff |
| SA-07 | Backup | Automatisierte tägliche Backups aller kritischen Daten. Backup-Rotation: 30 Tage. | 1) Tägliches Backup per CronJob; 2) Backup-Rotation: 30 Tage; 3) Restore erfolgreich verifiziert | T1: CronJob db-backup aktiv; T2: Backup-Datei vorhanden |
| SA-08 | Single Sign-On (SSO) | Zentrales Identity-Management über Keycloak OIDC. Alle Dienste (Nextcloud, Vaultwarden, Website, Docs) authentifizieren über denselben Keycloak-Realm. | 1) Keycloak OIDC-Clients für Nextcloud, Vaultwarden und Website konfiguriert; 2) Redirect-URIs korrekt; 3) Nextcloud SSO-Login leitet zu Keycloak weiter; 4) Talk erbt Nextcloud-OIDC-Session | T1: Nextcloud OIDC-Client in Keycloak konfiguriert; T2: SSO-Login Redirect zu Keycloak; T3: Cross-Service-SSO funktioniert |
| SA-09 | — | Übersprungen (Invoice Ninja aus dem Stack entfernt). | — | — |
| SA-10 | MCP-Endpunkt-Absicherung | ForwardAuth-Proxy schützt alle MCP-Server-Endpunkte. Zugriff nur mit gültigem Bearer-Token (aus mcp-tokens Secret). Unauthentifizierte Anfragen erhalten HTTP 401. | 1) ForwardAuth-Proxy deployed und running; 2) Anfragen ohne Token → 401; 3) Anfragen mit gültigem Token → 200; 4) mcp-tokens Secret vorhanden | T1: ForwardAuth-Proxy-Pod running; T2: Anfrage ohne Token → 401; T3: Gültiges Token → 200 |
