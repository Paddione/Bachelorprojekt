# observability-remediation — Implementation Plan (P4: alertmanager-secret-fix)

This is Partial **P4 (alertmanager-secret-fix)** of 5: **P1 logging-pipeline-fixes → P2
service-health-goals → P3 agent-tracing → P4 alertmanager-secret-fix → P5 tests**.

P4 is the odd one out: it is not a code change but a **documented manual secret-reseed
runbook**. The root cause (live-verified 2026-07-24, see design.md) is that the
`alertmanager-pushover` Secret in namespace `monitoring` holds empty (0-byte)
`PUSHOVER_USER`/`PUSHOVER_TOKEN` values, so the prometheus-operator silently drops the
`pushover` and `workspace-alerts` `AlertmanagerConfig` resources
(`mandatory field userKey is empty`) — every critical alert (disk-full, CrashLoop, 5xx)
currently routes to the no-op `null` receiver even though Kubernetes reports
`Reconciled: True`/`Available: True`.

**No agent can complete this partial autonomously** — it requires real Pushover
credentials that only the repo owner has. `dev-flow-execute` MUST pause here and ask
the human operator for the values rather than inventing placeholder secrets.

## Task 1 — Confirm the secret is actually empty (diagnostic, no changes)

```bash
kubectl --context fleet get secret alertmanager-pushover -n monitoring \
  -o jsonpath='{.data.PUSHOVER_USER}' | base64 -d | wc -c
kubectl --context fleet get secret alertmanager-pushover -n monitoring \
  -o jsonpath='{.data.PUSHOVER_TOKEN}' | base64 -d | wc -c
```

**expected: currently prints `0` for both** (confirmed live 2026-07-24). If either
value is already non-zero when this task runs, the secret was fixed out-of-band —
skip to Task 3 to confirm the operator accepted it.

blocking question to the user here (e.g. via `AskUserQuestion` or an equivalent
prompt) requesting the real `PUSHOVER_USER` and `PUSHOVER_TOKEN` values. Do not

## Task 3 — Verify the operator accepts the config (post-deploy)

After the sealed secret is applied (via the normal Flux reconciliation or
`task workspace:deploy ENV=mentolder` break-glass path — this partial does not
introduce a new deploy mechanism), confirm the prometheus-operator stops rejecting
the `AlertmanagerConfig` resources:

```bash
kubectl --context fleet logs -n monitoring deploy/monitoring-operator --tail=200 \
  --since=10m | grep -i "mandatory field" || echo "no mandatory-field warnings — OK"
```

**expected: FAIL before the reseed** (the `mandatory field userKey is empty` lines
are present, as diagnosed live 2026-07-24) — **expected: PASS (no output, "OK"
printed) after Task 2's reseed lands and the operator reconciles**. This is a live
cluster check, not a CI-runnable test — no `bats`/`vitest`/`playwright` runner
applies here, which is why P5 (tests) carries the formal automated test obligation
for this change set; this task is the human-verified counterpart for the one
partial that cannot be automated.

Then confirm the effective route is no longer the `null` receiver:

```bash
kubectl --context fleet exec -n monitoring alertmanager-monitoring-alertmanager-0 -c alertmanager -- \
  amtool config show --alertmanager.url=http://localhost:9093 2>/dev/null | grep -A2 "receiver: pushover" \
  || echo "pushover receiver not found in effective config — investigate"
```
